# 任务分配与执行偏差分析（多次调用支持）

## 现状概览
- 编排流程由 `workflow_order` 固定顺序驱动，每个 agent 在主循环中只执行一次。
- 计划阶段 `_llm_decompose_tasks` 仅为 `workflow_order` 内的 agent 生成 per-agent 任务说明（映射结构），`order` 字段未参与执行排序。
- `add_bgm` 属于执行阶段硬编码后置分支，在 `audio_generator` 之后条件触发，绕过计划阶段产出。
- 视频合成需要多次调用（合成成片 + 合入 BGM），但计划阶段只规划一次。

## 偏差与风险
- 任务分配未覆盖“同一 agent 多次调用”，导致 `add_bgm` 被执行期临时追加，破坏“任务前置分配”的规范。
- 执行期临时任务缺少一致的 spec 约束，LLM 可能认为“已有成片 + BGM 可用 = 无需处理”，导致 `mix_type_mismatch`。
- 计划阶段与执行阶段职责混杂，后续扩展（如 add_voiceover / 复合混音）只能继续硬编码分支。
- 计划产物无法表达“条件任务”，使编排可审计性与复现性降低。

## 根因定位
- `_llm_decompose_tasks` 产出结构是 `AgentType → spec`，假设每个 agent 仅调用一次。
- `workflow_order` 为固定列表，不支持“任务序列/步骤”级别的复用。
- `add_bgm` 以执行期分支实现，未在计划阶段生成明确任务目标与边界。
- 合成代理的规划提示未强制“背景音乐请求存在时必须执行混音”的业务约束。

## 修复思路
### 方向 A（推荐，最小改动但满足前置分配）
- **计划阶段产出条件任务 spec**：为 `video_composer_bgm_mix` 生成明确指令/目标/边界（来自 LLM 或规则化模板）。
- **执行阶段仅消费计划产物**：`add_bgm` 触发时，从预生成 spec 读取并执行，不再临时构造任务。
- **保留固定顺序**：仍按 `workflow_order` 执行主流程，条件任务作为“预分配的附加步骤”触发。

### 方向 B（完整演进）
- **改为任务序列**：Plan 产出 `[{step_id, agent, spec, when}]`，执行阶段按序列遍历，支持同一 agent 多次调用。
- 删除所有执行期硬编码分支，统一通过计划序列驱动。

### 过渡路线（A → B）
1) 先固化 `conditional_tasks` 的结构与执行入口，确保条件任务在计划阶段前置产出（方向 A）。
2) 引入 `task_sequence` 的兼容入口：执行阶段优先使用序列，缺失时回退到 `workflow_order + conditional_tasks`。
3) 将执行循环切换为“任务序列驱动”，`workflow_order` 仅作为回退序列。
4) 删除执行期硬编码分支（如 add_bgm），全部由计划产物驱动。

### 业务约束补强
- 在合成代理的规划提示中明确：当存在背景音乐请求且资源可用时，必须进行混音；仅在混音完成后才可判定完成。
- 该约束应以业务语义描述，避免暴露工具名/参数名。

## 行动计划（含优先级）
### P0（必须）
1) **计划阶段预生成 add_bgm 任务 spec**  
   - 产出结构中新增 `video_composer_bgm_mix`（或 step_id）  
   - `conditional_tasks` 视为必需产物；缺失即失败（不再静默回退）  
   - 触发时读取该 spec 执行，删除临时构造 task 的逻辑  
   - 验收：`add_bgm` 执行日志显示使用预生成 spec

2) **合成代理提示补强（业务约束）**  
   - 明确“背景音乐请求存在 → 必须执行混音 → 完成条件为混音产物写入 SoT”
   - 验收：出现 `bgm_requested=true` 时必触发混音工具调用

### P1（高优先）
3) **条件任务的统一载体**  
   - 规划产物支持 `conditional_tasks`，可扩展至 add_voiceover  
   - 验收：条件任务由计划阶段产出且可审计

4) **下载可靠性与超时策略**  
   - 统一文件下载超时/重试策略（避免 BGM 下载失败导致任务终止）  
   - 验收：下载超时率降低，可重试成功

### P2（中优先）
5) **任务序列化改造（方向 B）**  
   - 增加 `task_sequence` 入口（优先执行序列，缺失则回退）
   - 将执行循环从“agent 列表”替换为“任务序列”
   - 删除执行期硬编码分支（add_bgm/add_voiceover）
   - 验收：同一 agent 可多次出现且顺序可审计

## 验收与观测
- 触发 `add_bgm` 时，composer 必须产生混音产物（有 mix_receipt + 新 final_video）。
- `mix_type_mismatch` 不再出现。
- 日志中可追溯：任务 spec 来源（plan vs fallback）、背景音乐路径是否进入上下文。
