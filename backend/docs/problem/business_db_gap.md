# 业务库缺位导致审计库“兼容层”风险

## 现状
- 尚无独立的业务数据库/后端消费事件，当前本地 SQL 库兼作审计与业务兼容层。
- 场景/资源等业务数据经事件持久化写入现有 DB（原本定位为审计），未与审计视图分层。

## 偏差与风险
- 边界混淆：审计库写入业务内容（Scene/Resource），违背“审计最小投影”原则。
- 迁移成本：未来接入业务库时需拆分/迁移场景/资源数据，增加改造成本。
- 双轨不一致：记忆/事件与 DB 可能出现字段/粒度不一致，增加维护负担。

## 建议方向
- 短期：在现有事件 payload 中仅保留最小业务索引（scene_number、简要描述、资源 URL/path），控制写库粒度。
- 中期：引入独立业务后端/库，通过同一事件流消费场景/资源，审计库回归仅存任务状态/最终产物索引。
- 文档与状态：在 STATUS 中跟踪业务库接入进度，明确拆分计划。
