# 问题记录：任务序列规划缺失（agent 只能规划一次）

## 现象
- 编排器主循环按固定 `workflow_order` 执行，每个 agent 只执行一次。
- `add_bgm` 等“同一 agent 二次调用”的任务只能通过执行期硬编码分支追加。
- 计划阶段 `_llm_decompose_tasks` 只输出 per-agent 映射，无法表达“任务序列”或“条件任务”。

## 影响
- 任务分配不前置，执行期补丁式追加任务，降低可审计性与可复现性。
- 后续新增类似 `add_voiceover`、多段混音会继续堆叠硬编码分支。
- LLM 规划与实际执行存在结构性偏差，易出现“任务已完成但未执行关键步骤”的误判。

## 根因
- 计划输出结构为 `AgentType -> spec`，默认每个 agent 仅参与一次。
- `workflow_order` 为固定列表，执行阶段无法表达重复调用。
- 条件任务（如 add_bgm）未纳入计划产物，只能运行时追加。

## 修复方向（需规划）
- 方案 B（架构演进）：Plan 阶段输出“任务序列/条件任务”，执行阶段只消费计划产物，允许同一 agent 多次出现。
- 兼容路径：保留固定顺序，但新增 `conditional_tasks` 作为计划产物，替代执行期硬编码分支。
