# Subtask Snapshots
## 1. 旧 quick run 的“发现”和“继续查看”当前只是 read-model reattach
- 首页发现当前未完成 quick run 的入口是 `GET /tasks/quick/current?session_id=...`，见 `backend/app/api/v1/endpoints/tasks.py`。
- 前端发现到 `existingRun` 后，点击“继续查看当前任务”只会执行：
  - `setCurrentRequest(request)`
  - `setQuickRuntime(existingRun.runtime)`
  - `setCurrentStep('processing')`
  见 `src/components/forms/VideoRequestForm.tsx` 的 `handleContinueExistingRun()`。
- 该路径没有任何 resume/requeue API 调用，因此不会主动恢复后端执行。

## 2. 真正会恢复后端执行的当前只有 gate decision submit
- quick workspace 里的 `approve/revise/replan` 通过 `POST /tasks/{taskId}/runtime/script/decision` 提交，见 `src/components/preview/QuickModeWorkspace.tsx` 和 `src/lib/api.ts`。
- 后端在 `submit_script_gate_decision(...)` 里会：
  - 写控制层 decision/session 状态
  - 调 `_schedule_task_execution(...)` 重新入队
  - 返回新的 authoritative runtime view
- 所以“继续查看”与“继续执行”当前是两条完全不同的路径。

## 3. 本次日志证据已经表明：用户这次没有真正触发 resume
- 最新 API 重启后，`backend/logs/mas/mas_workflow.log` 只显示：
  - 两次 `GET /api/v1/tasks/quick/current?...`
  - WebSocket connected
  - WebSocket subscribed to task `5330d74a-0059-4c47-a5dd-25278eb905e4`
- 没有新的：
  - `POST /api/v1/tasks/.../runtime/script/decision`
  - `Task process_video_task[...] received`
  - `Loaded runtime session ...`
- 这说明用户看到的是旧 task 的恢复视图，而不是一次新的后台恢复执行。

## 4. 当前 read-model 仍会把旧 non-terminal run 暴露为“当前未完成任务”
- `_find_unfinished_quick_task_for_session(...)` 的判定规则是：
  - 找最新 quick runtime session
  - 只要 runtime status 不是 terminal（`completed/failed/cancelled`），就返回为 current run
- `RESUMING`、`RUNNING`、`QUEUED` 都会被视为 unfinished。
- 这意味着一个旧的、实际没有 worker 正在执行、但 runtime 状态仍停留在 non-terminal 的 run，会持续被前端发现并恢复为“当前任务”。

## 5. 当前最值得防的不是“小补丁”，而是再长出第二套恢复 owner
- 不要把“继续查看”偷偷做成前端直接 resume。
- 不要让 queue/broker 信号反向成为业务真相。
- 不要把 WM 持久化成 resume authority。
- 不要新增一层散乱 helper/adapter 把 snapshot、artifact、runtime view 各自映射一份。
- 正确 owner 仍然应该是：
  - queue: re-dispatch transport only
  - control-plane: resume/checkpoint authority
  - artifact: content fact SoT
  - WM: per-run rehydratable context

# Avoid Repeat
