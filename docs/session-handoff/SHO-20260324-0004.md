  - 仓库里又长出了 summary projection、compatibility projection、frontend/helper read-model、Task fallback、trajectory/context 之类多种解释路径

- 这导致的问题不是简单的多写冲突，而是：
  - SoT 对象边界不够清楚
  - projection 与 read-model 容易被消费方误当真值

- 当前 canonical MAS 口径必须继续保持：
  - `shared WM / WorkflowState` 只承载 episode/domain facts 与 deliverable facts
  - `workflow_session / node / attempt / gate / decision` 承载 runtime/control-plane SoT
  - published deliverables 是 stage-boundary snapshot / authoritative boundary carrier，不是另一套 live runtime SoT
  - agent trajectory (`obs_records` / agent WM facts) 是 agent-local planning context，不是 workflow stage truth
  - `runtime view` 是给 orchestrator / API / frontend 的正式 read model
  - `workflow_overview`、`MASStateView`、`Task.status/current_step/progress`、compatibility diagnostics projection 只能是 summary / projection / fallback surface，不能重新长成行为真值

## 2. 当前真正存在的问题不是“很多个并列 SoT”
- canonical 上本来就有至少四类不同语义对象：
  - `episode/domain facts`：shared WM / WorkflowState / deliverable facts
  - `runtime execution state`：`workflow_session / node / attempt / gate / decision`
  - `stage-boundary carrier`：published deliverable refs + payloads + `session.input_payload.published_deliverables`
  - `agent-local trajectory/context`：Agent WM `facts` / `obs_records`
- 问题在于这些对象外面又叠加了多条 projection/read-model 路径：
  - `workflow_overview`
  - `MASStateView`
  - `Task.status / current_step / progress_percentage`
  - frontend `runtimeReadModel`
  - compatibility diagnostics projection
  - shared-WM projected published-deliverable refs
- 所以要审计的不是“删到只剩一个真值”，而是“不同问题必须指向唯一正确的语义对象，projection 不得越权”。

## 3. 建议冻结的 SoT / Projection 矩阵
- 问题：`当前 run 的 live 状态是什么？`
  - 正式答案：`RuntimeSessionService.build_runtime_view_for_task(_sync)` 返回的 runtime view
  - 禁止答案：`workflow_overview.status`、`MASStateView.status`、`Task.status` 推理、compatibility projection
- 问题：`某个 agent/scene/project 已产出了什么交付物？`
  - 正式答案：shared WM 下的 deliverable facts，例如 `scene_outputs.*`、`project.scene_scripts`、`project.final_video`
  - 禁止答案：从 `Task.output_metadata` 或 UI summary 反推
- 问题：`下游 stage 应该消费哪个正式输入版本？`
  - 正式答案：published deliverable ref / payload + runtime carrier (`session.input_payload.published_deliverables`)
  - 允许的过渡 fallback：显式、可诊断的 projection fallback
  - 禁止答案：默认回读 stale live WM 来假装 boundary 一致
- 问题：`agent 在迭代里观察过什么、计划过什么？`
  - 正式答案：Agent WM `facts` / `obs_records`
  - 禁止答案：把 trajectory 当 workflow stage 或 runtime state
- 问题：`前端怎么显示当前状态？`
  - 正式答案：消费 runtime view，再用 [src/lib/runtimeReadModel.ts](src/lib/runtimeReadModel.ts) 做纯展示层推导
  - 禁止答案：再从 `Task.status`、`workflow_overview`、`MASStateView` 拼一套状态机
