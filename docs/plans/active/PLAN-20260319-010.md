## 1. Current Problem
- 近期真实运行与独立诊断脚本表明，当前 orchestrator 前置规划的瓶颈已经前移到 `_llm_decompose_tasks()`，而不是 `concept_planner`。
- 诊断脚本 [backend/scripts/diagnostics/probe_orchestrator_planning.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/scripts/diagnostics/probe_orchestrator_planning.py) 已证明：
- `candidate_selection` 使用同一模型成功返回，耗时约 `22s`
- `decomposition` 使用同一模型与近似输入规模，却在约 `115s` 超时失败
- 当前 `decomposition` 的输出 contract 混合了三层语义：
- `agent capability prior`
- `dynamic task assignment`
- `subagent local planning`
- 这种混合导致两个问题同时出现：
- orchestrator 输出过重，生成成本高
- subagent 接收到的 `task` 语义不清晰，边界容易漂移到 `how`
- 当前 `task_spec` 字段包含：
- `instruction`
- `goal`
- `expected_output`
- `boundaries`
- `scope`
- `order`
- `runtime_overrides`
- `conditional_tasks`
- 其中一部分字段与 `selection` 阶段重复，另一部分则越过 orchestrator 边界，开始替 subagent 预做执行层规划。
- Post-successor refresh:
- 本计划的核心边界目标仍有效，但其 closeout 口径已经落后于后续收敛：
  - `PLAN-20260323-016` 已冻结 canonical supporting-capability host 为 `ContextContractAssembler`，`ExecutionBoundaryAssembler` 仅剩兼容别名。
  - `PLAN-20260324-017` 已冻结 SoT / projection / compatibility-only 边界。
  - `PLAN-20260324-019` 已冻结 backend authority contract / frontend consumer discipline。
- 因此，本计划当前需要先完成 governance refresh，再判断是否进入最终 closeout；不能继续按 2026-03-19 的原始叙述直接关单。

## 2. Target
- 保留 `llm-centric orchestrator` 的两步结构，不回退成规则中心或硬编码路由：
- `candidate selection`
- `decomposition`
- 纠正两步之间的职责边界：
- `selection` 负责选择参与的 subagent，并说明选择理由
- `decomposition` 负责为已选 subagent 生成本轮 `assignment contract`
- `subagent` 负责基于 assignment contract、上下文、能力先验与工具 schema 做本地规划与执行
- 将 `decomposition` 从“重的 per-agent execution spec”收敛成“可被 subagent 使用的任务指令 contract”，而不是执行步骤说明。
- 在第一轮重构中，先验证最小 assignment contract 是否能显著减轻 `_llm_decompose_tasks()` 的负担，并保留后续增强空间。

## 4. Architecture Principles
- P1 Preserve Two-Step LLM Planning:
- 保留 `candidate selection + decomposition` 两步结构，不降级为单步路由或规则中心。

- P2 Boundary Clarity:
- orchestrator 只负责：
- `who`
- `what`
- `deliverable`
- `constraints`
- `priority/order`
- subagent 负责：
- `how`
- 工具选择
- 内部步骤
- 执行反思

- P3 Separate Prior from Assignment:
- agent 的静态能力、典型交付与抽象职责属于 capability prior，不应在每次 decomposition 中重复生成。
- decomposition 只表达本轮任务的动态 assignment 增量。

- P4 Selection Reasons Stay in Selection:
- `selected_reason` / `selection_rationale` 只属于 selection 阶段，不进入 decomposition contract。

- P5 Contract First, Not Micro-Plan:
- decomposition 必须输出可被 subagent 直接消费的任务 contract，而不是 tool-level 或 step-level 执行方案。

- P6 Minimal First Pass:
- 第一轮先以最小必需字段验证边界与性能，再决定是否按需增加 `dependencies`、`acceptance_criteria` 等增强字段。
- 若存在 runtime activation 所需的附加结构，它必须被明确降级为 bounded appendix，而不是重新膨胀 core per-agent assignment contract。

- P7 Successor Guardrails:
- 本计划 closeout 必须服从 `016/017/019` 已冻结的 canonical host、SoT / projection matrix 与 authority-contract 边界。
- 不允许为了保持旧文案一致，而反向删除当前已经被后续架构吸收的有效 contract 结构。

## 5. Implementation Approach
- 冻结 selection、decomposition、subagent local planning 三层边界。
- 将 orchestrator decomposition prompt/schema 从“执行指令生成”收敛成“assignment contract 生成”。
- 第一轮最小 decomposition core contract 固定为：
- `agent`
- `run`
- `mission`
- `deliverable`
- `constraints`
- `order`
- `runtime_hints`
- 第一轮明确移出 core per-agent assignment contract 的字段：
- `selected_reason`
- `instruction`
- `goal`
- `expected_output`
- `boundaries`
- `scope`
- `conditional_tasks`
- `dependencies`
- `acceptance_criteria`
- 其中：
- `selected_reason` 保留在 selection 阶段
- `dependencies` / `acceptance_criteria` 作为后续增强字段，暂不纳入第一轮
- `conditional_tasks` 若仍存在，只能作为 top-level runtime-activation appendix，被 control-plane / standby-activation 路径消费，而不是重新进入 subagent 主 assignment contract
- orchestrator 保持 `run/order/runtime_hints` 对 queue / standby / execution boundary 的现有驱动能力。
- 强化 subagent 对 assignment contract 的显式消费路径，使 `task` 不再只是弱语义 metadata，而是进入 ReAct PLAN 上下文时具有清晰 contract 含义。
- 使用 planning probe 复测 selection 与 decomposition 的时间、输出长度和错误形态，判断重构是否解决当前超时问题。
- Closeout refresh note:
- 当前 mainline 语义以 `runtime_hints` 为 primary contract；repo 内残留的 `runtime_overrides` 仅按 compatibility debt 记录，不作为本计划“未完成主语义切换”的证据。
- 当前 canonical supporting-capability host 以 `ContextContractAssembler` 为准；历史 `ExecutionBoundaryAssembler` 引用仅保留为实现演进痕迹。

## 7. Risks / Blockers
- 如果只为了压缩输出长度而删字段，而不修正语义边界，会把问题转移成隐式缺信息。
- 如果 `task` 仍只作为弱 metadata 存在，即使 decomposition 变轻，也不算真正打通 llm-centric orchestrator。
- 如果在第一轮就把过多增强字段重新塞回 contract，超时和边界漂移都容易回归。
- capability prior 目前仍较薄；本轮只能先把“每轮重复生成的动态 assignment”收敛到合理粒度，不能一次完成 prior/skill 全量升级。
