## 1. 当前问题
- `032-035` 已经完成并提交；当前问题不是旧 blocker 回归，而是 quick mode 首页“继续查看当前任务”与控制层恢复执行语义之间的新缺口。
- 最新证据已经闭合：
  - [SHO-20260329-0013.md](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/docs/session-handoff/SHO-20260329-0013.md) 明确指出，API 重启后只有 `GET /tasks/quick/current` 与 WebSocket subscribe，没有新的 runtime decision POST，也没有新的 `process_video_task[...] received`。
  - [tasks.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/api/v1/endpoints/tasks.py) 的 `GET /quick/current` 只返回当前 runtime read-model，不做 resume/re-dispatch。
  - [VideoRequestForm.tsx](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/src/components/forms/VideoRequestForm.tsx) 的 `handleContinueExistingRun()` 只恢复 `currentRequest`、`quickRuntime` 与前端步骤，不触发任何控制层命令。
  - [QuickModeWorkspace.tsx](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/src/components/preview/QuickModeWorkspace.tsx) 里真正会恢复后端执行的只有 script gate `approve/revise/replan`。
- 当前“继续查看当前任务”仍有局部价值：
  - 浏览器刷新后继续看同一 run。
  - `waiting_gate` run 重新进入工作台，然后继续提交 gate decision。
  - 真实仍在执行中的 run 重新订阅 WebSocket / polling。
- 但对“non-terminal 且实际没有 worker 在跑”的旧 run，当前入口会把旧 runtime view 重新挂回前端，用户容易误以为后台也会继续执行。

## 2. 架构边界与硬约束
- queue / worker / broker 只负责 transport 和 execution container，不拥有 resume 业务语义。
- runtime / session / node / attempt / gate / decision / checkpoint 的 SoT 只允许归 control-plane。
- artifact / published deliverable 只承载内容事实，不承载 resume authority。
- WM 只是可重建工作区，不得升级为跨 host 恢复 owner。
- 不允许把 snapshot 做成第二套业务模型，不允许持久化整份 MAS working memory，不允许新增散乱 helper / adapter 旁路。
- 不允许把前端“继续查看”偷偷改成“恢复执行”；resume 必须是 control-plane explicit command。
- 不允许让 queue 活性信号越权成为业务真相；若使用队列信息，也只能作为 control-plane read-model / guard 的 transport 诊断输入。

## 3. 方案判定与冻结结论
### 3.1 备选项判断
| 选项 | 结论 | 原因 |
| --- | --- | --- |
| `1. 纯文案 / 产品语义修正` | 不足 | 只能让当前入口更诚实，不能补齐 stalled run 的恢复能力 |
| `2. control-plane explicit resume 命令` | 单独实施风险过高 | 没有“何时允许 resume”的 authoritative guard，容易对真实仍在执行的 run 重复 re-dispatch |
| `3. runtime read-model 区分 running vs stalled/resumable` | 单独实施不完整 | 只能告诉前端“当前不是活跃执行”，但没有恢复执行的单一路径 |
| `4. 最小组合` | 选中 | 需要 `2 + 3`，并配一个很小的 `1` 做文案对齐 |

### 3.2 冻结后的最小组合
- 选定方向：`control-plane explicit resume command + runtime read-model 区分 stalled/resumable`，前端只做语义对齐，不承担恢复 owner。
- 文案层仅做语义对齐：
  - 首页入口不再暗示“点击即可恢复执行”。
  - 若当前只是 reattach 旧视图，应明确表达为“查看当前任务”或等价语义。
- read-model 层必须显式区分：
  - `waiting_gate`：继续查看即可，真正恢复依然走 gate decision。
  - `actively_running`：继续查看即可，前端只重连 WS / polling。
  - `stalled_resumable`：允许用户显式点击“恢复执行”，但不能由 `quick/current` 隐式触发。
- explicit resume 命令由 control-plane 所有：
  - 输入是 task/session 的 authoritative runtime state。
  - 输出是新的 authoritative runtime view。
  - transport 层只负责 re-dispatch，不解释 resume 业务语义。

### 3.3 Phase 1 contract freeze
- 当前 repo 没有 heartbeat 或运行中 lease；control-plane 可稳定读取的事实仍是：
  - runtime session / node / attempt / gate / decision
  - `task.updated_at` / `session.updated_at`
  - transport handle `output_metadata.celery_task_id`
- 当前 repo 也没有任何地方在 quick/project job 成功或失败后统一清理 `celery_task_id`，所以该字段只能作为 transport probe key，不能直接表达“仍在执行”。
- 因此本计划冻结为“control-plane projection + conservative transport diagnostics”：
  - `queue` 可以提供 per-task transport probe 输入。
  - `control-plane` 负责把 runtime facts、freshness、transport probe 折叠成一个 read-model projection。
  - 若 probe 证据不足，projection 必须返回 `resume_unknown`，而不是乐观地给出 `can_resume=true`。

### 3.4 runtime read-model projection contract
| 字段 | 类型 | 含义 | 冻结结论 |
| --- | --- | --- | --- |
| `resume_control.state` | enum | 当前 run 在首页/工作台的恢复语义 | 必需 |
| `resume_control.can_resume` | bool | 是否允许显式 resume | 必需 |
| `resume_control.reason_code` | string | 稳定原因码，供前端 copy 和 fail-fast 使用 | 必需 |

- `resume_control.state` 冻结为四档：
  - `waiting_gate`: 当前 run 需要人工 gate decision，不允许首页 resume。
  - `view_only_running`: 当前 run 只允许继续查看，不允许显式 resume。
  - `resume_available`: 当前 run 已被判定为 stalled，可显式 resume。
  - `resume_unknown`: 当前 run 非终态，但控制层无法证明 transport 已不活跃，因此暂不开放 resume。
- `reason_code` 冻结为最小稳定集合：
  - `awaiting_gate_decision`
  - `transport_active`
  - `runtime_recent`
  - `stalled_runtime_no_live_transport`
  - `transport_state_unknown`
- 该对象是 runtime view projection，不得回写数据库，不得变成新的持久状态机。

### 3.5 projection 判定规则冻结
1. 若 `active_gate.status == awaiting_human`，则：
   - `state=waiting_gate`
   - `can_resume=false`
   - `reason_code=awaiting_gate_decision`
2. 若 transport probe 明确给出 live（例如 active / scheduled / reserved），则：
   - `state=view_only_running`
   - `can_resume=false`
   - `reason_code=transport_active`
3. 若 transport probe 不为 live，且 runtime/task freshness 仍在配置阈值内，则：
   - `state=view_only_running`
   - `can_resume=false`
   - `reason_code=runtime_recent`
4. 若 transport probe 明确为 not-live，且 runtime/task freshness 超出配置阈值，则：
   - `state=resume_available`
   - `can_resume=true`
   - `reason_code=stalled_runtime_no_live_transport`
5. 若 freshness 已超阈值，但 transport probe 仍无法证明 not-live，则：
   - `state=resume_unknown`
   - `can_resume=false`
   - `reason_code=transport_state_unknown`

### 3.6 transport probe contract freeze
- probe 只允许回答 transport 侧问题：这个 queue handle 当前是否仍处于 live container 中。
- probe 允许的输入与证据：
  - `celery_task_id`
  - Celery inspect `active/scheduled/reserved`
  - Celery result backend terminal state（若可稳定取得）
  - debug in-process runner 的本地活跃分支（若显式启用）
- probe 的输出只允许是：
  - `live`
  - `not_live`
  - `unknown`
- 明确禁止：
  - 通过 `celery_task_id` 是否存在来直接判定 live
  - 把 probe 结果持久化到 runtime session / node / attempt
  - 让 queue/broker 字段直接驱动业务状态迁移

### 3.7 explicit resume command contract freeze
- endpoint 冻结为独立 control-plane 命令：`POST /tasks/{task_id}/runtime/resume`
- 最小响应面与 script gate decision 保持同风格：
  - `message`
  - `task_id`
  - `runtime`
- 该命令的前置 guard：
  - task / runtime session 存在且为 quick mode
  - runtime 非 terminal
  - `resume_control.state == resume_available`
  - 不允许绕过 `waiting_gate`
- 该命令的 side effects：
  - control-plane 先重算 authoritative `resume_control`
  - 若允许 resume，则把 session 切到 `RESUMING`，保留当前 node / attempt anchor
  - 调 `_schedule_task_execution(...)` 做 re-dispatch
  - 返回新的 authoritative runtime view
- 该命令的禁止行为：
  - 不新增第二条 queue-owned 状态线
  - 不伪造 node/gate/decision 迁移
  - 不复用 script decision endpoint 语义

### 3.8 frontend contract freeze
- “继续查看当前任务” 永远只做 view reattach，不得隐式触发 resume。
- 只有 `resume_control.state == resume_available` 时，前端才允许展示单独的“恢复执行” CTA。
- 当 `state == resume_unknown` 时：
  - 允许继续查看旧 runtime view
  - 不展示 resume CTA
  - 文案必须明确“当前无法确认后台是否仍在执行”

## 4. 单一路径语义冻结
1. `GET /tasks/quick/current` 继续只负责“发现当前 workspace 的 latest relevant quick run + 返回 authoritative runtime read-model”。
2. 前端点击“继续查看当前任务”时，仍然只做 view reattach：
   - 恢复 `currentRequest`
   - 恢复 `quickRuntime`
   - 重连 WebSocket
   - 恢复 polling
3. `runtime read-model` 新增一个 control-plane 投影，用来表达该 run 当前是：
   - 仅可查看
   - 等待 gate decision
   - 可显式 resume
4. 只有当 read-model 投影明确给出“可显式 resume”时，前端才展示独立的“恢复执行”动作。
5. `POST /tasks/{task_id}/runtime/resume` 之类的 explicit resume 命令由 control-plane 校验：
   - task / runtime session 存在且 non-terminal
   - 当前不处于 `waiting_gate`
   - 当前没有可确认的 live transport execution
   - 当前 runtime anchor 满足恢复前提
6. 校验通过后，control-plane 显式调用 re-dispatch；queue 只作为 transport 执行容器。
7. 前端继续通过现有 WS / polling 消费 authoritative runtime view，不引入第二套前端状态机。

## 5. read-model owner 冻结
- `WorkflowSession.status` 继续保留现有生命周期含义：`queued/running/waiting_gate/resuming/completed/failed/cancelled`。
- 本计划不新增新的持久化 runtime status 字段，不把 `stalled` / `resumable` 落成新的 DB SoT。
- `stalled/resumable` 只允许作为 control-plane 派生 read-model：
  - 基于当前 runtime lifecycle 锚点
  - 基于 task queue handle / transport 活性诊断
  - 基于 gate / attempt / decision 等已有控制层事实
- 当前仓库里并没有原生 `stalled` / `resumable` 持久状态；若需要这类表达，只能作为 runtime view projection，而不是修改主状态机 owner。

### 5.1 projection anti-expansion guard
- `resume_control` 不是 canonical runtime status，不是新的 workflow lifecycle，也不是 queue/worker/orchestrator 的推进依据。
- `resume_control` 只允许承担两类职责：
  - runtime/read-model 对外解释层
  - explicit resume command 的 eligibility guard
- 明确禁止以下扩展方向：
  - 把 `resume_control.state` 持久化到数据库或 artifact
  - 让 worker、queue、WebSocket consumer、前端 store 依据 `resume_control` 推进 node/gate/attempt
  - 把 `resume_control` 扩成通用业务状态总线，例如承载 `paused/stuck/recovering/...` 全套主语义
- 若未来产品确实需要新的主线生命周期语义，必须回到 control-plane canonical model 增补领域状态/命令契约，而不是继续膨胀 projection。
- 任何新增 projection 字段都必须继续满足：
  - 可由 canonical facts 纯函数重算
  - 不落库
  - 不成为 transport 或 orchestrator 的 SoT

## 6. explicit resume owner 冻结
- explicit resume 不属于前端，不属于 WebSocket hook，不属于 polling hook。
- explicit resume 入口必须落在 control-plane API / service：
  - endpoint 负责接收用户意图并返回 authoritative runtime view。
  - `RuntimeSessionService` 一类 control-plane service 负责判定 resume eligibility。
  - `_schedule_task_execution(...)` 仍然只是 transport re-dispatch 调用点。
- script gate `approve/revise/replan` 继续保留现有 resume 语义；本计划不把它们与首页 continue CTA 混成同一入口。

## 8. 风险与护栏
- 风险 1：
  - 直接把首页按钮改成自动 resume。
  - 护栏：continue CTA 永远不是 resume owner。
- 风险 2：
  - 只加 resume API，不做 read-model 区分。
  - 护栏：没有 eligibility/read-model guard 的 resume 不允许上线。
- 风险 3：
  - 为了做 read-model 区分，新增持久化 `stalled` / `resumable` 状态字段。
  - 护栏：该计划只允许派生 projection，不新增第二套持久状态机。
- 风险 4：
  - 把 queue 活性探测变成业务 SoT。
  - 护栏：queue 只提供 transport 诊断和 re-dispatch 执行容器，不解释 node/gate/decision 语义。
- 风险 5：
  - 用 WM / artifact / snapshot 去承接 stalled resume owner。
  - 护栏：resume owner 只允许在 control-plane。
- 风险 6：
  - 把 `celery_task_id` 是否存在直接解释成 live execution。
  - 护栏：handle 只是 transport probe key；没有 inspect/result 佐证时只能给 `unknown`。
- 风险 7：
  - 为了提升恢复率，在 transport 证据不足时乐观开放 resume。
  - 护栏：证据不足时返回 `resume_unknown`，而不是 `resume_available`。

## 10. Residual Gap After Review
- `POST /tasks/{task_id}/runtime/resume` currently guards via `resume_control`, then flips runtime status to `RESUMING` and re-dispatches; it does not consume a new generic stale-run continuation checkpoint.
- The only authoritative continuation consume path that orchestrator recognizes today remains the script-gate branch, so non-gate stale runs still lack a control-plane-owned checkpoint carrier.
- `resume_control.reason_code` precedence has drifted from the frozen contract because freshness is evaluated before transport-live proof.
- This plan no longer claims semantic closure. The remaining architecture work is to freeze `carrier / consumer / reason precedence / completion gate` in successor plan `037`.
