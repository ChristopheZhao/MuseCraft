## 1. 当前问题
- 当前 MAS 主线联调暴露了 `workflow_gates` 的 schema drift：运行时读路径在查询 `WorkflowGate` 时会显式选出 `scope` 列，但实际数据库表结构里没有该列，导致 orchestrator 在读取 gate projection 时直接失败。
- 报错位置：
  - [orchestrator.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/agents/orchestrator.py)
  - [runtime_session_service.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/services/runtime_session_service.py)
- 根因不是 project-level，也不是业务数据为空，而是 ORM 模型与 Alembic schema 不一致：
  - [workflow_runtime.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/models/workflow_runtime.py) 中 `WorkflowGate` 已声明 `scope` 与 `diagnostics`
  - [8b1f0c2d4e5f_add_workflow_runtime_tables.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/alembic/versions/8b1f0c2d4e5f_add_workflow_runtime_tables.py) 建表时未创建这两列
- 当前任务是把这条 drift 修正回 schema contract；不允许通过查询裁列、兼容分支、silent fallback 去绕过错误结构。

## 2. 架构边界与硬约束
- 本计划只处理 MAS 主线的 runtime gate schema alignment，不 reopen `024/027/028`，也不引入新的 project-level 范围。
- Hard rules:
  - 不新增 query-side compat seam；不得通过“少查几列”或条件分支绕过缺列数据库。
  - 错误设计按 schema contract 修正；不保留 silent fallback。
  - `workflow_gates` 的持久化 contract 以当前 ORM 模型为准；迁移必须把数据库拉回该 contract。
  - 若发现不只 `scope` 缺失，必须在同一 schema-alignment 阶段一并处理，避免修一列再炸一列。
  - 本计划不顺手改 `RuntimeSessionService` / `OrchestratorAgent` 业务逻辑；除非为验证迁移所需，不改 runtime 读写语义。

## 3. 范围
### 3.1 Scope in
- `workflow_gates` 表结构与 `WorkflowGate` ORM 模型对齐
- 缺失列审计与补迁移
- focused validation / governance sync

### 3.2 Scope out
- project-level workflow / read-model / host 逻辑
- single-episode runtime 行为改造
- frontend / websocket / polling 消费层
- 与本次 schema drift 无关的 controlled-transition cleanup

## 6. 风险与护栏
- 风险 1：
  - 只补 `scope`，遗漏 `diagnostics`，导致修一列再炸一列。
  - 护栏：Phase 1 以 ORM-vs-migration 全量对齐为验收，不按单报错列修补。
- 风险 2：
  - 为了临时跑通，把 runtime query 改成不查缺列，形成 compat seam。
  - 护栏：本计划禁止 query-side workaround；只允许 schema migration 修正。
- 风险 3：
  - migration 只补列不 backfill，导致历史 gate 记录投影不稳定。
  - 护栏：Phase 1 明确要求最小 backfill。
