# Plan: HITL script-gate resume path control-plane alignment
- Plan ID: PLAN-20260327-031
- Type: fix
- Status: completed

## 1. 当前问题
- 当前 MAS 主线在 script HITL 审批后能够正确恢复到后续节点，但恢复链路存在明显的控制面冗余：
  - 审批提交后会重新入队并进入新的 queued execution host
  - host bootstrap 期间会重新初始化 tool registry、event bus、orchestrator 与各 leaf agent
  - 更关键的是，orchestrator 在读取到 `script_resume_action="approve"` 之后，仍先执行 `_llm_select_candidate_agents(...)` 与 `_llm_decompose_tasks(...)`，直到后面才把 `concept_planner` / `script_writer` 放入 `skip_agents`
- 这导致审批通过后的恢复语义没有被当成“控制层已决定的 continuation”，而是被错误地重新当成了一次 top-of-flow 规划入口。
- 当前根因不是：
  - script gate boundary 本身错误
  - queue/worker 容器错误拥有业务语义
  - script 节点被业务上重复执行
- 当前根因是：
  - HITL 的 live runtime ownership 在控制层已经 materialize 为 `gate / decision / session.status=RESUMING`
  - 但编排层在消费该 authoritative resume state 之前，又先跑了一轮 LLM planning
  - repo 已经持久化了 `workflow.task_specs` 等 orchestration projection，但 approval resume 没把它们当 continuation authority 使用

## 2. 架构边界与硬约束
- 本计划只处理 single-episode MAS mainline 的 HITL resume-path 对齐，不 reopen project-level 范围，也不把 [PLAN-20260327-029.md](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/docs/plans/active/PLAN-20260327-029.md) 或 [PLAN-20260327-030.md](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/docs/plans/active/PLAN-20260327-030.md) 混入实现范围。
- HITL 的层级归属冻结如下：
  - 治理层：定义 review / closeout criteria
  - 门控层：产出 review boundary、gate result、human decision input
  - 控制层：主写 `workflow_session / node / attempt / gate / decision`，并承接 `approve / revise / replan` 后的 runtime transition
  - 编排层：消费控制层给出的 authoritative runtime state，不得把 `approve` continuation 重新解释成新的 top-level planning 起点
- Hard rules:
  - 不新增 queue-side、worker-side 或 API-side compat seam 去掩盖 resume-path 设计问题。
  - queue / worker 仍只做 transport / execution container，不得拥有 HITL 业务语义。
  - 不在 gate 层直接推进 session lifecycle；session/gate/decision 的 live state transition 仍只允许控制层主写。
  - 不通过“继续保留前置 LLM 规划、只是在执行队列里 skip”这种半兼容方式收口。
  - 不把 `approve` / `revise` / `replan` 混成一个统一分支；三种 action 必须保持显式语义差异。
  - 如需复用 persisted orchestration projection，只允许把它作为 authoritative continuation input 使用；不得把 diagnostics-only snapshot 偷偷升级成新的控制面 owner。

## 2.1 Phase 1 冻结结果
- `approve`：
  - 属于控制层 continuation，不属于新的 top-level planning 入口。
  - authoritative continuation 应直接从 post-script 阶段恢复，禁止再次执行 `_llm_select_candidate_agents(...)` 与 `_llm_decompose_tasks(...)`。
  - `script_review_contract` 在 `approve` 时应继续被清空；恢复输入不应再携带 review guidance。
  - authoritative continuation input 冻结为 persisted `workflow.task_specs` + `workflow.conditional_tasks` 的反序列化结果；`workflow.diagnostics.compat.activation_pool_snapshot` 仍只算 diagnostics，不得升级成 owner。
- `revise`：
  - 属于 node-scoped script continuation，不属于 workflow-level replanning。
  - authoritative target 继续冻结为 `script_writer`，与 [script_review_contract.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/services/script_review_contract.py) 中 `target_agents=["script_writer"]` 保持一致。
  - `revise` 必须继续携带 `script_review_contract` 进入 `script_writer`，但不应重新触发 candidate selection / task decomposition。
- `replan`：
  - 属于 workflow-scoped continuation，允许重新进入 candidate selection / task decomposition。
  - authoritative target 继续冻结为 `concept_planner + script_writer`，与现有 `script_review_contract` 和 `invalidation_scope="workflow"` 保持一致。
  - 只有 `replan` 可以合法重新打开 top-level planning；`approve` 与 `revise` 都不应走该路径。
- Phase 2 实现边界：
  - 允许新增从 persisted `workflow.task_specs` / `workflow.conditional_tasks` 回读 authoritative continuation queue 的 loader / adapter。
  - 不允许把 queue host、gate layer 或 diagnostics projection 升级成 continuation owner。

## 3. 范围

- `RuntimeSessionService` / orchestrator resume-path 的边界对齐
- persisted orchestration projection 是否可作为 approve continuation 输入的收敛

### 3.2 Scope out

- 目标：
  - 冻结 HITL 的架构归属与本轮根因，确认本轮问题属于控制层 continuation 语义没有在编排入口前被消费，而不是 gate 层或 queue 层 owner 错位。
- 执行清单：
  - [x] 对照架构文档确认控制层 / 门控层职责边界
  - [x] 定位 script review gate open / decision submit / worker resume / orchestrator skip 的调用链
  - [x] 确认 host bootstrap 重新执行是容器层预期行为，但不是本轮主 bug
  - [x] 确认 `_llm_select_candidate_agents(...)` / `_llm_decompose_tasks(...)` 发生在 `approve` skip 之前
  - [x] 确认 `workflow.task_specs` 已有持久化 projection，但 approval resume 尚未把它作为 authoritative continuation queue 使用

### Phase 1: approve continuation contract freeze
- 目标：
  - 冻结 `approve / revise / replan` 三种 HITL action 的 authoritative continuation contract，并决定 approve 路径应如何绕开冗余前置规划。
- 执行清单：
  - [x] 明确 `approve` 的恢复语义：直接进入 post-script continuation，而不是重新做 candidate selection / task decomposition
  - [x] 明确 `revise` / `replan` 的恢复语义：保留重新规划入口，但只在 action 语义需要时触发
  - [x] 盘点 persisted `workflow.task_specs` / candidate activation projection 的可用性，决定 approve continuation 的 authority source
  - [x] 冻结不修改 queue/worker ownership 的实现护栏

### Phase 2: control-plane driven resume-path implementation
- 目标：
  - 把 `approve` resume-path 前移到编排入口之前，使控制层 authoritative continuation 在进入 LLM planning 前即生效。
- 执行清单：
  - [x] 在 orchestrator resume 链路中前移 `approve` continuation 分支
  - [x] 对 `approve` 路径避免 `_llm_select_candidate_agents(...)` 与 `_llm_decompose_tasks(...)` 冗余调用
  - [x] 保持 `revise` / `replan` 语义正确且显式
  - [x] 保持 queue host、gate layer、runtime SoT ownership 不被重新发明或越层

### Phase 3: focused validation and governance sync

  - [x] 补 focused backend tests / probes，覆盖 `approve` 不再重跑前置 LLM planning

## 6. 风险与护栏
- 风险 1：
  - 误把 host bootstrap 冗余当成本轮主根因，转而去做 queue/worker 基础设施重构。
  - 护栏：本计划只修 control-plane continuation consumption，不做 host/container 体系重构。
- 风险 2：
  - 为了减少调用，直接把 HITL `approve / revise / replan` 三个 action 压平成同一 resume 分支。
  - 护栏：Phase 1 必须冻结三种 action 的显式语义差异，再进入实现。
- 风险 3：
  - 复用 `workflow.task_specs` 时越权，把 diagnostics projection 偷升级成新的 runtime SoT。
  - 护栏：只允许把既有 projection 作为 continuation input 使用；live gate/session/decision ownership 仍在控制层。
- 风险 4：
  - 通过在执行队列继续 skip 来掩盖冗余前置规划，使 bug 看似不影响业务但持续耗时。
  - 护栏：Phase 2 验收要求明确覆盖“`approve` 之前不再触发前置 LLM planning”。

## 7. 状态日志
- 2026-03-27T14:14:42Z created this new MAS-mainline fix plan after resume-path review showed the current script HITL chain is architecturally a control-plane continuation problem rather than a gate-layer ownership problem: script review boundary publication and gate opening are aligned, queue/worker host restart remains a transport/container concern, but `approve` continuation is consumed too late because orchestrator still performs candidate selection and task decomposition before applying runtime skip semantics. Phase 0 therefore freezes the stream to control-plane-driven resume-path alignment only, keeps `029` and `030` out of scope, and records that persisted `workflow.task_specs` already exist as orchestration projection but are not yet treated as authoritative continuation input for approval resume.
- 2026-03-27T14:23:46Z completed Phase 1 without entering implementation. The continuation contract is now frozen as follows: `approve` is a control-plane continuation that must resume from post-script execution without top-level planning and should consume deserialized `workflow.task_specs` plus `workflow.conditional_tasks` as authoritative input; `revise` is a node-scoped script continuation that keeps `script_review_contract` and targets only `script_writer`; `replan` is the only workflow-scoped action that may legitimately re-enter candidate selection and task decomposition. `workflow.diagnostics.compat.activation_pool_snapshot` remains diagnostics-only and may not be promoted into continuation ownership.
- 2026-03-27T14:38:40Z completed Phase 2 without entering Phase 3. Added a narrow orchestration-state loader for persisted `workflow.task_specs` / `workflow.conditional_tasks`, moved `approve` and `revise` onto a control-plane-driven resume branch that bypasses top-level candidate selection and task decomposition, kept `replan` on the existing replanning path, and preserved queue/worker plus gate ownership boundaries. Focused verification passed for the three direct resume cases: approve skips fresh planning, revise reruns only `script_writer`, and replan still re-enters top-level planning.
- 2026-03-27T14:56:40Z corrected the remaining Phase 2 control-plane leakage before entering closeout. Architecture review found that `approve` continuation still let `orchestrator.py` directly mutate live runtime state (`node/session/task + db.commit()`), which violated control-plane ownership even though top-level planning had already been bypassed. The fix moved that `APPROVED -> COMPLETED` and `RESUMING -> RUNNING` transition into `RuntimeSessionService.consume_script_approval_continuation_sync(...)`, left orchestrator as a consumer of the control-plane command only, and added focused verification for both the service-level transition and the three orchestrator HITL resume cases.
- 2026-03-27T15:20:03Z completed Phase 3 and moved the plan to awaiting user confirmation without lifecycle closeout. Governance review now records that approve continuation no longer mutates runtime live state from the orchestration layer, focused validation covers both the control-plane transition and the three HITL resume paths, queue/worker remain transport-only, and the stream is ready for explicit user acceptance before any final closeout.
- 2026-03-27T15:23:11Z completed lifecycle closeout after explicit user confirmation. The plan remains architecture-first: HITL approval/revise/replan continue to respect governance/gate/control/orchestration layering, queue and worker remain transport-only, and the corrected control-plane continuation boundary is now accepted as the new baseline for subsequent MAS-mainline integration work.
