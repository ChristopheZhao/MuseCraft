## 1. 当前问题
- `036` 已经把 quick mode 的“继续查看当前任务”与“恢复执行”分开，并且守住了 queue/worker transport-only 的架构边界。
- 但 `036` 的 explicit resume 仍然不是 generic checkpoint resume：
  - [tasks.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/api/v1/endpoints/tasks.py) 当前只根据 `resume_control.can_resume` 放行，然后把 session 标成 `RESUMING` 并重新投递。
  - [runtime_session_service.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/services/runtime_session_service.py) 的 `mark_session_resuming_sync(...)` 只改 session/task 状态，不装载新的 continuation anchor。
  - [orchestrator.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/agents/orchestrator.py) 里真正会消费 checkpoint 的仍只有 script-gate continuation 路径。
- 因此，对“non-gate 且 stale 的 quick run”，当前实现更准确的名字是 guarded re-dispatch，而不是 control-plane checkpoint resume。
- `resume_control.reason_code` 也与 `036` 冻结的解释顺序不一致：当前实现会先给出 `runtime_recent`，只有超 freshness 阈值后才 probe transport live/not-live。

## 2. 架构边界与前置真相
- `035` 已经冻结：`attempt.continuation_checkpoint` 是 continuation planning-state 的唯一 carrier；不得再引入 Shared WM projection、snapshot business model 或其他第二套 SoT。
- queue/worker/broker 继续只负责 transport/execution container，不拥有 runtime continuation 语义。
- runtime/session/node/attempt/gate/decision/checkpoint 的 SoT 继续只允许归 control-plane。
- `resume_control` 仍然只是 read-model projection + explicit command guard，不得升级为持久化主状态线。
- `036` 当前只作为 implementation snapshot / governance anchor，相关 checkpoint 提交为：
  - `4d41ccd chore(plan): align governance for PLAN-20260329-036`
  - `9191e56 checkpoint(wip): known-bad implementation snapshot before PLAN-20260329-036 next phase`

## 3. 方案判定与冻结结论
### 3.1 备选项判断
| 选项 | 结论 | 原因 |
| --- | --- | --- |
| `1. 保留当前 guarded re-dispatch，只改文案/命名` | 不足 | 可以让语义更诚实，但不能补上“resume 依赖 canonical continuation checkpoint”这一架构缺口 |
| `2. 为 generic stale run 建 control-plane continuation checkpoint 合同` | 选中 | 这是唯一能让 explicit resume 真正建立在 control-plane SoT 上的路径 |
| `3. 让 queue/worker 活性探测直接主导恢复语义` | 禁止 | 这会把 transport 重新抬升为业务真相 owner |

### 3.2 continuation carrier freeze
- `attempt.continuation_checkpoint` 继续是唯一 continuation carrier。
- 若 generic stale resume 需要额外 anchor 元数据，必须扩展在同一个 `continuation_checkpoint` 对象里，而不是新增 snapshot 表、artifact ref、WM projection 或第二个 checkpoint store。
- `037` 冻结的最小 carrier 扩展方向：
  - `continuation_checkpoint.version` 升级为 `2`，因为当前 validator 只保留 `version/decision_id/candidate_agents/task_specs/conditional_task_specs`，无法稳定承载 generic anchor 元数据。
  - `anchor_type`: 区分 `gate_decision` 与 `runtime_checkpoint`
  - `node_key`: 绑定当前 continuation 所属节点
  - `attempt_id`: 绑定当前 continuation 所属 attempt
  - `decision_id`: 仅 `gate_decision` continuation 允许存在；`runtime_checkpoint` 必须为 `null`
  - `candidate_agents / task_specs / conditional_task_specs`: 延续 `035` 的 authoritative planning payload，不新增平行字段
- schema 约束冻结为：
  - `anchor_type=gate_decision` => `decision_id` 必填，且 current session anchor 必须指向 gate 对应 attempt
  - `anchor_type=runtime_checkpoint` => `decision_id` 必须为空，且 `node_key/attempt_id` 必须和 `session.current_node_key/current_attempt_id` 一致
  - validator 负责保留并返回上述 anchor 字段；不允许再像当前实现那样把 extra keys 静默丢掉

### 3.3 continuation consumer freeze
- orchestrator/control-plane 必须新增 generic continuation consume path，而不是继续把 script-gate path 当成唯一 resume mainline。
- 最小 consumer contract：
  - `RuntimeSessionService` 新增 generic loader，例如 `load_active_continuation_sync(...)`，统一校验 `session.current_node_key + session.current_attempt_id + attempt.continuation_checkpoint`
  - 当前 `load_active_script_continuation_sync(...)` 只能保留为 script-gate specialization，不能继续当唯一 mainline consumer
  - `POST /runtime/resume` 不能再直接调用 `mark_session_resuming_sync(...)`；必须先通过 generic loader/consumer 完成 checkpoint 装载与 eligibility 校验
  - 没有 checkpoint、checkpoint 校验失败、checkpoint anchor 与当前 session anchor 不一致时，`POST /runtime/resume` 必须 fail-fast，不能退回裸 `RESUMING + re-dispatch`
  - `mark_session_resuming_sync(...)` 只能发生在 authoritative continuation consumer 成功之后，作为状态迁移后置动作，而不是替代 consumer 本身

### 3.4 runtime projection freeze
- `resume_control` 继续是 projection，不落库，但其 contract 必须与 continuation owner 一致。
- 本计划冻结新的 projection 判定流程：
  1. 若 `active_gate.awaiting_human`，返回 `waiting_gate / awaiting_gate_decision`
  2. 否则先做 transport probe；若 `live`，返回 `view_only_running / transport_active`
  3. 若 transport 非 `live` 且 runtime freshness 未超阈值，返回 `view_only_running / runtime_recent`
  4. 若 freshness 已超阈值但 transport probe 为 `unknown`，返回 `resume_unknown / transport_state_unknown`
  5. 若 freshness 已超阈值且 transport probe 为 `not_live`，但当前 anchored attempt 不持有可验证 checkpoint，返回 `resume_blocked / missing_continuation_checkpoint`
  6. 仅当 freshness 已超阈值、transport probe 为 `not_live`、且 checkpoint 校验通过时，返回 `resume_available / stalled_runtime_with_checkpoint`
- 只有当以下条件同时成立时，才允许 `resume_available`：
  - runtime 非 terminal
  - transport probe 明确 `not_live`
  - runtime freshness 已超阈值
  - 当前 anchored attempt 持有可验证的 canonical `continuation_checkpoint`
- 若 transport 未能证明 `not_live`，即使 checkpoint 存在，也只能返回 `resume_unknown`。
- 若 transport 已证实 `not_live`，但当前 anchored attempt 没有可验证 checkpoint，则必须返回 `resume_blocked`，不能继续把它归到 `resume_unknown` 或 `resume_available`。

### 3.5 explicit resume command freeze
- `POST /tasks/{task_id}/runtime/resume` 继续保留为单一显式入口，但语义必须升级为 “consume canonical continuation checkpoint”。
- 最小命令合同：
  - endpoint 重算 authoritative `resume_control`
  - 仅当 `resume_control.state == resume_available` 时，才允许进入 resume path
  - command 进入 control-plane consumer，装载当前 anchored checkpoint，并在成功后再切 session 到 `RESUMING`
  - `_schedule_task_execution(...)` 仍然只是 transport re-dispatch，不拥有 continuation 解释权
- 明确禁止：
  - 没有 checkpoint 时先把 session 置成 `RESUMING` 再说
  - 为了兼容 generic stale run 而伪造 gate decision / decision_id
  - 通过 queue/broker/result backend 直接构造 continuation business meaning

### 3.6 completion gate freeze
- `037` 不允许像 `036` 一样在“实现可工作但语义未闭合”时直接标成 completed。
- 关闭条件必须同时满足：
  - continuation carrier / consumer / reason precedence 已按单一路径实现
  - focused automated tests 通过
  - validation ledger 补齐
  - 用户显式确认

## 5. 风险与护栏
- 风险 1：
  - 沿用当前 `mark_session_resuming_sync(...) + _schedule_task_execution(...)`，只是再补条件判断。
  - 护栏：没有 canonical continuation consumer 的实现，不得宣称完成 generic resume。
- 风险 2：
  - 为 generic stale resume 新增第二个 checkpoint/snapshot 存储面。
  - 护栏：唯一 carrier 继续是 `attempt.continuation_checkpoint`。
- 风险 3：
  - 为了复用 script-gate 路径，伪造 decision binding。
  - 护栏：gate decision 与 generic runtime checkpoint 必须由 `anchor_type` 区分，不得伪造 `decision_id`。
- 风险 4：
  - 把 `resume_control` 再次扩成主状态线。
  - 护栏：projection 继续只服务于 read-model 和 command guard，不落库、不推进执行。
- 风险 5：
  - 为了提高恢复率，让 transport unknown 也开放 resume。
  - 护栏：transport 未证明 `not_live` 时，一律不能开放 resume。
