## 1. Current Problem
- 经过架构复盘，当前 single-episode 仍存在主线不统一的问题：
- [orchestrator.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/agents/orchestrator.py) 已被收为 `llm-decision-centric` 的 MAS 主线，而 `runtime_kernel.py` 曾以独立 live path 方式推进单集运行态。
- 这不是“多主线可选路由”，而是一个错误的过渡状态：`task queue / worker / API` 不应在多条 single-episode 主线之间路由。
- [single_episode_harness_architecture_20260311.md](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/docs/architecture/single_episode_harness_architecture_20260311.md) 已冻结新的架构前提：
- `OrchestratorAgent` 是 single-episode 的唯一 MAS 主线
- `runtime substrate` 是控制层底座与 runtime SoT 主写层
- `EpisodeRuntimeKernel` 不是主线候选，而是待退役的过渡实现
- 近期线上运行进一步暴露了编排层残余偏差：
- `workflow_order` 仍被错误地当作 planning 候选集来源，经 [orchestration_state_adapter.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/services/orchestration_state_adapter.py) 直接投影成 `available_agents`
- `_llm_decompose_tasks()` 又对该全集做 coverage 校验，导致 `voice_synthesizer` 这类可选 agent 被误提升为必填 task spec，进而在进入 step 1 之前就 fail-fast
- 这与当前架构目标冲突：single-episode 的第一步应该是 orchestrator 基于任务与能力上下文做 agent selection，而不是拿固定骨架反向限定 LLM 编排

## 2. Target
- single-episode 只保留一条 MAS 主线：`OrchestratorAgent`
- `EpisodeRuntimeKernel` 不再作为独立单集执行宿主存在，并在生产依赖移除后直接退役
- `workflow_session / node / attempt / gate / decision` 继续保留为控制层 runtime SoT，并只服务 orchestrator 主线
- 本轮不重构 memory system，只冻结 state/memory 边界，禁止新增 runtime/control state 回流到 Shared WM / `WorkflowState`
- hotfix 和主线统一问题分层处理：`005` 的局部回归修复不再承担 single-episode 主线统一职责
- `workflow_order` 不再作为 planning 骨架、候选集来源或 execution queue 包含语义来源存在
- orchestrator 在 decomposition 前先完成 LLM 驱动的 candidate agent selection，再由 LLM 基于 candidate agents 生成 `task_specs`
- selection prompt/template 外置，不在 orchestrator 代码中内嵌；state adapter 只承担 projection/persistence，不再承担 candidate 决策

## 4. Deviation Analysis
- Expected behavior:
- `task queue / worker / API` 只启动 single-episode 主线；`OrchestratorAgent` 负责 MAS 编排；控制层只承载 runtime SoT；门控层和治理层服务主线但不形成第二条执行链。
- Actual behavior:
- `OrchestratorAgent` 和 `EpisodeRuntimeKernel` 当前都能推进单集运行态，形成 dual live path。
- 当前虽然 dual live path 已被切除，但编排入口仍残留固定骨架偏差：
- orchestrator 通过 `workflow_order -> available_agents -> task_specs coverage` 这条链驱动 decomposition
- 结果是固定主线列表被错误地当成 LLM planning 的候选全集，像 `voice_synthesizer` 这样的 agent 会因为漏 spec 而在前置阶段直接失败
- Root cause:
- 之前的重构重点落在 “把 orchestrator 拉回 llm-decision” 上，但没有把 single-episode 全局主线约束写成可执行硬边界。
- `EpisodeRuntimeKernel` 作为 runtime/control-plane 过渡实现，没有被及时收回控制层边界，反而一度长成第二条单集执行链；当前生产路径已切走，但文件仍残留为过渡 wrapper。
- `runtime/session/gate` 基础设施与 orchestrator 主线尚未统一回同一套 single-episode engine。
- `workflow_order` 作为过渡期固定主线列表没有及时退化为非绑定 prior，反而继续承担了 candidate selection 与 queue 排序语义。
- state adapter 将其直接投影成 `available_agents`，覆盖了 orchestrator 本应负责的动态 candidate selection。

## 5. Implementation Approach
- 先冻结主线，再退役 kernel，不允许继续在“主线二选一”的前提下做局部修补。
- 保留控制层真正需要的 runtime substrate：
- `workflow_session`
- `workflow_node_state`
- `workflow_node_attempt`
- `workflow_gate`
- `workflow_gate_decision`
- 删除 `EpisodeRuntimeKernel` 中任何继续构成第二主线或误导长期保留的逻辑：
- 固定产品阶段推进
- 独立单集 live path
- 偏业务的 gate 内容拼装与阶段串联
- 本轮不重做记忆系统，只设边界保护线：
- runtime/control state 不回流到 Shared WM / `WorkflowState`
- memory/context 继续服务事实视图、上下文装配、结果沉淀
- Shared WM 中已存在的 orchestration facts 如需保留，只能作为 projection，不得作为 runtime control SoT
- 在编排入口引入 LLM 驱动的 candidate agent selection：
- 单向链路固定为 `registered_agents -> candidate_agents -> task_specs -> execution_queue/standby`
- `workflow_order` 不再参与 candidate 生成、coverage 全量校验或 execution queue 包含语义
- agent selection 不使用 if/else 业务硬规则决定 `voice_synthesizer` 等 agent，而是由 orchestrator 向 LLM 提供先验上下文：
- task traits
- `audio_contract`
- video/audio capability snapshot
- 必要 runtime facts
- selection prompt/template 外置，不在 orchestrator 源码中内嵌长字符串模板；代码只负责上下文装配与 schema/contract 调用
- 不保留兼容层、双写逻辑或空壳 helper；职责替换完成后直接删除 `workflow_order` 相关冗余实现

## 5.1 Architecture Principles
- P1 Mainline Principle：
- single-episode 的唯一 MAS 主线固定为 [orchestrator.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/agents/orchestrator.py)
- `task queue / worker / API` 只能启动主线，不得再承担“选择哪条 single-episode 主线”的职责

- P2 Control-Layer Principle：
- [runtime_session_service.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/services/runtime_session_service.py) 属于控制层 runtime substrate
- 控制层负责 `workflow_session / node / attempt / gate / decision` 的主写与状态迁移
- 控制层必须与编排层解耦，不得自行推进完整 episode 主流程

- P3 No-Second-Mainline Principle：
- `runtime_kernel.py` 不是主线候选
- `EpisodeRuntimeKernel` 只能作为短期退役过渡残片存在；当生产依赖已切除后，应优先删除而不是长期保留 wrapper

- P4 Runtime-Model Principle：
- runtime substrate 必须服务 orchestrator 主线的真实执行语义
- 不允许再用 legacy 固定产品阶段模型作为 single-episode runtime SoT 的正式表达
- 若 runtime view 不能准确表达主线节点、gate、resume/replan 语义，则不得判定 `U2/U4` 完成

- P5 State/Memory Boundary Principle：
- Shared WM / workflow facts 只承载事实、投影、诊断，不承载 runtime control SoT
- `workflow.session.*`
- `workflow.node.*`
- `workflow.attempt.*`
- `workflow.gate_decision.*`
- 属于禁止回流到 Shared WM 的控制态前缀

- P6 Candidate-Selection Principle：
- single-episode 编排的第一步是 orchestrator 发起的 agent selection，而不是固定骨架约束下的 task decomposition
- `workflow_order` 不得作为 `available_agents` 来源
- `available_agents` 必须来自 orchestrator 基于当前任务、contract、capability、runtime facts 的动态 selection 结果
- selection 决策由 LLM 完成，代码只提供先验上下文与 contract 校验

- P7 Prompt Externalization Principle：
- candidate selection 与 decomposition 的 prompt/template 不得内嵌在 orchestrator 代码里
- 代码层只保留 schema、上下文装配与调用入口；模板资产应外置到现有 prompt/config 体系

## 7. Quality Review Gates
- QG-1 Mainline Gate：
- single-episode 的正式 MAS 主线只能是 `OrchestratorAgent`
- 若发现任何文档、代码、API 仍将 kernel 视为主线候选，则计划不得进入实现关闭

- QG-2 No Second Live Path：
- 不允许存在 `EpisodeRuntimeKernel` 独立推进单集主流程的路径
- 若发现 kernel 仍串联产品阶段链或仍被生产/测试路径依赖，必须继续整改；本轮验收要求是退役完成

- QG-3 Runtime SoT Gate：
- `workflow_session/node/attempt/gate/decision` 必须继续作为控制层 runtime SoT
- runtime substrate 必须准确表达 orchestrator 主线语义；若仍保留 legacy 固定阶段蓝图作为正式 SoT，则不得关闭计划

- QG-4 State/Memory Boundary Gate：
- Shared WM 中的 `workflow.*` 只允许 facts/projections
- 若发现 `workflow.session.* / workflow.node.* / workflow.attempt.* / workflow.gate_decision.*` 回流 Shared WM，则计划不得关闭

- QG-5 Verification Gate：
- 必须同时具备以下 focused proof：
- `approve` 主线恢复
- `revise` 主线重进 script
- `replan` 主线重分解/重进 script
- `/tasks/{id}/status` 与 `/tasks/{id}/runtime` 一致且语义准确
- 若验证只覆盖“路由切换”而未覆盖 runtime model 语义，则不得判定收尾
- 若 runtime control state 再次回流到 orchestrator private loop、Shared WM 或 `WorkflowState`，则视为未收口

- QG-6 Closeout Regression Gate：
- script retry 成功后必须重新进入 review gate
- conditional runtime activation 必须应用 conditional task spec
- `runtime_overrides` 不得在 planning/adapter 边界静默丢失
- cancel_task 对无 runtime session 任务不得 silent no-op
- 若任一项失败，则不得关闭计划

- QG-7 Candidate Selection Gate：
- `workflow_order` 不得再出现在 `available_agents` 生成链路中
- candidate selection 必须由 orchestrator 发起 LLM 决策，不得用 if/else 业务规则直接决定 `voice_synthesizer` 等 agent 是否入选
- selection/decomposition prompt/template 不得内嵌在 orchestrator 代码中
- execution queue / standby 的包含语义不得依赖固定主线骨架
- 若仍存在 `workflow_order -> available_agents -> coverage` 链路，则计划不得关闭
