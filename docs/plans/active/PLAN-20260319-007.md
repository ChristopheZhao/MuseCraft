## 1. Current Problem
- single-episode MAS 主线已经回到 orchestrator，后端 `/tasks/{id}/runtime` 也能正确返回 `current_node_key / nodes / active_gate`。
- 但前端仍并存两套状态模型：
- [QuickModeWorkspace.tsx](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/src/components/preview/QuickModeWorkspace.tsx) 使用 `quickRuntime` 渲染 runtime/gate 面板。
- [AgentOrchestrator.tsx](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/src/components/agents/AgentOrchestrator.tsx) 与 [RealTimeProgress.tsx](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/src/components/progress/RealTimeProgress.tsx) 仍使用 [useAppStore.ts](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/src/store/useAppStore.ts) 中静态 `agents` 作为主状态源。
- 结果是后端已经进入 `script_review` gate，但前端节点图与进度条未同步到 runtime session 语义，用户无法可靠判断流程是否暂停在 gate、也难以验证 `approve / revise / replan` 是否生效。

## 2. Target
- Quick 前端只保留一套 runtime 主状态源：`quickRuntime`。
- 前端必须能清晰显示 single-episode 主线节点状态、当前节点、`script_review` gate 与恢复后的节点推进。
- 在不做整套前端重构的前提下，让用户可以通过前端完成 script 审核并观察主线继续推进，支持 MAS 单集流程跑通。

## 4. Architecture Principles
- P1 Single Runtime Source:
- Quick workspace 内的主流程节点、gate、当前节点，只能由 `quickRuntime` 驱动。
- 静态 `agents` 只能保留为辅助遥测，不得继续充当前端主流程状态源。

- P2 No Second Frontend Workflow Model:
- 前端不得再维护一套独立于 runtime session 的静态 workflow node 模型。
- 若 UI 需要展示 pipeline，必须从 `quickRuntime.nodes` 推导。

- P3 Gate Visibility:
- `script_review` gate 必须在前端显式可见、可操作、可验证恢复结果。
- 用户不能依赖后端日志判断是否已暂停到 gate。

- P4 Minimal-Scope Fix:
- 本轮只修“让单集 MAS 流程可观察、可审核、可推进”的最小前端问题。
- 不借机扩成 store 大改或全站状态模型重写。

## 5. Implementation Approach
- 让 [useTaskPolling.ts](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/src/hooks/useTaskPolling.ts) 与 [useWebSocket.ts](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/src/hooks/useWebSocket.ts) 继续以 `quickRuntime` 为 runtime SoT。
- 将 [AgentOrchestrator.tsx](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/src/components/agents/AgentOrchestrator.tsx) 改为由 `quickRuntime.nodes` 渲染，不再以静态 `agents` 为流程主视图。
- 将 [RealTimeProgress.tsx](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/src/components/progress/RealTimeProgress.tsx) 改为由 `quickRuntime` 推导阶段、进度和 gate 状态。
- 保留 [QuickModeWorkspace.tsx](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/src/components/preview/QuickModeWorkspace.tsx) 的 script gate 面板，并增强提交后恢复状态的可见性。
- 不再保留冗余 workflow mapping / duplicate progress semantics；移除或降级已失去主状态职责的静态流程依赖。

## 6. Stage Breakdown
### Stage A: Runtime Source Freeze
- 目标：冻结 `quickRuntime` 为前端 single-episode runtime 唯一主状态源。
- 完成标志：前端主流程节点 UI 不再依赖静态 `agents` 列表。

- [x] A1 Runtime Source Audit
- 盘点哪些组件仍在用静态 `agents` 驱动主流程状态。
- 明确保留为辅助遥测的字段与必须退场的字段。

### Stage B: Node UI Rebinding
- 目标：让节点视图与 runtime session 节点语义一致。
- 完成标志：节点图、当前节点、node status 均由 `quickRuntime.nodes/current_node_key` 推导。

- [x] B1 AgentOrchestrator Rebinding
- [AgentOrchestrator.tsx](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/src/components/agents/AgentOrchestrator.tsx) 改为渲染 runtime nodes。
- 使用 `node_key -> label/icon` 的轻量映射，仅作展示，不作为流程骨架。
- `active_gate`/`awaiting_human` 在对应节点上高亮。

- [x] B2 RealTimeProgress Rebinding
- [RealTimeProgress.tsx](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/src/components/progress/RealTimeProgress.tsx) 改为由 `quickRuntime` 计算：
- 当前阶段
- 已完成节点数
- 当前 gate 状态
- 总体完成度
- `scenesPlanned/imagesGenerated/videosGenerated` 仅作辅助指标，不再作为流程主状态判断。

### Stage C: Gate UX Tightening
- 目标：让 script 审核暂停与恢复过程在前端可见可验证。
- 完成标志：用户能通过前端明确看到 `waiting_gate(script)`，提交 decision 后能看到恢复中的节点变化。

- [x] C1 Script Gate Visibility
- [QuickModeWorkspace.tsx](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/src/components/preview/QuickModeWorkspace.tsx) 中 `script` 节点的等待审核状态必须显著显示。
- 提交 `approve / revise / replan` 后展示“已提交，等待主线恢复”而不是静默停留。

- [x] C2 Refresh Semantics Check
- 校正 polling / WS 触发时机，确保 gate 决策后一个合理轮询周期内能刷新到新 runtime。
- 不新增第二套刷新协议。

### Stage D: Focused Verification
- 目标：证明前端已能支撑单集 MAS 流程从 script gate 继续推进。
- 完成标志：有 focused 验证覆盖 gate 等待、审批提交、恢复后节点变化。

- [ ] D1 Waiting Gate Rendering Proof
- 验证 `active_gate.gate_name=script_review` 时，节点与 workspace 均显示等待审核。

- [ ] D2 Approve/Resume Proof
- 验证提交 `approve` 后，前端 runtime 视图会从 `script/pending_gate` 过渡到后续运行节点。

- [ ] D3 Revise/Replan Visibility Proof
- 验证 `revise / replan` 提交后，前端仍以 `quickRuntime` 展示主线变化，而不是停留在旧静态节点图。

## 7. Risks / Blockers
- 当前最主要风险不是后端主线错误，而是前端旧静态节点模型掩盖真实 runtime 状态，导致用户无法推进 HITL。
- 若在实现中继续同时维护 `agents` workflow 与 `quickRuntime` workflow，两套状态会再次分叉，计划视为未完成。

## 8. Quality Gates
- QG-1: 前端主流程节点 UI 仅由 `quickRuntime` 驱动。
- QG-2: `script_review` gate 在前端显式可见且可提交 decision。
- QG-3: 提交 decision 后，节点状态能在合理时间内同步恢复。
- QG-4: 不新增第二套静态 workflow 兼容层。
- QG-5: Focused frontend verification 覆盖 waiting_gate、approve/resume、revise/replan visibility。
