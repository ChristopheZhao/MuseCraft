## 1. Current Problem
- Symptom:
- 当前 [orchestrator.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/agents/orchestrator.py) 同时承担规则编排、gate、runtime probing、fallback/controller 等职责。
- 历史上 [video_generator.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/agents/video_generator.py) 曾直接读取 `workflow.plan/audio_route` 并 patch tool calls，暴露了 leaf-agent 边界失效问题；当前该问题已在 `P1-2` 修复，但其模式需要作为后续 agent 清理的反例持续约束。
- 当前 `Task + shared memory` 不能充当真正的 runtime control-plane，quick mode 仍是 whole-task one-shot，project mode 也无法复用单 `episode` 内核。
- `_execute_impl` 仍在主循环中主动拉 gate、主动组织 runtime replan，并直接改写 `execution_queue/task_specs/standby_agents`，说明门控触发模型和 orchestrator-subagent 主从通信协议还未收敛。
- Impact:
- `HITL` 无法在正确边界上落地，只能变成 UI/后端补丁。
- 新 agent、新 provider、新 gate 都会继续把逻辑堆进 orchestrator 或 agent。
- 从偏预定义 MAS 迁向更自主 MAS 的演进路径被阻断。
- Reproduction:
- 检查 [orchestrator.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/agents/orchestrator.py#L1331)、[orchestrator.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/agents/orchestrator.py#L1579)、[orchestrator.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/agents/orchestrator.py#L1941)、[orchestrator.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/agents/orchestrator.py#L2061)。
- 检查 [video_generator.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/agents/video_generator.py#L324)、[video_generator.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/agents/video_generator.py#L378)。

## 2. Deviation Analysis
- Expected behavior:
- `orchestrator` 只负责有限骨架、planner 输出消费与下一步 `ExecutionIntent`。
- `control-plane` 单独持有 `workflow/node/attempt/gate/decision`。
- `gate/evaluator` 单独产出 `GateResult`。
- `leaf-agent` 只消费业务事实和显式 `ExecutionContract`。
- `gate` 由 control-plane 在节点边界触发，orchestrator 只消费 `subagent report + gate result + runtime state` 做 LLM 决策。
- orchestrator 与 subagent 之间存在清晰的主从通信协议；结构化消息体只是协议载荷，不替代交互协议本身。
- Actual behavior:
- orchestrator 仍是集中式策略引擎 + gate 代偿器。
- leaf-agent 已开始反向理解 control-plane 协议。
- runtime 缺少独立宿主，导致 quick/project 都没有统一单 `episode` kernel。
- 主循环仍有 `gate-inline-polling`，且当前通信基础设施只解决了“怎么传”，没有完整定义“何时触发、向谁回报、谁做决策”的主从交互协议。
- Root cause hypothesis:
- 历史兼容与功能迭代把编排、门控、执行、补救策略混在了一起，缺少显式 control-plane 和 contract 边界。

## 3. Implementation Approach
- Patch strategy:
- 采用“边界先行、内核先行、功能后挂”的重构路线。
- 先冻结错误边界和扩散点，再补 `single-episode runtime kernel` 与 `GateResult/ExecutionContract`，先完成 quick mode 最小 runtime/gate 验证，再继续 compatibility retirement，最后把更重的 HITL 与 project mode 复用建立在硬化后的新内核上。
- 本轮后半程进一步收紧为：删除 domain-specific plan builder / activation planner，避免把规则中心从 `orchestrator` 平移到 `policy service`；`policy` 仅保留约束、能力事实、gate facts 和少量硬约束，不再直接替 orchestrator 生成编排结果。
- 当前范围继续收紧为：移除 `_execute_impl` 中的 gate-inline-polling，把 gate 改成 boundary-triggered，并补齐 orchestrator / subagent / control-plane 的主从通信协议，使 orchestrator 进一步回到 `llm-decision-based orchestration`。
- Compatibility considerations:
- 顶层阶段骨架可在过渡期保留固定顺序。
- 外层 `Task` 和 project API 暂时保留，对内逐步切换到 `workflow_session` 驱动。
- `workflow.plan/audio_route` 在迁移窗口内可保留为内部兼容/诊断数据，但不允许新增 live 依赖。
- 所有阶段性兼容必须附带明确的 retirement 阶段；不允许长期保留“兼容归一化”并使其重新固化为规则中心。
- Rollback plan:
- 如某阶段引入回归，允许回退该阶段的 runtime/kernel 接入，但不恢复已冻结的错误边界扩散。

## 4. Overall Plan

### Phase 0: 边界冻结与专项审查
- 目标：
- 冻结 orchestrator 内新的 agent-specific 分支、gate/probing 逻辑和 agent 侧 control-plane 泄漏。
- 增加一轮显式架构审核，防止“伪去规则化”重演。
- 输出：
- 边界禁区清单
- 专项审查清单
- 架构审核 Gate 与通过标准
- 受影响文件分级
- 完成标志：
- 新代码不再新增 `workflow.plan/audio_route` 消费点
- 不再新增 orchestrator 内 gate/fallback 分支
- 已完成一轮“反伪去规则化”架构审核，确认不是仅把全局变量切成内部变量

### Phase 1: Single-Episode Runtime Kernel 骨架
- 目标：
- 引入 `workflow_session / workflow_node_state / workflow_node_attempt / workflow_gate / workflow_gate_decision` 最小 runtime 实体。
- 输出：
- runtime 数据模型
- `EpisodeRuntimeKernel`
- `Task -> RuntimeKernel` 调用链
- 完成标志：
- `tasks.py` 和 `task_queue.py` 不再直接把 orchestrator 当 runtime owner
- 出现可查询的 session/runtime summary

### Phase 2: Gate / Evaluator 抽离
- 目标：
- 把媒体 probing、delivery check、quality gating 从 orchestrator 抽出，统一产出 `GateResult`。
- 输出：
- evaluator/gate service
- `GateResult` contract
- orchestrator 侧 gate result 消费路径
- 完成标志：
- orchestrator 不再直接 `ffprobe`
- orchestrator 不再直接拼 route/signal 作为 gate 结果

### Phase 3: Leaf-Agent Contract Cleanup
- 目标：
- 引入统一 `ExecutionContract`，去掉 leaf-agent 对 control-plane 协议的反向依赖。
- 输出：
- `ExecutionContract` assembler
- `video_generator` 去编排化
- `video_composer` 改为显式 contract 驱动的 mix/composition 模式执行
- 完成标志：
- `video_generator` 不再读取 `workflow.plan/audio_route`
- `video_composer` 不再自行推断主 mix 模式
- leaf-agent 不再 patch tool calls 来补编排参数
- `add_bgm` / `add_voiceover` / `static_context.requests` 降级为兼容输入，不再作为 live 主决策路径

### Phase 4: Quick Mode 最小 Runtime 适配与 Script Gate 验证
- 目标：
- 在新 kernel 上完成 quick mode 对 runtime/session/gate 的最小消费闭环，并用 `script gate` 验证 control-plane 与 gate 边界。
- 本阶段前端仅做最小适配，不扩展为完整 HITL 工作台。
- 输出：
- `script` gate runtime
- runtime API / websocket 扩展
- quick mode 单 `episode` 最小工作台
- 完成标志：
- quick mode 不再是单纯 one-shot 进度页
- `script` gate 可真实暂停、提交决策并续跑
- `final_compose` 进入 `video_composer` 时也只消费显式执行边界，不重新依赖兼容字段做 live 决策
- storyboard / scene video / richer review UX 不属于本阶段完成标志

### Phase 5: Compatibility Retirement 与边界硬化
- 目标：
- 删除迁移期兼容层，避免 legacy flags、兼容 seam 或 builder 归一化重新演化为规则中心。
- 输出：
- composer legacy flag retirement
- runtime compatibility seam retirement
- live-path boundary audit
- 完成标志：
- 新路径不再产出 `add_bgm / add_voiceover / compose_requested`
- `video_composer`、`orchestrator`、`runtime_kernel` 不再依赖 legacy flags 或 `workflow.plan/audio_route` 兼容 seam 做 live 决策
- `static_context.requests` 仅保留为派生展示/提示字段，或在无必要时删除
- 兼容 fallback 仅保留在明确隔离的迁移适配层，且不再处于 live 主路径

### Phase 6: Orchestrator Ownership 收口
- 目标：
- 在 compatibility retirement 之后继续迁出 residual ownership，避免当前阶段只完成 seam cleanup 而未完成真实分层。
- 执行原则：
- 本阶段采用 architecture-driven refactor，而不是 file-level decomposition。
- 任何函数迁移都必须先明确目标宿主属于已确认架构层之一：`orchestrator`、`control-plane/runtime kernel`、`gate/evaluator`、`policy service`、`execution-boundary assembler`、`state projection/event adapter`、`tool`、`leaf-agent`。
- 不允许为了缩短 `orchestrator.py` 而把逻辑平移到新的 util/helper 文件中；若迁移后的模块会重新成为规则中心，则视为不通过。
- 输出：
- tail-stage runtime ownership closure
- execution-boundary assembly cleanup
- orchestrator residual rule-center audit
- anti-llm-decision cleanup
- orchestration / gating / master-worker protocol convergence
- 完成标志：
- `script` gate 之后的 `storyboard / scene_video / compose / quality` 不再由 `orchestrator.execute()` 一把跑完，而是进入 kernel 可见的真实 node execution boundary
- `ExecutionContract` live 组装不再由 orchestrator 直接持有
- orchestrator 不再把 execution boundary 回写成 legacy-shaped request/control 字段作为 live path
- orchestrator 中残留的 audio-specific replan/controller 逻辑继续收缩，不再作为新的中心化规则宿主
- `_build_workflow_plan`、`_build_activation_pool`、`_resolve_agent_runtime_overrides`、`_build_audio_contract` 不再允许作为 orchestrator 常驻过渡 seam 保留，需在本阶段内明确迁出
- `audio_orchestration_policy.py` 不再直接产出 `active_pool / standby_pool / actions / decision_reason` 作为编排结果；`build_audio_preflight_plan`、`apply_audio_standby_activation` 这类 domain-specific plan/activation generator 不作为目标设计保留
- orchestrator 主循环不再每个 step 主动拉 gate、主动组织 runtime replan；相关 gate 改为 boundary-triggered，由 control-plane 在对应节点边界触发
- `_llm_decide_runtime_replan` 不再是 audio-domain gate 聚合点，而是收敛为通用 runtime decision 入口，消费 `subagent report + gate result + runtime state`
- orchestrator、subagent、control-plane 之间的主从通信协议已明确，不再依赖主循环内的隐式轮询和状态拼装
- `decision_basis`、`build_audio_route_payload` 等观测字段不再由编排状态宿主持有；若保留，仅能位于 observation/trace adapter，并且不得参与 live 决策
- 若仍存在明显的 `policy-first` 主循环硬编排，则本阶段不得视为完成，计划不得归档，必须继续定位其为何仍停留在主循环以及其正确宿主
- 若主循环仍主动拉 gate、主动组织 runtime replan，`P2-15` 必须判定失败，不允许以“局部 ownership 已改善”为由收口

### Phase 7: 下一期范围: Scene-Level Gates 与 Project Reuse
- 目标：
- 将更重的 HITL 能力和 project mode 复用建立在已硬化的 runtime/gate 边界之上。
- 本阶段明确不属于当前重构交付范围。
- 输出：
- scene-level gate runtime
- revise/retry/invalidation 规则
- project mode episode session 接入
- 完成标志：
- quick/project 共用同一单 `episode` runtime kernel
- `storyboard` 和 `scene_video` gate 可在 scene 级运行
- 下游 scene agent 只消费 approved artifact / node contract，不再靠隐式上下文推断主执行模式
- project mode 复用同一套 hybrid subagents 边界：编排层、控制层、门控层解耦，`HITL` 作为 gate capability 挂在 control-plane 上
  说明：`hybrid subagents` 是总架构表述；显式 execution boundary 只是当前实现边界约束，不替代总架构定义

### Current-Scope Completion Gate: 本轮交付核查
