## 2. Deviation Analysis
- Expected behavior:
  - A single authoritative final-video fact should exist before runtime completion, completion publish, persistence projection, and frontend review.
  - Retry success must use the same handoff path as initial success.
- Actual behavior:
  - Runtime summary can still fall back to `workflow_results.video_composer.final_video_url`.
  - Completion payload and persistence projection consume MAS `project.final_video`.
  - Current worktree moved final-video fact commit toward an orchestration-layer handoff path, but that handoff does not run on retry success.
  - Real acceptance shows a second defect in the same chain:
    - `video_composer` can finish on a no-tool PLAN round
    - the subagent-to-orchestrator completion envelope then collapses to a minimal subtask-complete receipt
    - the candidate deliverable refs that orchestrator currently promotes into `project.final_video` are no longer present in that handoff payload
- Root cause hypothesis:
  - `contract`
    - the authoritative final-video source was not frozen as one explicit contract across runtime and read-model projections
    - the repo also failed to separate two different contracts clearly:
      - shared-WM deliverable SoT
      - subagent-to-orchestrator completion / handoff receipt
  - `owner`
    - one user-visible fact was effectively owned by two paths:
      - runtime summary fallback from `workflow_results.video_composer`
      - persistence projection from `project.final_video`
  - `implementation`
    - the two paths diverged in behavior
  - `current-window regression`
    - the owner shift toward orchestration-owned shared-fact commit is directionally architecture-aligned
    - but the retry branch does not call the same commit path
  - `acceptance amendment`
    - the remaining compose-success failure sits on the handoff seam between:
      - the ReAct completion envelope emitted by the subagent
      - the orchestrator-owned promotion into shared WM
    - `react_agent.py` is therefore only a possible handoff-contract owner, not the authority owner of `final_video`

## 3. Architecture and Boundary Freeze
- `single authority`
  - final-video projections must derive from one authoritative runtime fact, not mixed runtime and persistence fallbacks
- `single owner`
  - the authoritative final-video fact must be committed through one orchestration-owned handoff path
  - leaf-agent return payloads are transient handoff inputs, not a parallel authority surface
- `subtask completion vs workflow completion`
  - `task_complete` means the current subagent has finished its local ReAct objective
  - workflow completion remains an orchestrator decision after downstream gates and SoT checks
- `handoff receipt vs shared fact`
  - shared WM `project.final_video` is the deliverable SoT
  - the subagent-to-orchestrator completion envelope is only the handoff receipt that lets orchestrator decide how to promote deliverables into shared WM
- `handoff owner split`
  - [backend/app/agents/react_agent.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/agents/react_agent.py) owns only the generic completion-envelope shape for ReAct subagents
  - [backend/app/agents/orchestrator.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/agents/orchestrator.py) owns promotion of accepted deliverables into MAS shared facts such as `project.final_video`
  - `quality_checker` remains a consumer and is not part of the fix owner path
- `runtime SoT vs read-model`
  - runtime fact ownership stays in MAS/control-plane/orchestration logic
  - task detail, resources, and frontend review consume explicit projections only
- `contract-first`
  - do not add new consumer-side fallback layers that hide owner drift
- `out of scope`
  - `quality_checker` semantics expansion
  - tool-registry startup warnings
  - queue/worker/control-plane redesign outside the final-video authority seam

## 3.1 Object Split
- `subagent completion receipt`
  - owner:
    - leaf-agent finalize path
    - explicit `orchestration_report` / subagent report envelope
  - allowed meaning:
    - the subagent completed its own local ReAct objective
    - the subagent can provide candidate deliverable refs for orchestrator consumption
  - forbidden meaning:
    - final deliverable authority
    - workflow terminal status
  - concrete surfaces in this stream:
    - `agent_output.final_video_path`
    - `agent_output.final_video_url`
    - `agent_output.mix_receipt`
    - `agent_output.orchestration_report`
- `shared WM deliverable SoT`
  - owner:
    - orchestrator-owned promotion into MAS shared facts
  - allowed meaning:
    - the only authoritative project-level final-video fact
    - the source consumed by downstream gates, completion projection, and persistence/read-model surfaces
  - concrete surfaces in this stream:
    - `project.final_video`
    - `project.final_video_mix`
- `workflow completion decision`
  - owner:
    - orchestrator plus runtime/control-plane completion publish path
  - allowed meaning:
    - the workflow may be marked completed only after SoT checks and downstream gates succeed
  - forbidden meaning:
    - any inference from `task_complete` or `plan_contract_task_complete`

## 4. Implementation Approach
- Patch strategy:
  - Keep the architecture-aligned direction that shared final-video fact commit is orchestration-owned, but make it single-path and complete.
  - Repair sequence:
    - preserve the already-landed 054 baseline for persistence/read-model convergence unless new evidence disproves it
    - freeze the exact subagent-to-orchestrator handoff contract for composer completion before changing code
    - repair the remaining gap first in `video_composer._finalize_success_results(...)`
    - on successful composer ACT, persist the minimal finalize-time boundary subset into agent-scope WM scratchpad
    - rebuild a memory-based completion receipt from agent-local execution evidence and that explicit scratchpad subset during finalize / handoff synthesis
    - do not read business receipt inputs back from `ExecutionState.input_data` during finalize
    - do not make `react_agent.py` carry forward previous ACT deliverables as implicit truth
    - keep orchestrator as the sole owner that promotes the accepted receipt into `project.final_video` and `project.final_video_mix`
    - widen to orchestrator or `react_agent.py` only if the composer-local repair proves contractually insufficient
    - keep frontend review on `detail/resources/runtime` only; no `/result` dependency
- Compatibility considerations:
  - Preserve the currently working offline compose flow and helper tests.
  - Do not reopen leaf-agent direct-shared-write designs unless review proves the orchestration-owned handoff is architecturally impossible.
  - `054` is a forward fix for new runs; it does not include retroactive backfill of already-completed historical task projections.
  - Any change that touches QC logic, provider/tool bootstrap, or broader runtime/controller semantics must split to a separate plan.
- Rollback plan:
  - If a current-worktree change is off-path or violates the frozen boundary, remove it from the repair stream rather than silently absorbing it.
  - If runtime verification disproves the architecture-aligned orchestration-owned handoff approach, stop implementation and amend the plan before changing owner direction.
