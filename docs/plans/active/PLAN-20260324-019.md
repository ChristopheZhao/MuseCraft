## 1. 当前问题
- `PLAN-20260324-017` 已经完成并保持关闭；本计划不 reopen `017`。
- `PLAN-20260324-018` 继续只负责 Jest / tooling 问题；本计划不吸收 `018` 的范围。
- 当前暴露出的新问题不是“前端要判断 SoT 真相”，而是：
  - 后端 contract 仍存在缺席态、双表面、松散表面或事件越权
  - 前端因此被迫承担状态路由、对象 fallback、双表面合并或可选 authority 分支
- 这类问题的根因在 backend contract，不在 frontend “判真值能力”。

当前已确认的高价值例子：
- quick mode live runtime：
  - active run 的 `/tasks/{id}/runtime` 仍可能返回 `Runtime session not found`
  - `src/hooks/useTaskPolling.ts` 因此保留 `404 -> coarse /status` 分流
- quick current / resumability：
  - `/tasks/quick/current` 仍以 `task + workflow_status` 双表面返回“当前可恢复 run”
  - 前端必须自己判断 `task && workflow_status`
- gate decision immediate state：
  - script gate decision response 中的 `workflow_status` 仍是 optional
  - 前端因此分支处理“有 runtime 就更新，没有就等后续 refresh”
- project mode status semantics：
  - `story_plan.episodes[].status` 与 `episodes_runtime[].status` 同时暴露
  - 前端当前通过 `runtime?.status || episode.status` 合并消费
  - 前端类型 `EpisodeStatus` 也复用了 editorial/runtime 两种语义
- project progress / reference progress：
  - `planning_status`、`character_references_status`、`planning_error` 等仍挂在 `global_settings`
  - 前端需要理解这些 string convention 才能做业务路由
- websocket terminal behavior：
  - `event.state` 仍可能在 runtime/project read-model refresh 之前直接驱动 terminal UI side effects

## 4. 问题分类
实现前固定按以下三类区分，避免误删：

- `backend-contract gap`：
  - backend 没有提供单一、稳定、权威的 contract/object 去回答业务问题
  - 修复动作发生在 endpoint、read-model、projection、event contract 边界
- `frontend authority-routing`：
  - frontend 因 backend contract 缺口而保留 fallback、双表面合并、状态路由
  - backend contract 收口后，这些 routing 才能删除
- `same-object presentation mapping`：
  - frontend 仅基于同一个 authoritative object 做展示映射
  - 这不是本计划删除目标，不得误当作 authority-routing 清理

## 5. 执行计划
### Phase A: 冻结问题到权威对象矩阵
- 为以下问题逐一冻结权威 contract：
  - quick live runtime
  - quick resumability / current run discovery
  - gate decision immediate state
  - project editorial/script status
  - project execution/runtime status
  - project planning progress
  - project reference-generation progress
  - terminal notification semantics
- 把现有 consumer site 分类为：
  - presentation mapping
  - controlled transitional fallback
  - backend-contract gap
- 本轮实现顺序在进入代码前冻结为：
  1. quick runtime contract
  2. `/tasks/quick/current` single authoritative contract
  3. gate decision response contract
  4. project mode status/progress contract
  5. websocket terminal semantics

### Phase B: backend-first 契约收口
- 1. quick runtime contract：
  - 去掉 active quick run 的 non-authoritative absence state
  - 对 active run 的 `/tasks/{id}/runtime` 返回 authoritative bootstrap/runtime 视图，而不是让 frontend 用 `404` 作为控制流
- 2. quick current contract：
  - 把 `/tasks/quick/current` 收敛为单对象 contract，而不是 `task + optional workflow_status`
- 3. gate decision contract：
  - 明确 endpoint 是“永远返回新 runtime”还是“只提交，前端必须 refresh”
- 4. project mode status/progress contract：
  - 先拆 editorial/script lifecycle 与 execution/runtime lifecycle
  - 再把 execution/runtime authority 收敛到 `episodes_runtime[*].status`
  - 用 typed projection 替代 `global_settings.*status/*error`
- 5. websocket terminal semantics：
  - 明确 event 是 bounded summary / refresh trigger，而不是 terminal authority
  - 如需短期兼容面，必须显式标注为短期债务，不得伪装为稳定架构

### Phase C: 删除 frontend authority-routing
- 删除 quick polling 中的 `runtime -> coarse status` fallback
- 删除 quick resumability 中的 `task + workflow_status` 双表面分支
- 删除 gate decision response 的 optional-runtime 分支
- 删除 project mode 中的 `runtime?.status || episode.status`
- 拆分 frontend type/read-model 中仍混合 editorial/runtime 语义的 status 类型
- 保留 same-object presentation mapping，不做误删

### Phase D: 验证与交接
- Backend:
  - quick runtime contract regression
  - `/tasks/quick/current` contract regression
  - gate decision response contract regression
  - project status/progress projection regression
  - websocket terminal semantics regression
- Frontend:
  - TypeScript compile
  - focused runtime/project consumer regression
  - 若 `018` 尚未修复，则不把 Jest 可执行性作为 `019` 本体验收门槛
- Documentation:
  - 记录任何仍保留的 compatibility surface
  - 明确其为短期债务，而非长期稳定架构

### Phase E: review 后补收口
- 1. `approved_script` contract semantics：
  - `runtime.approved_script` 只能表达“已批准脚本”
  - 未批准编辑必须停留在 editorial/draft surface，不能借道 runtime surface 泄漏给下游 agent context
- 2. project runtime absence semantics：
  - 前端不得把缺席的 runtime 直接映射成 `idle`
  - 要么 backend 确保 `episodes_runtime[*]` 对每个 episode 都有权威 execution surface，要么前端仅展示显式 unavailable / not_initialized，而不自行补业务状态
- 3. compat debt annotation：
  - `/tasks/{task_id}/status` 与前端 `getTaskCoarseStatus()` 若暂时保留，必须在文档与代码语义中明确为短期 compatibility surface
  - 不新增任何主线 consumer 回流到该 surface
- review 后补收口顺序冻结为：
  1. backend `approved_script` 语义收口
  2. project runtime absence / frontend residual fallback 收口
  3. compat debt 标注与回归验证刷新

### Phase F: re-review 后 fail-close 收口
- 1. `force_rerun` semantics：
  - `force_rerun` 只能表达“对已批准 episode 强制重跑 execution/runtime”
  - 不得表达“绕过 editorial approval 直接进入 execution-plane”
- 2. endpoint fail-close：
  - `POST /projects/{project_id}/orchestrate` 的预标记逻辑必须保留 `EpisodeEditorialStatus.APPROVED` 这一硬门
  - `force_rerun` 只能影响 runtime rerun eligibility，不得让未批准 episode 进入 `generating`
- 3. orchestrator fail-close：
  - `EpisodeOrchestratorAgent` 必须无条件拦住未批准 episode
  - 如果未来要支持显式 override，必须另立 contract/permission surface，不能继续借 `force_rerun`
- 4. regression：
  - 补 focused tests 覆盖 endpoint 与 orchestrator 两侧的 fail-close 行为
- re-review 后补收口顺序冻结为：
  1. endpoint `force_rerun` fail-close
  2. orchestrator `force_rerun` fail-close
  3. focused regression
  4. validation / delivery 刷新

## 6. 验收
- 已存在且活跃的 quick run，不再依赖 `404 Runtime session not found` 作为前端控制流分支。
- 没有 quick resumability flow 需要同时查看 `task` 和 `workflow_status` 才能判断 run 是否存在。
- gate decision response contract 已冻结，前端不再依据 optional runtime payload 做 authority 分支。
- 没有 project-mode frontend path 需要合并多个 backend status surface 才能回答同一个 execution-state 问题。
- 若 `EpisodePlan.status` 仍保留，它不再是 competing live execution status surface，并且语义被明确限定为 editorial/script lifecycle。
- `runtime.approved_script` 不再承载未批准草稿；任何下游读取到的 approved-script surface 都只表达 editorial 已批准内容。
- planning/reference progress 不再依赖 `global_settings.*status/*error` 这种松散 string convention 作为业务路由依据。
- websocket terminal event 不再单独决定 mainline UI 的 completed / failed 终态。
- project-mode frontend 不会因为 runtime 缺席而自行补出 `idle` 这类 execution-state。
- 若 `/tasks/{task_id}/status` compatibility surface 仍保留，它已被显式标注为短期债务，且不再被主线 authority-routing 消费。
- `force_rerun` 不会绕过 editorial approval gate；未批准 episode 不能仅凭该 flag 进入 execution-plane。
- `PLAN-20260324-017` 保持 completed，`PLAN-20260324-018` 保持 Jest/tooling scope；`019` 的验收不依赖 `018` 的 Jest 执行成功。

## 7. 风险与迁移
- quick runtime:
  - 任何把 `404 Runtime session not found` 当作控制流的客户端都需要迁移
- `/tasks/quick/current`:
  - response shape 可能需要从“双字段”向“单对象 / null”演进
  - 前端会有一次小范围 rename / type migration
- project mode:
  - 如果 `EpisodePlan.status` 被保留，必须重新定义为 editorial 问题，而不是 execution/runtime 问题
  - 这要求 backend 和 frontend 类型同时迁移
- planning/reference progress:
  - 从 `global_settings` 迁移到 typed projection 时，可能需要短暂兼容期
- websocket terminal semantics:
  - UI 即时性可能略有变化，但可以通过可靠 refresh 保持体验
