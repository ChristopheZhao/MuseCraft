## 1. 当前问题
- 当前成片偏“幻灯片化 / 特写化”，主要表现为单人或双人静态对峙、长时间面部或姿态展示、依赖推近/定格/抖动撑时长，故事推进弱。
- 该问题不是单一 agent bug，也不是 SoT 漂移，而是跨 `concept -> script -> image -> video` 的 scene semantics 与 prompt synthesis 失真。
- 真实运行证据已经闭合：
  - `[mas_workflow.log](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/logs/mas/mas_workflow.log#L11608)` 中 `CONSISTENCY_HINTS.visual_language` 明确给出“中近景和特写为主”的全局偏置。
  - `[mas_workflow.log](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/logs/mas/mas_workflow.log#L12167)` 中 image agent 为 5 个场景统一规划 `scene_opening_anchor + full_body`。
  - `[mas_workflow.log](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/logs/mas/mas_workflow.log#L12232)`、`[mas_workflow.log](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/logs/mas/mas_workflow.log#L12267)`、`[mas_workflow.log](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/logs/mas/mas_workflow.log#L12363)` 中最终 video prompt 均以面部/表情/眼神推进为核心。
  - `[mas_workflow.log](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/logs/mas/mas_workflow.log#L12459)` 对 scene 3 的 `PROMPT_COMBINED` 显示 prompt 虽然结构化，但缺少单一主导的 `main key`，而是并列 bullet 平铺。
- 影响范围：
  - 短视频无法形成“局部事件 -> 下一事件”的连续叙事。
  - `image_to_video` 首帧被海报化，后续视频天然退化为“静帧延展”。
  - 现有 prompt 结构被镜头词、opening-anchor、consistency append 共同放大，后续微调单条 prompt 收益有限。

## 2. 偏差分析
- 期望行为：
  - scene 应是可支撑目标时长的 `local_event`，而不是章节摘要或情绪功能句。[scene_contract_v2_freeze_20260329.md](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/docs/architecture/scene_contract_v2_freeze_20260329.md)
  - script / video prompt 应优先传达可见执行弧线，而不是镜头清单。[script_video_prompt_contract_v2_freeze_20260329.md](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/docs/architecture/script_video_prompt_contract_v2_freeze_20260329.md)
- 实际行为：
  - `concept` 先把 visual language 推向中近景/特写和强运镜。
  - `script` 未把全局叙事与目标时长显式写进单场景 prompt，`action_phases` 还被 `camera_hint` 主导。
  - `image` 默认把 scene 画成 `scene_opening_anchor` 的稳定静帧。
  - `video_prompt_builder/composer` 采用 append-only 的并列拼接，没有单一主导的 `scene thesis`，`prompt_mode` 和镜头词天然夺权。
- 根因假设：
  - authoritative scene semantics 不够强，未真正落成“能撑住 10s 的 local_event”。
  - prompt synthesis 采用平铺拼接策略，把弱 scene semantics 进一步摊平。
  - image 首帧策略和 `有图即 i2v` 的 mode policy 把上述弱语义放大成海报动画。

## 2.3 Plan Amendment A3: Mode Policy Regression Review
- Amendment trigger:
  - Follow-up provider probes showed that the same scene-4 hierarchical structured prompt does not fail uniformly across payload shapes.
  - `structured_no_flags + no image` failed with `InvalidParameter: Invalid content.text`, while the same structured prompt paired with `scene_4_image.jpg` succeeded on the same model and endpoint.
- Re-scoped diagnosis:
  - The priority blocker is no longer best described as generic `content.text` incompatibility.
  - The stronger evidence is that the recent mode-policy change moved certain scenes from a provider-successful `image_to_video` path into a higher-risk `text_to_video` path by stripping the reference image when `scene_strategy in {action_keyframe, climax_peak}`.
- Boundary ruling:
  - Do not treat this as proof that all structured prompts require inline flags.
  - Do not continue implementation on the assumption that `action_keyframe/climax_peak => text_to_video` is an accepted mainline policy.
  - Before further code changes, explicitly review whether generation mode may be derived from `image_purpose` at all, or whether that policy is itself a regression against the supplier contract.
- Amendment decision:
  - `PLAN-20260401-044` remains open.
  - The next accepted action is a review-first amendment of the mode-policy boundary, comparing old-success vs new-failure request shapes before any direct implementation.

## 2.4 Plan Amendment A4: Factual Input Boundary Restoration
- Amendment trigger:
  - The A3 probe confirmed that the same scene-4 structured prompt succeeds when the original reference image is preserved, and fails on the no-image `text_to_video` path introduced by the recent mode-policy change.
  - The current implementation derives provider mode from `image_purpose` and then strips `image_url` for `action_keyframe/climax_peak`, which changes the supplier request shape even though an upstream factual asset is present.
- Why amendment is required:
  - This is not a minor tuning disagreement; it is a boundary error.
  - The project’s governing principles require `Anti-Hardcoding`, `Tools are execution-only`, `Supplier-Agnostic`, and `Root-Cause First`.
  - The current mode policy violates those principles by encoding a semantic if/else rule in tool code and by silently discarding a factual upstream input asset.
  - Without an explicit amendment, any direct code change would look like ad-hoc implementation churn rather than a governed correction of a proven-erroneous design.
- Re-scoped diagnosis:
  - The regression is not merely “Doubao dislikes some prompts”.
  - The deeper issue is that execution-layer code was allowed to override upstream asset presence and rewrite the request path from image-backed generation to pure-text generation.
  - “去 PPT 化” drifted from root-cause repair into a tool-layer behavior shortcut: suppressing image-backed generation instead of fixing scene semantics, image semantics, and prompt semantics at their proper boundaries.
- Boundary ruling:
  - Factual assets are authoritative inputs. If `image_url` is present, tool-layer execution must preserve it rather than silently discarding it.
  - Tool-layer code must not derive `generation_mode` from semantic labels such as `action_keyframe` or `climax_peak` via hardcoded if/else rules.
  - Any future decision to ignore a provided image must come from an explicit upstream contract field, not from execution-tool inference.
  - The accepted default is now frozen as: `有图走图，tool 不丢图`.
- Amendment decision:
  - The previous `action_keyframe/climax_peak => text_to_video` policy is now recorded as a proven-erroneous design, not a compatibility branch to preserve.
  - The next implementation slice must restore factual execution mapping first, before any additional prompt/provider optimization work continues.

## 3. 实施方案
- Patch strategy:
  - `authoritative semantics repair`
    - 在 `script` authoritative output 中新增 `scene_thesis`，一句话定义本场视频的核心事件与观看重点。
    - 显式把目标时长与压缩后的全局叙事注入单场景脚本 prompt。
    - 把 `action_phases` 改成事件阶段，`camera_hint` 改为可选弱提示。
  - `prompt synthesis rebuild`
    - `video_prompt_builder` 改为 `Main Key -> Event Arc -> Motion Guidance -> Style / Continuity -> Technical Note`。
    - `video_prompt_composer` 从 append-only 改为层级合成器，不再把 `一致性要求` 作为尾部大块直接拼接。
    - `prompt_mode` 从主叙事指令降级为技术说明。
  - `image anchor repair`
    - 为 image 层新增 `frame_thesis`，定义这张图是 `opening_anchor`、`action_keyframe`、`climax_peak` 还是 `continuity_bridge`。
    - 扩展 `image_purpose`，停止默认未知用途落到 `scene_opening_anchor`。
    - `image_prompt_composer` 同步改成层级合成，而不是基础 prompt + consistency 尾巴。
  - `mode policy repair`
    - 恢复 factual execution mapping：有 `image_url` 就保留 image-backed 请求路径，tool 不再静默丢图。
    - 移除 `action_keyframe/climax_peak => text_to_video` 这类由语义标签驱动的 tool-layer hardcoded 分支。
    - 若未来需要显式忽略图像输入，必须由上游 contract 明确给出，而不是由 execution tool 推断。
- Compatibility considerations:
  - `scene_thesis`、`frame_thesis` 属于 contract 扩展，应在 authoritative 载体和消费侧同时接入，不保留长期空值兼容分支。
  - `scene_opening_anchor` 不再作为未知用途兜底；必要时只保留显式迁移期诊断，不保留 silent fallback。
  - 不允许在 `video` 或 `image` 消费端增加 case-specific workaround 来掩盖上游 scene 质量问题。
  - 不允许把 slideshow / close-up 风险判断重新包装成 runtime guardrail、context summary 或 payload metadata。
- Rollback plan:
  - 若某阶段改造引发生成能力显著退化，仅回滚该阶段变更，不回滚已冻结的 contract 诊断结论。
  - 回滚只允许恢复到上一个已验证阶段，不允许重新引入 append-only compatibility shim。
