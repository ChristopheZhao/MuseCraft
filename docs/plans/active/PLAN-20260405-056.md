## 2. Deviation Analysis
- Expected behavior:
  - `quality_checker` should inspect the authoritative final-video artifact through an owner-correct local-file boundary, not treat a public URL as a local filesystem path.
  - QC should derive technical metadata from the actual final artifact or from a stable metadata contract, rather than defaulting to `duration=0` / missing file size / missing format when the final video exists.
  - QC content analysis should receive the same approved script-stage concept/timeline facts that downstream media agents already use, or it should fail explicitly with owner-correct diagnostics if those boundary inputs are unavailable.
- Actual behavior:
  - `quality_checker._execute_impl(...)` reads `project.final_video` from MAS WM, but then sets `final_video_url = fv.get("url") or fv.get("path") or ""`.
  - `_analyze_technical_quality(...)` receives that string and calls `os.path.exists(...)` on it directly. For this workflow, that means QC prefers the public URL `/files/outputs/videos/蛮荒记预告_final.mp4` instead of the available local path, so file-integrity diagnosis runs against the wrong namespace.
  - `video_metadata` at QC is not authoritative:
    - `project.final_video` writer currently stores `path/url/storage`, not duration/file size/format metadata.
    - QC falls back to summing `scene_overview` durations only if `fv.metadata` is absent.
    - The actual QC output proves that fallback did not recover timeline data on this run, because it emitted `duration=0`, `scene_breakdown.total_scenes=0`, and AI analysis text claiming that original concept and final composition timeline were missing.
  - Upstream evidence proves richer facts existed earlier in the same workflow:
    - script published deliverable contains `concept_plan` and `scene_overview` with 5 scenes totaling 45.0s
    - temp context snapshots for `image_generator` and `video_generator` also contain the same scene overview
  - QC does not receive equivalent boundary context from `ContextContractAssembler`; only image/video/audio/voice/composer get published-boundary assembly.
- Root cause hypothesis:
  - Primary fault path:
    - `backend/app/agents/quality_checker.py`
    - owner defect is inside QC input consumption / diagnostic boundary:
      - public URL is consumed as if it were a local file path
      - absent `project.final_video.metadata` is treated as missing technical truth
      - QC has no published-boundary context assembly for concept/timeline facts
  - Possible upstream dependency:
    - `project.concept_plan` / `scene_overview` are not reliably available to QC through the same path QC currently reads, even though script-stage published payloads prove those facts were available upstream.
    - This is not yet evidence that runtime/control-plane or `054`/`055` ownership is wrong; it is evidence that QC lacks a correct consumer boundary for those facts.
  - Non-owner artifact that must not drive implementation:
    - `backend/app/agents/prompts/templates/quality_checker/quality_checker.yaml` is not the live prompt path for the current QC content analysis.
    - Actual QC prompt rendering goes through `BaseAgent.render_prompt(...)` -> `PromptManager` -> legacy `backend/config/prompts/agents/quality_checker.yaml`, plus `backend/app/config/prompts/agents/mas_system.yaml`.

## 3. Implementation Approach
- Patch strategy:
  - Keep this as a QC-boundary repair stream first.
  - Stage A:
    - freeze owner-correct evidence for:
      - `project.final_video`
      - `video_metadata`
      - `composition_timeline`
      - actual QC prompt/input surface
    - decide whether the owner fix stays inside QC boundary or requires a narrowly scoped upstream contract change
- Stage B:
  - implement only at the owner-correct boundary identified by Stage A
  - preferred repair order:
    - fix final-video inspection semantics inside QC
    - keep QC concept/timeline recovery inside MAS WM facts through adapter/context-assembler boundaries; do not bypass QC to published/temp artifacts
    - keep final-video technical metadata projection on the owner chain/service boundary, not in sub-agent finalize paths
  - Stage C:
    - run live verification against the real workflow shape and confirm the previous false diagnostics do not recur
- Compatibility considerations:
  - Runtime/session/node truth remains owned by the existing control-plane surfaces closed under `055`.
  - `project.final_video` authority remains with the current owner from `054`; this plan must not move final-video authority into QC.
  - Do not add case-specific hardcodes keyed to `/files/outputs/videos/蛮荒记预告_final.mp4`, workflow `9587...`, or one supplier-specific format string.
  - Do not silently coerce missing concept/timeline facts into "pass" if the contract is still unresolved; Stage B should either supply the right boundary inputs or emit explicit owner-correct diagnostics.
- Rollback plan:
  - If a proposed Stage B change requires broadening into runtime/control-plane, stop and split to a successor plan instead of widening `056`.
  - If a proposed fix weakens final-video authority or introduces consumer-side compatibility shims that mask missing contracts, reject it and keep Stage B blocked.
