## 1. 当前问题
- `PLAN-20260325-020` 与 `PLAN-20260325-021` 已 completed；single-episode mainline、runtime SoT、authority contract 的主线治理已基本回正。
- 当前剩余问题不再是 “是否已有 canonical path”，而是 repo 里仍残留若干会把系统重新拉回坏架构的实现入口：
  - supporting capability 在消费点自愈 boundary，而不是在 boundary fail-close
  - shared-WM / helper / adapter 仍保留第二套 boundary/readback 协议
  - first-party frontend 仍保留 coarse `/status` 与 legacy WS vocabulary 的 authority re-entry surface
  - diagnostics / facts-summary 与 runtime truth 的词汇边界还不够清晰
  - project wrapper 仍保留 `_schedule_project_plan(...)` 这类 whole-system host uniformity 保留项
- 这轮工作的目标不是再定义一版 MAS 架构，而是按已冻结的基线把 repo 逐步拉回正确实现，清掉旁路、双轨和模糊 ownership。

## 2. 架构目标与硬约束
- 本计划服从：
  - [single_episode_harness_architecture_20260311.md](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/docs/architecture/single_episode_harness_architecture_20260311.md)
  - [mas_architecture_alignment_note_20260323.md](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/docs/architecture/mas_architecture_alignment_note_20260323.md)
  - [mas_architecture_deviation_inventory_20260323.md](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/docs/architecture/mas_architecture_deviation_inventory_20260323.md)
- Canonical hard rules:
  - `ContextContractAssembler` 是唯一 boundary resolver；边界缺失必须在 assembler/contract boundary fail-close。
  - `memory_views`、`published_deliverable_adapter`、`observation adapter` 等 supporting capability 不是第五层。
  - supporting capability 允许只读消费 canonical facts / projection，但不得在消费点自行为缺失 boundary 做 legacy readback、bridge fallback 或第二套 protocol 补齐。
  - first-party frontend 只允许消费 runtime/project read-model；queue/coarse task status/legacy WS direct type 不是 authority。
  - queue / worker / API 继续只做 execution container，不拥有 MAS runtime semantics。
  - `project wrapper` 继续位于 harness 外；不得为求“统一”把 wrapper 逻辑倒灌回 single-episode orchestrator。
- Delete-first rules:
  - 不新增 helper/adapter/compat bridge 来“包住旧实现”。
  - 一旦设计已确认错误，优先删除或替换，不保留 silent fallback。
  - compat surface 若暂时必须保留，只能是显式、单点、可退出的边界保留项。

## 4. 总体策略
- 采用 `总计划 -> 阶段计划 -> 阶段执行清单 -> 阶段验证门禁` 的推进方式。
- 每一阶段只解决一种 ownership 偏差，避免把 backend boundary、frontend authority、project wrapper host 一起大改。
- 每阶段完成后都应有：
  - focused code cleanup
  - targeted verification
  - grep inventory refresh
  - 推荐的 checkpoint commit
- 本计划的执行顺序有意不同于通用 inventory 的“问题严重度排序”：
  - `project wrapper host` 当前在架构清单中仍是 `out-of-scope-retained`
  - 因此 `022` 先处理会污染 canonical harness 的 active-path / bridge / authority surface
  - whole-system host uniformity 放在最后作为独立 tail decision，不阻塞 single-episode harness 回正
- 顺序必须固定：
  1. 先关 active-path fail-open
  2. 再退 bridge / alias
  3. 再清 first-party authority surface
  4. 再清 diagnostics grammar
  5. 最后单独处理 project wrapper host

## 7. 风险与护栏
- 风险 1：
  - 为了平滑迁移再包一层 helper/adapter，产生新的 compat bridge。
  - 护栏：本计划默认 delete-first；不得新增新的长期 adapter。
- 风险 2：
  - Phase 1 直接删 fallback 后暴露出 hidden writer 缺口，导致局部流程短时失败。
  - 护栏：缺口必须在 writer/boundary 侧修，不允许回补消费点 workaround。
- 风险 3：
  - frontend surface 清理与 backend compat endpoint 删除顺序错误，造成 first-party 或外部消费者回归。
  - 护栏：先删 first-party binding，再决定 external compat surface 的退出节奏。
- 风险 4：
  - 把 project planning host 的问题与 single-episode MAS 主线问题混做，导致范围漂移。
  - 护栏：Phase 5 单独处理，且不得把 planning host 并入 orchestrator。
- 风险 5：
  - 验证不足导致 repo 看似更干净，实际 authority-routing 或 facts-summary 回归。
  - 护栏：每阶段都必须有 grep + focused tests + changed-file compile 三重门禁。
