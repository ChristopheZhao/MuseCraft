## 1. 当前问题
- 本轮真实 smoke 已证明 `script approve -> resume -> image` 的 MAS 主线恢复链在当前代码和真实 dev DB 上成立；失败样本来自 10:14 启动、长期存活的旧 `8005` API 进程，而不是新的 control-plane continuation 实现。
- 当前 dev harness 缺少“启动前残留进程治理”：
  - [start_dev_uv.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/scripts/start_dev_uv.py) 只在退出 `finally` 中做 best-effort `pkill`，启动前不会检测已有 `uvicorn app.main:app` / `watchmedo auto-restart` / `celery` 残留。
  - 这允许旧的 `8005` API、worker、beat 在同一 repo 下长期存活；开发者随后再手工起一个 `8006` 当前代码 API 时，两个本地控制面会并存，导致 smoke 命中 stale process。
- Phase 3 后的真实 operator retest 进一步表明，当前 launcher patch 还没有关闭 live exit contract：
  - 实时进程态显示 `start_dev_uv.py` 父进程已经退出，但 repo-local `uvicorn app.main:app --port 8005` 与 `celery beat` 仍然存活。
  - 最新 [mas_workflow.log](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/logs/mas/mas_workflow.log) 记录了 14:31 的新启动和 14:37 的运行态失败，但没有对应的 shutdown closeout，说明这次测试不是“完整受管退出”，而是 launcher 提前退场后留下 child process。
- 问题 owner 是 dev startup / harness governance，不是 MAS runtime control plane：
  - queue/worker 语义没有偏移
  - artifact/WM owner 没有回退
  - 需要治理的是本地进程单实例/残留检测与显式 operator diagnostics
- 同次日志中暴露出的 Doubao image size 400 属于独立的图像工具 contract 问题，已显式拆分到 successor [PLAN-20260328-034.md](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/docs/plans/active/PLAN-20260328-034.md)，不并入本计划。

## 2. 架构边界与硬约束
- 本计划是 [PLAN-20260328-032.md](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/docs/plans/active/PLAN-20260328-032.md) 的 dev-harness successor，只处理本地 API/worker/beat 残留进程治理，不 reopen continuation snapshot、queue semantics、runtime SoT、artifact ownership 或 WM 边界。
- Hard rules:
  - 不得把 dev harness 问题重新下沉成 MAS runtime / queue / worker 业务语义改造。
  - 不得引入广谱 `pkill` 作为默认启动策略；残留治理必须有明确作用域和显式 diagnostics。
  - 不得把 “当前 API 是哪一个” 交给端口碰运气或人工记忆；必须有启动前 preflight 护栏。
  - 不得为了清理 stale process 引入第二套守护进程/状态文件 owner；治理入口应收敛在现有 dev launcher。
  - launcher 只拥有 OS process discovery / spawn / stop contract；不得直接承担 app 内部 async task 的生命周期。
  - app lifecycle 只拥有进程内 async init / cleanup contract；不得反向依赖 launcher 的 broad kill 才能完成正常 shutdown。
  - queue/worker 仍然只是 transport/execution container；本计划只治理本地进程生命周期，不改 MAS 主线 owner。

## 3. 目标语义
- 开发者通过受管入口启动本地服务时，launcher 必须先判断：
  - 当前 repo 下是否已经存在受管 API/worker/beat/watchmedo 残留
  - 目标 API 端口是否已被同 repo 的旧 `uvicorn app.main:app` 占用
- 对检测到的残留，只允许两种明确行为：
  - 默认 fail-fast：打印 PID / 端口 / command / 建议动作，拒绝继续启动
  - 显式 opt-in cleanup：只清理当前 repo 的受管残留，再继续启动
- 用户不得在不知情的情况下同时命中 stale `8005` 与 current-code `8006` 两套本地控制面。
- `Ctrl+C` 的标准退出语义必须是：
  - 父 launcher 捕获中断
  - 按其自己创建并追踪的受管 process groups 做 scoped shutdown
  - 在有界等待后报告 residuals
  - 正常路径不依赖人工 `pkill`
- `pkill -u "$USER" -f '/short-video-maker/backend/'` 可以保留为临时 operator escape hatch，但只能作为异常恢复手段，不得成为受管 launcher 的正式退出 contract。
- `Task was destroyed but it is pending!` 这类日志不属于进程治理真相，而属于 app lifecycle hygiene 信号；其修复必须落在应用内 async init/cleanup owner，而不是 launcher 层。

## 5. 推荐实现切口
- 当前推荐方向，不是本阶段实现承诺：
  - 初始实现切口只允许修改 [start_dev_uv.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/scripts/start_dev_uv.py) 和 [test_start_dev_uv.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/tests/unit/test_start_dev_uv.py)。
  - [start_dev.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/scripts/start_dev.py)、[dev.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/scripts/dev.py) 本轮不改；它们继续作为旧入口/薄包装存在，不承担新的治理 owner。
  - `start_dev_uv.py` 内部需要新增一套单一 matcher/contract，统一用于：
    - 启动前 residual preflight
    - 显式 opt-in cleanup
    - `Ctrl+C` 后的 residual report
    - 退出期 scoped residual cleanup
  - matcher 只识别当前 repo 受管进程：
    - `uvicorn app.main:app`
    - `watchmedo auto-restart`
    - `celery -A app.services.celery_app worker`
    - `celery -A app.services.celery_app beat`
  - matcher 必须同时校验 repo anchor，避免把其他仓库的同名命令纳入 cleanup 范围。
  - 默认启动行为：
    - 发现 residuals -> fail-fast 并打印 diagnostics
    - 只有显式传入 cleanup opt-in 时，才执行 scoped cleanup 后继续启动
  - `Ctrl+C` 行为：
    - 先对 launcher 已追踪的 process groups 做有界 shutdown
    - 再用同一 scoped matcher 检查是否仍有 repo-local residuals
    - 若仍有 residuals，只打印 diagnostics，不自动 broad kill
  - app 层 follow-up 明确不在本切口内：
    - 不修改 [main.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/main.py)
    - 不修改 [monitoring_service.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/services/monitoring_service.py)
    - 不修改 [enhanced_ai_client.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/services/enhanced_ai_client.py)
    - 若 `Task was destroyed but it is pending!` 仍需治理，另开应用生命周期 successor

  - 新增 startup preflight tests，证明发现 residuals 时默认 fail-fast，且输出 PID / command / recommended action
  - 新增 opt-in cleanup tests，证明仅在显式 cleanup 模式下才会对 scoped residuals 执行 stop
  - 新增 shutdown tests，证明 `cleanup_processes(...)` 不再依赖 broad `pkill` 模式，而是先停已追踪 process groups，再走 scoped residual matcher

  - 使用显式 cleanup opt-in 再次启动，确认 stale repo-local residuals 被清理后才允许启动
  - 启动成功后单次 `Ctrl+C`，确认 launcher 退出时不再需要手工 `pkill` 才能让受管 API/worker/beat 停止
  - 若退出后仍看到 pending-task 警告，只记录为 app lifecycle follow-up，不在本切口内继续扩 scope

## 6. 风险与护栏
- 风险 1：
  - 直接用广谱 `pkill -f uvicorn app.main:app` 作为默认启动逻辑，误杀其他仓库或用户有意保留的本地进程。
  - 护栏：默认路径必须是 scoped detection + explicit diagnostics；cleanup 必须显式 opt-in 且限定当前 repo。
- 风险 2：
  - 为了处理 stale `8005` 或 `Ctrl+C` 退出不干净，顺手把 MAS runtime / queue / continuation 逻辑混进 dev harness。
  - 护栏：本计划只处理 launcher/process lifecycle，不改 runtime 业务 owner。
- 风险 3：
  - 保留“退出时 broad cleanup + 启动时不检查”的现状，导致同类问题反复出现。
  - 护栏：Phase 1 必须冻结启动前 preflight contract，不能只补文档提醒。
- 风险 4：
  - 把应用内 pending async task 的脏退出全部归因到 launcher，不去收口 import-time async init。
  - 护栏：Phase 1 已明确分层；launcher 只管 OS process groups，app lifecycle 另做 bounded follow-up。
- 风险 5：
  - 把当前环境里的 `pytest` runner 挂起误记成 launcher 实现失败，导致无谓返工。
  - 护栏：Phase 3 验证需显式区分“代码 contract smoke 通过”与“测试 runner 环境异常”；runner 问题不自动升级为本计划 scope。
- 风险 6：
  - 把实时任务失败里的 Doubao image size 400 一并塞回 launcher 计划，重新混淆 owner。
  - 护栏：`033` 只收敛 live exit/orphan；图像 provider capability / size normalization 单独进入 `034`。
