## 1. 当前问题
- [PLAN-20260325-022.md](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/docs/plans/active/PLAN-20260325-022.md) 已完成 canonical harness convergence 的 blocker cleanup；本计划不 reopen `022`。
- 当前剩余问题已不再属于 “single-episode MAS 是否回到正确实现” 的 blocker，而是 repo-level tail：
  - first-party frontend 仍保留少量 `event.state -> UI counters/telemetry` 的派生状态面
  - backend 仍保留 coarse `/tasks/{id}/status` compatibility surface
  - `project planning` 仍使用 endpoint-local thread host，whole-system host uniformity 仍未决
- 这轮工作的目标不是再次调整 MAS 主架构，而是把 residual authority surface、残余旁路和 wrapper-host 决策从 `022` 的 blocker closeout 中拆出来，做单独治理。

## 2. 架构边界与原则
- 本计划服从：
  - [single_episode_harness_architecture_20260311.md](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/docs/architecture/single_episode_harness_architecture_20260311.md)

- Hard rules:
  - first-party frontend 只消费 runtime/project read-model；WS event 只能做 refresh trigger 或显式 diagnostics，不再拥有额外业务 counters/state authority。
  - compatibility surface 若保留，必须显式标注 external-only / non-authoritative，并带退出条件；不得重新成为 first-party 主线路径。
  - `project wrapper` 继续位于 harness 外；若后续做 host uniformity，也只能收敛到独立 product-job/shared host，不能并回 single-episode orchestrator。
  - 不新增新的 helper/adapter/client-side fallback 来“温和兼容”旧状态面。
  - 不把 `022` 已完成的 blocker cleanup 重新拉回本计划。

## 5. 执行计划
### Phase A: first-party frontend runtime-read-model strictness
- 目标：
  - 去掉 first-party 前端里所有由 WS event.state 直接维护的业务 counters / status 派生面。
- 执行清单：
  - [x] 删除 `concept_plan_ready` / `image_assets_ready` / `video_assets_ready` 对 app-store counters 的写入
  - [x] 删除 `scenesPlanned` / `imagesGenerated` / `videosGenerated` store 字段及其 UI 消费，除非能由显式 runtime/project read-model 提供等价字段
  - [x] 保留 `workflow_completed` / `workflow_failed` 仅作为 notification + refresh trigger，不写新的业务状态
  - [x] 校正 `reset()`，避免任何残余 telemetry state 穿透到下一次 run
- 阶段验收：
  - first-party quick frontend 不再持有 WS-derived business counters
  - quick UI 的状态展示只由 runtime read-model 与局部 UI state 构成

### Phase B: coarse compatibility surface governance
- 目标：
  - 明确 `/tasks/{id}/status` 的退出策略，或将其降格为显式 external-only compat surface。
- 执行清单：
  - [x] 复核 repo 内 callers / docs / tests，确认没有 first-party 或 internal mainline 依赖
  - [x] 无内部主线路径依赖后，删除 endpoint 及其测试/文档口径，而不是继续保留 external compat wrapper
  - [x] 清理仍把该 endpoint 当标准任务状态接口的当前文档口径，并同步刷新 active deviation inventory
- 阶段验收：
  - repo 内不再把 coarse `/status` 描述成 canonical task runtime view
  - compat surface 的地位与退出条件明确

### Phase C: project-wrapper host successor decision
- 目标：
  - 让 `_schedule_project_plan(...)` 的长期归属不再模糊，但不污染 single-episode harness。
- 执行清单：
  - [x] 做出 host-uniformity decision：不在 `023` 内继续实现，改由独立 successor plan 单独治理
  - [x] 保持 `_schedule_project_plan(...)` 继续处于 project-wrapper / product-job retained scope，并在 active docs 中补上 successor pointer
  - [x] 创建独立 successor plan `PLAN-20260325-024`，专门处理 project-wrapper host uniformity
- 阶段验收：
  - project-wrapper host 的定位清晰
  - 不把 repo-level host uniformity 再次误写成 single-episode MAS drift

## 7. 风险与护栏
- 风险 1：
  - 为了保留 UI 体验，再造一层新的 frontend derived counters。
  - 护栏：若 runtime/project read-model 未提供等价字段，就删 UI，而不是重建 client authority。
- 风险 2：
  - `/tasks/{id}/status` 删除过快，影响外部 compat consumer。
  - 护栏：先完成 caller inventory 与文档降级，再做删除或 deprecation。
- 风险 3：
  - 把 project planning host 顺手并回 orchestrator，造成 wrapper/harness 边界倒灌。
  - 护栏：host uniformity 只能走独立 product-job/shared host 路线，不能并入 single-episode harness。
- 风险 4：
  - 用这个 successor plan 反向稀释 `022` 的 closeout 结论。
  - 护栏：本计划只处理 non-blocking residuals；`022` 继续保留 canonical harness convergence 的 closeout 语义。
