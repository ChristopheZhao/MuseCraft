## 1. Current Problem
- Symptom:
- Quick 模式联调时，后端实际执行的 run 与前端当前显示/订阅的 task 可能不是同一条。
- 常见表现：
- 后端日志显示旧 run 已推进到 `script gate`
- 前端仍停在全节点 `queued` / 灰色
- 人工审核弹窗不出现，导致无法确认 MAS 修改是否真实命中 HITL 节点
- Impact:
- 修改 MAS 代码后，用户点击再次提交时，前端可能切到一条新的 queued task，而 worker 仍在串行执行旧 run。
- 如果页面刷新或后端重启，前端又会失去对当前 run 的控制面，导致即使后端已到 gate，前端也无法继续确认。

## 2. Root Cause
- Quick 模式当前 UI 形态本质上是单 workspace / 单当前 run，但实现上缺少 run ownership 约束。
- 具体断点：
- [src/components/forms/VideoRequestForm.tsx](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/src/components/forms/VideoRequestForm.tsx) 创建任务时 `session_id` 仍传 `undefined`，前后端没有稳定 workspace 锚点。
- 前端只有一个 `currentRequest`，任何新建任务都会覆盖原来的绑定对象。
- worker 以 Celery 串行执行 quick run，旧 run 不会因为前端发起新测试而自动让位。
- 旧 quick backlog 一旦已经进入 broker，消费端 [backend/app/services/task_queue.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/services/task_queue.py) 仍会重新执行数据库中已终态的任务，导致旧 run 在 worker 重启后“复活”。
- [backend/app/services/celery_app.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/services/celery_app.py) 仍以 `update_state(state='FAILURE', meta={'error': ...})` 写入非标准 failure result；历史脏结果会在 worker 启动后先于业务 guard 触发 Celery backend 解码崩溃，阻断新 run 的实际执行。
- 因而出现：
- 前端显示 run B
- 后端执行 run A
- HITL gate 落在 run A，但前端订阅的是 run B
- 本轮新增定位：
- [src/components/forms/VideoRequestForm.tsx](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/src/components/forms/VideoRequestForm.tsx) 中“继续查看当前任务”只做本地 store 绑定，不会创建新任务；真实的新任务创建仍只来自表单 `handleSubmit()`
- 但 unfinished-run 选择层当前不是强门控，提示条下方的整张创建表单仍可直接提交，因此会出现“用户以为已继续查看，随后仍触发表单提交并切到新 run”的交互歧义
- 本轮进一步新增定位：
- `script gate` 打开后，orchestrator 在 `finally` 中调用 `mem_service.cleanup_workflow(wf_id)`；当前 [backend/app/agents/memory/short_term/service.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/agents/memory/short_term/service.py) 的 `cleanup_workflow/reset_workflow` 会删除该 workflow 下的全部短期记忆 scope，而不仅是 `agent:` scope
- 由于 `mas:<workflow_id>` 共享短期记忆也被一并删除，approve/resume 之后的 `build_image_generation_context()` / `build_video_generation_context()` 只能从空的 MAS WM 重新组装上下文，导致 `scene_info_ref` 文件内容为空，媒体阶段跨 run 断链

## 3. Target Semantics
- 本轮不做“自动强绑旧 run”，也不做“任意排队池”。
- Quick workspace 的正确语义定为：
- 一个 workspace 同时只拥有一个“当前有效 run”
- 页面进入时若发现未终态 run，只做“发现并提示”，不默认接管
- 用户可显式选择：
- `继续查看当前 run`
- `新建运行`
- 用户点击提交按钮时，语义固定为：
- 发起一次新的测试 run
- 若当前 workspace 存在旧的未终态 run，则旧 run 需要被取消/作废后再创建新 run
- Queue / worker 必须共享同一套“任务执行资格”contract：
- 入队前终态任务不得进入 broker
- broker 中的历史旧消息若对应任务已终态，也必须在消费端被短路丢弃，不能再次进入 MAS 主线
- Celery 结果后端也必须遵守标准异常 contract：
- 失败任务由真实异常驱动 `FAILURE`
- 不再手写不完整的 failure meta
- 历史脏 Celery result/backlog 在修复后需一次性清理，避免继续先于 worker guard 触发崩溃

## 4. Non-Goals
- 不改 MAS orchestrator / gate / subagent 主逻辑
- 不改 Celery 基础设施模型
- 不引入复杂调试策略开关矩阵
- 不把“恢复旧 run”和“新建测试 run”混成同一个隐式行为

## 5. Execution Checklist
- [x] R1 Stable workspace session identity
- 在前端 store / quick workspace 层生成并持有稳定 `session_id`
- 创建 quick task 时显式传递该 `session_id`
- 刷新页面后仍能识别同一 workspace

- [x] R2 Backend unfinished-run discovery
- 在后端增加按 `session_id` 查询最新未终态 quick run 的 focused lookup
- 该 lookup 只用于页面进入时的“发现并提示”
- 判定规则统一为：
- 优先看 latest runtime session 是否为非终态
- 若无 session，再退回 `Task.status` 是否为非终态

- [x] R3 Frontend resume-or-new prompt hard gate
- 在 quick workspace 初始化时调用 unfinished-run lookup
- 若发现未终态 run，展示轻量选择层：
- `继续查看当前任务`
- `新建运行`
- 不自动绑定旧 run，避免用户修完代码后仍被默认拉回旧错误 run
- 在用户做出选择前，unfinished-run 选择层必须成为强门控：
- 不渲染或禁用下方整张创建表单
- 不允许在“未选择”状态下触发 `handleSubmit()`
- 只有点击 `新建运行` 后，表单才恢复为可编辑、可提交状态
- `继续查看当前任务` 必须直接切到 processing/workspace，并停止输入页的后续 submit 路径

- [x] R4 Submit means new run
- 用户点击提交按钮时，一律按“新建测试 run”处理
- 若当前 `session_id` 下存在旧未终态 run：
- 后端先取消/作废旧 run
- 再创建新 task
- 前端绑定新 task_id

- [x] R5 Reliable old-run cancellation
- 先在任务执行元数据中持久化 `celery_task_id`
- queue 时写回任务记录，并对已终态任务增加 queue 前护栏
- 提交新 run 时精确 revoke 旧 celery task，并将旧 task/runtime 标记为 `cancelled`
- 避免只改 DB 状态、旧 worker 仍在后台继续跑

- [x] R6 Lightweight submit dedupe
- 在 [src/components/forms/VideoRequestForm.tsx](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/src/components/forms/VideoRequestForm.tsx) 中增加 `isSubmitting`
- 仅用于单次请求未返回期间的防重入
- 这一步不承担 run 语义，只防止一次点击打出多次 `POST /tasks/`

- [x] R7 Unified execution eligibility policy
- 抽出统一的 quick task 执行资格判断，不再把终态判定散落在 create/queue/worker 各处
- 判断顺序统一为：
- 若存在 latest runtime session，则以 runtime terminal/non-terminal 为主
- 若无 runtime session，再回退到 `Task.status`
- 生产端与消费端复用同一套 policy，而不是手写两份状态判断

- [x] R8 Consumer-side terminal short-circuit
- 在 [backend/app/services/task_queue.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/services/task_queue.py) 的 `sync_process_video_task()` 入口增加终态护栏
- 对 `completed / failed / cancelled` 的历史 broker 消息直接记录并短路返回
- 终态旧消息不得继续进入 orchestrator / runtime session 恢复路径

- [x] R9 Celery failure-result contract fix
- 修正 [backend/app/services/celery_app.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/services/celery_app.py) 的失败上报逻辑
- 移除 `update_state(state='FAILURE', meta={'error': ...})` 这类非标准写法
- 失败时抛出标准异常，让 Celery 自己落标准 `FAILURE` result
- 保留 `PROGRESS` 更新；成功路径直接 `return result`，不再手写 `SUCCESS` state payload

- [x] R10 Stale Celery result/backlog hygiene
- 明确一次性开发环境清理步骤，清掉已写入的坏 Celery result / backlog
- 验证 worker 启动后不会再因历史 `exc_type` 缺失结果在 Celery backend 解码阶段崩溃

- [ ] R11 Focused verification
- 页面进入时，若存在旧未终态 run，前端应出现“继续查看 / 新建运行”而不是自动接管
- 选择“继续查看”时，前端能恢复到旧 run 的 runtime/gate，并且在该交互之后日志中不应出现新的 `POST /api/v1/tasks/`
- 选择“新建运行”或直接点击提交时，旧 run 被取消，新 run 成为当前唯一 run
- 修改 MAS 代码后重新提交，前端看到的是新 run 的节点推进和 HITL gate
- worker 重启或 broker 中存在历史消息时，已终态 quick task 不会再次进入 MAS 主线
- 旧 Celery backlog 不会再污染当前 workspace 的新 run 验证
- worker 启动后不会再先被历史 failure-meta 脏结果击穿，新的 quick run 能正常开始执行
- 在存在 unfinished-run 选择层时：
- 未点击 `新建运行` 前，不允许通过 Enter 键或其他隐式 form submit 创建新任务
- 只有明确点击 `新建运行` 后，才允许出现新的 `POST /api/v1/tasks/`

- [x] R12 Dev harness safe Celery output strategy

## 6. Test and Validation
- Backend focused checks:
- quick task 创建时写入稳定 `session_id`
- unfinished-run lookup 只返回当前 workspace 的最新未终态 run
- 提交新 run 时，旧 run 被 `cancelled`，新 run 获取新的 `task_id`
- 若旧 run 存在 `celery_task_id`，对应 revoke 被触发
- 历史 broker 消息命中已终态 task/runtime 时，worker 会直接 `skip`，不会再执行 orchestrator
- `process_video_task` 在业务失败时抛标准异常，不再写入非标准 `FAILURE meta`
- worker 在存在历史坏 result/backlog 时，清理后可稳定恢复，不再因 `exc_type` 缺失崩溃
- Frontend focused checks:
- 页面进入时发现旧 run 会弹出选择层，而不是自动绑定
- 选“继续查看”后，`currentRequest` 与 `/runtime` 绑定到旧 run
- 选“新建运行”后，`currentRequest` 切到新 run，不再显示旧 run 的 queued/gate
- `isSubmitting` 只拦一次请求内的重复点击
- Manual verification:
- 先启动一条 quick MAS run，到 `script gate` 或中途运行态
- 刷新页面，确认出现“继续查看 / 新建运行”
- 选择“继续查看”时，前端恢复到旧 run 的 gate/节点
- 修改代码后点击提交，确认旧 run 被取消，新 run 以新的 `task_id` 开始，前端看到的是新 run 的推进
- 重启 worker 或保留一条历史 broker 消息，确认已终态 `failed/completed/cancelled` 任务不会被重新执行
- 在清理坏 Celery result/backlog 后，重新验证一次“新建运行”链路，确认 worker 能从 `POST /tasks/` 后实际拉起新 run
