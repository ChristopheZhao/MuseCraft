## 1. Current Problem
- Symptom:
  - The current branch contains the completed MAS corrections but has not been published; it is 23 commits ahead of `main` / the locally cached `origin/main`, has no upstream, and the live remote cannot be queried because SSH host-key verification currently fails.
  - A clean install through `backend/requirements.txt` fails because it contains the nonexistent `hashlib3==2.1.2`, while `pyproject.toml` and `uv.lock` already form a different dependency graph.
  - All 15 Alembic revisions are ignored and untracked. The chain is linear, but several revisions and `alembic.ini` are MySQL-specific while runtime defaults, compose, CI, and the root environment example use PostgreSQL.
  - `backend/Dockerfile` installs from the broken requirements file and uses `curl` in its health check without installing it. Compose does not define an explicit environment-file contract for API/worker services.
  - Backend CI references three valid tests through incorrect paths. Frontend lint/type/unit pass, but integration and performance suites exceeded the local 180-second audit budget, and accessibility currently fails 28 of 31 tests.
  - README, backend README, `ENV_SETUP.md`, and the three environment examples disagree about database, environment-file location, and Docker commands. The root README also contains a clone placeholder and an unverified support address.
  - The repository lacks the usual contribution, security, conduct, changelog, and issue/PR guidance expected for public collaboration.
- Impact:
  - A new contributor cannot reliably install or start the project from the tracked repository.
  - Database initialization is not reproducible and the documented PostgreSQL path cannot consume the current MySQL-specific migration history safely.
  - GitHub Actions would fail or hang after publication, so CI badges or release-quality claims would be misleading.
  - Public users cannot distinguish the supported path from historical or local-only configuration.
- Reproduction:
  - `backend/requirements.txt:53` contains `hashlib3==2.1.2`.
  - `.gitignore` ignores `backend/alembic/versions/*.py`; `git ls-files backend/alembic/versions` is empty while 15 local revisions exist.
  - `.github/workflows/integration-tests.yml` references `tests/test_*_validation.py`, but the files live under `backend/tests/`.
  - Baseline on 2026-07-11: `npm run lint`, `npm run type-check`, and frontend unit tests passed; frontend integration/performance timed out at 180 seconds; accessibility reported 28 failed and 3 passed; `uv lock --project backend --check` passed when pointed at an isolated valid environment.

## 2. Deviation Analysis
- Expected behavior:
  - `pyproject.toml` plus the committed lock file are the backend dependency source of truth, and every install surface consumes that contract without a second drifting list.
  - One documented PostgreSQL-first path works from clean clone through schema migration and service startup; optional database support is not claimed unless it is independently tested.
  - CI references real tests, has bounded runtime, and reports actionable failures without stale suites dominating the signal.
  - Public docs describe one environment layout and one set of prerequisite/runtime commands.
  - The public repository includes concise collaboration and security policies and does not depend on ignored local marketing or secret files.
- Actual behavior:
  - Dependency, database, container, CI, and documentation surfaces evolved independently and now contradict each other.
  - Migration files were treated as local artifacts even though they are runtime schema source-of-truth assets.
  - Several frontend suites encode older UI copy/behavior and are not currently credible release gates.
- Root cause hypothesis:
  - Internal development optimized local iteration and accumulated parallel setup paths without a clean-clone acceptance contract.
  - The earlier open-source assessment mixed hard runtime blockers with optional history/branding cleanup, obscuring the minimum release boundary.

## 3. Implementation Approach
- Patch strategy:
  - Sprint 1 - reproducible backend and schema: make `pyproject.toml`/`uv.lock` authoritative, repair Docker installation/health behavior, track a PostgreSQL-compatible Alembic path, and define explicit environment propagation.
  - Sprint 2 - truthful CI: repair backend paths, classify frontend suites by current validity, fix test infrastructure/open handles and stale assertions needed for stable release gates, and avoid claiming checks that cannot pass.
  - Sprint 3 - public repository contract: converge README/environment guidance on one PostgreSQL-first setup, remove unverifiable placeholders, add minimum community/security files, and document external prerequisites including FFmpeg, Docker, Node, Python, and uv.
  - Sprint 4 - release acceptance and submission: validate clean tracked content, dependency lock, migration graph, backend focused/full tests as feasible, frontend lint/type/tests/build, secret-ignore boundaries, and Git diff hygiene; then create intentional commits. Remote push remains a separate external action after SSH trust and destination are confirmed.
- Compatibility considerations:
  - Existing MySQL runtime branches may remain as non-primary compatibility code, but public docs and migrations must not claim MySQL support unless the full chain is tested.
  - Existing production databases must not be silently restamped onto a replacement baseline; any migration consolidation must preserve an explicit upgrade/stamp policy.
  - Internal governance documents and the existing commit history are not removed or rewritten in this plan. A separate public-history/export decision requires explicit user approval because it changes provenance and recovery options.
  - Ignored real `.env`, backup files, business materials, generated media, and local marketing assets must remain outside Git; release artifacts must be produced from tracked files only.
- Rollback plan:
  - Keep plan/governance, backend reproducibility, CI, and public-doc/community changes in separate commits.
  - Do not rewrite branch history or force-push. If a migration or test-gate design proves invalid, revert that semantic slice without touching the completed MAS commits.

## 3A. Execution Contracts

### Slice A - Reproducible backend and schema
- What: a clean tracked checkout can resolve locked dependencies, build the backend image contract, and identify a PostgreSQL-compatible migration head without relying on ignored local files.
- Why: all higher-level documentation and CI claims depend on this runtime substrate being truthful first.
- Done signals:
  - `hashlib3` and dependency-list drift are eliminated; Docker consumes the authoritative dependency contract.
  - Alembic revisions needed by the application are tracked and the graph has one head.
  - Alembic derives its URL from configuration and no default release path requires MySQL-only DDL.
  - Compose passes documented environment configuration to every service that needs provider credentials.
- Evidence: `backend/pyproject.toml`, `backend/uv.lock`, `backend/Dockerfile`, `backend/docker-compose.yml`, `backend/alembic.ini`, `backend/alembic/env.py`, tracked revisions, lock/migration tests.
- Negative cases: no nonexistent package, ignored schema revision, hard-coded credentialed database URL, missing health-check binary, or silent production-database restamp remains.
- Suggested checkpoints: `dependency-sot`, `postgres-migration-chain`, `container-config-contract`.

### Slice B - Truthful CI
- What: checked-in workflows execute real, bounded tests that represent current product behavior.
- Why: CI must become evidence before public documentation or release metadata relies on it.
- Done signals:
  - Every workflow test path exists and its command is locally reproducible.
  - Frontend integration/performance/a11y jobs either pass after root-cause repair or are explicitly removed from required CI with a documented follow-up; they are not left as known-red gates.
  - Lint, type-check, unit tests, and production build pass.
- Evidence: `.github/workflows/*.yml`, Jest configuration/setup, repaired tests, command results.
- Negative cases: no ghost path, unbounded open handle, 28-failure stale suite, or green-by-skipping required tests remains.
- Suggested checkpoints: `workflow-path-integrity`, `frontend-test-signal`, `build-smoke`.

### Slice C - Public repository contract
- What: a new contributor sees one supported installation path and the minimum collaboration/security policy surface.
- Why: setup truth and contribution boundaries must match the validated runtime rather than internal historical instructions.
- Done signals:
  - Root and backend setup docs consistently describe PostgreSQL-first configuration and one backend environment-file location.
  - Clone/support placeholders and references to unavailable tracked assets are removed or replaced with truthful guidance.
  - `CONTRIBUTING.md`, `CODE_OF_CONDUCT.md`, `SECURITY.md`, changelog/release notes, and GitHub contribution templates exist with repository-appropriate content.
  - README documents Python 3.11, Node 18+, uv, Docker, and FFmpeg prerequisites and does not claim unverified CI or provider support.
- Evidence: README files, environment examples, community files, link/reference scans.
- Negative cases: no MySQL/PostgreSQL contradiction, mutually exclusive `.env` copy instruction, fake support address, or ignored marketing asset required for first render remains.
- Suggested checkpoints: `clean-clone-doc-contract`, `community-surface`, `tracked-asset-reference`.

### Slice D - Release acceptance and submission
- What: only validated, intentionally staged changes are committed, with remote publication gated on destination and SSH trust verification.
- Why: the local workspace contains real ignored secrets/business files, and remote state is currently unverifiable.
- Done signals:
  - `git status`, `git diff --check`, tracked-file secret scan, dependency/migration checks, selected backend tests, frontend checks, and build evidence are recorded.
  - Commits are split by governance, runtime/schema, CI, and public surface where practical.
  - No ignored local secret or business artifact is staged.
- Evidence: Git staged-file lists, validation outputs, commit log, remote/upstream verification.
- Negative cases: no whole-worktree archive, accidental `.env` staging, unverified force-push, history rewrite, or release tag is created.
- Suggested checkpoints: `release-candidate-audit`, `staged-content-review`, `remote-destination-verification`.
