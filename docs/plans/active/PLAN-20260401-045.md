## 1. Current Problem
- Symptom:
  - quick runtime repeatedly fails with `Workflow attempt <id> execution lease expired`
  - current observed failure surfaced on the `video` node after image repeat-planning was corrected
- Impact:
  - long-running image/video execution can continue at the business/tool layer but still be rejected by the control plane at attempt completion
  - runtime diagnostics currently preserve the expired-lease result, but not enough host-side keepalive failure context to identify the exact bridge failure mode
- Reproduction:
  - `task_id=46766c83-a414-44aa-a13f-5940e3841adb` failed on `video`
  - runtime evidence shows `last_heartbeat_at=2026-04-01T02:32:12+00:00`, `lease_expires_at=2026-04-01T02:37:12+00:00`, and session failure at `2026-04-01T02:48:16+00:00`
  - local artifacts show `backend/storage/temp/video_output/final_frame_1775011566470_c0860f38.jpg` at `2026-04-01 10:46:07`, which indicates business execution likely progressed after the lease had already expired
  - `task_db_id=1039` reproduced the same failure on `video` after `PLAN-20260402-048` repaired the upstream `video_generator` authority chain
  - runtime DB evidence for session `33` / attempt `47` shows `last_heartbeat_at=2026-04-02T14:32:39+00:00`, `lease_expires_at=2026-04-02T14:37:39+00:00`, and failure capture at `2026-04-02T14:44:22+00:00`
  - this replay proves the current production symptom is still a post-grant lease-renewal failure in the control plane rather than a `video_generator` business failure
  - a fresh replay on `task_db_id=1043` / session `37` / attempt `55` reproduced the same terminal failure after three accepted scene videos had already been generated
  - mid-run runtime projections showed `attempt 55` receiving at least two successful heartbeat renewals, but the completion-time authoritative view still failed with `lease_expires_at=2026-04-03T03:22:53+00:00` and `last_heartbeat_at=2026-04-03T03:17:53+00:00`
  - this replay proves the remaining failure is no longer "renewal never happened"; it is a stale validation read at completion time

## 2. Deviation Analysis
- Expected behavior:
  - queue/worker remains execution container only
  - orchestrator grants the attempt lease, activates host-side keepalive, and keeps the lease live throughout long-running node execution
  - `RuntimeSessionService` remains the sole authority for attempt liveness and may reject stale completions
- Actual behavior:
  - control-plane lease validation still works as designed
  - the host-liveness bridge itself can renew long-running attempts, but completion/failure validation previously ran on a stale repeatable-read snapshot taken from the long-lived task-queue/orchestrator SQLAlchemy session
  - failure therefore surfaced as a terminal runtime error even when the host had already renewed the lease successfully mid-run
- Root cause:
  - the fault is not in media-agent business logic or planner behavior
  - the fault is inside the control-plane freshness path that validates an attempt lease at completion/failure time
  - confirmed root cause:
    - quick runtime opens one long-lived SQLAlchemy `Session` in `task_queue.sync_process_video_task` and passes it through orchestrator execution
    - host-side heartbeat renewals commit through separate `SessionLocal()` instances
    - MySQL is running at `REPEATABLE-READ`, so the long-lived orchestrator/session transaction can continue to see the pre-heartbeat lease snapshot even after `db.expire_all()`
    - `RuntimeSessionService.complete_node_attempt_sync` and `fail_node_attempt_sync` therefore validate against stale lease timestamps and falsely reject a still-live attempt as expired

## 3. Implementation Approach
- Patch strategy:
  - treat this slice as a runtime host-liveness correction, not a `video_generator` fix
  - start with an architecture-level RCA for the lease model itself instead of assuming the current design is correct:
    - whether lease belongs at runtime attempt scope
    - whether `RuntimeSessionService` / orchestrator / execution-host ownership is the right split
    - whether TTL + heartbeat renewal is the right liveness model for long-running quick-runtime nodes
    - whether completion-time lease validation semantics are correct for the intended control-plane authority model
    - whether current timeout values are configuration mismatch or merely expose a deeper design / implementation fault
  - first localize the root cause inside the control-plane bridge before adding another repair patch
  - use architecture review, code-path tracing, runtime DB evidence, and focused cross-layer tests to identify which owner layer actually breaks:
    - keepalive activation scheduling
    - background heartbeat execution
    - heartbeat persistence / validation
    - unhealthy-state diagnostic publication / propagation
  - only after that RCA gate is closed, correct the owning implementation
- Compatibility considerations:
  - preserve `RuntimeSessionService` as the single SoT for attempt liveness
  - do not move lease semantics into media agents, tools, or queue status
  - do not treat `video_url`, final-frame artifacts, or `scene_outputs.*` as completion authority
  - do not reopen `PLAN-20260402-048` business-boundary repairs while investigating this runtime bridge failure
  - do not fold `PLAN-20260402-047` image-composer cleanup into this stream; `047` remains an unfinished parallel plan with its own boundary and acceptance
- Rollback plan:
  - if a host-side keepalive change destabilizes quick runtime, rollback only the keepalive/bridge patch
  - do not rollback by weakening control-plane lease validation or by introducing media-agent-side lease shims

## 4. Boundary Statement
- Queue worker and queued execution host are transport / execution containers only.
- `RuntimeSessionService` owns live runtime/session/node/attempt/lease semantics.
- Orchestrator owns the bridge between active node execution and host-side lease renewal.
- Media agents and tools must not own lease renewal, runtime completion authority, or artifact-based recovery logic.
- This plan is separate from `PLAN-20260401-044`; `044` remains the parallel slideshow/prompt-synthesis stream and must not be overwritten by this work.
- This plan must not absorb `PLAN-20260402-047`; `047` remains in_progress for image-composer diagnostic cleanup and is reference-only here.
- Root-cause localization is constrained to runtime/control-plane lease ownership. No queue-status authority, media-agent lease shim, or artifact-based recovery workaround is allowed.
- Architecture review is in scope, but only within those control-plane boundaries:
  - this plan may conclude that the current lease design itself is wrong or incomplete
  - if so, the remedy must be an explicit control-plane redesign, not a compatibility shim or a business-layer workaround

## 5. Ownership Clarification
- The queue layer is not the semantic owner of the heartbeat event.
- The execution-host keepalive loop is the event-source owner for `mid-run heartbeat stopped/failed` because it is the component that actively renews the lease and first observes renewal loss.
- Orchestrator is the bridge owner because it binds the leased attempt to the execution host, activates keepalive, and decides how host-liveness failures are surfaced during node execution.
- `RuntimeSessionService` is the authority owner because it alone defines whether the attempt lease is valid, expired, mismatched, or already cleared.
- Therefore the original production symptom must be treated as an execution-host / orchestrator / runtime bridge problem, not a queue problem and not a media-agent business problem.

## 5A. Stage A Architecture Verdict
- Lease verdict:
  - attempt-scoped lease remains architecturally valid as a control-plane fencing / ownership mechanism
  - `workflow_id` / task identity is not sufficient to replace it because workflow identity does not answer which live attempt currently owns completion rights
- Owner verdict:
  - `RuntimeSessionService` owns lease truth and validation
  - execution host owns keepalive event production only
  - orchestrator owns bridge activation and completion surfacing, but is not the semantic owner of lease truth
  - queue / worker / broker are transport only and must not own MAS runtime meaning
- Communication verdict:
  - Celery transport, event bus, and WebSocket progress projection are not lease authority and must remain diagnostics / projection paths only
  - there is no evidence that queue or WebSocket currently compete for runtime truth; the defect is inside the application-layer lease bridge
- Design-correction verdict:
  - the lease concept itself is not the primary design error
  - the current keepalive contract is under-specified and operationally misfit for long-running quick-runtime nodes because:
    - activation does not prove first-heartbeat success
    - blocked heartbeat can create a silent expired window before unhealthy state is visible
    - tool-visible lease semantics leak control-plane liveness too deeply into execution paths
    - completion failure receipts are too weak to distinguish activation, renewal, lifecycle-clear, and final-validation defects
- Required design direction:
  - preserve attempt-scoped lease at the control-plane authority layer
  - reduce lease visibility outside that layer to bridge diagnostics and explicit cancellation semantics
  - do not treat queue health, worker state, or produced artifacts as runtime completion authority

### Stage D verdict recorded
- Owner layer:
  - runtime validation freshness inside the control-plane path, specifically the long-lived SQLAlchemy session opened by `task_queue.sync_process_video_task` and reused by orchestrator attempt completion/failure transitions
- Primary defect class:
  - stale transaction snapshot under MySQL `REPEATABLE-READ`
- Concrete failing path:
  - `task_queue.sync_process_video_task` opens one sync `db` session for the whole run
  - `queued_task_execution_host` keepalive renewals use separate `SessionLocal()` sessions
  - `RuntimeSessionService.complete_node_attempt_sync` / `fail_node_attempt_sync` call `_fresh_session_node_attempt_sync(db, ...)` on that long-lived session
  - `db.expire_all()` refreshes ORM objects, but does not end the transaction, so the session keeps reading the old repeatable-read snapshot and validates against stale lease timestamps
- Proof bundle:
  - live replay `task_db_id=1043` / session `37` / attempt `55` showed multiple successful heartbeat receipts during `video`
  - final runtime failure still reported `last_heartbeat_at=2026-04-03T03:17:53+00:00` and `lease_expires_at=2026-04-03T03:22:53+00:00`, proving completion did not observe the later renewals
  - a live MySQL probe confirmed `@@transaction_isolation = REPEATABLE-READ`
  - a second live probe proved that one session continues to read the pre-heartbeat lease values after another session commits a heartbeat, and only sees the new values after `rollback()` clears the old transaction snapshot

### Stage E: repair design and bounded implementation
- Goal:
  - repair the owning defect without widening scope into `047`, `048`, queue authority, or media-agent workarounds
- Preferred repair shape:
  - keep long-running business execution and control-plane completion/failure validation off the same repeatable-read transaction snapshot
  - use a fresh short-lived control-plane session for post-execution attempt transitions, or otherwise end the old transaction before authoritative lease validation
  - do not change MySQL isolation globally as a first fix
  - do not weaken lease validation and do not move lease truth into tools, queue state, or artifacts
- Required constraints:
  - preserve `RuntimeSessionService` as attempt-liveness authority
  - do not move business truth into queue / worker state
  - do not let vendor tools own lease semantics
  - if design change is required, make it explicit at the control-plane boundary rather than adding a compatibility shim
- Exit gate:
  - repair plan names:
    - exact files/functions to change
    - exact invariants that will hold after the patch
    - exact regression tests and runtime replay needed for closure

- Invariants now enforced:
  - host-side keepalive renewals remain authoritative inputs into `RuntimeSessionService`
