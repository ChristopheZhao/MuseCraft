## 1. 当前问题
- 当前视频质量问题不是单一 prompt 质量波动，而是 `concept -> script -> image -> video` 四层 contract 同时失真。
- 已确认的根因链：
  - `scene` 粒度偏“章节摘要”，不是“可拍的局部事件”。
  - `script` 没有把剧情功能稳定下沉成时间尺度内的可见动作弧线。
  - `image` 首帧职责不够收敛，容易混入剧情解释和风格海报化。
  - `video prompt` 组装把抽象叙事、一致性 prose、动作信息混拼，且 `motion_beats` 实际存在字段错配丢失。
  - `scene sizing` 受 provider 单段时长能力反推，偏容量驱动，不是叙事密度驱动。
- 架构层同时存在未来扩展风险：
  - 当前最多 15s，未来可能到 20s；如果 scene 叙事结构写死在 10s prompt 模板里，能力升级会重新打破语义。
  - 一致性资产、上下文装配、WorkingMemory、runtime orchestration 若不重新钉死 owner 边界，后续会再次长出第二套 SoT 或静默 fallback。

## 2. 目标边界与硬约束
- 目标收敛：
  - `concept` 负责 episode 梗概、角色线、故事线、scene 功能与依赖关系。
  - `script` 负责把 scene 压成可拍的局部事件，并输出可扩展的动作阶段结构。
  - `image` 只负责 scene 开场状态首帧与视觉锚点。
  - `video` 负责把首帧和动作阶段演成镜头段，并始终遵守全局一致性与当前 scene 边界。
- 时长扩展约束：
  - 叙事能力必须按 `scene contract` 投影到 5s/10s/15s/20s，不允许把 10s 的死板秒表脚本复制到所有时长。
  - `action phases` 需要支持相对时序或权重分配，避免 provider 时长升级后重新设计语义。
- 架构硬约束：
  - `global consistency` 与 `local continuity` 必须分层，不得混用。
  - `ContextAssembler` 继续只读 fail-fast，不得变成补丁入口。
  - `MemoryWriter` 只负责标准化事实回写，不得拥有 runtime/planning 语义。
  - `consistency_tool` 只能做资产收集/投影，不得长成第二套状态机。
  - generator/evaluator 质控只能是 gate surface，不得变成第二个 owner 或自动重写主线。
  - 所有 `v2 contract` 必须是 authoritative carrier 的原位演进；不得新增平行 JSON、平行 memory slot、平行 runtime payload 作为第二条主线。
  - 在进入任何 contract 设计前，必须先冻结 owner/SoT 矩阵；未映射到单一 authoritative carrier 的字段不得进入 schema。
  - generator/evaluator 的输出只允许落到 plan file、validation ledger 或 review notes；不得写入 runtime input、published deliverables 或 WorkingMemory 主消费槽位。

## 2.1 纠偏治理原则
- 已执行阶段不可重写：
  - Sprint 1-3 的完成记录、证据和当时的验收结论保留原样，不通过直接改写历史来“假装从未发生”。任何新的根因发现必须以 amendment 追加，并注明对后续阶段的影响。
- 根因纠偏优先于继续推进：
  - 当已发现高危架构偏离时，必须先完成根因纠偏并重置剩余阶段前置条件；不得一边保留已知高危偏离，一边继续推进未实施 sprint。
- Read-model purity：
  - runtime/read-model 投影、查询选择器和 UI 探测路径不得承担 authoritative runtime mutation。任何 reconcile、failover、terminalization 都必须来自显式 control-plane mutation 入口。
- Single reconcile owner：
  - stale runtime reconcile 只能由单一控制面维护入口拥有；不得散落在 `build_runtime_view`、API discovery path、FastAPI lifespan 后台循环等高扇出入口。
- Error transparency over suppression：
  - runtime projection integrity error 不得在 current-run selector 或类似入口被静默吞掉并伪装成“没有当前 run”；异常必须显式 surfaced 或经受控 gate 处理。
- 执行前先重排剩余计划：
  - 对于尚未实施的 sprint，只允许通过 amendment 更新其前置条件、范围与验收；不得通过覆盖原文抹平偏离。

## 3. 目标结果
- 形成 `SceneContract v2`，支持 narrative-scalable 的局部事件表达，而不是写死秒表脚本。
- 形成 `ConsistencyAssetContract v2`，将 episode 级锁定、scene 级开场锚点、局部连续性区分为三个独立语义面。
- 形成 `VideoPromptContract v2`，使 video agent 消费的是结构化动作弧线，而不是 planning prose。
- 形成单一 owner 的编排/上下文/记忆边界，保证扩展到更长时长与更多 provider 时不回退。
- 形成可复用的阶段验收与轻量 generator/evaluator 质控模式。

## 4. 分阶段执行
### Sprint 1: Scene Contract v2 freeze
- 前置 gate：
  - 先冻结 owner/SoT matrix，明确 `concept/script/image/video` 的 authoritative carrier、read-model、validation surface 与禁止落点。
  - 先完成 `SceneContract v2` 字段到 authoritative carrier 的映射，未映射字段不得进入草案。
- 本阶段关闭内容：
  - 定义 `scene = 可拍的局部事件` 的单一语义。
  - 冻结 `opening_state / event_trigger / action_phases / end_state / global_locks_ref / continuity_ref` 这类可扩展字段方向。
  - 冻结时长投影原则：支持 5s/10s/15s/20s 的相对时序，不写死固定分秒模板。
- 为什么先做这个边界：
  - 上游 scene 语义不收敛，后续 script、consistency、video prompt 都只能继续 patch。
- 验收边界：
  - Done signals：
    - owner/SoT matrix 已冻结，并覆盖 contract 字段落点。
    - `concept/script/image/video` 对 scene 的职责定义一致。
    - 至少给出 3 个 scene 样例，覆盖 5s、10s、15s 或 20s 中任意两个时长，且结构不依赖写死秒表。
    - 明确什么是 scene 局部事件、什么是 episode 剧情功能，二者不再混写。
  - Evidence checklist：
    - owner/SoT matrix
    - SceneContract v2 字段草案
    - 3 个正例 scene 样例
    - 2 个反例说明：章节摘要型 scene、秒表硬编码型 scene
  - Negative cases：
    - `SceneContract v2` 通过新的 memory/runtime carrier 平行落盘
    - 仍允许 scene 只写“高潮/成长/铺垫”等功能词，没有可见事件
    - 将 10s 的分秒脚本直接当成固定 contract
  - Suggested checkpoint：
    - 通过一次 generator/evaluator 语义对齐检查后，才能进入 Sprint 2

### Sprint 2: Script and Video Prompt Contract rewrite
- 本阶段关闭内容：
  - `script` 输出从摘要型字段转向可执行动作阶段。
  - `video prompt` 主体改为“开场状态 + 动作阶段 + 镜头语言 + 结束状态”。
  - 明确 `text_to_video`、`image_to_video`、`continuity` 三类模式的 prompt 消费差异。
- 为什么单独成阶段：
  - 这是 planning 语言和执行语言的分水岭；如果和一致性或 orchestration 一起改，回归面会失焦。
- 验收边界：
  - Done signals：
    - `script` 输出 contract 明确支持 `action_phases` 或等价结构。
    - 最终 video prompt 不再直接把 `narrative_description` 当主体。
    - 明确 image-to-video 场景下文本重点写“如何动”，而不是重新描述整张首帧。
  - Evidence checklist：
    - ScriptContract v2 草案
    - VideoPromptContract v2 草案
    - 至少 3 组 prompt before/after 对照样例
  - Negative cases：
    - 仍把“烘托紧张感、铺垫转折、突出成长”直接作为视频主体 prompt
    - 仍要求 video prompt 依赖固定秒级脚本才能成立
  - Suggested checkpoint：
    - generator 生成候选 prompt，evaluator 按“可见动作、镜头可执行、一致性不喧宾夺主”三条进行 gate

### Sprint 3: Consistency Asset Contract convergence
- 本阶段关闭内容：
  - 收敛 `consistency_tool` 输出为结构化 locks，而不是大段 prose。
  - 将 `global consistency / local continuity / opening anchor` 三层彻底拆开。
  - 明确 image/video composer 如何消费这些资产。
- 为什么单独成阶段：
  - 一致性是全局约束，但不能再侵入 scene 主体动作语义；这一步不收敛，prompt 重写也会被噪声淹没。
- 验收边界：
  - Done signals：
    - ConsistencyAssetContract v2 明确包含 episode 锁定、scene 开场锚点、局部连续性。
    - 不再要求将角色小传、世界观说明直接拼进最终 prompt 主体。
    - image/video 两侧消费方式一致，不产生两套锁定语义。
  - Evidence checklist：
    - consistency assets 新 schema
    - image/video composer 消费矩阵
    - 至少 2 个跨 scene 连续性样例和 2 个全局一致性样例
  - Negative cases：
    - 用 continuity 去承担全局风格一致性
    - 用 global consistency 去覆盖当前 scene 的具体动作信息
  - Suggested checkpoint：
    - evaluator 对同一 episode 的 3 个 scene 进行资产注入审阅，确认锁定信息“够硬但不喧宾夺主”

  - Evidence checklist：

    - 读写路径重映射表
    - 至少 3 个负例测试：
      - runtime view 不写库
      - current-run selector 不终结 runtime
      - API 实例默认不成为 reconcile owner
    - 至少 1 个显式 reconcile 入口验证样例
  - Negative cases：
    - 在 read-model builder 内部调用 `mark_session_failed_sync()`
    - 在 selector 中吞掉 projection integrity error 并继续装作无当前 run
    - 让 FastAPI app instance 默认通过 lifespan 后台循环充当 reconcile owner
  - Suggested checkpoint：
    - 通过一次 generator/evaluator boundary gate，确认不存在第二条 runtime mutation 主线后，才能恢复原 Sprint 4

## 5. 质控方案：是否需要 generator/evaluator battle
- 结论：
  - 需要，但必须是轻量 gate，不是第二套 owner。
- 冻结用法：
  - 每个 sprint 只在阶段验收时启动一次 generator/evaluator 对抗式审阅。
  - generator 负责产出候选 contract / prompt / asset schema / boundary inventory。
  - evaluator 负责按本 sprint 的 done signals、evidence checklist、negative cases 给出通过/不通过与诊断。
  - 单个 sprint 最多允许 2 轮 battle；超过 2 轮说明 contract 仍未收敛，应回到人工调整，不继续自动循环。
  - battle 输出只允许写入 plan file、validation ledger 或 review note；不得进入 runtime payload、published deliverables、WorkingMemory 主槽位。
- 明确禁止：
  - evaluator 直接改写主方案
  - generator/evaluator 自行推进 lifecycle 状态
  - 让 battle 替代用户确认或正式 validation ledger

## 7. 风险与护栏
- 风险 1：
  - 把 scene contract 做成 prompt 模板别名，而不是上游语义 contract。
  - 护栏：Sprint 1 必须先冻结结构化 scene 语义，再允许改 prompt。
- 风险 2：
  - 把 consistency 继续做成长 prose，导致下游 prompt 被淹没。
  - 护栏：Sprint 3 只允许结构化 locks，不接受“更长的一致性提示词”作为完成态。
- 风险 3：
  - 为了快速打通链路，给 ContextAssembler/WorkingMemory 增加 fallback。
  - 护栏：Sprint 4 明确禁止新增第二套 owner 或静默补丁入口。
- 风险 4：
  - 把 20s 扩展实现成新的 case-by-case prompt hardcode。
  - 护栏：Sprint 5 必须证明扩展来自 contract 投影，而不是新模板分叉。
- 风险 5：
  - generator/evaluator battle 演变为隐式审批系统。
  - 护栏：battle 只出诊断与 gate verdict，不拥有执行主线与生命周期状态。
- 风险 6：
  - 将 runtime reconcile、runtime projection、API discovery 继续混写，形成 mutate-on-read 或第二条控制面主线。
  - 护栏：所有 reconcile 只能来自单一显式 control-plane mutation 入口；read path 和 API lifespan 默认只读。
