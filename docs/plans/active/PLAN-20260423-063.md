## 1. Current Problem
- Symptom:
  - The MAS runtime/read-model/queue boundaries have mostly converged, but the architecture audit still found residual boundary violations:
    - plan governance drift: `PLAN-20260402-048.md` says `in_progress`, while `PLAN_INDEX.json` says `draft`; `plan_ops doctor --json` reports broader index/doc/path drift.
    - LLM-facing orchestrator planning context still includes execution-only `workflow_state_id`.
    - `video_generation_tool_v2` still contains a hardcoded provider fallback to `zhipu_client`.
    - several agents/orchestrator paths still directly read or write Shared WM instead of going through a clearly owned context/writeback boundary.
    - some tool-internal subtool calls and fallback paths are soft or hidden from the ReAct/FC audit trail.
- Impact:
  - Project status cannot be treated as a reliable governance SoT until index/doc drift is triaged.
  - Runtime identity can leak into semantic planning, weakening the execution-envelope boundary established by the 048 repair.
  - Provider-specific fallback logic weakens supplier-agnostic behavior and can mask missing capability wiring.
  - Direct memory access and silent fallbacks make root-cause diagnosis harder and can regress the ContextAssembler/MemoryWriter separation.
- Reproduction / evidence anchors:
  - `python3 /home/zhaojj/.codex/skills/sdd-plan-maintainer/scripts/plan_ops.py doctor --root . --json` currently reports governance drift.
  - `docs/plans/active/PLAN-20260402-048.md` still has an unchecked focused runtime replay item.
  - `backend/app/agents/orchestrator.py` includes `workflow_state_id` in LLM-rendered `runtime_constraints_json`.
  - `backend/app/agents/tools/ai_services/video_generation_tool_v2.py` hardcodes `registry.get_tool("zhipu_client")` as a fallback.

## 2. Deviation Analysis
- Expected behavior:
  - `docs/plans/PLAN_INDEX.json` is the lifecycle SoT; plan headers mirror it, and meaningful drift is either fixed or explicitly classified.
  - Runtime/session/node/attempt/gate/decision state stays in the MAS control plane and execution envelope, not in LLM semantic planning text.
  - Supplier selection is config/service-interface driven. Missing provider capability fails fast with actionable diagnostics.
  - Agents consume context through ContextAssembler-style boundary objects and write durable facts through owned write paths, with direct Shared WM access either removed or explicitly classified as control-plane-owned shared fact publication.
  - Tool-internal fallback is explicit, diagnosable, and contract-bound; hidden subtool I/O is either promoted to explicit FC or documented as deterministic post-processing with telemetry.
- Actual behavior:
  - The high-risk runtime/read-model surfaces are largely aligned, but the remaining deviations are spread across plan governance, orchestrator planning prompts, provider fallback, memory write paths, and tool internals.
- Root cause hypothesis:
  - Previous fixes were intentionally bounded around urgent runtime failures, leaving older compatibility paths and governance debt in place.
  - Shared WM evolved into both MAS fact exchange and ad hoc agent persistence, so ownership is not uniformly expressed at module boundaries.
  - Some operational fallbacks were added to keep generation running before the newer fail-fast/error-transparency rules were frozen.

## 3. Implementation Approach
- Patch strategy:
  - Treat this plan as a governance cleanup umbrella with strict stage gates. Do not start broad rewrites before the current SoT and acceptance gates are stable.
  - Stage A fixes or classifies plan lifecycle drift, especially 048, and records which historical doctor issues are in/out of this plan.
  - Stage B removes LLM-facing runtime identity from orchestrator planning prompts while preserving execution-envelope access for agent/tool invocation.
  - Stage C replaces hardcoded provider fallback with configured service/provider resolution or fail-fast diagnostics.
  - Stage D inventories direct Shared WM reads/writes in live agents and splits them into approved MAS fact publication, ContextAssembler input, MemoryWriter output, or follow-on migration.
  - Stage E makes hidden subtool I/O and soft fallbacks auditable: either explicit FC/tool-chain calls, or tool-internal post-processing with structured diagnostics and tests.
  - Stage F runs focused runtime validation, including the pending 048 `video_generator -> video_composer` path.
- Compatibility considerations:
  - Avoid compatibility shims for known-wrong behavior. If a legacy path conflicts with the frozen boundary rules, replace it or split a breaking-change follow-on with explicit risk.
  - Do not collapse all historical `plan_ops doctor` issues into this plan unless they block current lifecycle trust. Historical archive/path cleanup can become a separate plan.
  - Preserve the current working quick-mode runtime/read-model behavior unless a boundary violation is directly proven.
- Rollback plan:
  - Each stage must remain independently revertible.
  - If a provider or memory-boundary change breaks a live flow, revert only that bounded stage and keep the governance/index corrections.
  - Do not revert unrelated user changes or pre-existing documentation edits.
