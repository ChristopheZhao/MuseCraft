## 1. 当前问题
- 症状:
- 音频编排语义分散在 orchestrator、门控信号、工具层逻辑三处。
- `strategy/mode/fallback` 与 `run_audio_agent/video_generate_audio` 布尔字段在承担路由职责，替代了计划模型。
- 影响:
- orchestrator 被弱化为变量路由器，新增 agent 时容易出现策略漂移和耦合膨胀。
- 模式名称与运行行为可能不一致，评审与排障成本上升。
- 复现路径:
- 检查 `backend/app/agents/orchestrator.py` 的音频路由和激活池构建。
- 检查 `backend/app/agents/tools/ai_services/video_generation_tool_v2.py` 的策略归一化与参数决策。
- 检查 `backend/app/agents/tools/video_composition/composition_tool.py` 的跨场景音频门控行为。

## 2. 偏差分析
- 期望行为:
- orchestrator 基于任务契约与能力事实单点决定 agent 参与和动作序列。
- 门控只输出结构化达标判定，不直接控制 agent 参与。
- 工具只执行显式参数并校验本地契约。
- 实际行为:
- 预定义策略别名和 mode 标签仍在影响主编排语义。
- 全局布尔路由将编排意图泄漏到非 orchestrator 层。
- 根因假设:
- 决策模型被表达为“模式 + 标志位”组合，而不是显式 `workflow.plan`。
- 兼容旧策略的别名逻辑保留过久，掩盖了边界归属问题。

## 3. 修复思路
- 修复策略:
- 引入契约驱动的 `workflow.plan`，并将其作为唯一参与决策来源。
- 从主决策链路移除 `strategy/mode/fallback` 语义，仅保留诊断 `reason` 字段。
- 门控统一为观测输出（`pass/fail/confidence/reason/metrics`），由 orchestrator 消费。
- 工具层只消费显式 action 参数与 provider 能力约束，不读取全局编排策略。
- 术语约束:
- 当前状态定位为 `Plan-Then-Execute`（一次规划为主，重规划能力不足）。
- 目标状态为 `Plan-and-Execute with adaptive replanning`（每次交付后可评估并重规划）。
- 兼容性考虑:
- 迁移窗口内可保留对旧路由字段的只读兼容，但不再作为决策源。
- 无显式 action 参数时，沿用 provider 默认能力行为，避免突变。
- 回滚方案:
- 若出现关键回归，临时恢复 route 驱动分支并通过单一开关关闭 plan-first 路径。

## 4. 阶段化执行方案
- [x] S1 (P0-A) 契约先行
- 交付物: `workflow.contract.audio` 写入与默认构建逻辑落地。
- 验收: contract 可从工作流输入读取并持久化，不依赖工具层策略。
- [x] S2 (P0-B) 计划化编排
- 交付物: `workflow.plan`（`active_pool/standby_pool/actions/decision_trace`）落地。
- 验收: `AUDIO_GENERATOR` 参与由 `workflow.plan` 优先决定，`video_generator` 可从 plan action 注入 `generate_audio`。
- [x] S3 (P0-C) 去模式主导
- 交付物: 参与决策链路不再依赖 `strategy/mode/fallback` 作为主语义。
- 验收: 编排分支基于 contract + capability + delivery evaluation，不新增全局布尔控制字段。
- [x] S4 (P0-D) 工具边界收敛
- 交付物: 工具层移除跨编排策略读取，仅执行显式参数和本地契约校验。
- 验收: `video_generation_tool_v2`/`composition_tool` 不再承担跨 agent 决策。
- [x] S5 (P1-E/P1-F) 自适应重规划与收口
- 交付物: orchestrator 在每次子 agent 交付后执行 `evaluate -> decide -> apply`。
- 验收: 支持 `continue/retry/activate_from_standby/abort`，并有 `replan_trace`、预算与去重控制。

## 5. 测试与验收
- 测试用例:
- Orchestrator 计划构建与 agent 参与测试（provider 支持/不支持原生音频）。
- Plan 优先路径测试（`workflow.plan.active_pool` 与 `workflow.plan.actions`）。
- 工具边界测试（执行路径不再读取全局编排策略）。
- 契约评估与 replan 触发测试（pass/fail 与动作映射）。
- 回归检查:
- 既有音频编排单测保持通过。
- 新增 agent 时不新增 `run_xxx/use_xxx` 全局布尔路由字段。
- 用户侧验收点:
- `native-only` 目标下不因工具层门控被隐式改路由。
- `hybrid` 目标下由 orchestrator 显式重编排实现恢复，而非工具层策略补丁。
- 术语一致性:
- 架构文档统一使用 `Plan-and-Execute` 与 `adaptive replanning`，不使用无出处术语。
