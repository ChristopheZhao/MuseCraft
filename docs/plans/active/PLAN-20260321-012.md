## 4. Implementation Approach
- Boundary-first strategy:
- 先定义职责拆分表，再做最小重构切片；禁止继续把 runtime/read-model 责任回填到 `task_queue.py`。
- Validated state-boundary summary:
- 队列状态、请求/调度状态、MAS runtime 状态、前端 read model 状态必须解耦，不能互相替代。
- `queue infra` 只处理 transport/scheduler 语义：enqueue、dequeue、ack、retry、revoke、worker health、queue depth。
- `Task` 只处理请求锚点、审计和粗粒度调度状态；不承载 node/gate/attempt 的运行时 SoT。
- `runtime/control-plane` 才持有 `workflow_session / node / attempt / gate / decision` 的状态主写权。
- 前端 current-run 必须消费 runtime/read model；`nodes` 只是阶段明细，不能反向推断队列是否卡住。
- External validation:
- 外部一手资料支持上述拆分：
- RabbitMQ queue 文档聚焦消息与消费者缓冲语义，而非业务工作流节点状态。
- Celery 官方 state 文档聚焦 task execution/result state，而非业务 runtime state machine。
- Temporal 的 task queue 与 workflow execution history 明确分离：队列负责路由，execution history 才是执行真值来源。
- CQRS 官方模式强调 UI 应消费 query/read model，而不是把异步命令/队列侧状态当展示真值。
- Intended ownership split:
- `task_queue.py` 保留：入队、revoke/cancel、消费入口、统一 execution eligibility policy 的接线。
- runtime/control-plane 承接：session/get-or-create、gate state、runtime write model、failure/completion 主写。
- execution host/bootstrap 承接：tool registry init、event bus reset、async loop/run harness、dispatch host。
- frontend runtime-first 收口：processing/current-run 主看 runtime session/read model，节点列表降级为阶段明细而不是顶层真值。
- Compatibility considerations:
- `011` 已被 superseded，仅保留为历史参考；本 plan 不以它作为当前任务判断依据，也不尝试覆盖或改写它。
- 本轮优先做最小可验证切片，不一次性推倒 Celery 或 orchestrator 主线。
- Rollback plan:
- 每个切片保持接口兼容与 focused regression；若 runtime/control-plane 抽离导致消费链失稳，可回退单个切片，不回退 `011` 已完成的战术修复。

## 4.1 Adherence Principles
- MAS / Queue decoupling principle:
- 后端队列层不得拥有 MAS 内部运行时语义；MAS runtime/control-plane 也不得向队列层暴露 node/gate/attempt 级内部协议作为调度前提。
- Queue responsibility boundary:
- `queue infra` / `queue adapter` 只负责 transport/scheduler 语义：enqueue、dequeue、ack、retry、revoke、worker health、queue depth、最小执行命令转发。
- Queue layer forbidden scope:
- 不允许在队列层直接持有或推进 `workflow_session / workflow_node_state / workflow_gate / workflow_gate_decision` 的业务状态。
- 不允许在 `task_queue.py` 中继续承载 `dispatch_generation()`、tool registry init、event bus reset、event loop lifecycle、runtime payload reconstruction、gate/apply/failure control logic。
- Runtime/control-plane ownership principle:
- `workflow_session / node / attempt / gate / decision` 的状态主写权唯一属于 runtime/control-plane；任何推进这些 SoT 的实现都必须显式归入控制层，而不是 queue consumer。
- API / command-plane principle:
- API 层负责创建任务、建立/查询 runtime view、提交 gate decision、触发 cancel/retry/replan 等显式命令；队列层不负责解释这些业务命令的语义。
- Resume semantics separation:
- `继续查看当前任务` 只是前端重新绑定已有 run 与 runtime view，不得被实现成新建任务或隐式恢复队列消费。
- `gate decision -> resume execution` 是 control-plane 命令：先写入 gate decision / runtime state，再触发一次新的执行调度。
- 历史 broker/backlog 消息被 worker 再次消费属于 queue regression，不属于合法的“resume”机制，也不能作为恢复语义被前端或后端依赖。
- Frontend runtime-first principle:
- 前端 current-run 只与 runtime/read model 交互，不以 queue 状态推断 MAS 节点状态。
- `Task` 只作为请求锚点与审计句柄，`nodes` 只作为阶段明细，不能反向充当 queue 健康证据。
- Supplier/container swap test:
- 任何设计若在把 Celery 替换为其他队列容器时会迫使 MAS runtime/control-plane 语义一起改变，则说明当前实现仍然过耦合，应回退并重新划边界。
- Review rejection rules:
- 代码评审中，凡出现“为了方便”把 runtime state writeback、gate 恢复、node 语义、agent bootstrap、UI 真值判断继续塞入 queue layer 的改动，默认拒绝。
- 凡让前端直接以 broker/worker/queue 语义判断 current run 的改动，默认拒绝。
