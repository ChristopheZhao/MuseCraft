## 1. Current Problem
- Symptom:
  - `PLAN-20260401-045` closed the repeated quick-runtime `execution lease expired` failure by fixing stale control-plane validation reads.
  - that RCA also exposed broader architectural debt:
    - orchestrator still owns too much runtime-control-plane routing and lifecycle glue
    - lease/keepalive semantics remain more visible in execution paths than ideal
    - test environments can confuse SQLite transport limits with MySQL runtime-authority semantics if the boundary is not made explicit
- Impact:
  - future contributors may misdiagnose runtime failures as media-agent bugs, queue bugs, or provider bugs instead of control-plane boundary bugs
  - orchestrator can continue to accumulate runtime authority concerns if the owner split is not explicitly slimmed
  - test harnesses may keep producing misleading results if SQLite is treated as behaviorally equivalent to the production MySQL runtime store
- Evidence baseline:
  - `PLAN-20260401-045` proved the closed production fault was a stale repeatable-read control-plane validation bug rather than failed lease renewal
  - the live replay on `task_db_id=1043` / runtime session `40` completed successfully after switching completion/failure/gate transitions onto fresh short-lived control-plane sessions
  - `PLAN-20260402-047` remains an unfinished image-composer diagnostics plan and does not cover runtime/control-plane governance

## 2. Deviation Analysis
- Expected behavior:
  - runtime/control-plane authority is clearly separated from business facts, queue transport, and frontend/read-model projection
  - orchestrator acts as coordinator/bridge, not as an ever-growing owner of runtime semantics
  - tools consume generic execution cancellation/liveness signals only where needed, without directly learning lease semantics
  - tests distinguish production-authoritative MySQL/runtime behavior from constrained SQLite unit-harness behavior
- Actual behavior:
  - the lease owner split is now understood, but the architecture is still documented mostly through RCA notes rather than a dedicated follow-up governance slice
  - orchestrator remains the visible surface for runtime transition failures and still directly routes multiple control-plane transitions
  - some lease lifecycle visibility remains deeper in the execution path than ideal
  - SQLite-based tests still need explicit care to avoid mixing transport/storage limits with production control-plane semantics
- Governance conclusion:
  - `045` fixed the defect, but did not complete the larger architecture cleanup
  - that follow-up should be tracked separately from both `045` closure and `047` image diagnostics work

## 3. Implementation Approach
- Patch strategy:
  - document the runtime state ownership matrix explicitly:
    - MAS/business facts SoT
    - runtime/control-plane SoT
    - queue/worker transport diagnostics
    - frontend/read-model projections
  - define the intended steady-state orchestrator role after `045`:
    - coordinator / bridge
    - not lease-truth owner
    - not queue-authority owner
    - not business-fact owner
  - inventory current orchestrator responsibilities and classify each one as:
    - keep in orchestrator
    - move behind a dedicated control-plane facade/service
    - reduce to diagnostics-only responsibility
  - review lease-surface leakage into execution/tool paths and decide the minimal acceptable interface for long-running execution cancellation/liveness
  - define test-harness guidance:
    - MySQL-backed runtime authority checks for production-equivalent transaction semantics
    - SQLite-backed unit tests only for bounded semantics that do not depend on MySQL isolation or concurrent writer behavior
- Compatibility considerations:
  - do not reopen `PLAN-20260401-045` root-cause repair unless a new runtime fault is reproduced
  - do not absorb `PLAN-20260402-047`; image composer missing-owner diagnostics remain a separate active stream
  - do not weaken `RuntimeSessionService` authority or move runtime truth into queue/broker state
  - do not treat artifact existence or business-fact projections as substitutes for runtime completion authority
- Rollback plan:
  - if a proposed slimming step changes runtime authority semantics, stop and split that step into a dedicated design amendment before implementation
  - do not use compatibility shims that preserve known-bad authority boundaries just to keep old call sites alive

## 4. Boundary Statement
- This plan is a governance/refactor successor to `PLAN-20260401-045`, not a reopening of the lease-expiry bug fix itself.
- This plan must not consume `PLAN-20260402-047`; `047` remains in_progress for narrow image-composer diagnostics.
- Queue/worker/broker remain transport only.
- `RuntimeSessionService` remains the sole owner of runtime/session/node/attempt/gate/decision truth.
- MAS working memory and published deliverables remain business-fact surfaces, not runtime-control-plane authority.
- Any orchestrator slimming done under this plan must preserve current runtime behavior before attempting larger restructuring.
