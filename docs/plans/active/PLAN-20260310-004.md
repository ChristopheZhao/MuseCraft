## 1. Current Problem
- 当前 [PLAN-20260308-003.md](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/docs/plans/active/PLAN-20260308-003.md) 已完成大量边界清理，但 residual ownership 仍集中在 orchestrator 主循环及其相邻控制层：
- [orchestrator.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/agents/orchestrator.py#L464) 仍直接串起 `report -> gate -> decision -> apply`
- [orchestration_runtime_controller.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/services/orchestration_runtime_controller.py#L149) 仍会插队、补 fallback `task_spec`、修改 activation state
- [orchestration_protocol.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/services/orchestration_protocol.py#L56) 仍在缺少显式 `orchestration_report` 时静默构造默认 report
- 当前 focused tests 缺少 orchestrator 主循环真实链路的端到端覆盖，无法证明当前实现已达到 `more llm decision, less hard-coding`

## 2. Target
- 将 orchestrator 收拢到 `llm-decision-based orchestration`
- gate 由 control-plane/runtime 在节点边界触发，orchestrator 不再主动拉 gate
- runtime controller 只 apply decision，不再补造 fallback spec 或持有 queue 策略
- protocol 成为真实主从边界：缺少显式 report 时 fail fast，而不是静默默认化
- 以真实主循环链路测试作为收口门槛

## 4. Implementation Approach
- 以 architecture-driven refactor 为主，不做 file-level 拆分式修补
- 先稳定当前中间态，再让 protocol 强边界在系统级生效，然后收 controller pure-apply，最后完成主循环 handoff 与真实链路验证
- 允许保留串行 `execution_queue` 作为过渡骨架，但不允许继续在主循环内轮询 gate 或直接组织 replan
- `base.py` 中的最小 `orchestration_report` 自动注入只作为 contract bootstrap；不能替代显式 `boundary_event / gate_triggers / artifacts`
- `orchestration_control_plane.py` 在 controller 纯 apply 化之前仅视为 draft 宿主，不计入 ownership 已迁移证据
- 任何质量审核未通过时，动作必须是继续 root-cause analysis、更新本计划并补充修复子任务，不允许以“局部 seam cleanup 已完成”或“已有草稿模块”判定收口

## 6. Goal Coverage Audit
- 目的：
- 防止计划只覆盖局部 seam cleanup，而未覆盖目标架构收口
- 审核项：
- `llm-decision-based orchestration` 是否有明确宿主迁移路径，而不是只新增 helper / draft 模块
- `boundary-triggered gate` 是否落实为 control-plane/runtime 边界触发，而不是仍由主循环主动拉 gate
- `runtime controller pure apply` 是否有明确删除 fallback spec / queue / activation 推导的任务
- `protocol fail-fast` 是否落实为系统级行为，而不是仅文件内抛错
- `main-loop true-chain verification` 是否有 focused tests 与关闭门槛
- `de-rule-centering` 是否已区分主链路 blocker 与 residual closeout，避免以 residual 未清误判整个主链路未成立
- `gate/orchestration decoupling` 是否明确到“主链路已解耦”与“类职责仍待纯化”的差异
- `planning contract hardening` 是否覆盖 expected agent coverage，而不是仅删除 fallback helper
- `gate contract hardening` 是否覆盖 unknown gate trigger 的显式失败语义
- `no-gate semantics hardening` 是否已从隐式 `None/else` 语义收成显式 envelope
- `decision action hardening` 是否已去掉 missing/unknown `action` 的默认 `continue`
- `gate authority hardening` 是否已限制 known gate 只能由合法 agent/boundary 组合触发
- `report field validation hardening` 是否覆盖 malformed report fields，而不是仅缺 report 才失败
- 失败处理：
- 若任一目标没有对应执行项或完成标志，必须先更新本计划再继续实现，不允许带着目标缺口推进

- 若仍存在双宿主或 helper 平移，必须继续 root-cause analysis，补明确迁移任务
- QG-2 强边界审核：
- 检查 protocol violation 是否真正 fail fast，主循环是否仍吞 boundary error
- 若未通过，必须继续修复错误传播链，不能以“protocol 文件已改”收口
- QG-3 Controller 纯 apply 审核：
- 检查 controller 是否仍补 fallback spec、仍推导 queue / activation / replan state
- 若未通过，必须继续把这些逻辑迁出或显式化为 apply payload
- QG-4 主循环审核：
- 检查主循环是否仍直接 `report -> gate -> decision -> apply`
- 检查 runtime decision failure 是否仍被静默降级为 `continue` 或 warning-and-continue
- 若未通过，必须继续收主循环 ownership，不能以 control-plane draft 模块存在作为完成证据
- QG-5 验证审核：
- 检查 focused tests 与代码核查是否同时通过
- 检查 workflow-level negative proof 是否补齐，而不只是 helper-level 负例
- 检查 expected agent coverage 缺 spec 时是否显式失败，而不是 omission=skip
- 检查 unknown gate trigger 是否显式失败，而不是 silent no-gate
- 检查 no-gate 是否来自 control-plane 显式 envelope，而不是 orchestrator 本地默认继续
- 检查 missing/unknown `action` 是否显式失败，而不是 default continue
- 检查 known gate authority 是否有显式校验
- 检查 malformed report fields 是否显式失败
- 若未通过，必须继续补测试或修正实现，不允许将计划转为 `completed` / `awaiting_user_confirmation`

## 8. Regression Checks
- 不再保留 orchestrator 主循环里的 `gate-inline-polling`
- 不再保留 controller 侧 fallback spec 生成
- 不再保留 protocol 缺失时的默认成功态
- 不再以局部 seam cleanup 作为架构收口证明
- 不再以新增 `control_plane` 草稿模块或 helper 拆分作为 ownership 已迁移的证据
