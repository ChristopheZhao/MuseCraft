- Symptom:
  - `PLAN-20260403-049` closed after freezing the runtime ownership model and landing one bounded `C1` slice for post-execution transitions only.
  - orchestrator still directly owns the next residual runtime-control-plane choreography seam:

- Impact:
  - the post-`045` target role for orchestrator remains only partially achieved because pre-execution runtime choreography is still local orchestration code instead of a dedicated bridge/facade boundary
  - future resume/bootstrap work can accidentally re-couple runtime authority, keepalive handling, or queue-transport details if this seam is not isolated deliberately
  - widening `OrchestrationRuntimeTransitionFacade` without a fresh boundary contract would risk creating a new mixed-responsibility orchestration god-object
- Evidence baseline:
  - `PLAN-20260403-049-stage-b-responsibility-inventory.md` marked runtime resume/load and attempt bootstrap as `move-behind-facade`, while keepalive activation/deactivation remained a separate residual surface

    - `_start_runtime_attempt(...)`

## 2. Deviation Analysis
- Expected behavior:
  - orchestrator acts as coordinator/bridge only and delegates runtime resume/bootstrap choreography through a narrow orchestration-facing control-plane facade/service
  - `RuntimeSessionService` remains the sole authority for runtime/session/node/attempt/gate/decision truth
  - post-execution transitions stay in the existing `C1` facade, while pre-execution resume/bootstrap logic is isolated behind its own clearly bounded surface
- Actual behavior:
  - orchestrator still loads authoritative resume state and performs attempt bootstrap sequencing directly
  - adjacent keepalive activation/deactivation calls remain nearby in the file, increasing the risk of accidental scope expansion if the next slice is not frozen tightly
  - the existing `OrchestrationRuntimeTransitionFacade` already carries post-execution choreography plus script-review boundary publishing, so it should not automatically absorb another residual seam
- Boundary conclusion:
  - `C2` is a follow-on bounded slice, not a continuation of `049`
  - `C2` should narrow only the resume/load/bootstrap seam unless an explicit review amendment proves a slightly wider surface is still single-purpose
  - keepalive/liveness redesign is not part of this plan's default scope

## 3. Implementation Approach
- Patch strategy:
  - freeze one bounded extraction seam around:
    - runtime checkpoint/script-state loading for resume decisions
    - authoritative resume task-spec loading
    - attempt bootstrap/start coordination
  - move that seam behind one dedicated orchestration-facing resume/bootstrap facade/service instead of leaving it inline in orchestrator
  - keep post-execution runtime transitions in `OrchestrationRuntimeTransitionFacade`; Stage A review rejected widening that class for `050`
- Target files:
  - `backend/app/agents/orchestrator.py`
  - `backend/app/services/orchestration_runtime_resume_bootstrap_facade.py`
  - `backend/tests/unit/test_orchestrator_runtime_mainline.py`
- Non-targets:
  - `backend/app/services/runtime_session_service.py`
  - `backend/app/services/execution_host_lease.py`
  - `backend/app/services/queued_task_execution_host.py`
  - `backend/app/services/orchestration_runtime_transition_facade.py`
  - queue/worker/broker/websocket/artifact truth surfaces
  - product-path keepalive activation/deactivation redesign
  - `PLAN-20260401-045` reopen or any `PLAN-20260402-047` scope
- Compatibility considerations:
  - preserve the `045` correctness fix and do not reintroduce stale control-plane validation reads
  - preserve the `049` owner matrix: runtime truth stays in `RuntimeSessionService`, queue remains transport-only, and frontend/read-model projections remain projections only
  - keep SQL/session ownership explicit: pre-execution resume/bootstrap consumes the caller-owned current session, while post-execution fresh-session transitions remain isolated in the existing transition facade
  - keep MySQL-authoritative vs SQLite-bounded validation claims explicit; SQLite unit results do not prove MySQL transaction semantics
  - keep runtime control-plane persistence separate from MAS/agent memory: runtime facts may be exposed as stable contracts, but any memory association must flow through the memory application layer interfaces rather than promoting memory state into runtime authority
- Rollback plan:
  - if the extraction starts to absorb keepalive/liveness semantics, queue-host behavior, or broader resume semantics than the listed seam, stop and split the work into another follow-on plan instead of widening `C2`
  - do not add compatibility shims that preserve the known-bad "orchestrator owns too much runtime choreography" boundary just to keep old helper shapes alive

## 6A. Runtime / Memory / SQL Freeze
- Runtime persistence and MAS/agent memory are separate architecture layers even when both happen to use SQL-backed storage.
- Runtime/session/node/attempt/gate/decision truth remains control-plane data owned by `RuntimeSessionService`.
- MAS memory remains an application-layer context/memory facility accessed through `ContextContractAssembler`, `MemoryWriter`, `MemoryServices`, and memory interfaces.
- Runtime-to-memory association is allowed only through explicit stable contracts and application-layer interfaces.
- See `PLAN-20260403-050-runtime-memory-sql-boundary-freeze.md` for the frozen owner map, verification matrix, and SQL solution-design order.
