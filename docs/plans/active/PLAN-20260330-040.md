## 1. 当前问题
- 症状：
  - rerun 后 image agent 仅部分场景可生成，场景 1、2、3、5 持续收到 Doubao `InputTextSensitiveContentDetected`，多轮重试和 fallback prompt 仍失败。
  - 图片生成日志中的 `PROMPT_COMBINED(generate_image)` 仍然是“场景梗概 + 叙事提纲 + 角色小传 + 一致性大段 prose”的组合，而不是首帧静态图像描述。
  - `scene_info_ref` 已经提供了 `opening_state`、`event_trigger`、`action_phases` 等结构化字段，但 image prompt 没有以 `opening_state` 为根来构图。
  - 当前 image path 默认把任务理解成“场景首帧”，但仓库里 image generation 还承担角色参考图等用途；规划层没有给 image agent 明确的 image purpose / task direction。
- 影响：
  - image pipeline 在当前 provider 上会被内容安全直接拦截，导致 episode 的首帧资产缺失。
  - 即使生成成功，prompt 也会偏向 scene summary / montage card，而不是可直接生成静态 keyframe 的 still-image prompt。
  - 该问题是 `038` 交付后的 rerun follow-up 缺陷，但不应回填为 `038` 主线实施内容。
- 发现来源：
  - 2026-03-30 rerun 的 `backend/logs/mas/mas_workflow.log`
  - `backend/storage/temp/context/image_generator_e31e6a52-0c0e-4047-9cee-86c578c2ab92.json`

## 2. 根因分析
- 安全链路根因：
  - `apply_prompt_safety()` 当前是 advisory-only，明确“原样返回 prompt”，并不在 provider 调用前做实质改写。
  - 规则级 `sanitize_with_locks()` 只有极少量词表替换，覆盖不到当前 prompt 中的大量 provider 敏感触发词。
  - `image_generation_tool._generate_image()` 当前只在 `except ToolError` 分支里尝试 reactive rewrite，但 provider adapter 实际抛出的是原始 `RuntimeError`；结果是 provider-sensitive failure 在 boundary 正规化前就越过了 rewrite 分支，最后只在外层被统一包装成 `image_generation_failed`。
  - `ps_is_sensitive_error()` 目前依赖 message/code 的字符串 marker，而不是在 tool / adapter boundary 上显式正规化 provider diagnostics；对 `InputTextSensitiveContentDetected` 这类 provider code 的识别属于偶然字符串命中，不是稳定 contract。
  - ReAct 层给出的 fallback_prompt 也不是 contract-bound 的安全改写，而是临时简化语句，仍包含动作/冲突/剪辑性描述，且没有形成真正的 provider-specific safe retry 闭环。
- Prompt contract 根因：
  - image path 当前缺少显式的 `image purpose / task direction` 输入，导致“scene opening anchor”“character reference”“scene reference / key art”这几类任务在 prompt 组装阶段没有分流规则。
  - `_create_image_prompt_from_scene()` 仍以 `visual_description` + `narrative_description` 为主体，并拼入角色原型/物种/叙事角色等结构化设定。
  - `image_autoprompt.jinja2` 模板本身也是 scene-outline 模板：`场景描述 / 补充要素 / 角色设定 / 风格指导 / 情绪与氛围 / 道具与细节`。
  - `image_prompt_composer_tool` 在基础 prompt 之后又注入整段一致性说明，其中 `opening_anchor` 还继续带入 `氛围` 和 `镜头起点`，进一步把静态图像 prompt 拉成了 scene card。
  - 对于 scene 5 这类 montage / mixed scene，当前 image prompt 仍试图描述“快速切换”“标题闪现”这类只有视频才成立的语义，而不是单张 poster/keyframe。

## 3. 目标边界
- 本计划只处理 image-agent 两类根因修复：
  - provider-sensitive error 的安全闭环
  - image-purpose-aware still/reference prompt contract 收敛
- 本计划关闭以下问题：
  - image provider 明确返回 `InputTextSensitiveContentDetected` 时，本地 safety/rewrite 没有真正接管
  - image prompt 仍然缺少明确的任务方向，导致 scene still、角色参考图、montage key art 等不同目标没有被 contract 化区分
  - scene still prompt 仍然是 scene outline / montage prose，而不是静态构图
  - consistency 注入在 image path 中过重，侵入 still prompt 主体
- 本计划不做：
  - video prompt / video provider 质量修复
  - scene/script 上游 contract 重写
  - 泛化 AIGC safety 平台重构
  - runtime / queue / orchestrator 治理

## 4. 实施方案
- Patch strategy A: 关闭 sensitive-error remediation 闭环
  - 在 tool / prompt-safety boundary 先把 provider 原始异常正规化成稳定 diagnostics，再识别 `InputTextSensitiveContentDetected` 这类 provider 显式敏感错误；不得依赖 call site 临时判断或偶然字符串命中。
  - 保持 supplier-specific 触发条件配置化或集中在 prompt-safety/rewrite 边界，不在 agent/ReAct 层散落关键词判断。
  - rewrite/retry 必须真正使用被改写后的 prompt，并记录清晰 telemetry；若 rewrite 仍失败，向上层 surfaced provider code、rewrite_attempt、safety diagnostics，而不是无效重试。
  - fallback prompt 只能作为 contract-governed safe prompt surface，不能继续让 agent 即兴生成未受控的动作型字符串。
- Patch strategy B: 收敛 still-image first-frame prompt contract
  - 在现有 image path 上增加显式 `image_purpose / task_direction` 语义，用于区分至少两类当前已知目标：`scene_opening_anchor` 与 `character_reference`；允许后续受控扩展到 `scene_reference` / `key_art`，但不在本计划内泛化实现更多种类。
  - 当 `image_purpose=scene_opening_anchor` 时，prompt root 优先消费 `opening_state` / `opening_anchor` / `scene opening composition`，只在缺失时回退到 `visual_description`。
  - 当 `image_purpose=character_reference` 时，prompt root 应转向角色稳定特征、服饰、道具、正面/全身/头像等参考图约束，而不是 scene narrative。
  - scene still prompt 主体只保留单帧静态构图必要信息：主体、环境、姿态/状态、光线、色调、风格、必要道具。
  - `narrative_description`、`creative_intent`、`camera motion`、`action_phases`、`motion_beats` 不再作为 still-image prompt 主体；最多只通过压缩后的“画面焦点”转译为静态描述。
  - consistency 注入压缩成短锁定，并按 `image_purpose` 选择不同锁定面：scene still 侧重 opening anchor，reference 图侧重角色稳定特征与标志道具；不再拼接长段角色小传、节奏 prose、镜头起点。
  - montage / mixed scenes 必须先在 image contract 层投影成单张 still 的 dominant composition，而不是把“快速切换/标题闪现”直接送给图像模型。
- Compatibility considerations：
  - 保持 `scene_info_payload.scenes_to_generate[]` 为 authoritative carrier，不新增 image-only parallel schema。
  - 与 `038` 交付的 scene contract v2 保持一致：image path 是消费 `opening_state`，不是重写上游 scene 语义。
  - 与现有 project character reference 资产面保持兼容：项目级角色参考图仍可复用 `image_generation` tool，但 `040` 只要求在 image tool / prompt contract 层证明 `character_reference` 方向成立；除非先 amendment，不扩写到 project planning / orchestrator 主线。
  - 允许保留 provider-specific safety diagnostics，但不允许把供应商差异扩散到 agent planning 层。
  - `image_purpose / task_direction` 只是 image agent 的显式上下文输入，不得演化成固定重试次数、固定工具顺序、或 agent-level 决策表；image agent 仍应保留 ReAct 迭代自主性。
- Rollback plan：
  - 若 still-image contract 收敛导致现有成功 case 回归，仅回滚本计划触及的 image prompt / safety boundary 文件，不回滚 `038` 的 scene/video contract 主线。

## 6. 测试与验收
- 必要自动化：
  - synthetic sensitive-error regression：provider raw exception / normalized ToolError 中携带 `InputTextSensitiveContentDetected` 时，都能在 boundary 触发 rewrite path
  - rewrite telemetry regression：触发后能看到 rewrite attempt，不是原 prompt 重放
  - scene 1 prompt contract regression：当 `image_purpose=scene_opening_anchor` 时，根提示词以 `opening_state` / 静态构图为主体，而不是 `narrative_description`
  - character reference regression：当 `image_purpose=character_reference` 时，tool-level prompt contract 主体不会再混入 scene narrative / montage prose，且无需引入新的 project/image mainline
  - scene 5 prompt contract regression：不会再输出“快速切换/标题闪现/混剪”这类视频剪辑描述作为 still-image prompt 主体
  - consistency compression regression：image composer 注入的是短锁定信息，而不是长段角色/节奏 prose
- 手工验收：
  - 从日志可清楚区分“provider returned sensitive error”与“local rewrite attempted / not attempted”的责任边界
  - scene still 样例 prompt 在阅读上明确是“单张首帧图”，reference 样例 prompt 明确是“参考图方向”，两者不会混淆
  - 修复不引入第二套 image contract 或新的 memory/runtime owner
- 完成信号：
  - rerun 样例中的 sensitive-error 链、outline-like image prompt 链都有明确根因修复路径与回归证据
  - 新窗口 review 认为本计划仍是 narrow remediation，且未扩成泛化 provider/runtime/content 改造
  - `040` 的功能修复已完成，但若按架构规范做最终 closeout，则还必须消除 still/reference prompt contract 中重复的内容级硬编码；该收口工作已拆分为 `PLAN-20260330-041`，在其完成或被用户显式豁免前，本计划不得标记为通过验收
