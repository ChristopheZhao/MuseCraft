## 1. 当前问题
- [PLAN-20260325-024.md](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/docs/plans/active/PLAN-20260325-024.md) 已完成 bounded host-uniformity governance：它已确认 endpoint-local planning host 不是长期终局，但也确认这条线不能在原计划边界内直接迁移。
- 当前真正阻塞 project planning host 迁移的，不再是单点 `_schedule_project_plan(...)`，而是更大的 project-level substrate 问题：
  - `ProjectStateRepository` / project read-model 仍是 process-local，API 与前端直接依赖它
  - queue/worker dispatch 仍按 `GenerationMode` 分支，缺少 host-neutral `job_kind` / `handler_key` contract
  - `TaskType.SCRIPT_WRITING` 已被 project planning 之外的写作/记忆逻辑复用，不能充当新的 planning worker authority key
- `quick` 与 `project` 是并列的产品入口模式：用户可以直接进入 quick，也可以进入 project；`project` 不是 quick 之上的控制平面。
- `project-level` 内部确实包含对 single-episode MAS 的复用：每个 episode 的生成仍走 `OrchestratorAgent` 主线，但这种复用必须通过预定义的 project workflow / product job 步骤完成，而不是让 `project-level` 再长成一层新的 MAS。
- 本计划的目标不是把 `project-level` 做成 “MAS 之上的 MAS”，而是把它收敛成 multi-episode product wrapper 所需的显式 authority facts、shared read-model projection 与 host-neutral project job substrate，让 project planning host 迁移具备前置条件。

  - `project-level` 是 multi-episode wrapper / product coordination surface，不是新的 MAS 层，也不是 “MAS 之上的 MAS”。

  - 是否复用 `Agent` 类只是实现细节，不是架构分层结论；不得因此把 project-level 误塑造成第二套 agent runtime。

## 4. 执行计划
### Phase A: project authority-facts / read-model split decision
- 目标：
  - 先定义 project-level authority facts，再定义 shared read-model projection，避免把 `ProjectStateRepository` 直接放大成新的 project control surface。
- 执行清单：
  - [x] 盘点 `ProjectStateRepository` 的现有写入点、读取点、前端依赖与 process boundary
  - [x] 冻结 placeholder project creation、planning progress、character reference progress、story-plan foundation、episode aggregate / project summary 的 authority facts surface
  - [x] 决定 project read-model 的 shared persistence / projection 方案，并明确哪些字段只是 projection、哪些字段是 authority facts 的稳定映射

### Phase A Findings (2026-03-27)
- 当前 `ProjectState` / `ProjectStateRepository` 是一个混合的 project-level authority + read-model host-local 对象，而不是纯 projection：
  - planning path 会写 `progress.planning` / `progress.character_references`
  - `SeriesPlannerAgent` 会整体重建并保存 `ProjectState.story_plan`、`style_profile`、`character_bible`
  - `project_service.update_episode_script(...)` 会更新 episode editorial status、approved script、runtime stale/idle state
  - `EpisodeOrchestratorAgent` 会读取同一对象做 selection、approval/runtime gating，并回写 `episodes_runtime`、`completed_episodes`、`total_cost` / `total_tokens`
  - `GET /projects/{id}` 与前端 polling / project workspace 则直接把这组数据当成 authority read-model 消费
- 因此，`027` 的 shared-visibility 目标不能只覆盖 planning host 迁移；它必须覆盖目前被 project planning、project orchestration 和 `/projects/{id}` 共用的同一组 project facts。
- Phase A 冻结的 project authority facts surface 分为四组：
  - project foundation facts：`project_id`、`mode`、`story_plan`、`global_settings`、`style_profile`、`character_bible`
  - project operation facts：`progress.planning`、`progress.character_references` 的 `status/task_id/error`
  - episode aggregate facts：`episodes_runtime.*`、approved script、workflow task id、output assets、runtime error、per-episode aggregated cost/tokens
  - project summary aggregate facts：`completed_episodes`、`total_cost`、`total_tokens`
- Phase A 冻结的当前用户可见 read-model contract 是：
  - `create_project(...)` 继续立即返回 placeholder project，让前端能立刻进入 project workspace
  - `/projects/{id}` 继续稳定暴露 planning/character-reference progress、story-plan episodes、episode runtime aggregate、character bible、completed-episodes / cost summary 等 project workspace 所需字段
  - frontend 继续轮询这一 authority read-model，不新增 second-state logic
- Phase A 决策：
  - 目标形态应是“显式 project authority surface + shared read-model projection”，而不是简单把当前 in-memory `ProjectStateRepository` 复制到 worker 可见位置
  - 当前 in-memory `ProjectStateRepository` 只可被视为 host-local MVP implementation，不应被升级为新的 project-level runtime/control-plane
  - 进入 Phase B / Phase C 的前提是：任何 shared persistence / projection 方案都必须同时满足 planning host migration、`orchestrate_project(...)` gating、`EpisodeOrchestratorAgent` 读取、以及 `/projects/{id}` / frontend polling 的同源可见性；否则只能继续 defer，不能半迁移

### Phase B: project workflow / host-neutral job substrate decision
- 目标：
  - 把 `project-level` 收敛成预定义 workflow / product job，对外与 quick 并列，对内通过显式 job steps 复用 single-episode MAS，同时让 queue/worker 保持语义中立。
- 执行清单：
  - [x] 定义 project planning / project workflow job 的 `job_kind` / `handler_key` contract
  - [x] 决定复用现有 queue substrate 还是抽出同级 project-job adapter，但不把 scheduler 写成 project/MAS 语义 owner
  - [x] 明确 `TaskType`、`GenerationMode`、handler registry、project workflow step 各自只承担什么职责，并明确 quick/project 只是产品模式而非 scheduler authority 语义

### Phase B Findings (2026-03-27)
- 当前 queue/worker substrate 仍显式绑定 generation 主路径：
  - `TaskQueueService` / `sync_process_video_task(...)` 通过 `resolve_generation_mode(...)` 解析 `mode`
  - quick 路径会读取 `RuntimeSessionService`
  - `run_generation_in_host(...)` 直接按 `GenerationMode.QUICK/PROJECT` 分支到 `OrchestratorAgent` / `EpisodeOrchestratorAgent`
  - Celery 入口名仍是 `process_video_task`
- 因此，`027` 不做 repo 级 queue neutralization，也不在本计划内改写现有 quick/project episode generation 主路径。
- Phase B 决策：
  - 复用现有 Celery broker / worker container / DB-task enqueue 这一层 transport substrate，但不复用 generation-specific worker entrypoint 作为 project planning host。
  - project-level job 采用独立的 project workflow contract：
    - `job_kind`：标识 transport-neutral job family，例如 `project_workflow`
    - `handler_key`：标识具体 worker handler，例如 `project_planning`
  - project planning / 后续 project workflow job 应通过 sibling project-job adapter / host entry 承载，而不是继续塞进 `process_video_task -> sync_process_video_task -> run_generation_in_host(...)` 这条 generation-only host。
  - `GenerationMode` 继续只承担现有 quick/project generation 主路径中的产品模式选择，不扩展成 project planning dispatch authority。
  - `TaskType` 继续只承担 coarse task metadata / UI / analytics 分类；`TaskType.SCRIPT_WRITING` 明确不得作为 project planning worker dispatch key。
  - handler registry / workflow step 只表达 project-level product job 的执行入口与步骤，不表达 MAS runtime authority；single-episode MAS 仍只在需要生成 episode 时由 project workflow 显式调用。
- Phase B implementation gate:
  - `027` 后续 Phase C 只允许为 project planning host migration 增加 project-job adapter / handler contract
  - 不允许顺手重构现有 `TaskQueueService.queue_task(...)` quick/project generation 分流、`RuntimeSessionService` quick path、或 `run_generation_in_host(...)` 的 generation 主线路由
  - 如果后续需要把整个 queue substrate 从 `GenerationMode` 完全抽离，应拆成 `027` 之后的独立 successor，而不是并入本计划

### Phase C: minimal migration slice or narrower implementation successor

  - 在 authority/read-model split 与 neutral job substrate 已冻结后，给 planning host migration 留下最小可执行入口，同时不改写 single-episode MAS 主线。
- 执行清单：
  - [x] 迁移 `_schedule_project_plan(...)` 到 sibling project-job adapter / host entry，或拆出更窄 implementation successor
  - [x] 保持 `/projects` 立即返回 placeholder project 的产品行为
  - [x] 保持 project-level 只通过预定义 workflow 调用 single-episode MAS，而不是在 project 层新增第二套 agent runtime / control-plane
  - [x] 补齐 focused backend tests / docs / inventory sync

### Phase C Findings (2026-03-27)

  - 显式 project authority store / shared read-model projection
  - project planning、project orchestration、`/projects/{id}`、frontend polling 对同一组 project facts 的读写切换
  - sibling project-job adapter / host entry 与 planning host migration

## 6. 风险与护栏
- 风险 1：
  - 误把 project-level substrate 改造成 “MAS 之上的 MAS”。
  - 护栏：project-level 只做 multi-episode wrapper / product workflow coordination；它可以调用 single-episode MAS，但不得新增第二套 runtime/control-plane。
- 风险 2：
  - 为了迁移 planning host，把 queue/worker 写成 project 或 MAS 语义 owner。
  - 护栏：dispatch contract 只允许 host-neutral `job_kind` / `handler_key`，不允许 scheduler 猜业务 authority；quick/project 只能作为产品入口或 workflow 选择，不得成为 queue authority 语义。
- 风险 2b：
  - 为了“顺手统一”，把现有 generation queue 主路径一并拖进 `027`，导致计划边界膨胀成 repo 级 queue 重构。
  - 护栏：`027` 只新增 project-job adapter / sibling host entry，不改写 `process_video_task` / `sync_process_video_task` / `run_generation_in_host(...)` 的现有 generation 主线路由。
- 风险 3：
  - 迁移 shared visibility 时把 authority facts 与 projection 混成新的 project-level SoT，或破坏 `/projects` 的即时 placeholder/轮询行为。
  - 护栏：Phase A 先冻结 authority facts 与当前产品行为，再决定 persistence/projection 方案。
