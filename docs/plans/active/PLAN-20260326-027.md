## 1. 当前问题
- [PLAN-20260325-024.md](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/docs/plans/active/PLAN-20260325-024.md) 已完成 bounded host-uniformity governance：它已确认 endpoint-local planning host 不是长期终局，但也确认这条线不能在原计划边界内直接迁移。
- 当前真正阻塞 project planning host 迁移的，不再是单点 `_schedule_project_plan(...)`，而是更大的 project-level substrate 问题：
  - `ProjectStateRepository` / project read-model 仍是 process-local，API 与前端直接依赖它
  - queue/worker dispatch 仍按 `GenerationMode` 分支，缺少 host-neutral `job_kind` / `handler_key` contract
  - `TaskType.SCRIPT_WRITING` 已被 project planning 之外的写作/记忆逻辑复用，不能充当新的 planning worker authority key
- 本计划的目标不是把 `project-level` 做成 “MAS 之上的 MAS”，而是为 multi-episode product wrapper 提供显式 shared read-model 与 job substrate，让 project planning host 迁移具备前置条件。

- Hard rules:
  - `single-episode MAS mainline` 已回正；本计划不得 reopen `022/023/025/026`，也不得把 project-level substrate 倒灌回 `OrchestratorAgent` / runtime control plane。
  - `project-level` 是 multi-episode wrapper / product coordination surface，不是新的 MAS 层，也不是 “MAS 之上的 MAS”。
  - queue / worker / thread host 只能是 transport/execution container；不得拥有 project 或 MAS 业务 authority。
  - project read-model 必须是显式 shared projection；frontend 继续消费 authority read-model，不新增 second-state logic。
  - 是否复用 `Agent` 类只是实现细节，不是架构分层结论；不得因此把 project-level 误塑造成第二套 agent runtime。

## 4. 执行计划
### Phase A: project read-model shared visibility decision
- 目标：
  - 为 project-level read-model 明确 shared visibility 方案，避免 worker 与 API 各写各读。
- 执行清单：
  - [ ] 盘点 `ProjectStateRepository` 的现有写入点、读取点、前端依赖与 process boundary
  - [ ] 决定 project read-model 的 shared persistence / projection 方案
  - [ ] 冻结 placeholder project creation、planning progress、character reference progress 的 authority surface

### Phase B: host-neutral job substrate decision
- 目标：
  - 让 queue/worker 保持语义中立，只按 job handler 执行 project jobs。
- 执行清单：
  - [ ] 定义 project planning job 的 `job_kind` / `handler_key` contract
  - [ ] 决定复用现有 queue substrate 还是抽出同级 project-job adapter，但不把 scheduler 写成 project/MAS 语义 owner
  - [ ] 明确 `TaskType`、`GenerationMode`、handler registry 各自只承担什么职责

### Phase C: minimal migration slice or narrower implementation successor
- 目标：
  - 在 shared visibility 与 neutral job substrate 已冻结后，给 planning host migration 留下最小可执行入口。
- 执行清单：
  - [ ] 迁移 `_schedule_project_plan(...)` 到新的 substrate，或拆出更窄 implementation successor
  - [ ] 保持 `/projects` 立即返回 placeholder project 的产品行为
  - [ ] 补齐 focused backend tests / docs / inventory sync

### Phase D: governance closeout
- 目标：
  - 在实现或显式 defer 完成后，收口治理资产并确认剩余 retained item。
- 执行清单：
  - [ ] 同步 plan / validation / inventory / PLAN_INDEX
  - [ ] 记录 focused validation 与 residual risk
  - [ ] 等待用户确认后再完成 lifecycle closeout

## 6. 风险与护栏
- 风险 1：
  - 误把 project-level substrate 改造成 “MAS 之上的 MAS”。
  - 护栏：project-level 只做 multi-episode wrapper / product job coordination，不新增第二套 runtime/control-plane。
- 风险 2：
  - 为了迁移 planning host，把 queue/worker 写成 project 或 MAS 语义 owner。
  - 护栏：dispatch contract 只允许 host-neutral `job_kind` / `handler_key`，不允许 scheduler 猜业务 authority。
- 风险 3：
  - 迁移 shared visibility 时破坏 `/projects` 的即时 placeholder/轮询行为。
  - 护栏：Phase A 先冻结当前产品行为，再决定 persistence/projection 方案。
