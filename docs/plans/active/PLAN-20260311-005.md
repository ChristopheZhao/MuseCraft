## 1. Current Problem
- follow-up review 在 current MAS contract hardening 之外发现 3 个生产语义回归：
- [task_queue.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/services/task_queue.py#L116) 当前把排队任务统一送进 `EpisodeRuntimeKernel`，但 quick/runtime-kernel 路径尚不支持 voice/BGM 相关执行，存在把音频需求任务静默做成无声成片的风险。
- [runtime_kernel.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/services/runtime_kernel.py#L383) [runtime_kernel.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/services/runtime_kernel.py#L424) [runtime_kernel.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/services/runtime_kernel.py#L464) 当前对 `scene_video / compose / quality` 阶段返回值没有 success contract 校验，partial 结果仍会被记成 `COMPLETED` 并推进后续阶段。
- [tasks.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/api/v1/endpoints/tasks.py#L24) 当前把 API 进程内线程执行建立在 `try import` 上，而不是显式 debug 开关；在常规部署中会默认绕过 worker 隔离与重试。

## 2. Target
- 音频能力不完整的 quick/runtime-kernel 路径不得再吞掉需要 voice/BGM 的任务。
- runtime kernel 只在阶段显式成功时标记 node/session `COMPLETED`，partial/failure 必须显式失败并停止后续阶段。
- API 进程内 task runner 只允许在显式 debug 配置下启用，默认仍走 worker 队列。
- 以上修复完成后，当前 MAS 重构才能进入整体收口判断。

- mode dispatch / targeted unit tests
- Out of scope:

- 为 runtime kernel 新增完整 voice/BGM 节点编排
- 扩展 project mode 产品能力

## 4. Implementation Approach
- 先修 root cause，不加“异常输入继续往下跑”的兼容分支。
- unsupported audio-capable task 不能继续进入当前 kernel 路径；在 capability 未补齐前必须走已有标准编排入口，而不是 silent downgrade。
- stage success contract 在 kernel/runtime-session 边界统一收紧，不允许用“返回了 dict”替代“阶段成功”。
- API 本地线程 runner 必须由显式配置控制，不能由 import 成功与否隐式启用。
- 本计划优先做止血与语义正确性，不把“补齐完整 audio runtime pipeline”塞进当前热修复范围。
