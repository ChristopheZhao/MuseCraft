## 2. Canonical Boundary Freeze
- Top-level vocabulary:
- 当前 canonical 顶层架构只允许：编排层 / 控制层 / 门控层 / 治理层。
- Supporting capability rule:
- `runtime substrate` 属于控制层底座；`Context/Contract Assembler`、published deliverable projection、leaf agents 都不是新层。
- Single-mainline rule:
- `single-episode` 的唯一 MAS mainline 仍是 `OrchestratorAgent`；任何 project wrapper、runner、历史 orchestrator 变体都不得维持并列主线语义。
- Runtime SoT rule:
- `workflow_session / node / attempt / gate / decision` 的主写仍必须留在控制层；编排层不得以 helper/builder/projection 的名义重新主写 live execution state。
- Contract guardrail:
- `ExecutionIntent` / `ExecutionContract` 不得重新吸收 `activation_pool`、`standby_pool`、`route_source`、`decision_reason` 这类 compatibility 语义；任何 execution queue / standby / insertion 信息若仍有必要存在，也只能作为编排层与控制层之间的内部协作对象，而不是 leaf-facing contract 字段。

- `memory_views.py` 的 per-path builder 分散状态。
- `published_deliverable_adapter.py` 与 orchestrator startup projection bridge。

## 4. Dependency and Execution Order
- Relationship summary:
- 本 successor 不是单点修复，而是一个 serial conformance chain：必须先清显式越层，再让 supporting capability 正式落位，随后才能安全 retire compatibility seam，最后才处理 project/noise residuals。
- Required serial order:
- 1. Phase A: freeze `Context/Contract Assembler` 的宿主接口与 ownership cut，并清掉编排层的显式越层 ownership。
- 2. Phase B: 收敛 stage-input / execution-contract assembly 与 published deliverable resolution。
- 3. Phase C: retire compatibility seam，切换到控制层 read model / runtime truth。
- 4. Phase D: 清理 secondary mainline 与 project-wrapper residual noise。
- Parallel allowance:
- hotspot inventory、调用链 tracing、测试设计可以并行。
- 代码修改必须按 A -> B -> C -> D 串行推进；Checkpoint A 未通过前，不进入 compatibility seam retirement。
- Phase boundary note:
- Phase A 只允许冻结 assembler host/interface、迁出 orchestrator 的明确越层调用点、并把调用关系接到新的 supporting-capability 边界；per-path builder convergence、published-deliverable resolution unification 与 leaf-facing contract shape 收敛都属于 Phase B，不得在 Phase A 中顺手扩大实现面。

## 5. Phase A: Orchestrator Residual Ownership Extraction
- Objective:
- 先清 active mainline 上已经明确越层的职责，避免后续只是把错误 ownership 移位置。

### 5.1 A1 Script Review Boundary Split
- 拆解 [orchestrator.py:379](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/agents/orchestrator.py#L379) `_open_script_review_gate(...)`：
- deliverable payload build / publish 属于 supporting capability + persistence 边界。
- runtime payload mutation、attempt completion、gate lifecycle 推进属于控制层。
- shared WM projection 只能作为显式 adapter/writer 能力存在，不再混在 orchestrator gate helper 里。

### 5.2 A2 Stop Orchestrator-Local Builder Routing
- 让 [orchestrator.py:1863](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/agents/orchestrator.py#L1863)、[orchestrator.py:1935](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/agents/orchestrator.py#L1935)、[orchestrator.py:1970](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/agents/orchestrator.py#L1970)、[orchestrator.py:2023](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/agents/orchestrator.py#L2023) 不再按 agent type 直接挑 `build_*context(...)`。
- orchestrator 只表达 orchestration intent / target agent / runtime decision；stage-input / contract assembly 改由显式 assembler capability 接管。
- 这里的“接管”仅指 ownership relocation 到正式 assembler 边界，不代表在 Phase A 里完成所有 builder 合流；Phase A 不得借此把 queue/standby/route compatibility 语义重写进新的 leaf-facing contract。

### 5.3 A3 Checkpoint A
- `OrchestratorAgent` 主循环不再同时承担：
- published deliverable publish + runtime payload writeback + WM projection + gate open 的混合职责。
- per-agent builder routing 与 stage-input persistence。
- 验收标准：
- 编排层只保留 orchestration decision、agent activation、control-plane 调用。
- active mainline 中的明确越层项全部转出 orchestrator。

## 6. Phase B: Context/Contract Assembler Placement
- Objective:
- 让 `Context/Contract Assembler` 从“缺位的 supporting capability”变成正式宿主，而不是继续散落在 `memory_views`、adapter bridge 和 orchestrator helper 中。

### 6.1 B1 Freeze the Host
- 决定是升级 [context_assembler.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/services/context_assembler.py) 还是以新宿主替代它，但目标必须明确：
- 宿主负责 stage-input resolution、execution-contract assembly、published-deliverable-backed boundary input normalization。
- 旧式 guidance/continuity-only assembler 不得再继续占用正式 `Context/Contract Assembler` 名义。
- Phase-B-only rule:
- builder convergence、deliverable resolution unification 和 contract shape normalization 在本阶段完成；如果需要保留 execution queue / standby / insertion 之类的内部 orchestration artifact，必须把它们限制在 orchestrator <-> control-plane 内部对象中，不得进入 `ExecutionContract` 或其他 leaf-facing payload。

### 6.2 B2 Converge Per-Path Builders
- 收敛 [memory_views.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/agents/adapters/memory_views.py)：
- image/video 已较接近正式 boundary input，但仍是 per-path builder。
- voice/media 仍有 live-WM-first residual，需要收敛到同一 stage-boundary model。
- assembler capability 必须能显式区分：
- formal stage-boundary truth
- live overlay / iterate facts
- diagnostics-only fallback

### 6.3 B3 Make Deliverable Resolution Explicit
- 收敛 [published_deliverable_adapter.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/services/published_deliverable_adapter.py) 与 [orchestrator.py:643](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/agents/orchestrator.py#L643)：
- runtime payload -> shared WM projection 不再作为 orchestrator 启动时的隐式 bridge。
- published deliverable ref resolution / projection 必须进入显式 capability contract，并带可见诊断。

### 6.4 Checkpoint B
- 仓库中存在唯一明确的 `Context/Contract Assembler` 宿主。
- stage-input / execution-contract 组装不再依赖 orchestrator per-path helper 拼装。
- published deliverable resolution 不再通过 hidden bridge 进入 active path。

## 7. Phase C: Compatibility Seam Retirement
- Objective:
- 让 compatibility seam 退回 diagnostics 或彻底删除，不再作为 active status / decision / read path 的事实来源。

### 7.1 C1 Runtime Read-Model Convergence
- 修改 [orchestrator.py:1401](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/agents/orchestrator.py#L1401) `get_workflow_status(...)`：
- 不再依赖 `workflow.activation_pool` 等 compatibility projection。
- 优先消费控制层 runtime read model / session projection。

### 7.2 C2 Shrink or Retire OrchestrationStateAdapter
- 收缩 [orchestration_state_adapter.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/services/orchestration_state_adapter.py)：
- `workflow.plan`、`workflow.activation_pool`、`workflow.audio_route` 不能继续作为 active path 的运行事实来源。
- 若保留投影，必须明确降级为 diagnostics-only，并且不再被 active decision/status path 回读。

### 7.3 C3 Resolve Gray-Zone Ownership Explicitly
- 对 execution queue / standby / insertion logic 给出明确归属：
- 若属于 orchestration intent shaping，则回到编排层显式对象。
- 若属于 control-plane apply support，则进入控制层的显式 contract/payload。
- 不允许继续藏在 compatibility adapter 里混合持有。
- 无论最终归属如何，这些对象都不得重新命名后进入 leaf-facing `ExecutionIntent` / `ExecutionContract`；它们不是 leaf agent 需要理解的控制语义。

### 7.4 C4 Converge Gate Handling Vocabulary
- 对 [runtime_session_service.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/services/runtime_session_service.py) human review path 与 [orchestration_control_plane.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/services/orchestration_control_plane.py) boundary-triggered gate path 做 vocabulary/contract 收敛。
- 目标不是重写 owner，而是让 gate result / decision handling 共享 canonical contract vocabulary。

### 7.5 Checkpoint C
- `workflow.plan / workflow.activation_pool / workflow.audio_route` 不再被 active path 当作真值。
- status/API/read path 以控制层 runtime read model 为准。
- compatibility seam 已退回 diagnostics 或被删除。

## 8. Phase D: Secondary Mainline and Project-Wrapper Noise Retirement
- Objective:
- 在 mainline 边界已经稳定后，再清掉 project wrapper、dormant runner、parallel orchestrator variants 制造的第二套语义噪音。

### 8.1 D1 Thin the Project Wrapper

- `workflow_manager.create_workflow()` 等旧 `WorkflowState` foundation 语义不得继续暗示第二套 episode engine。

### 8.2 D2 Retire Dormant Mainline Candidates

### 8.3 Checkpoint D

- project mode 被清楚地表达为 wrapper，而不是 sibling engine。
- 历史 runner/orchestrator 变体不再制造正式架构歧义。
