## 2. Canonical Object Freeze
- Facts SoT:
  - shared WM / `WorkflowState`
  - episode/domain facts and deliverable facts such as `project.scene_scripts`, `scene_outputs.*`, `project.final_video`
- Runtime SoT:
  - `workflow_session / workflow_node_state / workflow_node_attempt / workflow_gate / workflow_gate_decision`
  - authoritative live execution and control-plane state
- Stage-boundary carrier:
  - published deliverable refs/payloads
  - `session.input_payload.published_deliverables`
  - authoritative input carrier across stage boundaries, but not another live runtime state machine
- Agent-local trajectory:
  - agent WM `facts` / `obs_records`
  - local planning/observation context, not workflow-stage truth
- Projection / read-model / diagnostics surfaces:
  - runtime view is the formal read model for runtime SoT
  - `workflow_overview`, `MASStateView`, `Task.status/current_step/progress_percentage`, completion event summaries, and compatibility snapshots are projections or diagnostics only

## 3. SoT / Projection Matrix
| Business question | Canonical object | Allowed projection / read-model | Forbidden fallback / overreach |
| --- | --- | --- | --- |
| What is the live status of the current run? | `RuntimeSessionService.build_runtime_view_for_task(_sync)` over runtime substrate | frontend `src/lib/runtimeReadModel.ts`; `/tasks/{id}/runtime`; `/tasks/quick/current.workflow_status` | `workflow_overview.status/progress/current_step`; `MASStateView.status/progress/current_step`; `Task.status/current_step/progress_percentage` as live-run truth |
| What deliverable facts already exist for the episode/project? | shared WM / `WorkflowState` deliverable facts | completion/event summaries and UI views derived from facts | `Task.output_metadata`, runtime summary payloads, or generic workflow summaries as authoritative deliverable SoT |
| What input version should the downstream stage consume? | published deliverable ref/payload plus `session.input_payload.published_deliverables` | explicit compatibility projection only when labeled as transitional and diagnosable | silently falling back to stale live WM; treating shared-WM projected refs as boundary SoT |
| What did the agent observe or plan in its iterations? | agent-local `facts` / `obs_records` | context editor / truncation receipts | treating agent trajectory as workflow-stage truth or runtime-state evidence |
| How should the frontend display quick-run status? | runtime view + pure display-layer helpers | `runtimeReadModel.ts`; WS-triggered refresh of runtime endpoint | rebuilding state from `Task.status`, `/{task_id}` detail, `/{task_id}/status`, `workflow_overview`, or `MASStateView` |

## 4. Inventory of SoT Ambiguity
### 4.1 Wrong-Object Reads
- `backend/app/api/v1/endpoints/tasks.py`
  - `GET /{task_id}` exposes coarse `Task` summary fields that can be mistaken for current-run truth
  - `GET /{task_id}/status` returns both `Task` coarse fields and `workflow_status`, preserving a dual-read surface
- `src/lib/api.ts`
  - `ApiClient.getTaskStatus()` actually calls `GET /tasks/{task_id}`, so the frontend fallback path currently reads task detail rather than runtime truth
- `backend/app/agents/adapters/state/mas_state.py`
  - `MASStateView` still answers `status/progress/current_step`, which is semantically adjacent to runtime read-model vocabulary

### 4.2 Transition Bridges With Boundary Overreach
- `backend/app/services/context_assembler.py`
  - `resolve_published_stage_payload(...)` still falls back from runtime carrier to shared-WM projected deliverable refs
  - `project_runtime_payload_deliverables(...)` still projects runtime carrier refs back into shared WM
- `backend/app/services/published_deliverable_adapter.py`
  - keeps the shared-WM deliverable projection bridge alive as a readable alternate boundary source
- `backend/app/agents/adapters/memory_views.py`
  - can still load published script-stage payloads via shared-WM projection bridge when assembler does not explicitly inject boundary payload/views

### 4.3 Summary Surfaces That Can Be Mistaken For Truth
- `backend/app/agents/orchestrator.py`
  - still initializes and updates `workflow_overview` during active execution
  - still carries `get_workflow_status(...)` as a summary surface, even though runtime view already exists
- `backend/app/services/workflow_completion_adapter.py`
  - publishes `mas_state` in completion payloads; this is acceptable only as completion summary, never as live runtime truth
- `backend/app/models/task.py`
  - `status`, `current_step`, `progress_percentage` remain necessary coarse transport fields but must not drive quick-mode runtime semantics

### 4.4 Controlled Diagnostics That Must Not Be Promoted
- `backend/app/services/orchestration_state_adapter.py`
  - `workflow.diagnostics.compat.*` is a controlled diagnostics projection and must not be re-read as behavior truth
- `src/lib/runtimeReadModel.ts`
  - valid as display-only runtime projection helper; must not grow new business-state fallback reads

## 5. Convergence Strategy
- Freeze object ownership first, then delete readers:
  - runtime questions read runtime view only
  - boundary questions read published deliverable carrier only
  - facts questions read shared WM only
  - trajectory questions read agent WM only
- Retire bridges before adding anything new:
  - stop projecting runtime carrier into shared WM for active-path consumption
  - remove shared-WM projection fallback from active boundary resolution
  - downgrade any unavoidable interim bridge to explicit diagnostics with `fallback_reason`
- Quarantine summary surfaces:
  - `workflow_overview` becomes retirement target, not a shadow state machine
  - `MASStateView` loses runtime-like fields or is explicitly limited to facts-summary vocabulary
  - coarse task status fields remain compatibility/transport only
- Frontend rule:
  - quick-mode runtime display and completion/failure side effects must pivot on runtime view first
  - task detail/status endpoints can remain only for coarse metadata and non-runtime result lookup

## 6. Execution Order
- Phase A: freeze the matrix in code and API contracts
  - narrow quick-mode runtime reads to `/runtime` and `/quick/current.workflow_status`
  - stop frontend fallback from using task-detail status as live state
- Phase B: converge stage-boundary carrier
  - remove active-path dependence on shared-WM projected deliverable refs
  - retire orchestrator startup projection of runtime payload deliverables into shared WM
  - keep only explicit diagnostics if a controlled transition must remain
- Phase C: quarantine or retire summary surfaces
  - demote or rename `workflow_overview` so it cannot answer live-status questions
  - strip runtime-like vocabulary from `MASStateView` or move it to facts-summary naming
  - constrain completion summaries to terminal/event contexts only
- Phase D: validate and document controlled residuals
  - enumerate retired bridges
  - enumerate any remaining controlled transition states and their consumers

## 7. Test and Validation
- Backend:
  - task/runtime endpoint unit tests
  - runtime-session and published-deliverable context assembly tests
  - orchestrator runtime mainline regression tests
- Frontend:
  - TypeScript compile for touched runtime UI/helper files
  - focused runtime polling/helper validation where available
- Audit gates:
  - no active quick-mode runtime path should require `workflow_overview`, `MASStateView`, or shared-WM deliverable projection
  - compatibility diagnostics must remain diagnostics-only and non-authoritative

## 8. Current Convergence Outcome
- Retired from active path:
  - `workflow_overview` live writes/readers in `backend/app`
  - orchestrator startup projection of runtime-input published deliverables into shared WM
  - script-review boundary write-side projection of published script refs into shared WM
  - `ContextContractAssembler.resolve_published_stage_payload(...)` fallback from runtime carrier to shared-WM projected refs
  - frontend quick polling fallback that read `GET /tasks/{task_id}` detail as if it were runtime status
  - `/tasks/quick/current` fallback that exposed a task without a runtime view/session as a resumable quick run
- Still present as controlled transition:
  - `ContextContractAssembler.project_runtime_payload_deliverables(...)` remains as an explicit compatibility bridge helper, but has no active mainline caller
  - `published_deliverable_adapter.py` shared-WM projection helpers remain for explicit compatibility/test paths only
  - `memory_views.py` shared-WM published-deliverable loader remains for direct legacy/dormant callers only; active mainline injects boundary payloads via `ContextContractAssembler`
  - `GET /tasks/{task_id}/status` remains as a coarse task projection, explicitly marked `projection_role=compatibility_coarse_task_status` and `runtime_authoritative=false`
  - completion/event `facts_summary` remains a terminal summary surface, not live runtime truth
  - quick polling may still inspect `/tasks/{task_id}/status` during explicit runtime-session bootstrap lag, but it no longer drives terminal completion/failure side effects from that coarse projection
