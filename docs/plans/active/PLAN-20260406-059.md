## 1. Requirement Analysis
- User intent:
  - 在任何无限画布实现前，先完成必要性、交互收益、runtime truth、前后端 contract、架构成本的综合评估，并把这类探索从 [PLAN-20260406-058.md](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/docs/plans/active/PLAN-20260406-058.md) 的已完成 Sprint 2 基线中剥离出来。
- Scope in:
  - 以 [PLAN-20260406-058.md](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/docs/plans/active/PLAN-20260406-058.md) 的 Sprint 2 stable workspace 为 baseline，识别它已经解决的表达问题与仍未解决的高价值场景。
  - 评估 quick-mode 是否真的需要 canvas 作为 primary workspace、secondary workspace 或 diagnostic surface。
  - 定义 canvas 需要承载的 UI / interaction model：pan / zoom / focus / select、branch navigation、revision / HITL path、artifact lineage、overview vs detail 导航。
  - 盘点现有 runtime/read-model 能表达与不能表达的 graph semantics，并明确哪些语义只能由 control-plane/read-model 提供。
  - 评估前端依赖、性能、移动端、可访问性、Playwright 验证与维护成本。
  - 给出最终 verdict：`reject`, `keep as spike only`, 或 `promote to implementation planning`。
- Scope out:
  - 不在本计划里替换 stable quick workspace，不把无限画布继续塞回 `058`。
  - 不在本计划里修改 backend runtime contract、control-plane 语义、queue/worker 行为。
  - 不在 discovery 阶段实现 production canvas 或默认路由替换。
  - 不允许前端基于 `order_index`、`skipped`、`revision_index` 等线性投影自行发明 graph truth。
- Acceptance criteria:
  - 形成书面 necessity verdict，明确“为什么需要 / 为什么不需要”。
  - 形成 runtime/read-model contract gap inventory，并将候选 metadata 分层为 required / optional / not justified。
  - 形成前端 UI/interaction 与 architecture assessment，覆盖依赖、性能、测试、移动端、隔离策略。
  - 明确是否值得进入 bounded spike；若值得，说明进入条件、non-goals 与 kill criteria。
  - 输出后续动作建议：停在结论、进入 spike、或新开正式实现/contract 计划。

## 2. Functional Decomposition
- Track A: Problem framing and product necessity
  - 定义 Sprint 2 baseline 已经解决的问题，以及仍未覆盖的用户任务。
  - 区分“current-focus workspace 问题”与“graph-navigation 问题”，避免容器先行。
- Track B: UI and interaction model
  - 定义 canvas 中节点、边、分组、gate、artifact、revision 的产品语义，而不是先选库再找用法。
  - 评估 primary vs secondary surface、desktop vs mobile、overview vs detail 的交互模型。
- Track C: Runtime truth and contract boundary
  - 对照 [runtimeReadModel.ts](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/src/lib/runtimeReadModel.ts) 与 [index.ts](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/src/types/index.ts) 盘点现有 read-model 边界。
  - 建立 truth-source map，标出哪些语义已有字段支撑，哪些属于 contract gap，哪些推断被明确禁止。
- Track D: Frontend architecture and validation feasibility
  - 评估 React Flow 或等价方案、渲染性能、手势冲突、无障碍、Playwright 验证与 behind-flag 隔离策略。
- Track E: Governance outcome
  - 将 discovery 输出收敛为明确 verdict，并说明是否需要 successor plan。

## 3. Implementation Approach
- Data flow:
  - 以 Sprint 2 stable workspace 和现有 runtime read-model 为起点，先证明“现有工作台是否不足”，再判断是否需要空间化容器。
  - 对每类候选 canvas semantic 建立 source-of-truth map：`existing field`、`missing contract`、`forbidden inference`。
  - 只有在 spike gate 被显式放行后，才允许以 isolated route 或 explicit flag 做 bounded prototype，而且仍然只能消费现有 projection。
- API/contract changes:
  - 本计划本身不落地 API/contract 变更，只输出候选 contract 清单与优先级。
  - 预期需要评估的候选 metadata 包括：`graph_version`、`edges[]`、`branch provenance`、`node execution lineage`、`skip/invalidation provenance`、`artifact lineage`、`group/phase metadata`、`current_path_id` 等。
  - 任何真实 contract 变更都必须在 verdict 之后另开实现计划，不得在 discovery 阶段混入。
- Migration or compatibility notes:
  - [PLAN-20260406-058.md](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/docs/plans/active/PLAN-20260406-058.md) 的 Sprint 1/2 baseline 继续作为默认稳定路径。
  - discovery 输出不得改写当前 processing 主路径，也不得重新引入固定流水线骨架。
  - queue/worker/broker 仍然只是诊断面，不是业务进度 truth。

## 4. Technical Solution
- Key design choices:
  - `contract-first`, not `library-first`
  - 以 Sprint 2 stable workspace 为对照基线，而不是默认画布一定更优
  - 先判断画布是否更适合作为高级/诊断工作台，再判断是否可能成为默认工作台
  - 用阶段 verdict 驱动推进，并保留明确的 `no-go` 结果
- Trade-offs:
  - 前置分析会拉长启动时间，但能避免做出“看上去更强、实际上不 truthful”的 graph UX
  - 若画布仅适合作为 secondary surface，会带来双工作台维护成本，但可保住 quick path 简洁性
  - 更强 contract 需求会抬高后端改造门槛，但这是保持 runtime truth 的必要代价
- Risks and mitigations:
  - Risk: 被“新容器”的视觉吸引力误导，把审美偏好当成产品必要性
    - Mitigation: necessity gate 只接受 unresolved user jobs / navigation pain 作为推进依据
  - Risk: 前端从 `order_index`、`skipped`、`revision_index` 等字段倒推出 graph
    - Mitigation: 产出 forbidden-inference 清单，并把 contract gap review 设为硬门
  - Risk: discovery 演变成实现，污染稳定路径
    - Mitigation: 输出限定为文档结论和可选 spike gate；任何 spike 都必须 isolated / behind-flag
  - Risk: 依赖、性能、移动端、测试成本被低估
    - Mitigation: architecture assessment 必须覆盖 rendering、gesture、accessibility、Playwright、fallback 成本

## 6. Test and Acceptance
- Unit tests:
  - discovery 阶段默认无新增产品代码测试；若后续批准 isolated spike，再补最小状态映射与 layout helper 覆盖。
- Integration tests:
  - 不以集成测试替代 contract review；若进入 spike，必须保持 behind-flag，并补 Playwright Firefox 路径验证。
- Manual verification:
  - 复用 Sprint 2 的浏览器证据作为 baseline，对照现有工作台是否已经满足核心用户任务。
  - 逐项回答：
    - 现有 workspace 具体缺了哪些高价值任务？
    - 这些缺口是否需要 canvas 才能解决？
    - 现有 contract 是否足以 truthful render？
    - 若不足，最小 contract 扩展是什么？
- Acceptance gate:
  - 仅当 `baseline gap`、`interaction brief`、`contract gap`、`architecture assessment`、`verdict` 五类 deliverable 完成且结论一致，本计划才可进入 execution decision。
  - 本计划的 done signal 不是“出现一个画布 demo”，而是“有足够证据决定做或不做”。
