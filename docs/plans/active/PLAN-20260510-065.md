## 1. Requirement Analysis
- User intent: 将最新母亲节视频暴露出的角色身份漂移、创作质量与系统评分偏乐观问题纳入独立优化项，先明确架构 owner 与边界，再计划实施。
- Scope in: quick 单 episode 主链的角色身份契约、图片/视频生成上下文、prompt asset 归一化、质量检查评分、端到端验收。
- Scope in: 复用现有 project-mode character_bible/reference image 思路，但不把 project runtime 语义直接搬进 quick runtime。
- Scope out: 不处理 064 的 runtime resume/checkpoint 稳定性问题；不把前端或 Celery 作为本问题 owner；不切换视频模型供应商作为默认修复；不通过提示词局部硬编码掩盖数据契约缺失。
- Acceptance criteria: 相同母亲节任务重新生成时，母亲/孩子在允许年龄变化内保留稳定视觉锚点，系统评分不能在角色连续性失败时仍给出 Excellent。
- Acceptance criteria: `scene_info_ref`、`consistency_tool`、image/video prompt composer 与 quality checker 都能看到同一个结构化身份契约，并输出可追踪 diagnostics。

## 1.1 Research Baseline
- Evidence run: task `1080`, workflow `e47efe5f-6e5f-4735-8f32-ff175f349e08`, final video `backend/storage/outputs/videos/mothers_day_final.mp4`, system quality score `92`.
- Manual review: 技术指标基本正常，但 contact sheet 显示母亲和孩子跨场景身份连续性不足，观感更像六段同主题插画而非同一组角色的连续故事。
- Current upstream facts: `concept_plan.content_elements.characters` 已包含 `mother` / `child` 的 `visual_identity`、`signature_outfit_or_props`、`visual_traits` 等信息。
- Current contract gap: `concept_plan.roles` 在该 run 中为空；`consistency_tool._extract_character_assets` 主要读取 `concept_plan.roles`，因此全局 stable traits 未进入 prompt asset。
- Current scene gap: `script_writer` 写入 `characters_present` 和 `character_descriptions`，但存在 `list(set(...))` 式去重，顺序不稳定；prompt composer 后续只取少量描述，容易丢掉关键角色锚点。
- Current prompt gap: image/video prompt composer 已注入 `character_lock`，但锁内容主要压缩为“妈妈、孩子”和少量场景描述，缺少 canonical id、年龄阶段、稳定锚点、允许变化与禁止漂移。
- Current quality gap: `quality_checker` 内容评分主要基于场景数量、技术指标、叙事/合规摘要；角色身份连续性没有成为评分硬约束，导致 manual 75-80 的视频仍被系统评为 92。

## 1.2 Architecture Owner And Boundary
- Owner: MAS 内容契约与上下文投影层，具体落点是 concept/script stage 的结构化角色事实、`ContextContractAssembler` 的 `scene_info_ref`、`consistency_tool` 的 prompt asset 边界，以及 quality checker 的验收合同。
- Not owner: Frontend 只消费 read model 和展示诊断；Celery 只负责任务 transport/resume；image/video provider tool 只执行被授权的生成能力。
- Boundary rule: 角色一致性不是消费端 prompt patch；必须在契约边界归一化和校验，再由 composer/tool schema 消费。
- Boundary rule: 供应商能力通过 tool schema/config 暴露；当前 `image_generation_tool.generate_image` 未暴露 reference image 输入，因此 reference image 是后续能力扩展，不是第一阶段必须依赖项。
- Boundary rule: 质量评分要显式记录 `role_continuity_score`、`identity_drift_findings`、`fallback_reason`；不能用静默 fallback 把缺失事实当作通过。

## 2. Functional Decomposition
- Contract module: 定义 workflow-scoped `character_identity_bible` / `visual_identity_locks`，包含 canonical id、display names、aliases、stable anchors、age-stage variants、signature props、allowed changes、negative drift notes、reference assets。
- Planning/script module: 将 `content_elements.characters`、`role_analysis_tool` 输出和 per-scene script facts 归一为有序 `character_constraints_struct`，避免无序 set 和纯文本截断作为主合同。
- Context module: `ContextContractAssembler` 和 `scene_info_ref` 将全局角色锁与 per-scene cast 一起投影到 image/video generation context。
- Consistency asset module: `consistency_tool` 从结构化合同提取角色锁，优先 canonical/global facts，再叠加场景年龄/状态变化。
- Prompt composer module: image/video prompt composer 消费结构化锁，保持 prompt neutrality，不写供应商名和参数范围。
- Quality module: quality checker 增加角色连续性、叙事角色一致性、风格/画幅一致性的评分与 gate。
- UX/read-model module: 前端可选展示质量诊断，不从队列或 broker 信号推断业务质量。

## 3. Implementation Approach
- Data flow: concept planner 产出或补齐 `character_identity_bible` -> script writer/role analysis 产出 ordered per-scene cast -> MemoryWriter 发布脚本阶段产物 -> ContextAssembler 持久化 `scene_info_ref` -> consistency_tool 组装 prompt assets -> image/video composer 注入角色锁 -> quality_checker 验收。
- API/contract changes: 在 `scene_info_payload` 中新增 `character_identity_bible`、`scene_character_locks`、`quality_expectations.role_continuity`，并让旧 `characters_present` / `character_descriptions` 只作为兼容读字段。
- API/contract changes: `consistency_tool.get_prompt_assets` 输出合同增加角色 identity fields、allowed_variation、reference_assets、diagnostics。
- Migration or compatibility notes: 不保留“错误设计”的静默兼容；旧纯文本字段可以保留读取，但必须标记 `source=legacy_text` 并在缺少结构化锁时触发 prominent diagnostics。
- Migration or compatibility notes: reference image 作为 P1/P2 能力扩展，必须先确认 provider capability；当前阶段可先落地文本身份锁和评分闭环。

## 4. Technical Solution
- Key design choice: 先修 contract boundary，再调 prompt。否则会重复出现“上游有事实、下游只拿到名字”的损耗。
- Key design choice: 年龄变化要显式建模为 allowed stage variant，例如 child: young -> teen -> adult，mother: adult -> elderly；不能简单要求所有镜头同一外观。
- Key design choice: 质量评分要 fail-visible。如果缺少可评估角色合同，结果应降级为 needs_human_review 或低置信评分，而不是 Excellent。
- Trade-off: 第一阶段不强依赖 reference image，可更快闭环；代价是角色脸部一致性仍受生成模型影响。
- Trade-off: 增加角色合同会提升上下文长度，需要 context editor/预算策略控制，不应把完整人物小传无剪裁地注入每个场景。
- Risk: 过强锁定可能伤害剧情时间跨度。Mitigation: 使用 stable anchors + allowed variation，而不是固定单一脸龄/服装。
- Risk: provider 不支持图像参考。Mitigation: capability-gated tool schema；缺失时记录 `reference_asset_unavailable`，不静默假装使用。

- Unit tests: `consistency_tool` 能从 `content_elements.characters` 构造 stable traits；缺失结构化合同输出 diagnostics；不依赖 `concept_plan.roles` 非空。

- Unit tests: image/video prompt composer 在场景 1-6 都包含对应角色 canonical lock、年龄阶段和关键道具/服饰锚点。
- Unit tests: quality checker 在角色连续性失败或缺少评估合同情况下不能返回 Excellent。
- Integration tests: `build_image_generation_context`、`build_video_generation_context`、`scene_info_ref` 都携带同一 `character_identity_bible`。
- Integration tests: prompt safety rewrite 保留 `consistency_locks`，不会删除角色身份锚点。

- Manual verification: `ffprobe`/blackdetect/freezedetect/audio checks 保持通过，角色优化不能牺牲基础技术质量。
