# Plan: Backend memory bootstrap downgrade-path fix for quick runs

## 1. Current Problem
- Symptom:
  - Real quick-mode runs can fail before the first workflow node with `OSError: [Errno 5] Input/output error`.
  - The traceback points to [backend/app/agents/memory/long_term/manager/memory_manager.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/agents/memory/long_term/manager/memory_manager.py), inside `LongTermMemoryManager._start_background_tasks()`.
  - The issue was exposed during the live browser validation attempt for [PLAN-20260406-058.md](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/docs/plans/active/PLAN-20260406-058.md), but the owner is not the frontend amendment.
- Impact:
  - Real quick-mode end-to-end validation can be blocked even when the frontend fix is correct.
  - User-submitted quick runs may fail before entering the first runtime node, producing a backend-owned false start rather than a business-logic failure.
  - The defect weakens confidence in the queued execution host bootstrap path because an optional memory-maintenance downgrade currently has the power to abort the whole orchestrator startup.
- Discovery evidence:
  - Recent successful runs on `2026-04-06` prove the same quick orchestrator path can complete normally:
    - task `1064` initialized memory services and completed successfully
    - task `1071` completed successfully and served a final video download
  - The failing run on `2026-04-07` was task `1072`, which failed inside the memory bootstrap downgrade path before the first workflow node.

## 2. Deviation Analysis
- Expected behavior:
  - In the sync Celery execution host, optional long-term-memory background tasks should degrade safely when no running event loop is available.
  - A missing running loop should not abort `LongTermMemoryManager` construction or quick orchestrator startup.
  - When consolidation and cleanup are disabled, the manager should not probe async startup state unnecessarily.
- Actual behavior:
  - [backend/app/services/queued_task_execution_host.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/services/queued_task_execution_host.py) creates and sets an event loop, but `OrchestratorAgent.create_default()` runs before `loop.run_until_complete(...)`, so `asyncio.get_running_loop()` in the manager constructor raises `RuntimeError: no running event loop`.
  - [backend/app/agents/memory/long_term/manager/memory_manager.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/agents/memory/long_term/manager/memory_manager.py) catches that `RuntimeError`, but then uses raw `print(...)` in the downgrade branch.
  - Under the failing worker/runtime context, that `print(...)` raises `OSError: [Errno 5] Input/output error`, which bubbles through [backend/app/agents/memory/managers/management.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/agents/memory/managers/management.py), [backend/app/services/memory_provider.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/services/memory_provider.py), and [backend/app/agents/orchestrator.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/agents/orchestrator.py), ultimately failing the queued task.
- Root cause classification:
  - Primary owner:
    - backend memory bootstrap implementation
  - Not the primary owner:
    - frontend quick-mode result UI
    - runtime/read-model contract
    - queue/control-plane semantics
  - Core defect type:
    - logic-implementation bug in the downgrade path
  - Secondary design smell:
    - the manager probes loop state even when both maintenance features are disabled, expanding the surface area of a non-essential side effect during construction

## 3. Implementation Approach
- Boundary freeze:
  - Keep this as a bounded backend fix.
  - Do not change frontend behavior, runtime/read-model schema, control-plane semantics, or queue routing meaning.
  - Do not redesign the memory subsystem; only repair the unsafe bootstrap downgrade path.
- Patch strategy:
  - In [backend/app/agents/memory/long_term/manager/memory_manager.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/agents/memory/long_term/manager/memory_manager.py):
    - short-circuit `_start_background_tasks()` when both `enable_consolidation` and `enable_cleanup` are false
    - keep `_consolidation_task` and `_cleanup_task` at `None` in disabled or no-loop cases
    - replace raw `print(...)` with a logger-based or fully best-effort diagnostic that cannot abort manager construction
  - Preserve the current guarantee that background maintenance is optional and must not own quick-run startup success.
- Regression strategy:
  - Add focused backend coverage for manager construction in a sync context with no running event loop.
  - Cover both:
    - disabled maintenance flags
    - enabled flags but no running loop available
  - Assert the manager constructs successfully and leaves task handles unset rather than raising.
- Non-goals:
  - No refactor of memory feature ownership
  - No event-loop redesign for the queued execution host
  - No compatibility layer for unrelated frontend download behavior
