## 1. Current Problem
- Current delivery scope:
- 本计划只覆盖 `012` 之后新暴露出的两个强关联架构问题：
- A. quick 模式下非-artifact 阶段成果没有正式、稳定、可跨执行段引用的发布形态。
- B. `dispatch` 不是正式架构层，却仍混入 quick/project 入口分流、memory composition 与 orchestrator 装配职责。
- Symptom:
- `script approve -> resume -> image/video` 已可进入 media-agent entry，但 `scene_info_ref` 仍可能落成空 payload；根因不再是 queue/runtime 边界，而是阶段成果与 shared WM 边界没有收住。
- `dispatch_generation()` 仍在 quick 主线入口 `build_memory_services()`，导致 execution boundary 与 memory lifecycle 错误耦合。
- Impact:
- 如果继续让 gate/downstream 直接依赖 live shared facts，则 `with HITL` 与 `wo HITL` 无法在 `approve` 条件下保持一致。
- 如果继续保留 `dispatch` 的 per-dispatch memory composition，则 shared WM 会被入口层错误拥有，后续任何 execution boundary 都可能再次暴露类似问题。
- Scope guard:
- 本计划不回退 `012` 的 queue/runtime decoupling 结论。
- 本计划不讨论 `EpisodeOrchestratorAgent` 的 project/multi-episode 收口，只聚焦 quick 模式。

## 2. Dependency and Execution Order
- Relationship summary:
- 两个任务不是独立平行任务，而是一条根因链上的相邻收敛点。
- 非-artifact 阶段成果问题定义了“跨 gate / resume / downstream 应引用什么正式输入”。
- dispatch 问题定义了“谁不该拥有 memory composition，以及 quick 主线入口应收回到什么边界”。
- Dependency:
- 先收住非-artifact 阶段成果，才能避免在 dispatch 收口时把 live shared WM 错当作边界真值继续保住。
- 如果先改 dispatch，很容易退化成“让新的 memory composition root 继续维持 shared WM 充当 stage boundary input”，从而把错误边界固化下来。
- Required serial order:
- 1. 先完成任务 A：非-artifact 阶段成果正式发布/引用模型。
- 2. 再完成任务 B：dispatch 拆解、memory composition root 收口、dispatch 去层化/退役判断。
- Parallel allowance:
- 架构分析、职责 inventory、候选宿主评估可并行。
- 代码实现必须串行：任务 A 的 Checkpoint A 通过后，任务 B 才能进入实现阶段。

## 3. Architectural Decisions Frozen by This Plan
- Shared WM boundary:
- `shared WM` 继续作为 MAS 当前执行段的 live iterate-facts SoT，不被削弱，也不承担跨执行段的正式阶段成果宿主。
- Stage deliverable boundary:
- 节点边界需要统一的“阶段成果发布/引用模型”；artifacts 属于该模型的一类，非-artifact 结构化成果属于另一类。
- SoT rule:
- 不新增平行可写业务 SoT。
- live facts SoT 仍是 shared WM；阶段成果是从 shared WM 在节点边界发布出来的冻结版本，供 gate/resume/downstream 稳定引用。
- HITL equivalence rule:
- `with HITL` 和 `wo HITL` 在 `approve` 条件下必须消费同一份已发布阶段成果；`approve` 只能放行，不能改变下游正式输入来源。
- Dispatch rule:
- `dispatch` 不得继续承担 memory composition、runtime graph 装配或 single-episode 主线路由语义。
- 若职责迁出后 `dispatch` 只剩无意义转发，则应退役，而不是为保留抽象而保留文件。

## 4. Task A: Non-Artifact Stage Deliverables
- Objective:
- 将 concept/script 等非-artifact 成果从“仅存在于 live shared facts”收敛为“统一阶段成果发布/引用模型”的正式成员。

### 4.1 A1 Contract Freeze
- 明确统一上位模型：阶段成果发布/引用模型，而不是“媒体 artifacts 一套、文本成果另一套平行系统”。
- 明确 artifacts 只是该模型的一类；非-artifact 结构化成果是另一类。
- 明确 control-plane 只存引用元数据，不存整份 payload。
- 首轮实现直接冻结正式成果宿主决策：
- 正式阶段成果宿主采用“独立 published deliverable record + `payload_ref`”模式。
- `WorkflowNodeAttempt.output_artifacts`、`WorkflowNodeState.artifact_refs`、`WorkflowGate.artifact_refs` 只保留引用元数据，不升级为 payload 宿主。
- `Resource` 继续只承接媒体/文件资产，不承接结构化文本阶段成果。

### 4.2 A2 Script Deliverable Publication Slice
- 以 quick `script` 节点为首个落点。
- 不重新发明业务变量；直接从现有 shared WM facts 中发布该节点已有成果。
- 第一版 payload 最小集合固定为：
- `project.concept_plan`
- `scene_overview`
- `project.scene_scripts`
- 同时确定统一引用元数据最小集合：
- `deliverable_type`
- `scope_type`
- `scope_id`
- `attempt_id`
- `revision_no`
- `payload_ref`
- `summary`
- `is_candidate`
- `is_approved`

### 4.3 A3 Gate and Downstream Switch
- `script` gate 不再直接引用 `shared_fact` 作为正式审核对象。
- `wo HITL` 下游 media 输入切换为读取已发布 `script` 阶段成果。
- `with HITL + approve` 下游 media 输入切换为读取同一份已发布 `script` 阶段成果。
- shared WM 保留为当前执行段投影/协作视图，但不再作为 gate/downstream 的正式边界输入。

### 4.4 A4 Builder and Adapter Placement
- 阶段成果发布 builder 属于 MAS 内部 adapter/builder 层，不属于 control-plane，不属于 queue，不属于 dispatch。
- 新执行段读取阶段成果后，通过 adapter/projection 回到 shared WM 当前执行段视图。
- 禁止把该 builder 做成 per-node control-plane 变量集或 HITL 特判集合。

### 4.5 Checkpoint A
- `script` 节点已经具备统一阶段成果发布/引用机制。
- `with HITL` 和 `wo HITL` 在 `approve` 条件下可证明读取同一份正式成果来源。
- `gate artifact_refs` 不再直接指向 live `shared_fact`。
- 验收标准：
- 没有新增平行 SoT。
- 没有把文本成果硬塞进 `Resource` 模型。
- 没有为每个 HITL 节点单独增加一套控制变量。

## 5. Task B: Dispatch Convergence and Retirement Decision
- Objective:
- 拆除 `dispatch` 对 memory composition 与主线装配的越界拥有，恢复 quick 主线入口边界，并判断 `dispatch` 是否仍有保留价值。

### 5.1 B1 Dispatch Responsibility Inventory
- 明确 [mode_router.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/core/mode_router.py) 当前职责：
- 共用入口
- quick/project 分流
- `build_memory_services()`
- orchestrator/coordinator 构造
- 为每项职责标注正式归属：
- 极薄启动 helper
- MAS 内部 mainline bootstrap/composition root
- project-mode 外层 wrapper
- 应删除的历史残留

### 5.2 B2 Memory Composition Root Decision
- 明确 quick 主线中的唯一 memory composition root。
- 要求：
- 位于 MAS 内部，不位于 queue/worker，不位于 dispatch。
- 所有 active-path 组件消费同一份注入的 `MemoryServices`。
- 去除 active live path 中 `memory_services or build_memory_services()` 的隐式默认装配。
- 该收口必须与任务 A 的阶段成果发布/读取机制兼容，不能再把 live shared WM 当边界真值保住。
- Active-path audit 范围固定包括：
- [orchestrator.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/agents/orchestrator.py)
- [orchestration_control_plane.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/services/orchestration_control_plane.py)
- [orchestration_runtime_controller.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/services/orchestration_runtime_controller.py)
- [orchestration_state_adapter.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/services/orchestration_state_adapter.py)
- [orchestration_observation_adapter.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/services/orchestration_observation_adapter.py)
- [workflow_completion_adapter.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/services/workflow_completion_adapter.py)
- [script_stage_runner.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/services/script_stage_runner.py)
- [post_script_stage_runner.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/services/post_script_stage_runner.py)

### 5.3 B3 Quick Entry Refactor
- queue host 与 direct quick path 不再通过 `dispatch` 进行 memory composition。
- quick 主线入口只启动同一 `OrchestratorAgent` 主线，不再承担 mode 级主线路由语义。
- project-mode 外层包装与 quick 主线入口进一步拆离，避免继续通过 quick 入口保存历史双宿主结构。

### 5.4 B4 Dispatch Retirement Decision
- 当职责迁移完成后，重新判断 `dispatch` 是否还有剩余价值。
- 若只剩无意义转发，则退役 `dispatch` 抽象。
- 若仍保留一个极薄 helper，也必须明确：
- 不 build memory
- 不装配 runtime/control-plane graph
- 不再承担 single-episode 主线路由
- 若 `dispatch` 被退役或语义被大幅收缩，必须同步更新 [single_episode_harness_architecture_20260311.md](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/docs/architecture/single_episode_harness_architecture_20260311.md) 的组件映射与偏差描述，避免代码与架构文档再次分叉。

### 5.5 Checkpoint B
- quick 主线入口不再拥有 per-dispatch memory composition。
- active live path 不再允许 `dispatch`、queue host、外层包装任意一处隐式重建 memory graph。
- `dispatch` 已被收缩为可证明合理的极薄 helper，或已退役。
- 验收标准：
- `approve -> resume` 不再依赖 per-dispatch memory continuity 偶然成立。
- memory composition root 在架构归属上可明确落位。

## 6. Integration Gate
- Checkpoint A 通过前，不允许进入任务 B 的代码实施。
- Checkpoint B 通过前，不允许宣告 shared WM / stage deliverable / dispatch 三者边界收敛完成。
- 最终验收标准：
- `script approve -> resume -> image/video` 读取同一份已发布阶段成果。
- quick 主线入口不再拥有 memory composition。
- `with HITL` 与 `wo HITL` 在 `approve` 条件下路径一致，仅差 gate 放行，不差正式输入来源。
