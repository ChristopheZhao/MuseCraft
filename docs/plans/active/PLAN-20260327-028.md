## 1. 当前问题
- [PLAN-20260326-027.md](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/docs/plans/active/PLAN-20260326-027.md) 已完成 project-level 边界收敛，并在 Phase C 明确：
  - 当前 `ProjectStateRepository` 是 host-local 的混合 authority/read-model 对象
  - project planning、project orchestration、`/projects/{id}` 与 frontend polling 共用同一组 project facts
  - queue/worker 只能复用现有 Celery transport/container 层，不能重写现有 generation mainline
- 因此，真正的最小实现不再是“把 `_schedule_project_plan(...)` 从 thread 搬到 worker”这么简单，而是需要一条更窄但完整的 implementation stream：
  - project authority surface / shared persistence / projection
  - project reader/writer migration
  - sibling project-job adapter / planning host migration
- 本计划承接这条实现线；`027` 继续只负责治理 closeout，不在本计划内 reopen single-episode MAS mainline 或 repo-level queue rewrite。
- `project-level` 不是当前仓库的长期主线；`028` 的定位是 bounded unblocker：只把 project mode 收敛到可稳定联调的 workflow/wrapper 外层包装，然后回到 `single-episode MAS mainline` 与前后端联调主线，不继续扩展 project-level 体系。

## 2. 架构边界与硬约束
- 本计划服从：
  - [single_episode_harness_architecture_20260311.md](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/docs/architecture/single_episode_harness_architecture_20260311.md)

  - [PLAN-20260326-027.md](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/docs/plans/active/PLAN-20260326-027.md)
- Hard rules:
  - `project-level` 仍是 multi-episode workflow / product-job surface，不是新的 MAS 层，也不是 “MAS 之上的 MAS”。
  - `project-level` 只通过预定义 workflow 调用 single-episode MAS；不得新增第二套 runtime / control-plane / gate semantics。
  - `project authority surface / authority store` 只承接 [PLAN-20260326-027.md](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/docs/plans/active/PLAN-20260326-027.md) Phase A 已冻结的 project wrapper facts 与 read-model projection：project foundation facts、project operation facts、episode aggregate facts、project summary aggregate facts；不得承接任何 single-episode runtime / gate / decision authority。
  - queue / worker / Celery 只负责 transport/execution container；不得拥有 project 或 MAS 业务 authority。
  - `GenerationMode` 继续只属于现有 quick/project generation 主路径；`TaskType` 继续只做 coarse metadata / UI / analytics，不得承担 project planning dispatch authority。
  - 现有 `process_video_task` / `sync_process_video_task` / `run_generation_in_host(...)` generation 主路径不在本计划内重写。
  - 新的 sibling project-job host entry 不得读取 `RuntimeSessionService`、不得准备 single-episode MAS dispatch payload、不得调用 `run_generation_in_host(...)`；它只能按显式 `job_kind` / `handler_key` 调度 project workflow handler。
  - project read-model 必须保持 authority projection 语义；frontend 不新增 second-state logic。

## 4. 执行计划
### Phase A: authority store and projection implementation
- 目标：
  - 引入显式 project authority surface / shared persistence / projection 基础能力，替代 host-local in-memory repository，同时保持它只是 project wrapper facts 的 authority/projection substrate，而不是新的 project control-plane。
- 执行清单：
  - [x] 设计并实现 project authority store abstraction 与 shared backing
  - [x] 将 authority surface 明确限制为 `027` Phase A 冻结的四组 project wrapper facts，不引入 single-episode runtime/gate/decision authority
  - [x] 定义/实现 `/projects/{id}` 所需 projection serialization
  - [x] 保持 placeholder project creation、project summary、polling contract 不变

### Phase B: project reader/writer migration
- 目标：
  - 将 planning、project orchestration、project services、project reads 全部迁到 shared authority surface，但不把这些迁移演化成新的 project orchestration authority 面。
- 执行清单：
  - [x] 迁移 planning path、project read endpoints、project service 写入
  - [x] 迁移 `EpisodeOrchestratorAgent` / character reference service 对 project facts 的读写
  - [x] 去除对 host-local in-memory repository 的 active-path 依赖

### Phase C: sibling project-job adapter and planning-host migration
- 目标：
  - 在不动 generation mainline 的前提下，把 project planning 从 endpoint-local thread host 迁到 sibling project-job host entry。
- 执行清单：
  - [x] 实现 project-job adapter / handler contract
  - [x] 将 `GenerationMode` / `TaskType` 排除在新的 planning dispatch authority 之外，只使用显式 `job_kind` / `handler_key`
  - [x] 新增 project planning worker host entry 并迁移 `_schedule_project_plan(...)`
  - [x] 保证新的 project-job host 不读取 runtime session、也不调用 `run_generation_in_host(...)`
  - [x] 保持 project-level 仍通过预定义 workflow 调用 single-episode MAS

### Phase D: focused validation and governance closeout
- 目标：
  - 完成 focused tests、inventory/docs 同步与用户确认门禁。
- 执行清单：
  - [x] 跑 focused backend validation / migration checks
  - [x] 同步 plan / validation / inventory / PLAN_INDEX
  - [x] 等待用户确认后再 closeout

### Exit Criteria: return-to-mainline gate
- `028` 只在以下条件全部满足时才算完成，并在完成后停止继续扩展 project-level 范围：
  - project mode 的 authority backing 已从 host-local in-memory repository 收敛到 shared durable backing，且 `create_project()` 写入的 placeholder project 可以立即被 `/projects/{id}` 读到
  - `/projects/{id}` 的 authority projection contract 保持稳定：placeholder project、planning/character-reference progress、story-plan episodes、episode aggregates、project summary、style profile、character bible 不回退
  - planning、script approval/editing、project orchestration 对同一组 project facts 的 active-path 读写已切到同一个 shared authority source，不再依赖进程内 `ProjectStateRepository`
  - project planning host 已迁到 sibling project-job entry，且新 host 不触碰 `GenerationMode`、`TaskType`、`RuntimeSessionService`、`run_generation_in_host(...)`
  - focused backend validation 通过，并补一条最小 frontend authority-consumer smoke + backend contract slice：覆盖 create project -> 轮询 planning -> 查看 project workspace projection -> planning 完成后触发 orchestrate 的 consumer path，但不把它写成完整 FE/BE 端到端证明
- 一旦达到以上门槛，后续工作默认回到 `single-episode MAS mainline` 的前后端联调/验证，不在 `028` 内继续扩展新的 project-level product/infrastructure scope。

## 6. 风险与护栏
- 风险 1：
  - 把 authority store / projection implementation 做成新的 project control-plane。
  - 护栏：只实现 `027` Phase A 冻结的四组 project wrapper facts 与 projection，不引入 single-episode runtime/gate/decision semantics。
- 风险 2：
  - 为了迁移 planning host，越界改写现有 generation queue 主路径。
  - 护栏：project-job host 必须是 sibling entry；不改 `process_video_task` / `run_generation_in_host(...)` 主线路由，且新的 host entry 不得读取 `RuntimeSessionService`。
- 风险 2b：
  - 为了图省事，重新借用 `GenerationMode` / `TaskType.SCRIPT_WRITING` 充当 planning dispatch key，导致 queue/worker 重新持有 project 语义。
  - 护栏：planning dispatch authority 只能来自显式 `job_kind` / `handler_key`；`GenerationMode` 与 `TaskType` 的边界保持和 `027` 一致。
- 风险 3：
  - reader/writer migration 过程中破坏 placeholder、project summary、frontend polling contract。
  - 护栏：每阶段都以 `/projects/{id}` 的 authority projection contract 作为验收基线。
- 风险 4：
  - `028` 完成后继续顺势扩展 project-level 范围，导致 project wrapper 反客为主，拖慢回到 MAS-level 主线与前后端联调。
  - 护栏：以上方 `Exit Criteria` 作为收口门槛；一旦 project mode 联调解阻完成，就切回 `single-episode MAS mainline` 的联调/验证，而不是继续新增 project-level 能力。
