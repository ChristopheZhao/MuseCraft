# Plan: 编排去规则化收口与边界清理

## 1. Current Problem
- Symptom:
- Orchestrator 仍存在规则变量驱动分支（`AUDIO_MIXING_MODE`、`VIDEO_AUDIO_POLICY/VIDEO_AUDIO_STRATEGY`）。
- `AUDIO_GENERATOR` 后仍有硬编码的 `VIDEO_COMPOSER add_bgm` 二次编排路径。
- 自适应重规划已落地，但触发点与可激活目标仍偏硬编码（仅 `VIDEO_GENERATOR/VIDEO_COMPOSER -> AUDIO_GENERATOR`）。
- 兼容层仍保留 legacy 双轨（`workflow.plan` 与 `workflow.audio_route` 的决策混用）。
- Impact:
- 编排决策权未完全收敛到统一 `evaluate -> decide -> apply` 闭环，存在反-agent 设计残留。
- 新增 agent/能力时，扩展成本高且容易重复引入规则分支。
- 运行行为可解释性下降，排障需要跨多处变量与兼容路径。
- Reproduction:
- 检查 `backend/app/agents/orchestrator.py` 中 `AUDIO_MIXING_MODE` 分支与 composer re-entry 分支。
- 检查 `_build_audio_contract` 的配置默认回退逻辑。
- 检查 `_decide_replan_action` 的固定 checkpoint 与固定 target 逻辑。

## 2. Deviation Analysis
- Expected behavior:
- Orchestrator 以任务契约与运行事实为唯一决策输入，统一输出下一步动作（continue/retry/activate/abort）。
- 门控只报告达标事实，不承担跨 agent 的参与控制。
- 工具层仅执行显式参数与本地校验，不承载编排策略。
- Actual behavior:
- 编排主链路已显著改善，但仍存在“规则变量 + 局部硬编码”并存。
- “重规划”与“后处理再编排”存在并行决策口，尚未完全统一。
- Root cause hypothesis:
- 历史兼容与稳定性需求导致临时规则长期留存，未在 S5 后继续做收口清理。

## 3. Implementation Approach
- Patch strategy:
- S1: 去除 `AUDIO_MIXING_MODE` 触发的 composer re-entry 硬分支，将 `add_bgm` 统一纳入 replan 动作（由 contract + facts 触发）。
- S2: `_build_audio_contract` 去配置变量决策驱动，改为“输入契约优先 + 明确常量默认 + 诊断标记”，不再读取 `VIDEO_AUDIO_STRATEGY`。
- S3: 泛化 `_decide_replan_action`：
  - 由 `workflow.plan.standby_pool` 与未满足 contract 项生成候选动作；
  - 保留预算与去重，不为每个 agent 增加全局 bool。
- S4: 收口 legacy 双轨：
  - `workflow.plan` 为唯一决策源；
  - `workflow.audio_route` 降级为诊断镜像；
  - 清理 `_should_run_audio_generator`、`_resolve_audio_orchestration_strategy` 等遗留调用路径。
- S5: 文档与测试同步，补充“无规则变量驱动”回归断言。
- Compatibility considerations:
- 迁移窗口保留 `workflow.audio_route` 只读诊断，禁止其反向驱动参与决策。
- 对外输入暂保留 `audio_policy` 字段，但内部统一归约到 `workflow.contract.audio.policy`。
- Rollback plan:
- 如出现线上回归，单开关回滚到“禁用 S3/S4 收口”的上一版本分支，不引入额外兼容 shim。

- Orchestrator：standby_pool 中多 agent 候选时，动作选择符合 contract 缺口与预算约束。
