## 1. Current Problem
- 最近一轮真实运行与诊断脚本表明，当前超时失败的主因不是编排主线错误，而是 `concept_planner` 的超时治理不符合真实耗时分布。
- [backend/app/agents/concept_planner.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/agents/concept_planner.py) 当前在 agent 内部使用 `_budgeted_timeout(deadline_ts, pending_calls)` 以“剩余时间 / 剩余调用数”均分请求预算。
- 该策略会在后段把 `scene_detail_batch_generation` 这类重请求压缩到明显不合理的预算，例如近期日志中已出现 `19s`、`7s` 级别的分配。
- 诊断脚本已证明这种预算不符合 provider 实际耗时：
- [backend/scripts/diagnostics/probe_zhipu_proxy.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/scripts/diagnostics/probe_zhipu_proxy.py)
- [backend/scripts/diagnostics/probe_zhipu_direct.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/scripts/diagnostics/probe_zhipu_direct.py)
- `skeleton` 轻请求在直连成功样本中约 `37.7s`，代理成功样本约 `40.5s`
- `scene` 重请求在 `45s` 预算下代理和直连均超时
- [backend/app/agents/tools/ai_services/zhipu_services.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/agents/tools/ai_services/zhipu_services.py) 当前 fallback 语义是：
- 先按系统代理跑完整 timeout
- 若超时，再按同样完整 timeout 直连重试
- 这会把一次请求的 wall-clock 放大为近似 `timeout + timeout`
- 当前超时控制已经出现明显的 per-agent 局部策略堆积：
- 通用层只提供 `BaseAgent.timeout_seconds`
- `concept_planner` 在内部自建一套均分预算
- provider 层又有独立 fallback timeout 语义
- 这些策略没有统一 contract，导致诊断、调优和失败语义都不稳定

## 2. Target
- 在不改变 MAS 编排/控制/门控主逻辑的前提下，统一 `concept_planner` 的超时治理层级：
- `agent hard deadline`
- `stage budget`
- `provider request deadline`
- `fallback shared deadline`
- 将 `concept_planner` 从“剩余调用数均分”改成“按阶段分配预算”。
- 将 provider fallback 从“双份完整 timeout”改成“共享单份 deadline”。
- 当预算已经不足以完成阶段时，显式 fail-fast，不再发注定超时的请求。
- 清理语义重叠或失去明确 owner 的超时字段，收敛成少量可解释的配置项。
- 保留诊断能力，使后续能够用日志和探测脚本直接解释每次 timeout 失败点。

## 4. Architecture Principles
- P1 MAS Logic Isolation:
- 不改变现有 MAS 主线、门控与 runtime 协作语义。
- 本轮只修正 timeout/deadline contract，不借机改变编排行为。

- P2 Layered Timeout Contract:
- 超时治理只允许存在四层：
- agent hard deadline
- stage budget
- provider request deadline
- fallback shared deadline
- 不允许继续扩展成每个 case/分支各写一套 timeout 逻辑。

- P3 Config over Constants:
- timeout 预算必须通过配置表达，不把阶段预算硬编码成不可调的魔法常量。
- 允许少量结构性 stage timeout 配置，不允许按业务 case 写 if/else 特判。

- P4 Fail-Fast over Doomed Requests:
- 当剩余预算已低于阶段最小 floor 时，应显式报错并停止，而不是继续发注定超时的请求。

- P5 Shared Deadline Fallback:
- provider fallback 必须消费同一份剩余预算，不能让 proxy/direct 分别拿一份完整 timeout。

- P6 Diagnostics First:
- 每次 provider 调用和阶段预算都必须能输出解释性诊断字段，便于用日志与探测脚本交叉验证。

## 5. Implementation Approach
- 冻结现有 timeout 字段语义，明确哪些保留、哪些重命名、哪些删除。
- 在 [concept_planner.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/agents/concept_planner.py) 中引入阶段预算分配函数，替换 `_budgeted_timeout(deadline_ts, pending_calls)` 的主导地位。
- 在 [zhipu_services.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/agents/tools/ai_services/zhipu_services.py) 中引入 shared-deadline fallback 语义，避免一次调用吃两份完整 timeout。
- 在 `scene_detail_batch_generation` 这类重阶段上增加最小 budget floor，预算不足时直接 fail-fast。
- 保留 `CONCEPT_PLANNER_TIMEOUT_SECONDS` 作为 agent 级 hard deadline；将真正有意义的阶段预算收敛为少量 `CONCEPT_STAGE_TIMEOUT_*`。
- 对 `LLM_FALLBACK_TIMEOUT_MIN/MAX` 等字段进行语义审查；若在 shared-deadline 模型下失效，则删除或重命名为真正的 guard 字段。
- 用探测脚本和真实主链运行复核修正结果。

## 6. Stage Breakdown
### Stage A: Timeout Contract Freeze
- 目标：冻结超时治理层级与配置 owner。
- 完成标志：明确保留/删除/重命名的 timeout 字段，并形成统一 contract。

- [x] A1 Timeout Inventory
- 盘点以下层级的 timeout 字段与实际消费位置：
- agent timeout
- provider default timeout
- fallback timeout
- concept internal timeout split
- 输出字段映射表：`field -> owner -> consumer -> keep/remove`

- [x] A2 Contract Freeze
- 定义最终保留的超时层级：
- `CONCEPT_PLANNER_TIMEOUT_SECONDS`
- provider default timeout
- 少量 `CONCEPT_STAGE_TIMEOUT_*`
- shared-deadline fallback guard
- 明确失效字段的去留策略

### Stage B: Concept Planner Stage Budget Refactor
- 目标：将 `concept_planner` 从“剩余调用数均分”改成“阶段预算”。
- 完成标志：`scene_detail_batch_generation` 不再被压缩到不合理预算。

- [x] B1 Stage Budget Introduction
- 引入阶段预算配置，至少覆盖：
- `skeleton`
- `style`
- `voice`
- `scene_batch`
- `finalize`

- [x] B2 Replace Equal-Split Logic
- 让 `concept_planner` 以 stage budget 为主，而不是 `_budgeted_timeout(deadline_ts, pending_calls)`。
- 总 deadline 仍保留，阶段预算不得突破 agent hard deadline。

- [x] B3 Stage Floor Fail-Fast
- 给 `scene_batch` 等重阶段设置最小 floor。
- 若剩余预算低于 floor，直接报 `insufficient_stage_budget` 或 `timeout_budget_exhausted`。

### Stage C: Provider Shared-Deadline Fallback
- 目标：修正 provider fallback 的 wall-clock 放大问题。
- 完成标志：proxy/direct 共用一份 deadline，不再双份 timeout。

- [x] C1 Shared Deadline Plumbing
- 将调用方传入的 request timeout / deadline 统一转成“剩余预算”语义。

- [x] C2 Fallback Refactor
- proxy 请求先消耗预算
- direct fallback 只能使用剩余预算
- 若剩余预算低于 guard，直接失败

- [x] C3 Guard Cleanup
- 审查 `LLM_FALLBACK_TIMEOUT_MIN/MAX` 是否仍有语义；若没有，删除或重命名为共享 deadline guard 字段。

## 7. Risks / Blockers
- 如果只调大 `CONCEPT_PLANNER_TIMEOUT_SECONDS`，而不修 stage budget 与 fallback 语义，问题只会被延后而不会消失。
- 如果 shared-deadline 没做好，provider fallback 仍会继续放大尾延迟。
- 如果 timeout 字段不收口，后续容易继续在其他 agent 内部复制一套局部规则。
- 供应商侧真实性能抖动仍可能导致重阶段超时；本轮目标是让失败更可解释、更可控，而不是承诺完全无 timeout。

## 8. Quality Gates
- QG-1: `concept_planner` 不再以“剩余调用数均分”作为核心 timeout 分配策略。
- QG-2: provider fallback 不再出现 `timeout + timeout` 双份 wall-clock 语义。
- QG-3: `scene_batch` 等重阶段预算不足时会 fail-fast，而不是继续打 doomed request。
- QG-4: timeout 配置层级清晰，语义重叠字段已清理或重命名。
- QG-5: 代理/直连探测结果与真实主链日志能够共同解释 timeout 行为。
- QG-6: 本轮改动不改变 MAS 编排、runtime、gate 的既有执行逻辑。
