## 2. Deviation Analysis
- Expected behavior:
  - authoritative scene/image facts should originate from `scene_info_payload.scenes_to_generate[]` and upstream prepared assets, not from helper-side lexical inference.
  - consistency owner should provide stable traits, scene cast, opening anchor, and continuity assets in structured form.
  - composer should synthesize prompts from those explicit facts.
  - helpers should remain limited to non-semantic formatting such as clipping, alias canonicalization, and whitespace cleanup.
- Actual behavior:
  - image mainline no longer writes back inferred scene strategy, and execution tooling no longer owns prompt construction.
  - the remaining mismatch is narrower: runtime payload contract excludes `image_purpose` / `frame_thesis`, while the composer can still consume them as explicit inputs.
  - structured character/consistency assets are now the primary mainline path, but repo-level helper debt still exists outside this restored slice.
- Root cause hypothesis:
  - earlier anti-slideshow repairs added helper-side heuristics to compensate for weak upstream fields.
  - later root-cause fixes introduced better upstream structure, but the helper/composer fallback paths were not removed.
  - Stage 1 removed most of that downstream drift, but Stage 2 owner closure never decided whether `image_purpose` / `frame_thesis` belong in `scene_contract` or should disappear from accepted image-mainline inputs entirely.

## 3. Implementation Approach
- Patch strategy:
  - Stage 1 removes helper-side semantic rewriting and lexical owner logic from the image mainline without compatibility shims.
  - Stage 2 closes the remaining owner decision for `image_purpose` / `frame_thesis` at the proper upstream boundary instead of leaving them as half-supported downstream inputs.
  - restore owner boundaries so that:
    - `scene contract` owns scene/image strategy fields
    - `consistency assets` own stable traits, scene cast, opening anchors, and continuity references
    - `image_prompt_composer_tool` owns prompt synthesis only
    - `image_generation_tool` owns execution only
  - consume structured consistency/character assets as the primary path and remove semantic fallback paths that guess “appearance” or “best still frame”.
- Compatibility considerations:
  - no compatibility layer should preserve proven-wrong helper-side owner logic.
  - if a required upstream field is genuinely missing, fail with explicit diagnostics or keep only the minimal non-semantic fallback explicitly allowed by contract.
  - this plan does not reopen `PLAN-20260401-044`; it continues boundary cleanup after that plan closed.
  - this plan does not absorb unrelated side-service debt such as `error_recovery.py` or generic provider-routing heuristics; those remain separate cleanup streams unless they directly block image mainline owner restoration.
- Rollback plan:
  - if a removal breaks image prompt assembly because upstream owner fields are absent, stop and amend the plan to add the missing contract field at the proper owner layer.
  - do not rollback by reintroducing helper lexical guessing or execution-tool semantic fallbacks.

## 4. Boundary Statement
- `scene_info_payload.scenes_to_generate[]` remains the authoritative runtime carrier for scene/image/video facts.
- until `scene_contract` explicitly accepts `image_purpose` / `frame_thesis`, image mainline runtime authority is limited to frozen scene fields plus structured consistency assets; downstream consumers must not treat those two fields as required or authoritative runtime contract inputs.
- `consistency_tool` may prepare structured assets, but helper/composer layers must not infer or rewrite primary semantics that belong to scene contract or consistency owners.
- `image_prompt_normalization.py` is allowed to do contract-bound formatting only.
- `image_prompt_composer_tool` owns prompt synthesis, not secondary scene-strategy selection.
- `image_generation_tool` is execution-only and must not own prompt construction or semantic repair.
