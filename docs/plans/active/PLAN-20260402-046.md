## 1. Current Problem
- Symptom:
  - image mainline still contains helper/adapter/composer logic that performs semantic selection and strategy inference instead of consuming authoritative upstream facts.
  - the clearest cases are:
    - `memory_views` infers and writes `image_purpose` / `frame_thesis` during payload assembly
    - `image_prompt_composer_tool` re-infers the same fields and uses helper heuristics to decide whether `opening_state` survives into the prompt
    - `image_prompt_normalization` helper functions such as `compress_character_description()` and `soften_still_language()` decide what counts as stable appearance or rewrite semantic text
    - `image_generation_tool` still falls back to helper-driven semantic compression when structured character constraints are missing
- Impact:
  - helper/adapter layers are acting like secondary semantic owners, which violates `Anti-Hardcoding`, `Tools are execution-only`, `Contract-First Normalization`, and `Root-Cause First`.
  - prompt behavior becomes difficult to trace because the final image prompt is partly determined by lexical heuristics rather than by explicit scene/consistency contract fields.
  - upstream owner improvements such as `scene_thesis`, `frame_thesis`, `stable_traits`, and structured consistency assets cannot become the real SoT while helper heuristics keep overriding or supplementing them.
- Reproduction:
  - [memory_views.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/agents/adapters/memory_views.py#L534) and [memory_views.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/agents/adapters/memory_views.py#L697) infer `image_purpose` / `frame_thesis` during context assembly.
  - [image_prompt_composer_tool.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/agents/tools/image_prompt_composer_tool.py#L355) re-runs the same inference and [image_prompt_composer_tool.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/agents/tools/image_prompt_composer_tool.py#L672) uses `contains_high_risk_action_language()` to choose which timepoint survives into the prompt.
  - [image_prompt_normalization.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/agents/tools/image_prompt_normalization.py#L209) rewrites semantic text and [image_prompt_normalization.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/agents/tools/image_prompt_normalization.py#L259) filters character descriptions with marker-based heuristics.
  - [image_generation_tool.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/agents/tools/ai_services/image_generation_tool.py#L742) falls back to `compress_character_description()` in the execution tool.

## 2. Deviation Analysis
- Expected behavior:
  - authoritative scene/image facts should originate from `scene_info_payload.scenes_to_generate[]` and upstream prepared assets, not from helper-side lexical inference.
  - consistency owner should provide stable traits, scene cast, opening anchor, and continuity assets in structured form.
  - composer should synthesize prompts from those explicit facts.
  - helpers should remain limited to non-semantic formatting such as clipping, alias canonicalization, and whitespace cleanup.
- Actual behavior:
  - image helper and adapter layers infer missing scene strategy and frame selection, then write those inferred values back into runtime-facing data.
  - execution tooling still contains compatibility prompt-building and semantic fallback paths instead of being execution-only.
  - structured character/consistency assets exist upstream, but image mainline does not fully consume them as the primary owner surface.
- Root cause hypothesis:
  - earlier anti-slideshow repairs added helper-side heuristics to compensate for weak upstream fields.
  - later root-cause fixes introduced better upstream structure, but the helper/composer fallback paths were not removed.
  - the system is now in a mixed-owner state: upstream contracts exist, but downstream helpers still behave like semantic repair layers.

## 3. Implementation Approach
- Patch strategy:
  - remove helper-side semantic rewriting and lexical owner logic from the image mainline without compatibility shims.
  - restore owner boundaries so that:
    - `scene contract` owns scene/image strategy fields
    - `consistency assets` own stable traits, scene cast, opening anchors, and continuity references
    - `image_prompt_composer_tool` owns prompt synthesis only
    - `image_generation_tool` owns execution only
  - consume structured consistency/character assets as the primary path and remove semantic fallback paths that guess “appearance” or “best still frame”.
- Compatibility considerations:
  - no compatibility layer should preserve proven-wrong helper-side owner logic.
  - if a required upstream field is genuinely missing, fail with explicit diagnostics or keep only the minimal non-semantic fallback explicitly allowed by contract.
  - this plan does not reopen `PLAN-20260401-044`; it continues boundary cleanup after that plan closed.
  - this plan does not absorb unrelated side-service debt such as `error_recovery.py` or generic provider-routing heuristics; those remain separate cleanup streams unless they directly block image mainline owner restoration.
- Rollback plan:
  - if a removal breaks image prompt assembly because upstream owner fields are absent, stop and amend the plan to add the missing contract field at the proper owner layer.
  - do not rollback by reintroducing helper lexical guessing or execution-tool semantic fallbacks.

## 4. Boundary Statement
- `scene_info_payload.scenes_to_generate[]` remains the authoritative runtime carrier for scene/image/video facts.
- `consistency_tool` may prepare structured assets, but helper/composer layers must not infer or rewrite primary semantics that belong to scene contract or consistency owners.
- `image_prompt_normalization.py` is allowed to do contract-bound formatting only.
- `image_prompt_composer_tool` owns prompt synthesis, not secondary scene-strategy selection.
- `image_generation_tool` is execution-only and must not own prompt construction or semantic repair.
