## 1. 当前问题
- 症状：
  - `040` 已经功能性关闭了 image provider sensitive-error 链和 still/reference prompt 漂移，但 post-implementation review 发现 still/reference prompt 的收敛仍依赖较多内容级字符串规则。
  - `backend/app/agents/tools/ai_services/image_generation_tool.py` 里仍维护了 `video_only_markers`、`high_risk_markers`、短语替换、角色描述压缩等规则，`backend/app/agents/tools/image_prompt_composer_tool.py` 里也存在近似的 still-clause / character-compression 规则。
  - 这些规则目前分散在 tool 与 composer 两侧，形成重复 owner，容易继续演化成 case-by-case patch。
- 影响：
  - 当前实现虽然没有长出第二条 image 主线，但与仓库的 `Anti-Hardcoding`、`Contract-First Normalization` 原则仍有偏差。
  - 重复的内容级规则会让 still/reference contract 变脆，后续小改动容易在 tool/composer 之间发生漂移。
  - 若不收口，这会成为 `040` 的架构验收阻塞项。
- 发现来源：
  - `040` 完成后的独立架构 review
  - `backend/app/agents/tools/ai_services/image_generation_tool.py`
  - `backend/app/agents/tools/image_prompt_composer_tool.py`

## 2. 根因分析
- `040` 的主目标是先关闭 rerun failure 和 still/reference contract 漂移，因此部分收敛逻辑直接落在现有 tool/composer 代码上，以确保 frozen scope 内快速闭环。
- still/reference 投影、静态图压缩、character reference 收敛目前没有一个清晰的单一 owner 边界，导致相近的内容级替换在多个消费点重复出现。
- 这些规则里有一部分是结构驱动的必要降噪，但也有一部分已经接近具体中文短语和 case-shaped prose 的内容级硬编码，不利于长期维护和验证。

## 3. 目标边界
- 本计划只处理 `040` 遗留的架构硬化问题：
  - still/reference prompt contract 的去硬编码收口
  - tool/composer 间重复规则的单一 owner consolidation
- 本计划必须保持：
  - 同一条 image 主线，不新增 parallel image workflow / carrier / consistency mainline
  - `image_purpose / task_direction` 继续作为现有 image context 的显式输入，而不是 agent-level decision table
  - `040` 已实现的 provider-sensitive rewrite closure 不回退、不泛化成通用 provider safety 工程
- 本计划不做：
  - video path 修改
  - scene/script 上游 contract 重写
  - project-level character-reference orchestration 重构
  - 新的 image runtime / memory / scheduler owner

## 3.1 执行前评审 Gate
- 新窗口在任何代码实现前，必须先 review 本计划和 validation ledger：
  - 确认任务仍是 image mainline 内的 contract hardening，而不是顺手扩成新的 image architecture 主题。
  - 确认修复方向是“单一 owner 的 contract-boundary normalization”，而不是在各消费点继续补新的词表和短语替换。
  - 确认不会因为去硬编码而引入第二套 still/reference pipeline、平行 carrier 或新的 policy center。
  - 确认 `040` 与 `041` 的关系保持为“功能修复”和“架构收口”两层记录，而不是把两者重新揉成一条主线。
- 若 review 认为 owner、范围、验收条件需要调整，必须先 amendment 本计划和 validation ledger，再开始实现。

## 3.2 Single-Owner Freeze
- 本计划中的“单一 owner”不是抽象口号，而是一个明确的 raw-to-contract normalization 实现边界：
  - 它负责把 still/reference 相关的 raw prose（scene opening、montage/mixed 描述、role-card character 描述、consistency asset 文本）投影为 contract-ready 的静态图像片段。
  - prompt root 与 consistency lock 两个消费面必须复用这一份 normalization owner；允许在各自层面做格式拼接，但不允许各自再维护独立的 marker/replacement/compression 规则中心。
- owner 可以是现有 image mainline 内新增的专用 helper / adapter，也可以收口在允许触达的单个现有文件中；但无论落点在哪里，都必须满足：
  - `image_prompt_composer_tool` 不再对 raw scene/asset 文本独立维护 still/reference marker、短语替换或角色压缩规则。
  - 若 owner 被抽到共享 helper，则 `image_generation_tool` 也不得保留第二套近似 lexical rules 作为并行兼容层。
  - “两个调用点各自保留近似实现，只是名字更像 helper” 不算完成本计划。

## 3.3 代码触达包络
- 默认允许触达：
  - `backend/app/agents/tools/ai_services/image_generation_tool.py`
  - `backend/app/agents/tools/image_prompt_composer_tool.py`
  - 与 still/reference contract consolidation 直接相关的 helper / config / tests
  - 本计划与 validation 文档
- 若实现需要触达 scene/script/video/orchestrator/runtime 代码，必须先 amendment；不得把这次 hardening 扩成跨主线重构。

## 4. 实施方案
- Patch strategy A: 建立单一 owner 的 still/reference normalization boundary
  - 将 still/reference prompt 的收敛逻辑集中到一个明确的 contract-boundary owner，避免 tool/composer 双方各自维护一套相近字符串规则。
  - owner 必须同时服务 prompt-root 与 consistency-lock 两个消费面；一个消费面调用 owner、另一个消费面继续保留独立规则，不算收口完成。
  - 优先保留结构驱动的投影与压缩；仅当结构信息不足时，才允许最小、集中、可审计的内容级降噪规则。
- Patch strategy B: 以 contract-first 收敛替代 case-by-case 短语修补
  - 去除或合并重复的 `video_only` / still-clause / character-compression 内容级规则，避免在多个调用点各自维护短语黑名单或替换表。
  - 若仍需要少量 lexical guard，必须集中到单一边界并具备明确职责说明；不得继续散落在 tool/composer 两套实现中，也不得以“shared helper + residual local guard”形式保留双中心。
- Patch strategy C: 在不扩 scope 的前提下保持 `040` 行为
  - `scene_opening_anchor` 仍以 `opening_state` / dominant still composition 为根。
  - `character_reference` 仍以稳定身份特征、服饰、标志道具为根。
  - montage/mixed scene 仍必须投影为单张 still，而不是重新放回视频剪辑 prose。
- Compatibility considerations：
  - 不新增新的 image schema/mainline；继续消费现有 `scene_info_payload.scenes_to_generate[]` 与 `image_purpose / task_direction`。
  - 不回退 `040` 的 safety closure、purpose-aware contract、consistency compression 成果。
  - 后续若要继续做 broader image architecture work，必须另开新计划，不得继续扩写 `041`。
- Rollback plan：
  - 若 centralized normalization 导致 `040` 的成功 case 回归，仅回滚本计划触及的 still/reference hardening 文件，不回滚 `040` 已完成的 provider-sensitive closure。

## 6. 测试与验收
- 必要自动化：
  - normalization-owner regression：针对 montage/high-risk opening/role-card 输入，owner 层单测能直接产出 prompt root 与 consistency lock 共享的静态化结果
  - still/reference normalization regression：集中化边界能覆盖 `scene_opening_anchor` 与 `character_reference` 两类方向，且输出保持 `040` 的功能行为
  - de-duplication regression：tool/composer 不再各自维护重复的 still/reference 内容级规则集
  - scene 5 regression：montage/mixed scene 仍只产生单张 still composition，不回退到视频剪辑 prose
  - consistency regression：compressed locks 仍保持短锁定，不回退为长段角色/节奏 prose
- 手工验收：
  - code review 可明确指出 still/reference normalization 的单一 owner，并说明另一个消费面只负责格式化/装配而非再次处理 raw prose
  - source review 不再能在 tool/composer 两侧同时找到近似的 still/reference marker 列表、短语替换表或角色压缩启发式
  - review 确认没有新增第二套 image contract / workflow / consistency mainline
  - `040` 的 closeout blocker 被清晰解除，不再需要以 duplicated hardcoding 作为保留风险
- 完成信号：
  - duplicated content-level hardcoding 不再分散存在于 image tool/composer 两侧
  - `040` 的功能修复记录与 `041` 的架构收口记录边界清晰，可独立归档
