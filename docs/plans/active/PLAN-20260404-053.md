## 1. Current Problem
- Symptom:
  - Existing completed tasks show a split between runtime completion and persisted read-model completion.
  - Concrete evidence: task `fab71508-3a2b-4c8f-a67c-4c63855c2b01` is `completed`, and its runtime view contains `summary_output.final_video_url=/files/outputs/videos/凡人修仙传2预告片_final.mp4`, but `task.output_metadata.final_video_url/final_video_path` are empty and persisted `resources` count is `0`.
  - Historical frontend path also attempted `GET /tasks/{id}/result`, but no such endpoint exists in the current backend API surface.
- Impact:
  - Runtime SoT and task/read-model projection diverge: the system may have successfully produced a final video, but downstream consumers still observe "no final result".
  - User-facing review flow cannot reliably open the completed asset even when the file already exists on disk.
  - This window already introduced adjacent runtime changes, so fix effectiveness cannot be evaluated safely without an explicit amendment boundary.
- Reproduction:
  - Static evidence already captured from the local database and runtime view:
    - latest completed task projection mismatch: `task.output_metadata.final_video_* = None`, `resources = 0`
    - runtime view mismatch: `summary_output.final_video_url` present on the same task
  - Backend process is currently down; any further reproduction should restart a clean backend only after this plan is accepted.

## 2. Deviation Analysis
- Expected behavior:
  - Once `video_composer` produces a final video, the completion path should persist final video references into the completion event payload, `task.output_metadata`, and `resources` projection.
  - Frontend completion polling should consume an existing read-model contract (`runtime`, `task detail`, or `resources`) rather than a removed or nonexistent result endpoint.
- Actual behavior:
  - [orchestrator.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/agents/orchestrator.py) writes `runtime.summary_output.final_video_url` using a fallback from `workflow_results.video_composer.final_video_url` when MAS `project.final_video` is absent.
  - [workflow_completion_adapter.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/services/workflow_completion_adapter.py) builds the completion event payload and final resource list from MAS `project.final_video` only.
  - Therefore a run can end with:
    - runtime marked completed and exposing `summary_output.final_video_url`
    - persistence event lacking `final_video_url/resources`
    - [data_persistence.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/services/data_persistence.py) never writing final refs into `task.output_metadata` or `resources`
  - Separately, the frontend completion path previously depended on `/tasks/{id}/result`, while [tasks.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/api/v1/endpoints/tasks.py) exposes `/{task_id}`, `/{task_id}/resources`, and `/{task_id}/runtime`, but not `/{task_id}/result`.
- Root cause hypothesis:
  - Primary backend root cause:
    - the runtime-completion fallback path and the persisted completion-event path do not use the same final-video authority.
    - runtime completion can succeed from `workflow_results`, while persistence still depends on MAS `project.final_video`.
  - Primary frontend root cause:
    - the completion UI was reading a stale/nonexistent endpoint contract.
  - Secondary observation, not yet justified as root cause for this fix:
    - this window introduced composer/QC/media helper changes that may affect runtime behavior, but the proven mismatch above already explains the original "video exists but result not synchronized" symptom.

  - [tasks.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/api/v1/endpoints/tasks.py#L411), [tasks.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/api/v1/endpoints/tasks.py#L491), and [tasks.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/api/v1/endpoints/tasks.py#L531) expose `/{task_id}`, `/{task_id}/resources`, and `/{task_id}/runtime`; there is no `/{task_id}/result`.
  - Database evidence on existing completed task `fab71508-3a2b-4c8f-a67c-4c63855c2b01`:
    - `RuntimeSessionService.build_runtime_view_for_task(...)` returns `summary_output.final_video_url=/files/outputs/videos/凡人修仙传2预告片_final.mp4`
    - the same task has `output_metadata={'celery_task_id': ..., 'workflow_session_id': 41}` and `resources=0`
- Strong inferences:
  - The original mismatch is most likely caused by runtime completion and persisted completion reading different final-video authorities:
    - runtime summary can succeed from `workflow_results.video_composer.final_video_url`
    - persistence still depends on MAS `project.final_video`
  - If `project.final_video` was missing or incomplete at completion publish time, runtime would look completed while `task.output_metadata` and `resources` would remain empty.
- Unknowns:
  - Whether the accepted fix should normalize the authority at `orchestrator`, `workflow_completion_adapter`, or both.
  - Whether any retained `video_composer` local changes are necessary to guarantee `project.final_video` exists before completion.
  - Whether the current widened `quality_checker` changes introduce an independent terminal-path regression on top of the original sync mismatch.
