## 1. 当前问题
- 当前 MAS 主线新的真实阻塞点不是 `029` 中的 schema drift，而是 `concept_planner` 在真实联调中因 GLM 并发/限流与 `proxy -> direct` 网络回退叠加，最终在 `scene_batch` 阶段触发 `timeout_budget_exhausted`。
- [PLAN-20260319-009.md](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/docs/plans/active/PLAN-20260319-009.md) 已经把 timeout 语义从“剩余调用数均分”收敛到“阶段预算 + shared-deadline fallback”；当前问题不允许通过回退旧方案或单纯放大总 timeout 掩盖。
- 当前仓库并不具备真正的 supplier-agnostic LLM multi-provider routing，也没有把 `concept_planner` 与具体 provider 解耦到可切换状态：
  - [llm_policy.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/agents/utils/llm_policy.py) 仍把非 `zhipu` provider 折叠回 `ServiceProvider.ZHIPU`
  - [service_interfaces.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/agents/tools/ai_services/service_interfaces.py) 目前只注册了 `zhipu` LLM 服务，没有 DeepSeek LLM 接入
  - [ai_config.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/core/ai_config.py)、[llm_policies.yaml](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/config/llm_policies.yaml)、[ai_config.yaml](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/config/ai_config.yaml) 也没有形成可直接切换到 DeepSeek 的完整 contract
- 因此，本轮若要推进 DeepSeek 方向，正确目标不是把 `concept_planner` 从“绑定 GLM”改成“绑定 DeepSeek”，而是先补齐 MAS-level 的 LLM provider routing contract，并让 `concept_planner` 对具体 provider 保持实现解耦；DeepSeek 只应作为首个 policy-selected primary provider 验证目标，而不是新的代码内绑定点。

## 2. 架构边界与硬约束
- 本计划是新的 MAS-level successor；不 reopen project-level 范围，也不把 [PLAN-20260327-029.md](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/docs/plans/active/PLAN-20260327-029.md) 的 lifecycle closeout 与本计划混 scope。
- Hard rules:
  - 严格遵守 supplier-agnostic / tools-first：provider 选择必须经由 policy/config/service registry contract，而不是 agent 内直接写供应商特判。
  - 不允许在 `concept_planner` 中引入 `if stage == ... then deepseek else glm` 之类的 stage-specific provider 分支。
  - 不允许把 `concept_planner` 或 concept-side helper 再默认绑定到任何单一 provider；本计划结束后，concept-side routing 至少应能在不改 agent 代码的前提下切换 provider/model。
  - 不新增 silent fallback；如果选定 provider/service 不可用，必须输出显式诊断，而不是静默退回别的 provider。
  - 不回退到 `009` 之前的“剩余调用数均分 / 递进压缩 timeout”方案；`009` 的阶段预算与 shared-deadline contract 继续保留。
  - queue / worker / Celery 仍只负责 transport/execution container，不得承接 MAS provider-routing 语义或 runtime lifecycle 语义。
  - 本计划只处理 LLM provider routing / DeepSeek 接入 / provider-agnostic concept routing，不顺手扩 scope 到 project-level、runtime control-plane 或 query-side compatibility seams。

## 2.1 Phase 0 冻结结果
- 当前真实调用链已经确认：
  - [concept_planner.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/agents/concept_planner.py) 先通过 `ai_config.get_model_for_agent("concept_planner")` 与 `get_fallback_model_for_agent("concept_planner")` 选出 primary/fallback model
  - 随后 `_invoke_concept_model(...)` 再通过 `self.get_llm("plan")` 取得 role handle，并把前述 model 显式传给 `llm.chat_completion(...)`
  - [base.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/agents/base.py) 的 role handle 来自 [llm_policy.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/agents/utils/llm_policy.py)；因此 provider owner 与 model owner 当前被拆在不同层
- 当前确认的 owner overlap / deviation：
  - `llm_policy` 名义上负责 role -> provider，但实现上把非 `zhipu` provider 全部折叠回 `ServiceProvider.ZHIPU`
  - `ai_config` 负责 agent -> model/fallback_model 与 model metadata，但 concept-side 并不校验这些 model 是否与 role handle 选出的 provider 一致
  - LLM service registry 在选定 provider `is_available() == False` 时会静默返回“任意其他可用 provider”，形成 cross-provider silent fallback
  - repo 里当前不存在 DeepSeek LLM service、DeepSeek provider config，或任何 `DEEPSEEK_*` 级别现成接线
- 本计划冻结的 single-owner contract：
  - role-level provider selection 必须由统一的 route contract 决定，不能继续让 `llm_policy` 选 provider、`ai_config` 另选 agent model、registry 再兜底改 provider
  - `ai_config` 只拥有 provider/model registry metadata、alias resolution、timeout/default metadata，不拥有运行时的隐式 provider 切换
  - service registry 只拥有 provider registration 与显式 availability diagnostics；禁止 cross-provider silent fallback
  - `concept_planner` 只消费 resolved route contract 与 model capabilities，继续拥有 stage budget / JSON response / diagnostics，不拥有 provider choice 语义
- Phase 1 入口门槛：
  - 后续实现必须先把 provider/model route 收敛为单一 resolved contract，且满足“切换 provider/model 不需要改 concept agent 代码”后，才允许进入 DeepSeek 接入与 routing cutover。

## 4. 执行计划
### Phase 0: routing audit and scope freeze
- 目标：
  - 冻结当前 `concept_planner -> BaseAgent role handle -> LLMPolicyManager -> service registry -> provider service` 的真实调用链，并明确 provider 选择的 single owner。
- 执行清单：
  - [x] 审计 `concept_planner` 当前主模型、备用模型、role handle 与 provider registry 的实际耦合点
  - [x] 明确 `ai_config`、`llm_policies.yaml`、service registry 各自拥有的 provider/model 语义边界
  - [x] 定义 “multi-provider support + DeepSeek primary routing” 的最小 contract，不引入 stage-specific if/else 或 silent fallback
  - [x] 冻结本计划与 `029`、project-level、runtime/control-plane 的边界

### Phase 1: provider routing substrate
- 目标：
  - 让 role-based LLM policy 与 provider resolution 真正支持多 provider，而不是把非 `zhipu` provider 静态折叠回 `zhipu`。

  - [x] 清理 LLM registry 层当前 “selected unavailable -> implicit other-provider fallback” 的不透明语义，改为显式诊断或显式配置驱动的后备策略
  - [x] 补足 provider selection 所需的 focused tests

### Phase 2: DeepSeek provider integration
- 目标：
  - 以 supplier-agnostic 方式接入 DeepSeek LLM provider，并把所需配置/诊断暴露到统一 contract；DeepSeek 在本阶段只是新增可选 provider，不直接成为 agent 内绑定。

  - [x] 在 `ai_config` / yaml config 中补齐 DeepSeek provider 与 model metadata
  - [x] 确保 JSON-only `chat_completion` 路径与 `response_format={"type":"json_object"}` 在 DeepSeek 路径上可用
  - [x] 为 provider startup diagnostics、missing env、selected-provider unavailable 等场景补测试或显式日志验证

### Phase 3: provider-agnostic concept routing cutover
- 目标：
  - 不改 `009` timeout contract 的前提下，让 `concept_planner` 的主规划调用对 provider 保持实现解耦，并通过 policy/config 把当前 primary 指向 DeepSeek。

  - [x] 通过 policy/config 让 `concept_planner` 的 planning role/primary provider-model 当前解析到 DeepSeek，而非在 agent 内写死
  - [x] 对 fallback model / provider 语义做 contract-first 收敛，避免跨 provider 隐式跳转或默认绑定
  - [x] 保留并补强 `concept_planner` 的 provider/budget diagnostics，确保失败仍可解释

## 6. 风险与护栏
- 风险 1：
  - 把 multi-provider 做成 agent 内 stage-specific 分支，破坏 supplier-agnostic 边界。
  - 护栏：provider 选择只能由 policy/config/registry contract 驱动；agent 只消费 role handle 与模型能力。
- 风险 2：
  - 继续保留 registry 层 “服务不可用就自动挑另一个 provider” 的隐式行为，形成 silent fallback。
  - 护栏：selected provider unavailable 必须给出显式错误/诊断；只有明示配置的 fallback 才允许发生。
- 风险 3：
  - 只做 DeepSeek 接入，不收敛 provider selection owner，导致 `ai_config`、`llm_policy`、service registry 三层继续语义重叠。
  - 护栏：Phase 0 明确 single-owner contract，后续实现按该 contract 收口。
- 风险 3b：
  - 把这轮工作做成“从绑定 GLM 切到绑定 DeepSeek”，导致 agent 层仍然不可切换。
  - 护栏：Phase 3 验收必须包含“切换 provider/model 不需要改 concept agent 代码”，DeepSeek 只作为当前 policy target。
- 风险 4：
  - 为了追求“尽快跑通”，重新回退到旧 timeout 方案或简单拉长总时限。
  - 护栏：本计划明确禁止 reopen `009` 之前的 timeout 分配设计；只允许在 provider routing 与 provider capability 层修正根因。
