## 1. Current Problem
- Symptom:
  - `PLAN-20260404-051` closed the runtime-vs-memory-vs-substrate governance split and concluded that the remaining SQL-adjacent open items live in the memory backend/config lane rather than the runtime SQL lane.
  - the current memory construction path exposes several backend-selection surfaces with partially overlapping naming:
    - `MEMORY_WORKFLOW_BACKEND`
    - `MEMORY_FACTS_BACKEND`
    - `MEMORY_STORAGE_BACKEND`
    - `MEMORY_BACKEND`
  - reviewed code paths show these settings do not currently map onto one explicit owner model:
    - `build_memory_services()` passes `MEMORY_WORKFLOW_BACKEND` into `build_memory_management()`
    - `build_memory_management()` independently selects the long-term store via `MEMORY_BACKEND`
    - short-term working memory currently resolves through `create_short_term_store(kind="memory")`
    - `MEMORY_FACTS_BACKEND` is declared in config but no active owner-path consumption was found in the reviewed construction path
- Impact:
  - operators and future maintainers can misread which backend knob actually controls which memory layer.
  - fallback behavior can silently weaken memory durability or backend semantics without threatening runtime authority, making backend debugging and support claims harder.
  - future memory cleanup could accidentally drift into runtime authority scope unless the memory-layer owner map and support matrix are frozen first.
- Evidence baseline:
  - `docs/plans/active/PLAN-20260404-051.md`
  - `docs/plans/active/PLAN-20260404-051-stage-b-claim-inventory.md`
  - `backend/app/core/config.py`
  - `backend/app/services/memory_provider.py`
  - `backend/app/agents/memory/managers/management.py`
  - `backend/app/agents/memory/storage/in_memory/factory.py`
  - `backend/app/agents/memory/long_term/stores/sqlite_store.py`

## 2. Deviation Analysis
- Expected behavior:
  - the memory stack should expose one clear owner map for:
    - workflow/facts backend selection
    - short-term working-memory backend
    - long-term store backend
  - each configured backend knob should have:
    - one defined consumer path
    - one defined fallback policy
    - one defined support-level claim
  - memory backend configuration must remain fully decoupled from runtime control-plane authority.
- Actual behavior:
  - memory backend selection is split across service construction, env defaults, and store-manager defaults.
  - short-term memory backend selection is effectively fixed to in-memory in the reviewed construction path, but this is not frozen as an explicit support statement.
  - long-term backend fallback to `DictMemoryStore` is possible, yet the support-level and operator-facing meaning of that downgrade are not frozen in one plan.
- Root cause hypothesis:
  - memory subsystem evolution added separate knobs for different layers over time, but governance did not freeze which knob owns which layer and which ones are legacy, active, or deferred.

## 3. Implementation Approach
- Patch strategy:
  - keep this plan governance-first and documentation-first.
  - complete Stage A now:
    - freeze the memory-layer owner map
    - freeze the env/config-to-construction-path map
    - freeze fallback semantics and support-level claims
    - freeze non-goals that keep runtime authority out of scope
  - defer any actual cleanup/refactor until Stage B classifies whether the next move is:
    - config-name normalization
    - fallback diagnostics hardening
    - backend-contract coverage hardening
    - or no implementation follow-on at all
- Target artifacts:
  - `docs/plans/active/PLAN-20260404-052.md`
  - `docs/plans/active/PLAN-20260404-052-validation.md`
  - `docs/plans/active/PLAN-20260404-052-stage-a-memory-governance-freeze.md`
  - `docs/plans/PLAN_INDEX.json`
- Non-targets:
  - `backend/app/services/runtime_session_service.py`
  - reopening `PLAN-20260404-051`
  - runtime SQL remediation
  - queue/broker/websocket/artifact truth
  - changing runtime/session/node/attempt/gate/decision authority ownership
- Compatibility considerations:
  - `RuntimeSessionService` remains the sole runtime authority.
  - memory backend changes may affect memory durability, continuity, and operability, but must not become runtime control-plane fallback.
  - no compatibility shim may preserve a known-confusing backend contract silently once a clearer model is chosen in a later stage.
- Rollback plan:
  - if a future cleanup proposal starts to mix runtime authority work with memory backend work, stop and split it instead of widening this plan.

- `PLAN-20260404-051` remains closed and reference-only.
