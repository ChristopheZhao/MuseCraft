# Plan: Provider-aware image size contract correction for image generation tools

## 1. 当前问题
- 最新真实 dev run 已证明当前 launcher 仍有 live exit/orphan 问题，但同一批日志还暴露出一个独立的运行态主故障：图像生成尺寸 contract 不满足当前 provider 约束。
- 最新 [mas_workflow.log](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/logs/mas/mas_workflow.log) 显示：
  - 当前 image provider 被选择为 Doubao。
  - `image_generator` 第一轮规划下发 `size="1024x1024"`，五个 scene 全部被 Doubao 以 400 拒绝，错误明确为 `image size must be at least 3686400 pixels`。
  - 第二轮规划虽然改成了 `1920x1924`，但 run 在此之前就被中断，说明当前系统并没有在工具 contract 边界上稳定提供 provider-compatible size 语义。
- 当前代码证据表明这不是单一场景 prompt 问题，而是图像工具 contract 层的系统性错位：
  - [image_prompt_composer_tool.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/agents/tools/image_prompt_composer_tool.py) 默认把 `size` 设为 `1024x1024`。
  - [image_generation_tool.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/agents/tools/ai_services/image_generation_tool.py) 对 `generate_image` 暴露“任意字符串”的 `size`，默认同样是 `1024x1024`。
  - [fc_param_guard.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/agents/utils/fc_param_guard.py) 仍只允许旧尺寸集合，而且违例仅 `warn`。
  - [doubao_services.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/agents/tools/ai_services/doubao_services.py) 目前只是透传 `size`，没有在 provider adapter 层做 capability-aware normalization 或 fail-fast diagnostics。

## 2. 架构边界与硬约束
- 本计划是从 [PLAN-20260328-033.md](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/docs/plans/active/PLAN-20260328-033.md) 中拆分出来的 successor，只处理 image size / provider capability contract，不 reopen launcher/process lifecycle、app lifecycle pending-task、MAS runtime/control-plane、queue/worker 或 continuation snapshot。
- Hard rules:
  - 不得把 provider 约束问题下沉成 launcher、任务重试或运行态人工操作问题。
  - 不得通过 prompt 文案、agent case-by-case if/else、或“遇到 400 再试更大尺寸”这种消费侧补丁掩盖 contract 缺口。
  - 不得把 Doubao 规则硬编码成新的全局业务模型；尺寸能力必须通过 provider config / adapter capability / tool schema 暴露。
  - 不得让 FC guard 继续只做 `warn` 而把明显无效的 provider 参数放进真实调用。
  - 不得为了兼容单一 provider 而破坏 supplier-agnostic 边界；修复应发生在 provider-aware tool boundary，而不是 agent prompt 中显式写供应商名。
  - artifact、runtime state、launcher/process lifecycle 与 app async cleanup owner 均不在本计划内。

## 3. 目标语义
- 图像生成的 `size` 必须在 tool contract 边界上成为 provider-aware 的受控参数，而不是由 agent 任意猜测的自由字符串。
- 当当前 provider 为 Doubao 时，系统必须在真实 provider 调用之前保证以下至少一项成立：
  - 只暴露 provider-compatible size 值；
  - 或把通用尺寸规范化为 provider-compatible size；
  - 或在 contract boundary fail-fast，并返回显式 diagnostics，拒绝发起注定失败的 provider 请求。
- image prompt composition 负责组合 prompt 与一致性资产，不负责偷偷持有过期的默认尺寸真相。
- provider adapter 负责把上层的规范化尺寸映射到供应商请求；adapter 不得静默吞错。
- FC exposure、guard、tool schema、provider adapter 必须形成单一路径 contract，而不是各自维护不同的尺寸真相。

## 5. 推荐实现切口
- 当前推荐方向，不是本阶段实现承诺：
  - capability host 复用现有 [service_interfaces.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/agents/tools/ai_services/service_interfaces.py)：
    - video 侧已经有 `PromptCapability / EnumCapability / VideoCapabilities / get_capabilities()`
    - `034` 的 canonical owner 应扩展在同一宿主上，为 VLM/image generation 增加对应 capability 表达，而不是把 size 列表塞回 `fc_param_guard` 或 prompt
  - 初始切口优先限制在以下边界文件：
    - [service_interfaces.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/agents/tools/ai_services/service_interfaces.py)
    - [image_prompt_composer_tool.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/agents/tools/image_prompt_composer_tool.py)
    - [image_generation_tool.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/agents/tools/ai_services/image_generation_tool.py)
    - [doubao_services.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/agents/tools/ai_services/doubao_services.py)
    - [zhipu_services.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/agents/tools/ai_services/zhipu_services.py)
    - [fc_param_guard.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/agents/utils/fc_param_guard.py)
  - 单一路径 contract 冻结如下：
    - `image_prompt_composer` 不再拥有 `1024x1024` 默认真相；如果上层未显式给 `size`，它只透传空值或请求值
    - `image_generation_tool` 成为唯一的 image size normalization / validation 边界：读取当前 selected VLM provider，对应加载 capability snapshot，选择 provider-compatible default，规范化 alias，并在无法规范化时 fail-fast
    - VLM provider adapter 只接收已经规范化后的 canonical size；adapter 可以做 provider-specific payload mapping，但不再承担 silent recovery
    - `fc_param_guard` 只消费与 `image_generation_tool` 相同的 capability snapshot 做 pre-execution enforcement；它不是 canonical size source
  - 推荐目标是把 `size` 变成 capability-backed contract，而不是把 Doubao 的错误字符串到处复制。

### 5.1 Focused Validation Bundle
- design-time checks:
  - provider capability host 必须是单一宿主；不得同时在 `fc_param_guard`、tool schema、provider adapter 里维护三份 size 列表
  - `image_prompt_composer` 去默认值后，默认 size 选择权必须收敛到 `image_generation_tool`
  - selected provider 为 Doubao 时，不允许再把 `1024x1024` 发到真实 provider
  - selected provider 为 Zhipu 时，现有 `1024x1024` / `1024x1792` / `1792x1024` 兼容集合不能被误伤
- planned focused tests:
  - provider capability resolution tests：选中 Doubao/Zhipu 时返回不同 size capability snapshot
  - normalization tests：通用输入经过 `image_generation_tool` 规范化后得到 provider-compatible canonical size
  - fail-fast tests：无效 size 在 tool boundary 被拒绝，并带显式 diagnostics，而不是等 provider 400
  - guard alignment tests：`fc_param_guard` 与 `image_generation_tool` 使用同一 capability source，不再出现一个 `warn` 一个默认放行的分裂

## 6. 风险与护栏
- 风险 1：
  - 只在 agent prompt 或 FC planning 上做提示，希望模型自己学会选“大图”。
  - 护栏：能力约束必须落在 tool schema / validation / provider adapter，不能把无效参数筛选责任留给 prompt。
- 风险 2：
  - 在 provider 返回 400 后做 silent retry 或 case-specific patch，掩盖上游 contract 缺口。
  - 护栏：先修 tool boundary，再决定是否存在受设计允许的 retry；错误不能被无痕吞掉。
- 风险 3：
  - 为了支持 Doubao，重新把 provider 名写进 agent 逻辑或 prompt。
  - 护栏：坚持 supplier-agnostic，provider 差异通过 capability / adapter 承载。
- 风险 4：
  - 把 launcher live-exit/orphan 或 app lifecycle pending-task 一并塞进本计划。
  - 护栏：这两个 owner 继续留在 [PLAN-20260328-033.md](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/docs/plans/active/PLAN-20260328-033.md)。
