## 1. 当前问题
- `PLAN-20260323-016`、`PLAN-20260324-017`、`PLAN-20260324-019` 已完成；single-episode mainline、SoT / projection、authority contract 的主线治理已经收口。
- 当前剩余问题不再是 runtime/control-plane 真值混乱，而是仓库里仍留有少量不符合 canonical harness 清洁边界的 residual implementation：
  - `backend/app/api/v1/endpoints/projects.py` 仍保留 `_schedule_episode_orchestration(...)` 这条 endpoint-local thread host，用于 project mode 下的 per-episode generation。
  - `backend/app/core/mas_*` 与 `backend/app/services/workflow_optimizer.py` 仍以可导入主树模块形式存在，保留了 parallel orchestration / dispatch / aggregation 语义岛。
  - 少量 residual summary / legacy alias surface 仍存在于 active tree 中，例如 `get_workflow_status(...)`、WS legacy terminal event alias、queued host 的 `result.get("workflow_status")` 兼容别名。
- 这些残余当前不构成 active runtime truth blocker，但它们会继续制造：
  - 第二种执行宿主印象
  - 平行 orchestrator 语义面的可导入错觉
  - “compatibility helper 仍是正式结构一部分”的认知噪音

## 2. 架构边界与本计划定位
- 本计划继续聚焦 `single-episode harness` 的 residual cleanup，不 reopen `016/017/019`。
- canonical 边界继续保持：
  - `OrchestratorAgent` 是 single-episode 的唯一 MAS mainline。
  - `RuntimeSessionService` / `OrchestrationControlPlane` / `OrchestrationRuntimeController` 等控制层宿主继续拥有 runtime SoT。
  - queue / worker / API 只允许作为 execution container / external scheduler。
- 本计划是 `016` 完成后的真正 residual-cleanup successor：
  - `016` 关闭时已完成 active-path conformance 与 secondary-mainline cleanup。
  - 当前新识别的问题不再是 active-path drift，而是 closeout 后 repo 树中仍残留的 second-host / parallel-island / legacy-noise surface。

## 3. 范围
- Scope in:
  - 统一 single-episode execution host，删除 project per-episode generation 的 endpoint-local thread host。
  - 退役 `backend/app/core/mas_*` legacy orchestration island 及其仅服务于该 island 的测试/依赖链。
  - 退役 `backend/app/services/workflow_optimizer.py` 这类无 active caller 的平行优化引擎。
  - 重新核对 remaining helper/alias surface，只保留明确需要的 compatibility/summary，并默认删除非必要残留。
- Scope out:
  - 删除 `SeriesPlannerAgent`
  - 删除 `_schedule_project_plan(...)`
  - project-level planning / character-reference generation host 统一
  - reopen `017` 的 SoT convergence
  - reopen `019` 的 authority-contract closure
  - `018` 的 Jest/tooling 问题

## 4. 明确保留与明确删除
### 4.1 明确保留
- `SeriesPlannerAgent`
- `_schedule_project_plan(...)`
- project planning / character reference generation 相关 `project_state.progress.*`

保留理由：
- 这部分属于 project / multi-episode wrapper 层，而不是当前 single-episode runtime governance 的删除目标。
- 当前共享 execution host 只覆盖 quick orchestrator 与 project episode generation，不覆盖 project planning job。

### 4.2 默认删除
- `backend/app/core/mas_orchestrator.py`
- `backend/app/core/mas_task_dispatcher.py`
- `backend/app/core/mas_result_aggregator.py`
- 与上述 legacy island 仅内部互相依赖且无 active app caller 的 `backend/app/core/mas_*` 链条
- `backend/app/services/workflow_optimizer.py`
- `_schedule_episode_orchestration(...)`
- 仅用于上述残余语义面的 legacy tests / dead imports / prompt assets / docs surface

删除原则：
- 若无 active app caller，则直接删除，不做 compatibility 保留。
- 若仍有短期依赖，必须在本计划内明确指出依赖项、退出条件和删除时机；否则不保留。

## 5. 残余问题分类
### 5.1 执行宿主分叉
- 文件：
  - `backend/app/api/v1/endpoints/projects.py`
  - `backend/app/services/task_queue.py`
  - `backend/app/services/queued_task_execution_host.py`
- 问题性质：
  - project 的 per-episode generation 仍走 endpoint-local thread host，而 quick/project queued execution 已有共享 host。
  - 这不属于 control-plane SoT 越权，但属于 execution-container duplication。

### 5.2 legacy orchestration island
- 文件：
  - `backend/app/core/mas_orchestrator.py`
  - `backend/app/core/mas_task_dispatcher.py`
  - `backend/app/core/mas_result_aggregator.py`
  - 以及其链式依赖的其他 `backend/app/core/mas_*`
- 问题性质：
  - 保留了 parallel orchestration / dispatch / aggregation 语义面。
  - 当前未见 active app caller，但仍在主树可导入，且被 legacy integration tests 引用。

### 5.3 residual summary / alias noise
- 文件：
  - `backend/app/agents/orchestrator.py`
  - `backend/app/services/queued_task_execution_host.py`
  - `src/hooks/useWebSocket.ts`
  - `src/types/index.ts`
- 问题性质：
  - 仍有少量 summary helper、legacy event alias 或 transport result alias。
  - 这类对象不应重新升级为 architecture blocker，但若无必要，应继续删除。

## 6. 执行计划
### Phase A: 冻结 residual inventory 与删除边界
- 目标：
  - 逐项确认 single-episode residual surface 是否仍有 active app caller。
  - 明确 project planning host 是本计划唯一受控排除项。
- 产出：
  - 删除清单
  - 保留清单
  - 每个保留项的短期依赖说明

### Phase B: 退役 legacy orchestration island
- 删除 `backend/app/core/mas_*` 平行编排岛与无 active caller 的 `workflow_optimizer.py`。
- 同步删除仅服务于这些 legacy modules 的 integration tests / dead imports / prompt assets / compatibility notes。
- 要求：
  - 不得把这些 legacy modules 迁到新的 active compatibility layer。
  - 若需要历史留档，只能进入 archive/legacy 区并禁止生产导入。

### Phase C: 统一 project per-episode execution host
- 把 project per-episode generation 从 `_schedule_episode_orchestration(...)` 切到共享 execution host。
- 保持 project endpoint 仍是 wrapper：
  - 创建任务
  - 标记 project runtime/editorial surface
  - 把执行委托给 shared host / queue path
- 删除 endpoint-local thread runner。

### Phase D: 删除非必要 residual helper / alias
- 复查以下对象是否仍有 active consumer：
  - `get_workflow_status(...)`
  - WS legacy event alias
  - queued execution host 的 `workflow_status` 兼容别名
- 原则：
  - 若仅是历史兼容噪音且无必要，直接删。
  - 若仍被 active path 依赖，则必须降级为显式 compatibility debt，并写清退出条件。

### Phase E: 验证与 closeout
- Backend:
  - repo grep 证明 single-episode mainline 只剩 canonical host/import surface
  - focused backend regressions for task queue / project contracts / orchestrator runtime mainline
- Frontend:
  - 确认未因 alias 删除而回归 WS/runtime consumer
- Governance:
  - 明确 `SeriesPlannerAgent` / `_schedule_project_plan(...)` 仍是 project-level retained scope
  - 若 closeout 后仍有 project-level wrapper host 问题，另开 successor，不回滚本计划边界

## 7. 验收标准
- `backend/app` 主树中不再保留 parallel single-episode orchestration island。
- project per-episode generation 不再使用 endpoint-local thread host。
- queue / worker / API 继续只作为 execution container，不主写 runtime/control-plane 语义。
- 未被证明必要的 helper / alias / legacy module 不再保留。
- `SeriesPlannerAgent` 与 `_schedule_project_plan(...)` 被明确记录为 project-level retained scope，而不是误判为 single-episode drift。

## 8. 风险与护栏
- 风险 1：
  - 删除 legacy island 时误伤仍被动引用的历史测试/工具模块。
  - 护栏：先做 active caller inventory，再删；测试依赖本身不构成保留理由。
- 风险 2：
  - 把 project planning host 误吸收到 single-episode cleanup，导致范围漂移。
  - 护栏：`SeriesPlannerAgent` / `_schedule_project_plan(...)` 明确排除，不在本计划删除。
- 风险 3：
  - 用新的 compatibility wrapper 替代旧 host，而不是消除 second host。
  - 护栏：默认 delete，不得新增中间 adapter 来“包住旧入口”。
