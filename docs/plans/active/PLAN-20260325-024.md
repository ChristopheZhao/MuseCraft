- 本计划的目标是把这条 retained host 路径治理成“明确长期 retained”或“独立 product-job/shared host 统一”二选一，而不是把它重新包装成 MAS blocker。

## 2. 架构边界与原则
- 本计划服从：
  - [single_episode_harness_architecture_20260311.md](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/docs/architecture/single_episode_harness_architecture_20260311.md)

- Hard rules:
  - `project wrapper` 继续位于 single-episode harness 之外，不能并回 `OrchestratorAgent` 或 runtime control-plane。
  - 若做 host uniformity，只能收敛到独立 product-job/shared host 路线，而不是重建第二条 single-episode mainline。
  - queue / worker / thread host 只能是 transport/execution container，不得偷偷承接 MAS runtime semantics。
  - 不新增 silent compat layer 来“先包住 endpoint-local thread host 再说”。

## 4. 执行计划
### Phase A: host inventory 与 ownership freeze
- 目标：
  - 明确 `_schedule_project_plan(...)` 当前触达的状态面、宿主职责和 retained 边界。
- 执行清单：
  - [ ] 盘点 project planning 入口、thread host、task/project-state 写入与错误上报路径
  - [ ] 区分哪些是 project-wrapper product job 语义，哪些绝不能回流进 single-episode harness
  - [ ] 冻结 retained path 的非目标与 shared-host 路线的最小边界

### Phase B: successor decision 与目标设计
- 目标：
  - 在 retained 与独立 shared-host 改造之间做出正式决策，并把理由写清。
- 执行清单：
  - [ ] 决定长期 retained 还是进入独立 product-job/shared-host 改造
  - [ ] 若 retained，写清 retained rationale、退出条件、禁止误读为 MAS drift 的依据
  - [ ] 若进入 shared-host 改造，定义宿主边界、生命周期、失败回报与测试面

### Phase C: implementation 或明确 defer
- 目标：
  - 让 host 问题留下可执行的实现入口，而不是继续模糊悬置。
- 执行清单：
  - [ ] 按决策落实现，或拆成更小的 implementation successor
  - [ ] 同步更新 architecture/deviation inventory/docs/tests
  - [ ] 记录 focused validation 与用户确认门禁

## 6. 风险与护栏
- 风险 1：
  - 为了追求“统一宿主”，把 project planning 倒灌回 single-episode harness。
  - 护栏：host uniformity 只能落到独立 product-job/shared host，不能并回 `OrchestratorAgent`。
- 风险 2：
  - 继续把 endpoint-local thread host 当成“暂时没问题”，却没有 exit criteria。
  - 护栏：若 retained，必须写清 retained rationale 与退出条件。
- 风险 3：
  - 让 queue/thread host 承接 project runtime semantics，形成新的控制面漂移。
  - 护栏：transport/execution container 与业务控制语义继续分离。
