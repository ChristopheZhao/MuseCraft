## 4. 执行计划
### Phase A: host inventory 与 ownership freeze
- 目标：
  - 明确 `_schedule_project_plan(...)` 当前触达的状态面、宿主职责和 retained 边界。
- 执行清单：
  - [x] 盘点 project planning 入口、thread host、task/project-state 写入与错误上报路径
  - [x] 区分哪些是 project-wrapper product job 语义，哪些绝不能回流进 single-episode harness
  - [x] 冻结 retained path 的非目标与 shared-host 路线的最小边界

### Phase A Findings (2026-03-26)
- `create_project(...)` 先创建 SQL `Task` 与占位 `ProjectState`，再由 `_schedule_project_plan(...)` 在 endpoint-local daemon thread 中新建 event loop 并调用 `_run_project_plan(...)`；该路径当前仍是独立 product-job host，而非 shared queue host。
- `_run_project_plan(...)` 当前会写两类状态：
  - SQL `Task.status/progress/output_metadata/error_message`
  - `project_state_repository` 上的 `progress.planning` / `progress.character_references`
- `SeriesPlannerAgent` 只生成/回写 `ProjectState.story_plan`、episode script drafts、character/style foundation，不直接创建 `RuntimeSession`、gate 或 single-episode control-plane state。
- `project` 模式下真正进入 single-episode engine 的路径仍是 `orchestrate_project(...) -> TaskQueueService.queue_task(...) -> queued_task_execution_host.run_generation_in_host(mode=project) -> EpisodeOrchestratorAgent -> OrchestratorAgent`；这条路径与 planning host 分离，且 shared host 只承接 transport/execution container 职责。
- Ownership freeze:
  - project planning host 只拥有 project-wrapper / multi-episode product-job 语义：story planning、project progress、optional character reference kickoff。
  - project planning host 不得创建或推进 single-episode runtime session、node/gate/decision、per-episode runtime SoT。
  - 若后续进入 host uniformity，实现目标只能是独立 product-job/shared host 承载 `SeriesPlannerAgent` 这类 project-level job，而不是把 planning 倒灌进 `OrchestratorAgent` 或 runtime control plane。
  - 当前 Phase A 不做 implementation，只把 retained path 的 authority 边界和非目标冻结下来，等待 Phase B 决策 retained vs. independent shared host。

### Phase B: successor decision 与目标设计
- 目标：
  - 在 retained 与独立 shared-host 改造之间做出正式决策，并把理由写清。
- 执行清单：
  - [x] 决定长期 retained 还是进入独立 product-job/shared-host 改造
  - [x] 若 retained，写清 retained rationale、退出条件、禁止误读为 MAS drift 的依据
  - [x] 若进入 shared-host 改造，定义宿主边界、生命周期、失败回报与测试面

### Phase B Decision (2026-03-26)
- 决策：
  - 不把 `_schedule_project_plan(...)` 维持为长期 retained host；`024` 正式选择进入“独立 product-job/shared host 收敛”路线。
- 决策理由：
  - 当前 planning path 仍是 endpoint-local daemon thread + local event loop 直接执行 `SeriesPlannerAgent`，见 [projects.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/api/v1/endpoints/projects.py)。这说明它是 process-bound host，而不是已有的 shared execution container。
  - 对比 `project` 生成主路径，episode orchestration 已通过 `TaskQueueService -> queued_task_execution_host -> EpisodeOrchestratorAgent -> OrchestratorAgent` 进入 shared host，且带有 `celery_task_id`、queue eligibility 与 worker bootstrap；planning path 不具备同等生命周期与观测约束。
  - 当前架构已明确 `project wrapper` 位于 single-episode harness 之外，且 `EpisodeOrchestratorAgent` 只是 multi-episode wrapper，不是第二条 orchestrator engine；因此 host uniformity 的正确收敛点是独立 product-job/shared host，而不是把 planning 倒灌进 MAS control plane。
  - 现有测试已覆盖 queued episode-generation contract，但未见 `_schedule_project_plan(...)` / `create_project(...)` 的 host-level 契约测试；把这条路径长期 retained 会冻结一条治理和验证都更弱的宿主方案。
- 目标设计冻结：
  - API 仍可同步创建占位 `ProjectState` 与 SQL `Task`，保证前端能立刻进入 project workspace。
  - planning 调度入口应改为 shared host ingress，而不是 endpoint-local `threading.Thread`；是否复用现有 `TaskQueueService` 或拆出同级 product-job queue adapter，留到 Phase C 实现时决定。
  - shared host 只负责 transport/execution container；进入 worker 后运行的是 project-level planning job，而不是 `RuntimeSessionService` 或 single-episode gate/runtime semantics。
  - planning job 继续只回写 SQL `Task` 状态以及 `project_state_repository.progress.planning/character_references` 与 `StoryPlan`/foundation 数据，不新增 per-episode runtime authority。
  - 失败回报继续落在 `Task.error_message` 与 `project_state.progress.*.error`；不引入 runtime session fallback、自愈写、或第二套 MAS state。
  - Phase C 的最小实现面应包含：dispatch 路由、worker host runner、`create_project` queue ingress、focused endpoint/worker tests，以及 inventory/docs 同步。

### Phase C1: project-state visibility 与 host-neutral job boundary
- 目标：
  - 先解决 shared host 实现前的两个真实前置约束：`ProjectState` 跨宿主可见性，以及 queue/worker 的语义中立 job boundary。
- 执行清单：
  - [x] 决定 `ProjectStateRepository` / project read-model 的 shared visibility 方案：同进程共享、数据库/缓存持久化、或其他显式可见性方案
  - [x] 定义 project planning 的 host-neutral job contract，使 queue/worker 只按 job handler 执行，不按 `project`/`MAS` 语义分支拥有业务 authority
  - [x] 判断当前仓库是否满足最小实现前提；若不满足，显式产出 defer/successor decision，而不是直接硬做 Phase C2

### Phase C1 Findings (2026-03-26)
- `ProjectStateRepository` 当前仍是 API 进程内内存仓库，而 `/projects/{id}` 与前端 polling 直接依赖这份读面；若把 planning job 直接迁到独立 worker/shared host，worker 侧对 `project_state_repository` 的写入不会天然对 API 进程可见。
- `create_project(...)` 现在会同步创建占位 `ProjectState` 并立即返回，前端随后通过 `GET /projects/{id}` 持续观察 `progress.planning` / `progress.character_references`。因此 host 迁移不能只改调度入口，必须同时保证 project read-model 的跨宿主可见性。
- 当前 queue/worker dispatch 仍按 `GenerationMode` 分支，并在 quick 路径耦合 `RuntimeSessionService`；`run_generation_in_host(...)` 也直接按 `quick/project` 分支实例化 `OrchestratorAgent` / `EpisodeOrchestratorAgent`。如果把 planning 继续塞进这条分支，会进一步把 scheduler 写成 project/MAS 语义 owner。
- `TaskType.SCRIPT_WRITING` 不是合格的 planning dispatch authority：project planning 当前复用该枚举，但 `script_writer` / `memory_writer` 也已将它用于其他写作语义，因此不能再把它当作 project planning 的唯一 worker handler 键。
- C1 决策：
  - shared visibility 的目标方案应是显式的 project read-model / repository shared surface，而不是“仍由 API 进程独占内存仓库、worker 旁路写另一份状态”。在当前仓库里，这更接近数据库/缓存持久化或其他可跨宿主读取的显式方案。
  - queue/worker 的目标 contract 应是 host-neutral `job_kind` / `handler_key` registry；queue 只投递 job，worker 只按 handler 执行，不按 `GenerationMode` 或 `TaskType` 猜 project/MAS 业务语义。
  - 当前仓库不满足直接进入 C2 最小实现的前提。若不先补 shared project read-model 与 neutral dispatch contract，就不应把 `_schedule_project_plan(...)` 机械迁到 shared host。
  - 因此 C2 的入口条件已冻结为：先具备 shared project-state visibility 方案与 neutral job boundary；否则 C2 应产出 explicit defer 或更窄 successor，而不是仓促实现。

### Phase C2 Findings (2026-03-26)
- `024` 不再尝试在本计划内直接迁移 planning host。C1 已证明真正阻塞点是更大的 project-level substrate 问题，而不是单点 host 改线：
  - `ProjectStateRepository` / project read-model 仍是 process-local
  - queue/worker 仍缺 host-neutral `job_kind` / `handler_key` dispatch contract
- 因此 `024` 通过 explicit defer 收口：
  - 保持本计划只覆盖 project-wrapper host uniformity 的 bounded governance
  - 不在本计划内扩成 project read-model persistence、queue neutralization、project job substrate 的全面改造
  - 创建新的 managed successor [PLAN-20260326-027.md](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/docs/plans/active/PLAN-20260326-027.md)，单独承接更大的 project-level work
- `024` 的 closeout 结论是：
  - 当前代码中的 endpoint-local planning host 仍是暂存现实，但它已不再被当作长期终局
  - `024` 已完成它应承担的治理职责：明确边界、否决长期 retained、识别前置约束、并把超出边界的改造显式拆出
  - 本计划已在用户确认后进入 `completed`；[PLAN-20260326-027.md](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/docs/plans/active/PLAN-20260326-027.md) 继续承接更大的 project-level substrate work

## 6. 风险与护栏
- 风险 1：
  - 为了追求“统一宿主”，把 project planning 倒灌回 single-episode harness。
  - 护栏：host uniformity 只能落到独立 product-job/shared host，不能并回 `OrchestratorAgent`。
- 风险 2：
  - 继续把 endpoint-local thread host 当成“暂时没问题”，却没有 exit criteria。
  - 护栏：若 retained，必须写清 retained rationale 与退出条件。
- 风险 3：
  - 让 queue/thread host 承接 project runtime semantics，形成新的控制面漂移。
  - 护栏：transport/execution container 与业务控制语义继续分离。
