## 1. Current Problem
- Symptom:
  - Some real quick-mode runs fail during memory store initialization with `sqlite3.OperationalError: unable to open database file`.
  - The failure happens after the worker has already moved past the old `LongTermMemoryManager` downgrade-path bug fixed by [PLAN-20260407-061.md](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/docs/plans/active/PLAN-20260407-061.md).
  - The traceback points to [backend/app/agents/memory/long_term/stores/sqlite_store.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/agents/memory/long_term/stores/sqlite_store.py) while executing `PRAGMA journal_mode=WAL;`.
- Impact:
  - The issue does not invalidate completed frontend or memory-bootstrap plans, but it can still block live backend validation on this machine.
  - Quick runs may fail before entering business execution even though control-plane and frontend behavior are otherwise correct.
- Evidence:
  - Previous failure:
    - task `1061` on `2026-04-06` failed with `unable to open database file`
  - Successful workaround:
    - later validation on `2026-04-06` used `MEMORY_SQLITE_PATH=/tmp/qc_live_memory.sqlite` and successfully initialized `SQLiteMemoryStore`
  - New reproduction:
    - task `1073` on `2026-04-07` failed with the same `unable to open database file` error after the `061` fix had already removed the old `Errno 5` choke point

## 2. Deviation Analysis
- Expected behavior:
  - The long-term memory store should initialize reliably under the local worker runtime without requiring control-plane changes.
  - Storage path selection and filesystem semantics should remain encapsulated inside the memory module.
- Actual behavior:
  - The default sqlite path resolves to a repo-local file under `backend/app/agents/storage/memory.sqlite`.
  - Historical logs show that this default path succeeds often, but intermittently fails with `unable to open database file`.
  - A prior validation run succeeded immediately after switching the memory sqlite path to `/tmp/qc_live_memory.sqlite`.
- Root cause hypothesis:
  - This is more likely a memory-module storage path / filesystem / WAL stability issue than a control-plane or permission-model issue.
  - The owner remains the memory storage layer, not runtime control, queue semantics, or frontend.

## 3. Implementation Approach
- Boundary freeze:
  - Keep this plan inside the memory/storage module.
  - Do not couple the fix to control-plane state, runtime contract, or queue transport logic.
  - Do not reopen `058`, `060`, or `061`.
- Candidate directions to evaluate:
  - operator/config-only stabilization around `MEMORY_SQLITE_PATH`
  - safer default sqlite path selection for local worker execution
  - memory-store-local diagnostics around sqlite path and WAL initialization
- Non-goals:
  - no control-plane fallback based on sqlite failures
  - no frontend workaround
  - no broad memory architecture redesign

## 4. Execution Checklist

- [ ] Decide whether the right fix is config guidance, default path policy, or store-local defensive initialization
