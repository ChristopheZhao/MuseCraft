## 2. Deviation Analysis
- Expected behavior:
  - Once a leaf-agent attempt completes successfully, that attempt lifecycle is closed and its lease is retired.
  - A later runtime-decision failure must be surfaced as a control-plane / decision-stage failure, not reclassified as a failure of the already-completed attempt.
  - The first failing reason at the runtime-decision boundary must remain the primary user-visible and diagnostic error.
- Actual behavior:
  - `orchestrator.py` completes the runtime attempt before running the post-execution runtime decision.
  - The same function keeps `current_attempt_id`, `current_attempt_lease_token`, and related attempt context alive across the decision phase.
  - When runtime decision evaluation raises, the broad outer `except` path reuses the already-completed attempt context and calls `fail_runtime_attempt(...)`.
  - Because the attempt lease was already cleared on successful completion, the secondary `active execution lease` error overrides the original runtime-decision failure.
- Root cause hypothesis:
  - Primary defect class:
    - post-execution phase boundary is not encoded explicitly in orchestrator control-plane choreography, so a completed attempt context is reused after retirement.
  - Secondary masking defect:
    - error handling violates error-transparency expectations by replacing the original runtime-decision failure with a later lease-validation failure.
  - Still-open root-cause branch inside the primary fault:
    - the repo proves `runtime_decision` was missing `action`, but Stage A must still determine whether the owning defect is:
      - malformed decision request assembly,
      - prompt / contract design that permits an under-specified JSON result,
      - provider / adapter / parser loss between raw response and `json.loads(...)`,
      - or a different control-plane envelope mismatch.

## 3. Implementation Approach
- Patch strategy:
  - Treat this as a runtime control-plane RCA and repair stream, not as a `video_composer` or lease-keepalive stream.
  - Stage A:
    - freeze primary vs secondary failure classification with direct evidence
    - capture the runtime-decision request/response surface needed to identify the owning defect
    - decide whether the primary root cause lives in decision request assembly, prompt/contract, provider/adapter parsing, or another control-plane boundary
  - Stage B:
    - repair phase-boundary and error-transparency handling so a completed attempt can no longer re-enter the same-attempt fail path
    - preserve the original runtime-decision error as the primary surfaced failure if Stage C is not yet fixed
  - Stage C:
    - apply the minimal owner-boundary fix identified in Stage A for the missing-`action` root cause
    - keep the repair within the runtime decision boundary unless Stage A proves another owner
  - Stage D:
    - rerun closest-mainline acceptance on fresh processes and a new task
    - confirm the workflow crosses the previous `video -> runtime decision` failure point without masking the primary error
- Compatibility considerations:
  - Preserve `RuntimeSessionService` as the only authority for attempt/session/node/lease truth.
  - Do not weaken lease validation or add silent fallback when a decision-stage failure occurs.
  - Do not reopen `054`, `045`, queue-host keepalive work, startup warnings, `quality_checker`, or historical backfill by default.
  - Do not infer runtime truth from artifacts, shared WM, queue state, or frontend read-models.
- Rollback plan:
  - If a proposed patch weakens lease authority semantics or moves runtime truth away from `RuntimeSessionService`, stop and split the work instead of merging a compatibility shim.
  - If Stage A proves the primary owner lies outside orchestrator/runtime-decision control-plane code, supersede this plan with a new owner-correct successor rather than widening `055` in place.

### Stage B: phase-boundary and transparency repair
- Goal:
  - ensure completed attempts cannot be re-failed and original runtime-decision errors are preserved
- Allowed changes:
  - orchestrator/control-plane sequencing
  - runtime transition facade usage
  - targeted diagnostics or exception propagation
- Forbidden changes:
  - weakening lease validation
  - media-agent fallback
  - queue/worker authority changes
- Exit gate:
  - targeted tests prove that a post-completion runtime-decision failure no longer surfaces as `active execution lease`
  - original runtime-decision failure remains visible

### Stage C: decision-root repair
- Goal:
  - repair the missing-`action` root cause at its true owner boundary
- Allowed changes:
  - prompt/request contract
  - provider/adapter parsing
  - control-plane request shaping
  - only the minimal code required by the Stage A owner verdict
- Exit gate:
  - targeted tests cover the previously failing decision path
  - no decision payload is accepted without explicit `action`
  - the repair does not reopen lease ownership drift

### Stage D: acceptance rerun
- Goal:
  - validate the repaired control-plane path on a fresh end-to-end run
- Required validation:
  - restart API/worker to avoid stale-code ambiguity
  - run one new closest-mainline task
  - record whether the workflow crosses the previous `video -> runtime decision` boundary
- Exit gate:
  - acceptance either passes the old blocker point or fails with a new primary error that is no longer masked
  - only then may any follow-on scope be considered
