# Plan: API/Worker 日志治理统一与可观测性收口

## 1. Current Problem
- 当前日志系统没有单一 owner：
- [main.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/main.py) 使用 `logging.basicConfig(...)` 并通过 middleware 再打一套 request access log
- [logging_utils.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/core/logging_utils.py) 运行时给 root logger/`uvicorn.access` 动态补 handler/filter
- [celery_app.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/services/celery_app.py) 在 worker init 阶段再次调用日志配置
- API 通过 `uvicorn app.main:app` 启动时，`uvicorn` 自身也有一套独立日志初始化
- 结果是：
- 终端主要刷 `uvicorn.access` 高频轮询日志
- `mas_workflow.log` 承担了大部分 agent/orchestrator/provider 业务日志
- 同一请求可能在 `uvicorn.access` 和 `app.main` middleware 中重复打印
- API 与 Celery worker 的日志 owner、handler 和 filter 规则不统一

## 2. Target
- 保留 `uvicorn app.main:app` 启动方式，但通过正式 `log_config` 统一 API 日志配置
- 保留 `uvicorn app.main:app` 启动方式，但通过集中式 `dictConfig` owner 统一 API 日志配置
- API 与 Celery 使用同一套日志治理原则与字段规范
- access logs、business logs、provider noise 分层配置
- 终端保留关键 MAS 主线日志，去掉高频 `/api/v1/tasks/{id}` 与 `/api/v1/tasks/{id}/runtime` 轮询噪音
- 不修改既有 MAS 编排/控制/门控逻辑，只做日志层解耦和治理

## 4. Architecture Principles
- P1 Single Logging Owner:
- 日志配置必须有单一 owner；禁止 `main.py`、`logging_utils.py`、`uvicorn`、Celery worker 各自运行时补配置

- P2 Access/Business Separation:
- HTTP access logs 只由 `uvicorn.access` 管理
- `agent.* / services.* / tasks_api / ZhipuLLMService` 等业务日志只由应用 logger 管理
- 禁止在 FastAPI middleware 中再维护第二套 access log

- P3 Startup Injection over Runtime Patching:
- API 日志应由集中式 `dictConfig` owner 注入，禁止 `main.py`/middleware/worker init 各自拼接配置
- Celery 日志应通过 Celery 官方 logging signal/config 注入
- 禁止运行时 patch root logger / `uvicorn.access`

- P4 MAS Logic Isolation:
- 日志治理不得改变 orchestrator、runtime session、gate、task queue 的执行逻辑
- 若需要更好的可观测性，只允许补充日志字段，不允许改业务分支

- P5 Noise Governance:
- 高频轮询 (`/api/v1/tasks/{id}`、`/api/v1/tasks/{id}/runtime`) 只能在 access 层过滤
- `tool_registry`、`httpx` 等默认下沉到配置驱动的较低噪音级别

## 5. Implementation Approach
- 新增统一日志配置模块，输出一份正式 `dictConfig`
- API 进程在应用 bootstrap 时只使用这一份集中式配置，不再依赖 `basicConfig()` 或 runtime patch
- 删除 [main.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/main.py) 中重复的 request logging middleware
- 将 [logging_utils.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/core/logging_utils.py) 收缩为：
- logging config builder / filters / formatters
- 不再直接向 root logger 动态加 handler/filter
- Celery 侧通过官方 logging signal/配置使用统一 formatter/handler 规则，不再在 worker init 中重复配置 root handler
- 保留 file logging，但统一由正式配置声明；后续若需进一步做多进程安全改造，再独立起一轮计划

## 6. Stage Breakdown
### Stage A: Logging Ownership Freeze
- 目标：明确谁拥有 API 与 worker 的日志配置权
- 完成标志：形成统一配置入口，不再依赖 `basicConfig()` + 运行时补丁

- [x] A1 Logging Entry Audit
- 盘点 API、uvicorn、Celery、`logging_utils.py` 当前各自如何初始化日志
- 明确哪些入口必须退役、哪些入口保留为正式注入点

- [x] A2 Unified Logging Config Design
- 设计统一 `dictConfig` 结构：
- console handler
- file handler
- access filter
- logger levels
- propagate 规则

### Stage B: API Logging Unification
- 目标：让 API 进程日志完全受统一 `log_config` 管理
- 完成标志：`uvicorn.access` 与应用业务日志在同一套配置下工作，无重复 access log

- [x] B1 Remove `basicConfig` / Middleware Access Log
- 删除 [main.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/main.py) 中的 `logging.basicConfig(...)`
- 删除 request access middleware 的主职责

- [x] B2 Wire Uvicorn `log_config`
- 通过 API bootstrap 与必要启动入口，确保 uvicorn 进程加载统一日志配置，且不再受 `basicConfig()`/middleware 影响

- [x] B3 Access Filter Governance
- 仅在 `uvicorn.access` 层过滤：
- `/api/v1/tasks/{id}`
- `/api/v1/tasks/{id}/runtime`
- 其他 access log 保留

### Stage C: Celery Logging Unification
- 目标：worker 不再单独 patch root/logger
- 完成标志：Celery 与 API 共用同一套 formatter/field 规范

- [x] C1 Celery Logging Signal Refactor

- [x] C2 Worker Noise Governance

- 目标：证明日志治理收口后，终端与文件都保留高价值主线日志，但高频噪音已消失
- 完成标志：开发联调时能直接在 console 跟踪 MAS 主线，不再被 polling 淹没

## 7. Risks / Blockers
- 如果继续保留 `main.py` middleware access log 与 `uvicorn.access` 并存，日志治理不会真正收口
- 如果 Celery 仍在 worker init 阶段 patch root logger，API/worker 行为仍会分叉
- 多进程共享 file handler 仍有潜在风险；本轮先统一 owner 与配置，不扩到更重的 queue/listener 方案

## 8. Quality Gates
- QG-1: `main.py` 不再使用 `basicConfig()` 作为主日志配置入口
- QG-2: FastAPI middleware 不再维护第二套 request access log
- QG-3: uvicorn 启动路径使用统一 `log_config`
- QG-4: Celery 不再在 worker init 中拼接 legacy root/file handler
- QG-5: 高频 task polling access logs 从 console 消失，但关键 MAS 业务日志仍可见
- QG-6: 本轮改动不改变 MAS 编排、runtime、gate 的既有执行逻辑

## 9. Current Focus
- 当前 focus：`A1 -> A2`，先冻结日志配置 owner 和正式接入点，再实施 API/Celery 收口
