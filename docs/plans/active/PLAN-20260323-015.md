## 1. Current Problem
- Current delivery scope:
- 本计划是 `PLAN-20260323-014` 的 successor correction plan，只处理 post-014 review 已确认的两个残余边界问题。
- Residual issues:
- A. image 路径仍同时暴露 live shared WM 的 `scene_overview/scene_scripts` 与 published-script-deliverable 驱动的 `scene_info_ref/scene_info_payload`，`with HITL / wo HITL` 的正式输入来源尚未彻底单一化。
- B. `PublishedDeliverableService` 仍在 service 内直接写回 `session.input_payload`，runtime/control-plane ownership 没有完全回收至调用方。
- Impact:
- 如果不纠偏 A，则 image agent 仍可能在同一轮读取两套阶段输入，`approve` 等价性保证不够硬，script-stage 边界仍有残余漂移。
- 如果不纠偏 B，则 deliverable persistence service 继续承担 runtime state mutation，和 runtime ownership 属于 control-plane 的原则存在张力。
- Scope guard:
- 本计划不回退 `014` 的 published-deliverable 主方向。
- 本计划不重新引入 `dispatch`、`mode_router` 或 queue/runtime boundary 讨论。
- 本计划不改变 shared WM 作为 live iterate-facts SoT 的既有定位。

## 2. Dependency and Execution Order
- Relationship summary:
- 任务 A 解决 image 主线路径仍保留双输入源的问题；任务 B 解决 deliverable persistence service 仍越界改 runtime state 的问题。
- Dependency:
- 两个任务都依赖 published deliverable 机制已存在，但代码实现可以串行执行以降低风险。
- Required serial order:
- 1. 先完成任务 A：image path boundary purity。
- 2. 再完成任务 B：deliverable runtime ownership repatriation。
- Parallel allowance:
- review、调用链 inventory、测试设计可并行。
- 代码实现按 A -> B 串行推进，避免在 service API 改动前后同时调整 image path。

## 3. Architectural Decisions Frozen by This Plan
- Stage-input uniqueness rule:
- `IMAGE_GENERATOR` 不得同时接收 live WM `scene_overview/scene_scripts` 与 published deliverable 驱动的 script-stage payload。
- 对 script->image 边界，正式输入来源必须唯一。
- Overlay rule:
- live shared WM 只允许继续提供非 stage-boundary overlay，例如 `scene_outputs.image` 回写后的补充视图；不得再提供 script-stage truth。
- Runtime ownership rule:
- `PublishedDeliverableService` 只负责 payload persistence、deliverable record persistence、deliverable ref construction/loading。
- `session.input_payload` 或其他 runtime/control-plane state mutation 必须由 `RuntimeSessionService` 或等价 control-plane 调用方负责。
- No compatibility shim rule:
- 不允许用“双路径都保留，但靠注释约定优先级”的方式收尾。
- 需要直接删除 image 路径上的 live-WM stage-boundary 注入。

## 4. Task A: Image Path Boundary Purity
- Objective:
- 让 image 路径只消费 published script deliverable 作为 script-stage 正式输入来源。

### 4.1 A1 Context Source Inventory
- 明确当前 image 路径的双输入源：
- [orchestrator.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/agents/orchestrator.py) 中 `_merge_media_context(include_roles=True)` 注入的 live `scene_overview/scene_scripts`
- [build_image_generation_context](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/agents/adapters/memory_views.py) 注入的 published-deliverable-based `scene_info_ref/scene_info_payload`
- 以此为基线冻结必须删掉的 live-WM 注入面。

### 4.2 A2 Remove Live Script-Stage Injection for IMAGE_GENERATOR
- 修改 image 路径，不再通过 `build_media_agent_context(... include_scripts=True ...)` 给 `IMAGE_GENERATOR` 注入 live `scene_overview/scene_scripts`。
- 如 image agent 仍需要角色/风格相关静态上下文，只允许注入不与 script-stage truth 重叠的字段，例如 `roles_context`。
- `IMAGE_GENERATOR` 的 script-stage 场景与脚本信息统一来自 published deliverable 驱动的 image context builder。

### 4.3 A3 Checkpoint A
- `IMAGE_GENERATOR` 路径上不再并行暴露 live `scene_overview/scene_scripts` 与 published script deliverable。
- `with HITL + approve` 与 `wo HITL` 在 image 阶段消费同一正式输入来源。
- 验收标准：
- image path 只剩一个 script-stage truth source
- video path 行为不被回退
- 不新增兼容层或隐式优先级规则

## 5. Task B: Deliverable Runtime Ownership Repatriation
- Objective:
- 让 deliverable persistence service 回归 persistence-only，runtime state mutation 交回 control-plane。

### 5.1 B1 Service API Tightening
- 修改 [published_deliverable_service.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/services/published_deliverable_service.py)：
- publish/approve API 只返回 deliverable/ref
- 不再在 service 内直接写 `session.input_payload`

### 5.2 B2 Control-Plane Writeback Rehoming
- 将 `session.input_payload` 中 published deliverable ref 的写回，迁移到：
- [runtime_session_service.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/services/runtime_session_service.py)
- 或等价的 runtime/control-plane 调用方
- 由 control-plane 在 publish/approve 的业务步骤中接管 payload mutation

### 5.3 B3 Checkpoint B
- `PublishedDeliverableService` 不再直接写 runtime/control-plane state
- runtime payload 写回只发生在 control-plane 层
- 验收标准：
- deliverable service 已回归 persistence-only
- published deliverable ref 仍正确进入 resume/read path
- 不引入新的双写点

## 6. Integration Gate
- Checkpoint A 通过前，不进入任务 B 的代码实施。
- 最终验收标准：
- image path script-stage truth source 唯一
- `PublishedDeliverableService` 不再直接改 `session.input_payload`
- `with HITL / wo HITL` 在 `approve` 条件下继续保持同一正式输入来源
