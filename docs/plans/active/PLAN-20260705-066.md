## 1. Current Problem
- Symptom:
  - Follow-up MAS review after completed `PLAN-20260423-063` still found boundary leaks that can hide runtime/control-plane failures:
    - legacy orchestrator decision failure can default to `proceed_next`;
    - `BaseAgent` can synthesize a default `orchestration_report`, masking agents that did not explicitly report boundary events;
    - media completion checks can still treat non-empty `scene_outputs.*` as enough evidence instead of using a scene-output acceptance contract;
    - `ScriptWriterAgent` remains a deterministic coded stage while still being described and routed like an autonomous native agent;
    - published payload / continuation checkpoint normalization still has paths that can collapse malformed input into `None`, ignore unknown keys, or lose typed diagnostics.
- Impact:
  - Runtime/session/node/gate decisions can proceed after undecidable control input, violating fail-closed control-plane ownership.
  - Downstream gates can believe an agent completed a boundary event that the agent never explicitly reported.
  - Partial or malformed media outputs can look complete to composer, quality, completion, or read-model surfaces.
  - Script writer ownership remains ambiguous, making future ReAct/native-agent migration harder and encouraging pipeline patches.
  - Contract-boundary errors become hard to diagnose because caller-side fallbacks hide the malformed carrier.
- Reproduction:
  - `backend/app/agents/orchestrator.py` contains a legacy decision fallback returning `proceed_next`.
  - `backend/app/agents/base.py` contains `_build_default_orchestration_report` / `_ensure_orchestration_report`.
  - `backend/app/agents/orchestrator.py` still has `_is_image_step_completed` and `_is_video_step_completed` checks based on `scene_outputs.*` non-emptiness.
  - `backend/app/agents/script_writer.py` subclasses `BaseAgent` and runs fixed internal tool-call stages, while ReAct/native contracts are elsewhere.
  - `backend/app/agents/adapters/memory_views.py`, `backend/app/services/context_assembler.py`, `backend/app/services/published_deliverable_service.py`, and runtime checkpoint tests are the initial contract-boundary audit targets.

## 2. Deviation Analysis
- Expected behavior:
  - Orchestrator/control-plane decisions fail closed when the model output, tool execution result, or decision envelope is missing, malformed, ambiguous, or unsupported.
  - Every subagent that participates in MAS runtime boundaries returns an explicit `orchestration_report`; missing report is a protocol violation, not an inferred success.
  - Media completion is evaluated from typed scene-output acceptance facts: expected scene set, per-scene accepted status, artifact/path/provenance requirements, and explicit diagnostics.
  - Script writer is either a true `ReActAgent` with observe-plan-act-reflect ownership or is honestly classified as a deterministic MAS stage/service with explicit boundary reports.
  - Published payload and continuation checkpoint adapters normalize at the boundary, preserve typed `reason_code`, and reject unknown or malformed carriers with strict diagnostics.
- Actual behavior:
  - The newer runtime decision path is mostly explicit, but legacy/fallback paths still contain fail-open behavior.
  - The protocol layer requires explicit reports, but `BaseAgent.execute()` currently enriches missing reports before the protocol sees them.
  - Some completion helpers still use weak semantic proxies over WM buckets.
  - Script writer retains deterministic internal flow while the surrounding architecture increasingly expects native-agent autonomy semantics.
  - Some adapters remain tolerant in ways that belong only to explicitly designed downgrade paths.
- Root cause hypothesis:
  - 063 correctly fixed urgent runtime/read-model failures, but compatibility defaults survived at lower layers.
  - Earlier code optimized for keeping generation alive, while the current AGENTS.md boundary rules now require fail-visible diagnostics.
  - Script writer predates the current native ReAct contract and was never reclassified after ReActAgent became the standard for generator-style agents.

## 3. Implementation Approach
- Patch strategy:
  - Treat this as a control-plane / native-agent / contract-boundary correction, not product polish.
  - Start with characterization tests that prove each P0/P1 leak, then make the smallest owner-correct change.
  - Prefer strict contract errors and typed diagnostics over consumption-site fallbacks.
  - Reclassify `ScriptWriterAgent` as deterministic MAS stage/service in this plan unless a narrow ReAct migration proves lower-risk during implementation review; do not let it continue to present as autonomous native-agent behavior.
  - Keep 064/065 role-continuity, frontend, README, CI, and OSS packaging changes out of this plan.
- Compatibility considerations:
  - Known-wrong fail-open behavior is not preserved behind compatibility shims.
  - Existing callers that depend on implicit `orchestration_report` must be updated to report explicitly or be marked deterministic/non-boundary.
  - If a broad ReAct migration for script writer would touch prompt/tool strategy deeply, split it to a follow-on and close this plan with the honest deterministic classification.
  - Queue/Celery remains transport only; runtime/session/node/gate/decision truth stays in MAS control plane.
- Rollback plan:
  - Keep changes in independently revertible slices: decision fail-closed, explicit report boundary, media acceptance, script writer classification, payload/checkpoint contracts.
  - If one slice destabilizes focused runtime tests, revert only that slice and keep the checkpoint commits as recovery anchors.
  - Do not revert the protected 065 checkpoint commits unless explicitly requested.
