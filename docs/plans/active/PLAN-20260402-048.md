## 1. Current Problem
- Symptom:
  - `video_generator` planned a real `video_generation.generate_with_continuity(...)` call in the latest run, but the call was rejected before execution because the FC arguments omitted `workflow_state_id`.
  - subsequent helper-only `video_prompt_composer.build_prompt` calls were projected into `successful_scene_numbers`, which then misled planner text into declaring all scenes complete.
  - `react_agent` accepted that plan-level completion verdict and ended the loop even though no accepted video delivery facts existed.
  - downstream `video_composer` then failed closed on missing `scene_videos`, revealing the upstream false-complete state rather than causing it.
- Impact:
  - end-to-end video generation can terminate without producing any accepted scene-video outputs.
  - execution-only runtime identity (`workflow_state_id`) is leaking into LLM-facing tool-call semantics.
  - progress projection is no longer trustworthy as a planner-visible read-model because helper success can be promoted into delivery success.
  - the current completion path can declare success without MAS SoT-aligned delivery receipts.
- Reproduction:
  - run the latest failing workflow where `video_generator` emits `video_generation.generate_with_continuity(...)` without `workflow_state_id`, then observe the ACT-stage rejection in [mas_workflow.log](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/logs/mas/mas_workflow.log#L15162).
  - continue the same run and observe helper-only `build_prompt` success with `VIDEO_PERSISTENCE summary: total=0` in [mas_workflow.log](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/logs/mas/mas_workflow.log#L15199).
  - observe the later false completion prose and `task_complete=True` acceptance in [mas_workflow.log](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/logs/mas/mas_workflow.log#L15313) and [mas_workflow.log](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/logs/mas/mas_workflow.log#L15328).

## 2. Deviation Analysis
- Expected behavior:
  - `workflow_state_id` should be a workflow-scoped execution binding injected by the execution layer, not an LLM-facing business parameter that the planner must remember to emit.
  - `execution contract` should remain execution-only and stay out of planner-visible semantic input.
  - completion should consume accepted delivery receipts or equivalent MAS SoT-aligned delivery facts, not helper/tool success or planner prose alone.
  - `progress read-model` should remain derived, rebuildable, non-authoritative, and source-mapped to authoritative receipts/artifacts only.
- Actual behavior:
  - `video_generator` validates `execution_contract.storage.workflow_state_id` against FC arguments before tool execution, effectively making a schema-optional field runtime-mandatory.
  - `plan_context` exposes `execution_contract` directly to planner input.
  - `progress_read_model` falls back to generic `executed_calls` and counts any successful scene-numbered helper call as a successful scene.
  - `react_agent` accepts `task_complete=True` from the normalized plan contract without reconciling against accepted delivery facts.
- Root cause hypothesis:
  - the execution-binding repair introduced in `d67c4df` fixed explicit routing requirements at the wrong layer by pushing execution-only bindings into planner responsibility.
  - the planner-continuity repair introduced in `07cf70d` corrected repeated-scene planning but used an overly broad receipt source, allowing helper success to become delivery truth.
  - no single delivery-acceptance boundary currently emits the only completion-consumable receipt surface, so completion still over-trusts planner text.

## 3. Implementation Approach
- Patch strategy:
  - move `workflow_state_id` and related execution-only bindings out of LLM-facing FC business parameters and treat them as execution-context inputs carried by the tool envelope.
  - keep `execution contract` in the control-plane / execution-boundary path only, removing its role as planner-visible semantic input.
  - define and consume a single delivery-acceptance surface for completion, distinct from raw tool/helper success.
  - narrow `progress read-model` to authoritative execution receipts and/or accepted delivery facts only, with explicit provenance metadata.
  - align `react_agent` completion handling so plan-level `task_complete` cannot override missing delivery acceptance.
- Compatibility considerations:
  - do not reopen `PLAN-20260402-047`; that plan remains scoped to image-composer missing-owner diagnostics and should stay independent.
  - do not add a planner-owned runtime contract object or reintroduce helper-side heuristic gates.
  - do not move queue/worker/broker state into progress authority.
  - preserve fail-closed downstream composer boundaries; the fix belongs upstream in execution binding, delivery acceptance, and progress authority.
- Rollback plan:
  - if execution-context injection proves insufficient for a specific tool path, stop and repair the execution-envelope carrier rather than restoring LLM-facing `workflow_state_id`.
  - if delivery acceptance cannot be represented narrowly, stop and amend the output-boundary design instead of letting completion continue to consume raw helper/tool success.
