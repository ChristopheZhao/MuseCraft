## 1. 当前问题
- 在 `034` 的 clean-stack live smoke 中，真实图像 provider 调用已经证明 image-size contract 正常，但 quick workflow 本身暴露出一个新的、独立的运行态缺口。
- 任务 `756a5a91-1d84-4f79-8b68-a3baf2fb9ec9` 的 authoritative runtime view 显示：
  - `status = failed`
  - `current_node_key = image`
  - `concept` 节点 `completed`
  - `script` 节点仍为 `queued`
  - `image` 节点 `failed`
  - 错误为 `Missing runtime-input published deliverable ref: workflow_id=756a5a91-1d84-4f79-8b68-a3baf2fb9ec9 node_key=script prefer_approved=True status=missing_runtime_input_ref`
- 同一轮 orchestrator/worker 日志显示：
  - `Starting workflow step 1/5: concept_planner`
  - `Completed workflow step 1/5: concept_planner`
  - `Starting workflow step 2/5: image_generator`
  - 随后在 `ContextContractAssembler` 组装 `image_generator` 上下文时，因缺少 `script` published deliverable ref 而 fail-fast
- 当前代码证据表明，失败不是 `ContextAssembler` 误报，而是上游 runtime/orchestration 让一个依赖 `script` published input 的消费者提前进入了执行队列：
  - [context_assembler.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/services/context_assembler.py) 对 `IMAGE_GENERATOR` / `VIDEO_GENERATOR` / `VOICE_SYNTHESIZER` / `AUDIO_GENERATOR` 明确要求 `script` stage payload，缺 ref 时按设计 fail-fast
  - [orchestrator.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/agents/orchestrator.py) 当前运行日志已证明 step 2 被排成了 `image_generator`
  - [orchestration_queue_policy.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/services/orchestration_queue_policy.py) 目前按 `task_specs.order/run` 纯排序，并不表达 `script -> image` 这类阶段前置依赖
- 进一步的 contract 审计还表明，当前 `approved` 语义也被写宽了：
  - [runtime_session_service.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/services/runtime_session_service.py) 在 gate 打开时就把 script candidate ref 写进 `session.input_payload.published_deliverables["script"]`
  - approve 时同一个 slot 又被覆盖为 approved ref
  - [context_assembler.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/services/context_assembler.py) 的 `prefer_approved=True` 当前基本只是诊断标签，并不会基于 `ref.is_approved` 做真实拒绝
  - 结果是 `session.input_payload.published_deliverables` 同时混入了 review preview candidate 和 downstream runtime input，扩大了 approved 的语义边界

## 2. 架构边界与硬约束
- 本计划是 [PLAN-20260328-034.md](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/docs/plans/active/PLAN-20260328-034.md) 的 successor，只处理 clean quick-run 下暴露的 `script -> image` runtime prerequisite gap。
- Hard rules:
  - 不得 reopen `034` 的 image-size/provider contract，也不得把真实出图成功的结论回滚成“仍需兼容修补”。
  - 不得把 `ContextAssembler` 的 `missing_runtime_input_ref` fail-fast 改成静默降级、Shared WM fallback 或 optional payload；缺 ref 诊断本身是正确的防线。
  - 不得把 queue/worker/broker 当成业务语义 owner；队列只负责 transport，不负责解释为什么 `image_generator` 可以在 `script` 之前跑。
  - 不得把 artifact/published deliverable 扩成新的执行语义 owner；artifact 继续只承载内容事实，执行前置关系应归 runtime/orchestration/control-plane。
  - 不得通过 prompt hardcoding、agent case-by-case 跳过、或 “image_generator 自己回头找 script” 这类消费侧补丁掩盖上游调度缺口。
  - 不得一开始就扩到 unrelated owners，如 launcher、provider config、app lifecycle、Celery state reset。

## 3. 目标语义
- 初始 quick run 中，任何依赖 `script` published payload 的消费节点都不得在 `script` 节点完成并发布可用 runtime input 之前进入执行。
- `ContextAssembler` 继续作为只读 contract enforcer：缺失 `script` published input 时 fail-fast；真正的修复必须发生在上游 runtime/orchestration prerequisite owner。
- `published_deliverables` 在 `session.input_payload` 上必须收敛为 approved-only downstream projection；candidate preview 应停留在 gate/attempt review surfaces，而不是进入 downstream runtime-input 面。
- LLM decomposition 仍可决定 candidate agents / per-agent mission，但不能破坏控制层已知的阶段前置关系。
- execution queue / standby shaping / runtime transition 必须与 node prerequisite contract 一致，不能出现 runtime view 里 `script` 仍 queued 而 `image` 已 running/failed 的状态组合。
- `task_specs / candidate_agents / conditional_task_specs` 必须只有一条 authoritative 主线；不得同时让 `continuation_checkpoint` 与 Shared WM projection 共同参与执行语义。
- `workflow.task_specs` / `workflow.conditional_tasks` 若不再承担必要主线职责，应在消费点迁出后直接删除，而不是以兼容名义保留成潜在第二主线。
- `ContextAssembler` 只允许承担 read-side boundary assembly 与 fail-fast contract enforcement；不得继续持有 planning/control state 读取或 execution contract 构造职责。
- adapter/helper 只能做纯转换、校验或数据整形；不得通过热路径回写把自己升级成 runtime/planning state owner。
- prerequisite enforcement 必须同时覆盖 control-plane 的三个入口：
  - initial quick-run 的 `build_execution_queue`
  - runtime activation 的 standby insertion / `activate_from_standby`
  - 实际 dispatch 前的 eligibility check
- `prefer_approved=True` 必须表达真实 contract，而不是诊断标签：
  - write side 只允许 approved ref 进入 `session.input_payload.published_deliverables`
  - read side 若收到未 approved ref，必须按同一 contract fail-fast
- 用户 `approve / revise / replan` 的语义必须保持收敛：
  - `approve` = 允许当前节点产物成为 downstream input
  - `revise` = 带反馈重跑当前节点 producer
  - `replan` = 带反馈重跑更上游 planning + 当前节点 producer
  - 它们都不是“直接改 memory 或 asset”的语义

- Phase 1 冻结后的最小方案方向：
  - control-plane write path 收紧 `session.input_payload.published_deliverables`，使其成为 approved-only downstream projection；不再在 gate-open 时把 candidate script ref 投影到 runtime input
  - review 期 candidate 继续停留在现有 gate `artifact_refs`、attempt outputs、gate facts / preview surfaces，不新增并行存储面
  - runtime/orchestration 在 initial quick-run 与后续 runtime activation 上共同守住 mandatory producer prerequisite：任何 script-consuming 节点都不能在 `script` producer 未完成且未形成 downstream-eligible input 前进入执行
  - prerequisite enforcement 的最小实现切口必须覆盖：
    - `build_execution_queue`
    - standby insertion / `activate_from_standby`
    - agent dispatch 前的最终 eligibility check
  - `ContextAssembler` 继续只读 fail-fast；`prefer_approved=True` 必须成为同一 contract 的 read-side enforcement，若 ref 未 approved 则拒绝解析，而不是作为新 owner 或 fallback
  - 当前最重要的设计判断不是“怎么让 image 继续跑”，而是“如何让 control-plane 同时守住 prerequisite 和 approved-only projection，而不引入旁路逻辑或第二套 SoT”。

## 6. 风险与护栏
- 风险 1：
  - 直接把 `ContextAssembler.required=True` 降成可选，或者补回 Shared WM/runtime_input 旁路。
  - 护栏：缺 ref 必须继续 fail-fast；它是症状观测点，不是根因 owner。
- 风险 2：
  - 让 LLM decomposition 继续自由下发 `image_generator` 排在 `script_writer` 前，而后再在消费侧打补丁。
  - 护栏：阶段前置关系必须在 runtime/orchestration owner 收口，不能靠消费者自救。
- 风险 2b：
  - 保留当前 candidate ref 进入 `session.input_payload.published_deliverables`，再寄希望于 loader 自己“偏好 approved”。
  - 护栏：approved-only projection 必须先在 write side 成立；不能把 candidate/approved 选择逻辑扩散成第二个 SoT。
- 风险 3：
  - 把 quick-run 缺口误归因为 queue/Celery 顺序问题。
  - 护栏：queue/worker 仅是 transport container；runtime 节点含义与可执行性归控制层。
- 风险 4：
  - 把 `034`、launcher、provider、pending-task 清理一起混进新修复。
  - 护栏：新计划只处理 `script -> image` prerequisite gap；其它 owner 均保持关闭。
- 风险 5：
  - 直接删除 Shared WM planning projection，却未先迁走 `runtime_hints / execution_contract` 消费点，导致 `VIDEO_GENERATOR` / `VIDEO_COMPOSER` 在运行时静默回退到默认行为。
  - 护栏：必须先把 `runtime_hints / execution_contract` owner 迁回 orchestrator/control-plane，再删除 Shared WM projection。
- 风险 6：
  - 以“兼容旧断言”名义保留 `workflow.task_specs`，使其再次长成第二条主线。
  - 护栏：若无明确且必要的 diagnostics-only 用途，Shared WM planning projection 直接删除，不保留空壳兼容层。
