## 1. Current Problem
- Current delivery scope:
- 本计划是 `PLAN-20260322-013` 的 successor correction plan，只处理 post-implementation architecture review 已确认的 residual drift。
- Residual issues:
- A. `published_deliverable_service` 已证明“阶段成果正式发布/引用”方向正确，但当前实现混入了 shared WM 读取与投影写回职责，越过了既定 memory adapter/writer 边界。
- B. `memory_views` 在 published script deliverable 缺失时仍 silently fallback 到 live shared WM，重新引入了“live WM 充当 stage-boundary truth”的旧路径。
- C. quick 路径已经不再依赖 `dispatch`，但 project 路径仍通过 `mode_router.py` + `EpisodeOrchestratorAgent(memory_services or build_memory_services())` 保留隐式 composition；`dispatch` 目前仍是 project-mode 的历史残留入口。
- D. 相邻 active-path drift 仍存在：`AudioDeliveryGateEvaluator` 还保留隐式 `build_memory_services()` 默认装配。
- Impact:
- 如果不纠偏 A/B，则 `with HITL` 与 `wo HITL` 在 `approve` 条件下仍无法保证由同一正式成果来源驱动，且发布失败会被 live-WM fallback 掩盖。
- 如果不纠偏 C/D，则 quick 虽已脱离 `dispatch`，但 project path 仍保留隐式 composition 漏口，`mode_router.py` 无法真正退役。
- Scope guard:
- 本计划不回退 `012` 的 queue/runtime decoupling 结论。
- 本计划不推翻 `013` 的 published-deliverable 主方向，只修正其边界落位。
- 本计划不重新定义 shared WM 为阶段成果宿主；shared WM 继续是 MAS 当前执行段的 live iterate-facts SoT。

## 2. Dependency and Execution Order
- Relationship summary:
- 任务 A 解决“阶段成果发布链路的边界不纯”，任务 B 解决“project path 的 implicit composition 与 dispatch 残留”。

- Dependency:
- 先完成任务 A，才能彻底冻结 `script -> media` 的正式输入来源，避免在删除 `dispatch` 之前仍保留 live-WM fallback 这种错误边界。
- 完成任务 A 后，任务 B 才能把 project path 收口到显式 composition root，并安全删除 `mode_router.py`。
- Required serial order:
- 1. 先完成任务 A：published deliverable boundary correction。
- 2. 再完成任务 B：project explicit composition + dispatch retirement。
- Parallel allowance:
- review、职责 inventory、候选宿主分析可并行。
- 代码修改必须串行：Checkpoint A 通过前，不允许进入任务 B 的实现。

## 3. Architectural Decisions Frozen by This Plan
- Shared WM boundary:
- `shared WM` 继续作为 MAS 当前执行段的 live iterate-facts SoT，不承担 gate/downstream 的正式边界输入职责。
- Published deliverable boundary:
- published deliverable 继续作为阶段边界的正式成果引用；它不是第二套 live SoT，也不是媒体专用机制。
- Deliverable service rule:
- `PublishedDeliverableService` 只负责 payload 落盘、deliverable record 持久化、deliverable ref 构造/读取。
- `PublishedDeliverableService` 不得直接持有 shared WM read/write adapter 职责。
- Adapter/writer rule:
- 从 shared WM 组装阶段成果 payload 与把 deliverable ref 投影回 shared WM，必须回到明确的 adapter/writer 层，不得继续混在 persistence service 中。
- HITL equivalence rule:
- `with HITL` 与 `wo HITL` 在 `approve` 条件下必须消费同一份已发布阶段成果。
- published deliverable 缺失时不得 silently fallback 到 live shared WM 掩盖问题。
- Dispatch rule:
- quick 主线不再保留 `dispatch` 抽象。
- project path 完成显式 composition 收口后，`mode_router.py` 的目标态是删除，不保留 rename 或 thin-wrapper 兼容层。

## 4. Task A: Published Deliverable Boundary Correction
- Objective:
- 保留 published deliverable 机制，但拆除其越界的 memory 读取与投影职责，并移除 downstream 的 live-WM fallback。

### 4.1 A1 Service Boundary Split
- 收缩 [published_deliverable_service.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/services/published_deliverable_service.py)：
- 保留：
- payload 持久化
- `WorkflowPublishedDeliverable` record 创建/更新
- deliverable ref 构造、payload 读取
- 移除：
- `_read_shared_fact`
- `_write_shared_fact`
- `project_deliverable_ref_to_shared_wm`
- `project_payload_deliverables_to_shared_wm`

### 4.2 A2 Stage-Output Builder and Projection Writer
- 新增或扩展一个明确的 stage-output builder：
- 从 shared WM 读取现有 `script` 阶段成果 facts：
- `project.concept_plan`
- `scene_overview`
- `project.scene_scripts`
- 组装统一 `script` published payload 后交给 `PublishedDeliverableService`
- 新增或扩展一个 projection writer：
- 将已发布 deliverable ref 投影回 shared WM 当前执行段视图
- 该 writer 属于 memory adapter/writer 层，不属于 persistence service

### 4.3 A3 Downstream Read Contract Tightening
- 修改 [memory_views.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/agents/adapters/memory_views.py)：
- image/video context builder 不再在 published deliverable 缺失时 fallback 到 live shared WM
- 若当前路径理论上必须已有 published `script` deliverable，则直接抛出显式错误或高可见诊断
- 保留 live `scene_outputs.*` 作为 overlay，但不再回退成 stage-boundary truth

### 4.4 A4 Checkpoint A
- `PublishedDeliverableService` 不再直接操作 shared WM
- published deliverable 的 payload 组装与 WM ref 投影均通过明确 adapter/writer 层完成
- image/video context builder 在 published deliverable 缺失时不再 silently fallback 到 live shared WM
- 验收标准：
- `with HITL + approve` 与 `wo HITL` 消费同一正式成果来源
- published deliverable 发布失败会显式暴露，不被 fallback 掩盖
- 不新增平行 SoT，不削弱 shared WM 的既有定位

## 5. Task B: Project Explicit Composition and Dispatch Retirement
- Objective:
- 收掉 project path 的 implicit memory composition，并直接删除 `mode_router.py`。

### 5.1 B1 Project Composition Root Freeze
- 修改 [episode_orchestrator.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/agents/episode_orchestrator.py)：
- `__init__` 必须显式接收 `memory_services`
- 提供 `create_default()` 作为显式 factory
- 删除 `memory_services or build_memory_services()` 模式
- project path 的 composition root 与 quick 一样必须显式可见，而不是藏在 wrapper/default constructor 内

### 5.2 B2 Remove Project Dispatch Call Sites
- project queued/direct path 不再调用 `dispatch_generation()`
- 调用方直接显式构造 `EpisodeOrchestratorAgent.create_default()` 或显式注入 `memory_services`
- 删除对 [mode_router.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/core/mode_router.py) 的生产路径依赖

### 5.3 B3 Delete mode_router.py
- `mode_router.py` 的目标态是删除：
- 不重命名
- 不保留 project-only thin wrapper
- 不保留兼容壳
- 删除后同步清理 import/call sites 与计划、架构文档映射

### 5.4 B4 Adjacent Explicit-Memory Cleanup
- 清理相邻 active-path drift：
- [audio_delivery_gate_evaluator.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/services/audio_delivery_gate_evaluator.py) 去掉隐式 `build_memory_services()`
- 快速复查 quick/project active path，确保不再新增 `memory_services or build_memory_services()` 漏口

### 5.5 Checkpoint B
- project path 已有显式 composition root
- `EpisodeOrchestratorAgent` 不再隐式 build memory
- [mode_router.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/core/mode_router.py) 已删除
- 相邻 active-path drift 已清理到与显式 composition 规则一致
- 验收标准：
- quick/project active path 均无隐式 memory composition
- `dispatch` 不再作为任何生产路径抽象存在
