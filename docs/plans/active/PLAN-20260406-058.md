## 1. Current Problem
- Symptom:
  - Quick mode 新建任务后，前端会先显示“恢复条件缺失 / reason: missing_current_attempt”，约 10-20 秒后才切入正常 agent/runtime 进度视图。
  - 结果评审页的“导出与分享”点击下载没有真实动作，用户看不到浏览器下载，也没有真实导出结果。
  - Quick mode 页面整体仍以固定四步条 + 固定节点网格呈现，用户感知更像预设流水线，而不是 MAS runtime 驱动的执行工作台。
  - Sprint 2 stable workspace 在真实 processing 态视口下仍存在布局与信息层级问题：中央工作区被左右栏挤压，长 runtime 字段打断阅读，右侧诊断栏权重过高，导致当前节点工作台虽然方向正确，但整体可读性不足。
  - 新证据显示 quick-mode completed 收口仍有残余缺口：部分 run 在 review/export 页会沿用上一条任务的成片状态，或在点击“下载当前成片”时直接报 `Failed to fetch`；播放器顶栏下载图标也没有真实动作。
- Impact:
  - 问题 1 会把新建 run 的正常启动阶段误显示成恢复异常，制造不必要的失败感和控制面误解。
  - 问题 2 直接阻断最终视频交付，属于用户可见的功能缺失。
  - 问题 3 会持续削弱产品对 MAS / runtime-driven orchestration 的表达，尤其在 voice/audio 被跳过时，固定节点展示会制造“工作流早就写死”的错误心智。
  - 新问题不会改变 runtime 真实语义，但会削弱 processing 页作为“当前节点工作台”的可读性与主次关系，属于 `058` Sprint 2 的残余验收缺口，而不是 canvas 需求。
  - 新问题会把 quick-mode 最终交付重新拉回“不稳定 / 偶发失败”的观感：用户可能看到上一条 run 的残留成片，或者在 runtime 已完成的前提下仍无法下载当前成片。
- Reproduction:
  - 问题 1：
    - Quick mode 提交新任务后，立即进入 `processing` 页面。
    - 在 runtime 还没有可恢复 attempt 的阶段，页面渲染 [src/components/preview/QuickModeWorkspace.tsx](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/src/components/preview/QuickModeWorkspace.tsx) 中的 `resume_blocked` 提示。
  - 问题 2：
    - 任务完成后进入 review/export 页面。
    - 点击 [src/components/video/ExportInterface.tsx](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/src/components/video/ExportInterface.tsx) 的导出按钮，没有真实下载；当前仅做本地模拟进度。
  - 问题 3：
    - Quick mode 顶部四步条和节点网格在运行期间始终以固定顺序表达。
    - backend runtime 节点蓝图在 [backend/app/services/runtime_session_service.py](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/backend/app/services/runtime_session_service.py) 中固定种下，frontend 目前又直接全量渲染这些节点。
  - 新问题：
    - 在 processing 态真实屏幕上查看 [src/components/preview/QuickModeWorkspace.tsx](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/src/components/preview/QuickModeWorkspace.tsx)，会看到三栏布局对中央面板形成挤压。
    - 右侧 `runtime snapshot / control plane / skipped path` 长期常驻，且原始字段值直接暴露，导致中心工作面板不够突出。
  - 新证据：
    - 先完成并下载一条 quick run，再切换到另一条 completed run。
    - 进入 review/export 后，播放器或导出区可能继续持有上一条 run 的 `finalVideoUrl`；点击“下载当前成片”时前端直接报 `Failed to fetch`。
    - 播放器右上角下载图标始终没有动作，因为当前并未接入真实下载 handler。

## 2. Deviation Analysis
- Expected behavior:
  - 新建任务进入 processing 时，应优先表达“runtime 正在启动/附着中”，而不是显示 resume failure。
  - 导出按钮至少应触发真实的最终视频下载；如果后端没有多格式导出能力，前端不得继续伪装 4K/1080p 导出流程。
  - Quick mode workspace 应以 runtime read-model 为主，而不是把固定节点蓝图直接等价显示成 MAS 真正执行图。
  - completed quick run 只有在 fresh 当前成片 URL 已解析成功后，才应进入 review/export；任务切换时必须使上一条 run 的成片状态失效。
  - 播放器与导出面板的下载入口应共享同一个真实下载行为，而不是一真一假。
  - Actual behavior:
  - [src/components/forms/VideoRequestForm.tsx](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/src/components/forms/VideoRequestForm.tsx) 在 `createTask` 返回后立即切到 `processing`，但 runtime polling 初期读到的 `resume_control.reason_code=missing_current_attempt` 会被 [src/components/preview/QuickModeWorkspace.tsx](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/src/components/preview/QuickModeWorkspace.tsx) 直接显示成橙色异常提示。
  - [src/components/video/ExportInterface.tsx](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/src/components/video/ExportInterface.tsx) 的 `handleExport()` 只是模拟进度条并弹成功通知，没有使用 `videoUrl` 触发下载，也没有调用任何后端导出接口。
  - [src/pages/HomePage.tsx](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/src/pages/HomePage.tsx) 保留了固定四步条，[src/components/agents/AgentOrchestrator.tsx](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/src/components/agents/AgentOrchestrator.tsx) 和 [src/components/progress/RealTimeProgress.tsx](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/src/components/progress/RealTimeProgress.tsx) 又将 runtime 节点以固定顺序网格/列表全量展示。
  - Sprint 2 当前实现虽然已经改成 `storyboard rail + current focus + side context`，但 [src/components/preview/QuickModeWorkspace.tsx](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/src/components/preview/QuickModeWorkspace.tsx) 仍使用固定三栏 `260px / fluid / 300px` 布局，右侧还长期展示 raw runtime diagnostics，导致在常见视口下中心工作区出现拥挤和内容断裂。
  - [src/hooks/useTaskPolling.ts](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/src/hooks/useTaskPolling.ts) 只要看到 runtime 进入 `completed` 就会停止轮询并切到 review/export，但没有保证 fresh `finalVideoUrl` 已经成功写入 store。
  - [src/store/useAppStore.ts](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/src/store/useAppStore.ts) 与 [src/components/forms/VideoRequestForm.tsx](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/src/components/forms/VideoRequestForm.tsx) 在任务切换时不会自动失效旧的 `finalVideoUrl`，导致新的 completed run 可能沿用上一条 run 的成片状态。
  - [src/components/video/VideoPlayer.tsx](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/src/components/video/VideoPlayer.tsx) 顶栏下载按钮没有 `onClick`，会制造“播放器也能下载”的错误预期。
- Root cause hypothesis:
  - 问题 1 primary owner：
    - frontend 对 `resume_control` 语义的消费过于字面，未区分“fresh-run bootstrap 阶段尚无 current_attempt”与“查看既有 runtime 时的恢复阻断”。
    - backend 当前 `missing_current_attempt` 投影本身不一定错误，但前端在新建任务后的前几轮 polling 里不应将它当作用户可见异常。
  - 问题 2 primary owner：
    - 导出 UI 是占位实现，不是真实交付实现。
  - 问题 3 owner split：
    - frontend 当前确有过度流水线化呈现问题；
    - 但 backend runtime contract 也确实仍是固定节点蓝图，没有 graph edges / branch provenance / subgraph metadata，因此前端即使升级为画布，也只能先做 runtime-driven projection，而不是“真实 graph”。
  - 新问题 primary owner：
    - Sprint 2 的信息架构方向是对的，但空间预算与信息层级仍未收口。
    - 这更像 stable workspace 的 layout/polish 缺口，而不是 contract 缺失或 infinite-canvas 必然性。
  - 新证据 primary owner：
    - owner 在 frontend quick-mode completed 收口链，而不是 backend 文件服务或 runtime contract。
    - 现有 run 的权威成片地址来自 `runtime.summary_output.final_video_url`；`tasks.output_metadata` 与 `/resources` 在这些案例里都没有提供 final-output 兜底。
    - 前端在 `completed` 边界上过早开放 review/export，并保留了跨任务的旧 `finalVideoUrl`，因此会把“fresh URL 未就绪”误表现成下载失败或旧视频残留。

## 3. Implementation Approach
- Technical freeze:
  - 本次 `058` 先固化为 `frontend-only` 方案，不修改 backend runtime contract、control-plane 语义或 queue/worker 相关逻辑。
  - frontend 只消费现有 runtime/read-model：`status`、`current_node_key`、`nodes`、`active_gate`、最终视频 URL/路径等已存在投影。
  - 问题 1 通过前端区分 fresh bootstrap 与 explicit resume 场景来修复，不要求 backend 改写 `missing_current_attempt` 的投影语义。
  - 问题 2 通过前端真实下载当前成片来修复，不引入伪导出流程，也不假设存在新的转码/多规格导出接口。
  - 新证据 fix 继续遵守 frontend-only 边界：completed 成片的唯一权威来源固定为 `runtime.summary_output.final_video_url/final_video_path`，`task detail` 与 `/resources` 只允许保留为 best-effort fallback，不反向推动 backend 持久化 contract 变更。
  - 问题 3 的第一阶段只做工作台信息架构与交互重构，不新增 graph edges / branch provenance / subgraph metadata。
  - 只有在 Sprint 3 评估确认需要真实 graph 表达时，才允许新开 follow-on contract plan。
- Patch strategy:
  - 按三个 sprint 分阶段推进，避免 bug fix 与 workspace 重构互相污染：
  - Sprint 1：Quick-mode bug fix
    - 修正新建任务启动阶段的 resume banner 呈现。
    - 将导出按钮改为真实下载最终视频，移除当前假的导出进度语义。
  - Sprint 2：Runtime-driven workspace reshape
    - 不沿用当前固定节点网格做“显隐版流水线”，而是重做 quick workspace 的信息架构。
    - 保留固定节点顺序这一 runtime 事实，但主视觉只围绕“当前节点 + 已发生节点 + 当前产物/诊断”组织。
    - 未来节点不预先占位；`skipped` 节点默认弱化或收纳；顶部四步条降级为产品壳层导航，不再冒充 runtime orchestration truth。
  - Follow-on discovery:
    - 无限画布已从 `058` 移出，改由 [PLAN-20260406-059.md](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/docs/plans/active/PLAN-20260406-059.md) 独立承接。
    - `058` 到此为止只覆盖 Sprint 1 与 Sprint 2 的 frontend-only 工作台修复与重构。
  - Sprint 2 amendment:
    - 基于真实 processing 态证据，补一次 bounded layout/information-hierarchy 收口。
    - 只允许调整 processing workspace 的列宽、诊断信息权重、顶部壳层占高和长字段展示策略。
    - 不引入 canvas，不修改 backend contract，不把 raw operator diagnostics 继续当产品主界面内容。
  - Sprint 1 residual amendment:
    - completed 收口只在 fresh 当前成片 URL 已解析成功后才允许进入 review/export 并停止轮询。
    - 任务切换或 attach 新 run 时，必须失效上一条 run 的 `finalVideoUrl`，防止跨任务残留。
    - 播放器顶栏下载入口与右侧导出入口必须统一到同一个真实下载 handler；若不保留双入口，则必须移除假下载按钮，而不是继续展示无动作图标。
- Compatibility considerations:
  - 继续遵守既有 runtime/read-model 边界：frontend 只消费 runtime projection，不重新推断 control-plane 语义。
  - Sprint 1 不引入新的 backend contract 依赖。
  - Sprint 2 默认不要求 backend 改 schema；若需要新增 graph metadata，必须另开 follow-on contract plan。
  - Sprint 3 若做 spike，优先 behind-flag 或 isolated component，不直接替换主界面稳定路径。
  - 不得从 Celery/worker/broker 状态推断业务进度；live truth 仍然只来自 runtime/read-model。
- Rollback plan:
  - Sprint 1 可独立回滚，不影响 runtime/control-plane。
  - Sprint 2 若 progressive reveal 影响可读性，可退回当前 runtime 节点消费逻辑而保留 bug fix。
  - Sprint 3 若 spike 证明收益不足，则只保留设计结论，不进入主线实现。

### Sprint 2: Runtime-driven workspace reshape
- Goal:
  - 将 quick workspace 从“固定铺满的预设流水线”改成“runtime 驱动的工作台”。
  - 渐进显示必须建立在新的工作台布局上，而不是当前固定网格的按状态显隐。
- Design-first requirement:
  - Sprint 2 开始前，先产出一个 bounded frontend design brief / wireframe 方向，不直接进入组件改造。
  - 设计方案必须回答三个问题：
    - 当前节点如何成为主焦点，而不是八个节点平均分配注意力。
    - 已发生节点、跳过节点、未来节点分别如何表达，避免继续暗示“固定必经流水线”。
    - 当前产物、诊断、HITL gate 如何与 runtime 节点并置，而不是被挤到次要区域。
- Scope:
  - [src/components/agents/AgentOrchestrator.tsx](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/src/components/agents/AgentOrchestrator.tsx)
  - [src/components/progress/RealTimeProgress.tsx](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/src/components/progress/RealTimeProgress.tsx)
  - [src/pages/HomePage.tsx](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/src/pages/HomePage.tsx)
  - 可能新增 runtime display helper，但不改 backend node blueprint
  - 允许重排为“当前节点焦点区 + 已完成节点轨迹区 + 侧边产物/诊断区”或等价布局
- Candidate design directions:
  - 方案 A：Focus workspace
    - 中央大卡显示当前节点、当前状态、当前产物预览与下一动作。
    - 已完成节点以左侧或顶部轨迹带呈现；未来节点不出现。
  - 方案 B：Execution storyboard
    - 已发生节点按时间顺序形成 storyboard / filmstrip。
    - 当前节点展开为 detail panel；适合视频制作叙事感。
  - 方案 C：Canvas-lite
    - 保留可平移缩放的空间感，但暂不宣称真实 graph。
    - 只在 Sprint 2 设计评审证明前两者不足时考虑进入实现。
- Design brief artifact:
  - [docs/plans/active/PLAN-20260406-058-sprint2-design-brief.md](/mnt/d/code/agent/opensource/vertical_application/short-video-maker/docs/plans/active/PLAN-20260406-058-sprint2-design-brief.md)
  - 当前已选定单一方向：`Focus workspace + storyboard rail`
- Acceptance:
  - 在 Sprint 2 实现前，先确定单一设计方向并记录取舍理由。
  - workspace 主视觉不再保留当前固定流水线骨架。
  - 只展示已到达节点与当前节点；未来节点不预先占位。
  - `skipped` 节点不再与主线节点同权呈现。
  - 顶部四步条被明确定位为产品导航层，而不是 agent orchestration 图。
  - UI 首屏能让用户感知“runtime 正在推进工作台”，而不是“预设 DAG 只是被涂状态”。
