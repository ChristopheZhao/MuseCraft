- [PLAN-20260327-031.md](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/docs/plans/active/PLAN-20260327-031.md) 已把 `approve` / `revise` / `replan` 的控制层 continuation 语义收敛出来，但 repo 当前“persisted `workflow.task_specs`”仍是名义上的：
  - [orchestrator.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/agents/orchestrator.py) 的 `_load_authoritative_resume_task_specs(...)` 在 resume 时要求读取 `workflow.task_specs`
  - [orchestration_state_adapter.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/services/orchestration_state_adapter.py) 的 `persist_task_specs(...)` / `load_task_specs(...)` 仍通过 Shared WM `read_shared_fact(...)` / `write_shared_fact(...)` 读写 `workflow.task_specs` 与 `workflow.conditional_tasks`
  - 因此 approve/revise/replan 的 continuation authority 仍部分挂在短期 MAS WM，而不是控制层持久化面
- 当前已经存在可复用的持久化事实面，但只覆盖了一半：
  - [runtime_session_service.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/services/runtime_session_service.py) 会在 decision submit 时持久化 `published_deliverables` ref、`script_review_contract`、`decision.invalidation_scope`
  - [published_deliverable_service.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/services/published_deliverable_service.py) 会把 script deliverable 落成稳定 payload ref
  - 缺的不是“把整份 WM 存下来”，而是一份最小的 control-plane continuation snapshot，以及唯一恢复链

## 2. 架构边界与硬约束
- 本计划是 [PLAN-20260327-031.md](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/docs/plans/active/PLAN-20260327-031.md) 的 successor，只处理 HITL approve/revise/replan 在跨 worker resume 场景下的 continuation authority 对齐，不 reopen provider-routing、project-level、queue infra 或其他既有计划范围。
- Hard rules:
  - queue / worker / Celery 仍只负责 transport / execution container，不得承接 continuation 业务语义。
  - runtime/session/node/attempt/gate/decision/checkpoint 的 SoT 只允许归控制层所有。
  - artifact / published deliverable 只承载内容事实，不承载执行语义。
  - WM 只能是可重建工作区，不得升级成跨 worker 恢复 owner。
  - 不把 snapshot 做成第二套业务模型；只允许最小 continuation projection。
  - 不持久化整份 MAS WM，也不新增旁路 persistence seam 或零散映射表。
- 恢复链必须单一路径：control-plane continuation snapshot -> published artifact refs / decision facts -> hydrate fresh WM -> continue orchestration。
- 结构化缺失必须 fail-fast 并输出显式 diagnostics，不做 silent fallback。

## 2.1 Phase 1 冻结结果
- continuation snapshot 冻结为“复用现有控制层 SoT + 一份缺失的最小 continuation projection”，而不是第二套业务模型。
- `approve` / `revise` / `replan` 共享同一恢复输入面，但只在 action 语义上分流：
  - `approve`：复用最新 decision 的 `action=approve`，继续走 post-script 队列，不重新做 top-level planning。
  - `revise`：复用最新 decision + `script_review_contract.target_agents=["script_writer"]`，只回到 script node continuation。
  - `replan`：复用最新 decision + `script_review_contract.target_agents=["concept_planner","script_writer"]`，允许重新进入 planning。

### 2.1.1 直接复用的现有控制层 SoT
| 逻辑字段 | 当前 owner / 来源 | 用途 | 结论 |
| --- | --- | --- | --- |
| `decision.id` | `WorkflowGateDecision.id` | 绑定 continuation projection 到最新人工决定，防止 stale snapshot | authoritative SoT；直接复用，不重复建模 |
| `resume_action` | `WorkflowGateDecision.action` | 区分 `approve` / `revise` / `replan` 语义 | 直接复用，不放入 snapshot body |
| `invalidation_scope` | `WorkflowGateDecision.invalidation_scope` | 区分 node-scoped 与 workflow-scoped continuation | 直接复用，不放入 snapshot body |
| `resume_anchor` | `WorkflowSession.status/current_node_key/current_attempt_id` + `WorkflowGate.attempt_id` | 指示 session 正处于 `RESUMING`，并锚定当前 gate/attempt | 直接复用，不重复建模 |
| `published_deliverables.script` | `WorkflowSession.input_payload.published_deliverables["script"]` | 作为 script 内容事实源，供 approve/revise/replan 后恢复使用 | 直接复用，不拷入 snapshot |
| `script_review_contract` | `WorkflowSession.input_payload.runtime_contracts.script_review` | 提供 revise/replan 的反馈、结构化约束、target_agents | 直接复用；`approve` 继续为空 |

### 2.1.2 必须新增的最小 continuation projection
| 逻辑字段 | 语义 | 最小内容 | 冻结结论 |
| --- | --- | --- | --- |
| `continuation.version` | projection contract version | 整数或短字符串版本号 | 必需；用于解析与 fail-fast |
| `continuation.decision_id` | projection 绑定元数据 | 最新 continuation decision 的 id | 必需；仅用于 stale 检查与单一路径绑定，不得升级为独立 SoT |
| `continuation.candidate_agents` | LLM 选出的有序 agent pool | 按 LLM 选择顺序保留的 agent 列表 | 必需；用于确定性重建 queue/standby 顺序 |
| `continuation.task_specs` | 当前 authoritative agent assignments | 复用现有 assignment schema，仅保留执行所需字段 | 必需；是 post-resume 编排语义主体 |
| `continuation.conditional_task_specs` | 后续 runtime activation 所需的 conditional assignments | 复用现有 conditional assignment schema，仅保留执行所需字段 | 必需；否则 resume 后无法保持同一 activation contract |

### 2.1.3 `task_specs` / `conditional_task_specs` 的冻结字段范围
- `continuation.decision_id` 只是 attempt-scoped checkpoint 内的 binding metadata mirror；权威人工决定始终是 `WorkflowGateDecision.id`。任何 reader 都必须先读取 latest decision，再校验 checkpoint 内 `decision_id` 是否一致；禁止反向把 checkpoint `decision_id` 当作 decision authority。
- `continuation.task_specs` 的每个 agent assignment 只允许保留这些字段：
  - `agent`
  - `mission`
  - `deliverable`
  - `constraints`
  - `order`
  - `runtime_hints`
  - `run`
  - `conditional_task_id`（仅当 runtime controller 已绑定 conditional task 时保留）
  - `trigger`（仅当 runtime controller 已把 conditional trigger 合并到 target spec 时保留）
- `continuation.conditional_task_specs` 的每个 conditional assignment 只允许保留这些字段：
  - `agent`
  - `mission`
  - `deliverable`
  - `constraints`
  - `trigger`
  - `runtime_hints`
- 明确排除：
  - `fallback_used`
  - diagnostics / compat snapshots
  - 任何 WM-only 事实

### 2.1.4 明确禁止持久化或重复建模的字段
| 字段 | 原因 | 结论 |
| --- | --- | --- |
| `resume_action` / `invalidation_scope` / `resume_from_node` | 已由 decision/session/node 控制层 SoT 持有 | 禁止重复存入 snapshot |
| `published_deliverable_refs` / `review_contract` | 已由 `session.input_payload` 持有，属于内容事实或 review contract | 禁止重复存入 snapshot |
| `execution_queue` / `standby_agents` / `skip_agents` | 可由 `candidate_agents + task_specs + action` 确定性重建 | 禁止持久化为 snapshot 字段 |
| diagnostics-only `activation_pool_snapshot` / `plan_snapshot` / `runtime_activation` | 这些 projection 明确是 diagnostics，不是 authority | 禁止升级为恢复真相 |
| MAS working memory 全量内容 | WM 只是可重建工作区，不是 checkpoint owner | 严禁持久化 |

## 2.2 Phase 2 冻结结果
- 恢复链冻结为“两个控制层写入时刻 + 一个恢复读取路径”，但它们属于同一 continuation object 的生命周期，不是第二套业务模型：
  - gate-open：把 normalized continuation plan checkpoint 写入控制层锚点
  - decision-submit：把最新 human decision 绑定到该 checkpoint，并把 session 切到 `RESUMING`
  - resume bootstrap：只从控制层 active continuation 读取，重建 fresh WM 后继续编排
- Phase 2 只冻结职责和时序，不冻结最终存储载体；具体落在哪个控制层字段或最小 schema 扩展，留到 Phase 3 决策。

### 2.2.1 单一路径恢复链
1. 初次 planning run 中，orchestrator 产生 `candidate_agents`、`task_specs`、`conditional_task_specs`。
2. script node 即将打开 human gate 前，orchestrator 只负责把这份 normalized continuation plan 作为参数传给控制层 API；orchestrator 自身不得直接把它写成 durable state。
3. 控制层在 gate-open 事务内持久化一个 pending continuation checkpoint，并把它锚定到当前 `session/node/attempt/gate`。这一步必须发生在 worker 返回 `waiting_gate` 之前。
4. 用户提交 `approve` / `revise` / `replan` 时，控制层读取最新 gate 对应的 pending checkpoint，绑定 `decision_id`，同时写入现有 SoT：`decision.action`、`decision.invalidation_scope`、`script_review_contract`、`published_deliverables` ref、`session.status=RESUMING`。
5. queue/worker 仅根据 execution eligibility 重新入队；`prepare_dispatch_payload_for_task_sync(...)` 继续只转发 `runtime_session.input_payload`，不推断 continuation 语义。
6. 新 host 启动后，orchestrator 创建 fresh WM workspace，然后先从控制层读取最新 runtime state 与 active continuation；不得先从旧 WM projection 取 authority。
7. 控制层 reader 按 `session.current_node_key + current_attempt_id + latest_decision.id` 定位 active continuation，做 stale / missing 校验；校验通过后返回 normalized continuation plan。
8. OrchestrationStateAdapter 只负责把 active continuation 中的 `task_specs` / `conditional_task_specs` hydrate 回 fresh WM projection，供当前 run 内已有 orchestrator / assembler 逻辑继续使用；它不是 durable owner。
9. ContextContractAssembler 继续从 `session.input_payload` 的 published deliverable refs 和 review contract 解析内容事实，独立于 continuation snapshot。
10. action 语义分流：
   - `approve`：控制层先消费 approval continuation transition，然后 orchestrator 依据 hydrated continuation 重建 queue，并跳过 `concept_planner` / `script_writer`
   - `revise`：orchestrator 加载 active continuation，注入 `script_review_contract`，只重新进入 script continuation
   - `replan`：orchestrator 先验证 active continuation 与 latest decision 一致，然后使旧 continuation 失效并重新走 candidate selection / task decomposition；旧 continuation 不再作为后续执行 authority

### 2.2.2 职责分配冻结
| 组件 | 允许职责 | 禁止职责 |
| --- | --- | --- |
| `RuntimeSessionService` / 控制层 runtime service | gate-open 持久化 pending checkpoint；decision-submit 绑定 active continuation；resume 时按 decision/gate/attempt 读取并校验 active continuation | 不得把 queue/worker bootstrap 语义吸进来；不得依赖 WM 本体读 authority |
| `OrchestratorAgent` | 生成 normalized continuation plan；在 gate-open 时把 plan 作为显式参数交给控制层；resume 时消费控制层返回的 active continuation | 不得直接把 continuation snapshot 写成 durable state；不得把 diagnostics/WM projection 当恢复真相 |
| `OrchestrationStateAdapter` | 规范化 continuation plan；把 active continuation hydrate 成当前 run 需要的 WM projections | 不得再作为跨 worker continuation 的 SoT owner |
| `ContextContractAssembler` | 用 published deliverable refs / review contract 解析内容事实并组装 leaf context | 不得承接 continuation 执行语义 |
| `TaskQueueService` / `queued_task_execution_host` | requeue、eligibility、host bootstrap、event loop 与 tool registry 初始化 | 不得判断或派生 approve/revise/replan 语义 |

### 2.2.3 fail-fast diagnostics 冻结
- 若 `session.status != RESUMING` 或 latest gate/decision 不成立，则不进入 continuation load 路径。
- 若 latest gate 对应的 pending/active continuation 缺失，直接报显式 `missing continuation checkpoint`，不回退 WM。
- 若 active continuation 的 anchor 与 `latest_decision.id`、`current_attempt_id`、`current_node_key` 不一致，直接报 `stale continuation checkpoint`。
- 若 `approve` / `revise` 加载出的 continuation 缺 `script_writer` assignment、`candidate_agents` 为空、或 `task_specs` 结构不合法，直接失败。
- 若 `revise` / `replan` 缺失或破坏 `script_review_contract`，直接失败；`approve` 仍要求 review contract 为空。
- 内容事实缺失继续由 `ContextContractAssembler` / deliverable resolution 显式报错，不由 continuation loader 兜底。

### 2.2.4 Phase 3 前保留未决项
- 最终 storage carrier 尚未决定：
  - 可选方向只允许是现有控制层锚点复用或最小 schema 扩展
  - 不允许落到 Shared WM、queue metadata、artifact payload 本体
- pending checkpoint 与 active continuation 的物理表示尚未决定：
  - 允许是同一 projection 的两个生命周期状态
  - 不允许变成两套并行业务模型

## 2.3 Phase 3 冻结结果
- 实现切口已经冻结：采用“最小 schema 扩展到现有控制层模型”，而不是继续复用语义不匹配的 JSON 杂项字段。
- 选定 carrier：
  - 在 `WorkflowNodeAttempt` 上新增专用 `continuation_checkpoint` JSON 列。
  - 不新增新表；不改 queue/worker；不把 snapshot 塞进 artifact payload 或 WM。
- 核心理由：
  - checkpoint 天然是 attempt-scoped，且必须在 gate-open 前持久化；`WorkflowNodeAttempt` 与 `gate.attempt_id` 锚点天然一致。
  - `WorkflowNodeAttempt` 不在 runtime read-model 对外暴露，避免把 continuation plan 泄漏到前端。
  - 专用列能避免把 `input_contract`、`gate.facts`、`input_payload` 这类已有字段语义污染成“万能 JSON 桶”。

### 2.3.1 方案选择结论
| 方案 | 结论 | 原因 |
| --- | --- | --- |
| `WorkflowNodeAttempt.continuation_checkpoint` 新增专用 JSON 列 | 选中 | attempt-scoped、控制层内聚、不暴露到 read-model、语义最干净 |
| 复用 `WorkflowNodeAttempt.input_contract` | 不选 | 会把“attempt 输入契约”与“post-gate continuation 生命周期”混成一体，后续 decision 绑定也要回写原始 input contract，语义发散 |
| 复用 `WorkflowGate.facts` | 不选 | [runtime_session_service.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/services/runtime_session_service.py) 会把 `gate.facts` 直接序列化到 runtime view，前端当前也消费 `active_gate.facts`，会把内部 continuation plan 泄漏到 read-model |
| 复用 `WorkflowSession.input_payload` | 不选 | 当前已承载内容事实、deliverable refs、review contract，并作为 dispatch payload 进入 orchestrator；继续混入执行语义会破坏 artifact/content 与 continuation 的边界 |
| 复用 `WorkflowSession.gate_policy` | 不选 | 虽未使用，但它是 session-level 泛化字段，不具备 attempt/gate 锚点；把 continuation 塞进去会形成隐式 shadow state |
| 继续使用 Shared WM projection | 禁止 | 已被最新失败证明不具备跨 worker durability |

### 2.3.2 专用 checkpoint 结构冻结
- `WorkflowNodeAttempt.continuation_checkpoint` 仅允许包含一个专用对象：
```json
{
  "version": 1,
  "decision_id": null,
  "candidate_agents": ["concept_planner", "script_writer", "..."],
  "task_specs": {
    "script_writer": {
      "agent": "script_writer",
      "mission": "...",
      "deliverable": "...",
      "constraints": [],
      "order": 2,
      "runtime_hints": {},
      "run": true
    }
  },
  "conditional_task_specs": {
    "bgm_mix": {
      "agent": "video_composer",
      "mission": "...",
      "deliverable": "...",
      "constraints": [],
      "trigger": "global_bgm_missing",
      "runtime_hints": {}
    }
  }
}
```
- 约束：
  - gate-open 时写入时 `decision_id=null`
  - decision-submit 时仅允许把 `decision_id` 绑定为最新 `WorkflowGateDecision.id`；不得复制 `action`、`invalidation_scope`、review contract 或 deliverable refs
  - `decision_id` 只作为 checkpoint 与 latest decision 的 binding metadata；resume reader 必须始终以 `WorkflowGateDecision.id` 为 authoritative SoT，并把 checkpoint `decision_id` 仅用作一致性校验
  - replan 后旧 checkpoint 仅作为历史 attempt 记录存在，不再作为新 run authority

### 2.3.3 具体实现切口冻结
1. schema:
   - 新增 Alembic migration，为 `workflow_node_attempts` 增加 nullable JSON 列 `continuation_checkpoint`
   - 不新增索引；读取路径始终经由 `attempt_id`
2. control-plane write path:
   - 在 orchestrator 侧把当前 normalized continuation plan 作为显式参数传入 `_open_script_review_gate(...)`
   - `_open_script_review_gate(...)` 再传给 `RuntimeSessionService.complete_script_attempt_and_open_review_gate_sync(...)`
   - `RuntimeSessionService.complete_script_attempt_and_open_review_gate_sync(...)` 在同一事务内把 pending checkpoint 写入目标 `WorkflowNodeAttempt.continuation_checkpoint`
3. decision binding path:
   - `RuntimeSessionService.submit_gate_decision_sync(...)` 读取 `gate.attempt_id -> WorkflowNodeAttempt.continuation_checkpoint`
   - 校验 pending checkpoint 存在且结构合法
   - 仅把 `decision_id` 更新为当前 `decision.id`
4. resume read path:
   - 新增控制层 reader，例如 `RuntimeSessionService.load_active_script_continuation_sync(...)`
   - reader 按 `session.current_node_key + session.current_attempt_id + latest_decision.id` 读取 latest gate 对应 attempt 上的 checkpoint
   - reader 负责 missing/stale/malformed fail-fast
5. orchestrator integration:
   - `OrchestratorAgent._load_authoritative_resume_task_specs(...)` 不再从 Shared WM 读取 authority
   - resume path 改为调用控制层 reader 获取 normalized continuation plan
   - `OrchestrationStateAdapter` 新增纯适配入口，把 checkpoint hydrate 到当前 run 的 WM projections；现有 `load_task_specs(...)` 降格为 WM-local helper，不再承担 cross-worker authority

### 2.3.4 focused validation bundle 冻结
- migration / schema:
  - migration review
  - 本地 dev DB 或测试 DB 上确认 `workflow_node_attempts.continuation_checkpoint` 存在
- service-level:
  - gate-open 会把 pending checkpoint 写入对应 script attempt
  - decision-submit 会把 `decision_id` 绑定到同一 checkpoint，但不复制 decision/action 语义
  - active continuation reader 会先读取 latest `WorkflowGateDecision.id`，再把 checkpoint `decision_id` 仅作为 binding metadata 做一致性校验
  - active continuation reader 对 missing checkpoint / stale decision / malformed schema 显式报错
- orchestrator-level:
  - `approve` 在 fresh WM 条件下可通过控制层 reader 恢复，不依赖旧 WM 的 `workflow.task_specs`
  - `revise` 同样走控制层 reader，并继续要求 `script_review_contract`
  - `replan` 读取旧 checkpoint 仅用于 anchor/合法性校验，然后重新走 candidate selection / task decomposition
- boundary checks:
  - `gate.facts`、`session.input_payload`、frontend runtime view 不新增 continuation checkpoint 泄漏
  - queue/worker tests 不增加 continuation 语义断言

## 6. 风险与护栏
- 风险 1：
  - 误把本轮问题重新归因到 queue/worker host bootstrap，转而做 transport/container 改造。
  - 护栏：本计划禁止 queue-side owner drift；queue/worker 只保留 transport 语义。
- 风险 2：
  - 为了“快速恢复”把整份 MAS WM 持久化，导致 WM 升级成跨 worker authority。
  - 护栏：snapshot 只能是最小 control-plane projection；WM 只能重建。
- 风险 3：
  - 把 published deliverable 或 artifact ref 直接升级成 continuation state owner。
  - 护栏：artifact 只承载内容事实；执行语义只能来自控制层 snapshot / decision / checkpoint。
- 风险 4：
  - 为了补缺失字段，再造一套与 session/node/gate/decision 并行的业务模型。
  - 护栏：字段冻结时必须标注 owner 与来源，禁止第二套业务模型。
- 风险 5：
  - 延续当前 `workflow.task_specs` 这种“名义 persisted、实际 WM-only” 的伪持久化设计。
  - 护栏：Phase 1/2 必须明确 authority source 与恢复链，不能只改命名。
