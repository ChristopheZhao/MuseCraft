## 2. 根因分析
- 根因 1：video prompt builder 虽然已经支持 `opening_state / event_trigger / action_phases / end_state`，但 production payload 仍大量走旧字段回退；结果 prompt 主体继续被 `visual_description / motion_beats / script_text` 主导。
- 根因 2：旧 `motion_beats` 本质上仍是摘要节奏，不是完整的局部事件动作链；builder 对它们的投影目前还会产生重复或近义堆叠，导致 10s clip 更像“同一意象展开”而不是“状态变化推进”。
- 根因 3：consistency block 分层方向是对的，但当前 prose 仍过重，压缩不够；动作推进在最终 prompt 中权重仍不足。
- 根因 4：当前质量问题主要发生在 video prompt surface，并不需要立即回到 scene/script 上游做第二轮大 contract 重构。
- 边界注记：即使 review 发现 `038` 整体仍未 completed，也不构成把 `042` 扩写为 `038` 后续 sprint 的理由；若要改 scene/script authoritative contract，必须先 amendment。

## 3. 目标边界
- 本计划只处理 video 质量的明显 follow-up 问题：
  - 旧 schema 回退下的时间叙事投影过弱
  - action arc 重复/梗概化
  - consistency prose 过重、稀释动作主体
- 本计划关闭以下问题：
  - scene 1/2 这类 10s clip 仍然更像“动态场景卡”而不是局部事件段
  - 旧 `motion_beats` 投影导致的重复动作描述和弱收束
  - 过长 consistency prose 抢占 prompt 主体
- 本计划不做：
  - 重开 `038`
  - image-agent、runtime、queue、scheduler 治理
  - 上游 scene/script contract 重写
  - provider-specific rendering tuning、大规模 A/B prompt 体系
  - 固定秒表式 per-second 规则表

## 4. 实施方案
- Patch strategy A: 强化 legacy scene payload 到 local-event prompt 的投影
  - 当上游仍提供旧 schema 时，在 builder 边界把 `motion_beats / script_text / visual_description` 投影成更明确的 `opening -> action progression -> end-state` 风格 prompt。
  - 投影必须优先消除重复、近义堆叠和“剧情提纲化”表述；让旧 schema 至少能产出一个可读的局部事件弧线。
  - 不引入固定秒表模板；仍然用 phase/progression 语言而不是硬编码 `0-3s/3-6s/...`。
- Patch strategy B: 降低非必要叙事 prose 的权重
  - `script_text` / `narrative_description` 只保留为辅助剧情细节，不得再次压过动作推进。
  - 当动作弧线已经存在时，应避免再拼一个重复的 summary bullet。
- Patch strategy C: 压缩 consistency block 到真正的短锁定
  - 保留 `global_style_lock / character_lock / opening_anchor / local_continuity` 的分层。
  - 但要压缩长段角色小传、环境 prose、镜头起点重复叙述，让 consistency 成为短锁定而不是第二主体。
- Patch strategy D: 加入最小质量门而不是过度优化
  - 仅对明显问题加 guard：动作重复、收束缺失、consistency 过重。
  - 不承诺 provider 最终渲染绝对稳定，只确保 prompt surface 相比当前明显更适合 10s 局部事件。
