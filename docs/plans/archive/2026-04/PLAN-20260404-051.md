## 1. Current Problem
- Symptom:
  - `PLAN-20260403-050` already completed its bounded runtime-resume/bootstrap seam and now sits at `awaiting_user_confirmation`; it is no longer the right place to keep discussing SQL governance or backend semantics.
  - `PLAN-20260403-050-runtime-memory-sql-boundary-freeze.md` froze the runtime vs MAS-memory vs SQL split, but there is not yet a separate managed plan that turns that freeze into a follow-on SQL governance/design program.
  - the repo still contains multiple SQL-adjacent facts that can be conflated unless they are governed explicitly:
    - runtime control-plane persistence in the app DB
    - SQLite-bounded unit/test evidence
    - SQLite as one optional MAS-memory backend
- Impact:
  - future SQL work can accidentally reopen `050`, blur runtime SQL claims with memory backend claims, and over-claim SQLite results as production runtime evidence.
  - memory backend work can drift into an implicit fallback for runtime/session/node/attempt/gate/decision authority unless the allowed interface is frozen first.
  - MySQL-authoritative runtime conclusions can be asserted without the right proof if the claim-to-evidence mapping stays implicit.

## 2. Deviation Analysis
- Expected behavior:
  - post-`050` SQL work should run under a new managed plan and keep `050`, `049`, `045`, and `047` as reference-only lifecycle artifacts.
  - every SQL-related conclusion should first be classified as one of:
    - runtime SQL claim
    - memory backend claim
    - SQL physical substrate claim
  - admissible evidence should be frozen before any runtime implementation or runtime-SQL verification slice starts.
- Actual behavior:
  - the runtime/memory/SQL freeze exists, but no standalone successor plan yet records the next-stage deliverables, verify gates, or successor-splitting rules.
  - multiple SQL carriers in the repo still make it too easy to reason from "uses SQL" to the wrong architecture conclusion.
- Root cause hypothesis:
  - earlier confusion came from mixing three different statements into one layer:
    - runtime control-plane truth is stored in SQL-backed tables
    - MAS memory may also use SQLite as one backend
    - tests may use SQLite as a bounded execution medium
  - those statements are simultaneously true, but they do not share one semantic owner or one admissible evidence class.

## 3. Implementation Approach

  - create an independent managed follow-on plan for SQL governance/design instead of reopening `050`.
  - complete Stage A now as documentation-only governance work:
    - re-confirm the three-layer authority map
    - classify runtime SQL claims vs memory backend claims vs substrate claims
    - freeze the MySQL-authoritative vs SQLite-bounded evidence policy
    - freeze the allowed runtime-facts to memory-association interface
  - keep runtime implementation code frozen in this plan until a later gate proves that a narrower successor slice is needed.
- Target artifacts:
  - `docs/plans/active/PLAN-20260404-051.md`

- Non-targets:
  - `backend/app/services/runtime_session_service.py` implementation changes
  - reopening or absorbing `PLAN-20260403-050`
  - reopening `PLAN-20260403-049`
  - reopening `PLAN-20260401-045` unless a new reproducible runtime fault appears
  - reopening or reabsorbing archived `PLAN-20260402-047`
  - queue/broker/websocket/artifact-existence truth
  - memory-backend redesign as a runtime authority fallback
- Compatibility considerations:
  - `RuntimeSessionService` remains the sole authority for runtime/session/node/attempt/gate/decision truth.
  - MAS memory remains an application layer behind `ContextContractAssembler`, `MemoryWriter`, `MemoryServices`, and memory interfaces.
  - SQL remains a physical substrate used by multiple subsystems rather than a single semantic owner layer.
  - no compatibility shim may preserve a known-bad mixed boundary just to move faster.
- Rollback plan:
  - if this follow-on starts to require runtime implementation before Stage B claim inventory and evidence classification are complete, stop and open a narrower successor plan instead of widening this governance slice.

- Stage C: bounded follow-on decision
  - either conclude governance is sufficient and close this plan, or open one or more narrower successor plans without merging them back into `050`.
