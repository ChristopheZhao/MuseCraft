## 1. Current Problem
- Symptom:
  - `PLAN-20260402-046` removed image-mainline owner drift, but missing accepted owner fields can still degrade into silent prompt fallbacks.
  - [image_prompt_composer_tool.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/agents/tools/image_prompt_composer_tool.py#L446) currently resolves non-reference scene root text by walking `opening_state`, `visual_description`, and `title`, then falls through to a generic `单帧静态画面`.
  - [image_prompt_composer_tool.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/agents/tools/image_prompt_composer_tool.py#L439) still resolves character-reference subject to a generic `角色` when `characters_present` and `title` are both missing.
  - [PLAN-20260402-046-validation.md](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/docs/plans/active/PLAN-20260402-046-validation.md#L21) closed owner restoration with one remaining unchecked gate: missing owner fields do not yet surface as explicit diagnostics.
- Impact:
  - upstream contract gaps are masked instead of being surfaced at the composer boundary.
  - operators cannot distinguish a valid minimal scene from a silently degraded prompt assembled from placeholder text.
  - future contributors may reintroduce helper-side repair if missing owner data is not rejected explicitly.
- Reproduction:
  - call `image_prompt_composer.generate` with a scene entry that lacks all accepted non-reference scene-root fields (`opening_state`, `visual_description`, `title`) and observe that prompt synthesis still succeeds with `单帧静态画面`.
  - call the character-reference path with no `characters_present` and no `title`, and observe that the subject silently collapses to `角色`.
  - [image_prompt_composer_tool.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/agents/tools/image_prompt_composer_tool.py#L213) only raises a generic `composed prompt is empty` error after synthesis, so the missing owner fields are not named directly.

## 2. Deviation Analysis
- Expected behavior:
  - once `PLAN-20260402-046` established that unsupported image-side fields are no longer authoritative inputs, any missing accepted owner fields should fail fast with explicit diagnostics at the composer boundary.
  - diagnostics should identify which owner fields are required for the current image purpose and which scene entry failed, instead of substituting generic placeholder text.
  - explicit caller-owned fallback, if still allowed, must not hide the missing-owner reason.
- Actual behavior:
  - non-reference scenes silently degrade through a fallback chain of `opening_state -> visual_description -> title -> 单帧静态画面`.
  - character-reference scenes silently degrade through `characters_present -> title -> 角色`.
  - the composer only exposes a generic empty-prompt failure when all downstream synthesis disappears; it does not report missing owner fields as the root cause.
- Root cause hypothesis:
  - the original permissive synthesis path was retained to keep prompt assembly alive while owner cleanup was still in flight.
  - `PLAN-20260402-046` correctly removed wrong owners and unsupported inputs first, but it intentionally deferred the remaining error-transparency step.
  - as a result, the current boundary is architecturally cleaner but still too forgiving when required owner data is absent.

## 3. Implementation Approach
- Patch strategy:
  - define the minimum accepted owner-field sets for image composer prompt roots after `PLAN-20260402-046`:
    - non-reference image mainline requires at least one normalized field from `opening_state`, `visual_description`, or `title`
    - character-reference flow requires at least one subject field from `characters_present` or `title`
  - replace generic placeholder fallbacks (`单帧静态画面`, `角色`) with explicit diagnostics that identify the missing owner fields and the affected `scene_number`.
  - ensure any explicit caller-owned `fallback_prompt` path does not erase the missing-owner diagnostic if it remains supported.
  - add narrow tests around these boundary failures without reopening helper inference or contract expansion.
- Compatibility considerations:
  - do not restore helper-side repair, lexical inference, or scene-payload override support.
  - do not promote `image_purpose` or `frame_thesis` into [scene_contract.py](/mnt/d/code/agent/Opensource/vertical_application/short-video-maker/backend/app/services/scene_contract.py#L58).
  - keep the change local to composer/tool boundary diagnostics; do not expand this plan into repo-wide error-framework cleanup.
- Rollback plan:
  - if explicit diagnostics reveal a true upstream contract gap that must be accepted product-wide, stop and amend the owner contract at the proper upstream layer.
  - do not rollback by reintroducing placeholder text or helper semantic repair.

## 4. Boundary Statement
- This plan continues governance after `PLAN-20260402-046`; it does not reopen the owner decision for `image_purpose` / `frame_thesis`.
- `scene_contract` remains unchanged; this plan only strengthens how the image composer reacts when accepted owner fields are absent.
- `image_prompt_composer_tool` is the only in-scope runtime owner for these diagnostics.
- `image_generation_tool`, `memory_views`, `consistency_tool`, and repo-level helper debt are out of scope unless a direct composer-boundary dependency is discovered.
