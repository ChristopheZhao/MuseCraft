### Phase 1: runtime read-path purity
- 目标：
  - 去掉 runtime read path 的控制面写副作用，让 runtime view builder 成为纯读路径。
- 边界：
  - 只处理 runtime read-model 的读路径纯度；不把 `project-wrapper host` 或其他 retained host 问题混入。
- 执行清单：
  - [x] 诊断 `build_runtime_view_for_task()` 里 runtime bootstrap / node repair 写侧职责的根因与承接点
  - [x] 把 runtime bootstrap / default-node invariant 移回明确的写侧 owner，删除读路径自愈
  - [x] 更新 `/quick/current` 与 `/{task_id}/runtime` 的缺失 session 行为，使缺失以显式 absent/diagnostic 形式暴露
  - [x] 增加 focused backend tests / probes，锁定 GET 路径不再创建 session 或 node
- 阶段验收：
  - GET runtime path 不再创建 `workflow_session`
  - GET runtime path 不再补写 default nodes
  - quick task 的 runtime session 创建仍只发生在显式写侧 owner 路径
- 验证：
  - changed-file backend `py_compile`
  - focused backend tests / probes for no-write-on-read
  - endpoint contract review for `/quick/current` and `/{task_id}/runtime`

### Phase 2: scene-info contract purity
- 目标：
  - 让 `scene_info_ref` 成为 image/video active path 的唯一 scene-info contract，移除静默 payload fallback。
- 边界：
  - 只清理 scene-info contract 边界；不借机增加新的 helper/bridge。
- 执行清单：
  - [x] 删除 assembler 中 `scene_info_payload` 的 active fallback 输出
  - [x] 让 `persist_scene_info_ref()` 在失败时提供显式诊断，而不是吞异常返回 `None`
  - [x] 为成功路径与失败路径补 focused tests / probes，锁定 contract-first 行为
- 阶段验收：
  - active assembler 输出要么包含有效 `scene_info_ref`，要么显式失败
  - `scene_info_payload` 不再作为 active fallback 注入 leaf-facing context
  - scene-info persistence failure 具备可审计诊断
- 验证：
  - changed-file backend `py_compile`
  - focused backend tests / probes for success + fail-close
  - grep review confirming active assembler no longer emits `scene_info_payload` fallback

### Phase 3: low-level compat seam retirement and composer contract-authority alignment
- 目标：
  - 删除 active path 中对 legacy composer keys 的低层 scrubber，并让 `execution_contract` 真正成为 `video_composer` 输入 authority，避免 supporting capability 继续吞旧 vocabulary 或二次推断 compose intent。
- 边界：

- 执行清单：

  - [x] 把 legacy composer key 的错误暴露保持在明确的 contract boundary，而不是低层 silent cleanup
  - [x] 让 `build_video_composer_context()` / active assembler path 按显式 `execution_contract` 投影 composer input，而不是根据 `project.final_video` 或 helper defaults 猜测 compose intent
  - [x] 补 focused validation，锁定 active path 不再依赖 helper inference 或 legacy seam
- 阶段验收：
  - active orchestration path 不再经过低层 legacy-key scrubber
  - 不新增新的 compat bridge / helper
  - legacy composer keys 只在明确 contract boundary 上被拒绝或诊断
  - `execution_contract` 成为 `video_composer` 输入 authority；supporting capability 不再二次推断 compose/bgm/voiceover
- 验证：
  - changed-file backend `py_compile`
  - focused tests / probes for composer execution boundary and contract-driven context projection
  - active-path review confirming composer static context is derived from `execution_contract` rather than helper inference

## 6. 风险与护栏
- 风险 1：
  - 为了让 GET 接口“更稳”，继续在读路径做 runtime repair。
  - 护栏：Phase 1 必须把 bootstrap/repair 还给写侧 owner；读路径只能读或显式报缺失。
- 风险 2：
  - 以“容错”为理由保留 `scene_info_payload` fallback 或吞异常。
  - 护栏：Phase 2 必须 fail-close with diagnostics，不得保留 silent fallback。
- 风险 3：
  - 为了兼容旧 composer 输入，在 supporting capability 增加新的 scrubber/bridge。
  - 护栏：Phase 3 只允许删除 seam 或把错误上提到明确 contract boundary。
- 风险 4：
  - 提前刷新 inventory，造成治理文档与实现状态再次脱节。
  - 护栏：Phase 4 必须在 Phase 1-3 完成后执行，且只记录已验证事实。
